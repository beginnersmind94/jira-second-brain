# NXT-70422 — FIN - Enforce Universal GL Mapping Integrity Rules

## Description
*As a* Child Nutrition Director configuring the Financial Oversight module, *I want* the system to automatically prevent me from creating a mapping that would corrupt the integrity of a journal entry, *So that* my exported ledger data is always audit-ready and I never produce self-canceling or misclassified transactions in Oracle or Tyler Munis.

h2. Prerequisites (already implemented — included for context)

*Base Type Integrity Rule:* A data point may only be mapped to a GL account whose base type matches its own (Revenue → Revenue GL, Expense → Expense GL, Asset → Asset GL, Liability → Liability GL). This is enforced at the dropdown level — when a user opens the GL account selector for a data point, only accounts matching that data point's base type appear. No cross-type account is ever selectable.

*Many-to-One Mapping is Permitted:* Multiple data points of the same base type may be mapped to the same GL account. This is not blocked, not warned on, and not restricted in any way. Districts commonly map all meal sales revenue data points to a single revenue GL account — this is valid and expected.

## Acceptance Criteria
h2. AC-1 — Debit-Credit Collision Prevention

Within any single GL entry template, no data point on the debit side may be mapped to the same GL account as any data point on the credit side of that same template.

This rule prevents a self-canceling journal entry — a net-zero posting that silently omits a transaction from the district's GL export, violating GAAP's requirement that every transaction be faithfully represented in the financial record.

*A mapping only saves if AC-1 is satisfied AND both prerequisites are satisfied. All three, every time, for every template type.*

h3. Enforcement Points

*At mapping save or update:* If saving or updating a data point's GL account mapping would create a debit-credit collision in any template that includes that data point, block the save and display an error.

*At template creation or update:* If a user creates or edits a template and selects a debit-side data point and a credit-side data point that are already mapped to the same GL account, block the save and display an error.

In both cases, the check must evaluate *every* debit-credit combination within the template, not just the pair being edited, to account for templates that carry multiple debit or credit lines.

h3. *Error Message*

"This mapping cannot be saved. Data points [Data Point A] and [Data Point B] are both mapped to the same GL account [GL Account Name] on opposite sides of the [Template Name] entry template. This would cause the journal entry to cancel itself out. Please resolve the conflict before continuing."

If more than two data points collide, list all of them: "Data points [A], [B], and [C] are all mapped to…"

*Visual Indicator*

When a collision is detected, highlight every conflicting data point row with a warning icon and a red/amber left border. The indicator stays until the conflict is resolved.

*Blocking Behavior*

Once a collision exists on a template, block all further mapping additions to that template — not just the conflicting rows. The user must resolve the existing conflict before adding or editing any other mappings. This prevents stacking multiple collisions that obscure the root problem.

h2. AC-2 — Select All Accounts for Manual Entry

* Given a user is on the Manual Entry page and needs to add GL accounts,
* When they want to make all accounts available,
* Then the system must provide a {{Select all accounts}} option that adds all GL accounts in a single action, rather than requiring each account to be added one row at a time.
* When {{Select all accounts}} is enabled, all GL accounts should be added as entry rows and the {{+ ADD ROW}} action should no longer be shown.
* When {{Select all accounts}} is turned off after rows have already been loaded, the system must display a confirmation warning before clearing the loaded rows and any entered amounts.
* The Manual Entry page must also provide an always-visible {{Filter entry rows}} field so users can filter the GL accounts already loaded into the entry.

Update on 3/31/2026:

AC-2 now includes the Manual Entry interaction details for {{Select all accounts}}. When unchecked, the page stays in normal manual-entry mode with {{+ ADD ROW}}; when checked, all GL accounts are added in one action and {{+ ADD ROW}} is hidden. If the user turns {{Select all accounts}} off after rows have been loaded, the system should show a short confirmation before clearing loaded rows and entered amounts; once confirmed, the page returns to manual mode. AC-2 also now includes an always-visible {{Filter entry rows}} field to filter accounts already in the entry.

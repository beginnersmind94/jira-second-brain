# NXT-70962 — Insights - KPI - MPLH - Drawer - PE

## Description
*I am a* district user
*that I want to* open the Meals Per Labor Hour (MPLH) details drawer from Insights
*so that I* can review the actual value, expected value, difference, and school-level detail behind the KPI.

----

h2. *Notes*

* This story applies to *Insights – PrimeroEdge data source*.
* MPLH (Meals Per Labor Hour) is part of the *front office KPIs* being integrated into Insights.
* This story covers the *MPLH Details Drawer only*.
* The user reaches this drawer by clicking:
** The *MPLH KPI card* on the dashboard
** The *MPLH value* in the School Performance grid
* The drawer UI should follow the same pattern used by existing Insights details drawers.

h3. *Data & Source of Truth*

* The source of truth for MPLH data is available in:
** *POS → Management → Meals Per Labor Hour*
* This must be used by developers and QA to validate MPLH calculations and displayed values.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure:
** PrimeroEdge: [https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109334|https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109334|smart-link] 
* The details drawer must use this stored procedure as the data source for MPLH values and school-level detail.

h3. *About This Metric*

* The drawer includes an *About this metric* section using existing system text.
* The text should be reused as-is from the current system content for Meals Per Labor Hour.

----

h2. *Out of Scope*

* SchoolCafé implementation (separate story)

----

h2. *User Flow*

# A user logs into PrimeroEdge.
# The user navigates to *Insights*.
# The user lands on the Insights dashboard.
# The user filters the dashboard by timeframe and/or site.
# The user clicks either:
#* The *MPLH KPI card*, or
#* The *MPLH value* in the School Performance grid
# The *MPLH Details Drawer* opens.
# The user views:
#* Drawer header with metric title and analysis date range
#* MPLH Overview section
#* Details by School section
#* About this Metric section
# The user reviews school-level breakdowns and metric definition details.
# The user closes the drawer and returns to the dashboard.

----

h2. *Requirements*

h3. *1. Drawer Access & Header*

* Clicking the MPLH KPI card must open the MPLH Details Drawer.
* Clicking the MPLH value in the School Performance grid must open the MPLH Details Drawer.
* The drawer header must display:
** *Meals Per Labor Hour*
** *Analysis for [date range]*
* The displayed date range must reflect the currently selected dashboard timeframe.

----

h3. *2. MPLH Overview Section*

* The drawer must include an *MPLH Overview* section.
* The overview section must display:
** *Actual Value*
** *Expected Value*
** *Difference*
* Values must use rate formatting of *0.00* where applicable.
* Difference must display both:
** Numeric difference
** Percentage difference, if supported by the existing drawer pattern
* The expected value comes from the benchmark configuration.

----

h3. *3. Details by School Section*

* The drawer must include a *Details by School* section.
* This section must display a grid with the following columns:
** School
** Breakfast
** Lunch
** Snack
** A La Carte
** Other
** -Adult Meals-
** -Adult Breakfast-
** -Adult Lunch-
** -Adult Snack-
** Meal Equivalents
** Labor Hours
** MPLH
* MPLH must be formatted as *0.00*.
* The section must reflect the dashboard filter context used when the drawer was opened.
* Tooltip:
** On the Other, field, we want to put a tooltip that works just like the tooltip on the performance chart.
** On hover over, show each additional count from the database that is not Breakfast, Lunch, Snack, a la carte in a label value pair (ex: Adult Breakfast: 0) in a vertical list like on performance chart.
** The objective is to give the user the remaining counts so that they can see how the numbers add up without having to show a massive report. Usually, these will be zero, but sometimes will have small numbers and we don’t want the user calling customer support saying the counts don’t add up and we don’t want to take up a lot of real estate on rarely used data.

----

h3. *4. About This Metric Section*

* The drawer must include an *About this Metric* section.
* This section must display the following text:

*Meals per Labor Hour = Equivalent Meals ÷ Actual Labor Hours*

*Meal Equivalents: Breakfast = 0.67 Lunch = 1.00 Snack = 0.33 Adult Breakfast = 0.67 Adult Lunch = 1.00 Adult Snack = 0.67 A La Carte = $0.24*

*(Pulled from configured meal equivalents on settings page of labor hours.)*

----

h3. *5. Data Integration*

* The details drawer must use the *PrimeroEdge MPLH stored procedure*.
* Data returned must support:
** Header date context
** Actual value
** Expected value
** Difference calculation
** School-level detail rows
** About-this-metric context, where applicable

----

h3. *6. Formatting & Consistency*

* The drawer layout must follow the same interaction and visual pattern as existing Insights details drawers.
* Numeric values must be formatted consistently with existing KPI detail patterns.
* MPLH values must be displayed as rates with *two decimal places*.

----

h3. *7. Performance Requirement*

* The details drawer must load without performance degradation.
* It must support large districts with many schools without excessive delay.

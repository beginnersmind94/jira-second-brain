# NXT-2569 — Understand and evaluate Object Dynamic Creation

## Description
*Why Dynamic Types?*

* The main purpose of the Dynamic Type is to bypass the static type checks and add more flexibility to the language. This makes it much easier to write (while being slower).
* Works well with non-static/ highly fluid data structures, unknown during compile time (Ex: ViewBag in [ASP.NET|http://ASP.NET]). Supports implicit type conversion, no explicit casting required.
* Improves readability because it avoids clutter with clear intent by avoiding plumbing code and custom classes.
* Interoperability with other dynamic languages like IronPython, IronRuby etc.
** Ability to leverage other language features that don’t exist in C# to elegantly solve the problem.
** Ability to use scripts that are written in other languages that are not in .NET
** Add user scripting. For example, business rules as script. As a downside, this may introduce security, data corruption and other risks.
* Simplifies responses from API calls when we don’t know what type of object to expect (or we don’t care). Also makes Data Access code easy to write with loose typing.
* Simplifies reflection code.
* A good solution for making generic solutions when speed isn’t the main concern.

*Why NOT Dynamic Types?*

* Static languages are considered to be faster because the types resolve during the compilation phase, improving the application performance.
* We can catch a lot of bugs even before the application runs using static types. This means fewer bugs in run time.
* Limited or no IntelliSense.
* Adds performance overhead of up to 100 ms. (~100 ns even with call site caching)
* Makes it error prone during run time unless there is adequate automatic test coverage because we are sacrificing compile time checks.

Given these observations, my inclination is to limit the use of Dynamic Objects/Types unless necessary. There may a be a few scenarios where it may be suitable, but for our product needs, I think sticking with well-defined models and classes, although old fashioned, will make for a more maintainable product. This will also give us well-documented API end points.

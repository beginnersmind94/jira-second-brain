# NXT-70959 — Insights - KPI - PNA - Drawer - PE

## Description
*I am a* district user *that I want to* open the Paid Not Applied (PNA) details drawer from Insights
*so that I* can review the actual value, expected value, difference, and school-level detail behind the KPI.

----

h2. *Notes*

* This story applies to *Insights – PrimeroEdge data source*.
* PNA (Paid Not Applied) is part of the *front office KPIs* being integrated into Insights.
* This story covers the *PNA Details Drawer only*.
* The user reaches this drawer by clicking:
** The *PNA KPI card* on the dashboard
** The *PNA value* in the School Performance grid
* The drawer UI should follow the same pattern used by existing Insights details drawers.

h3. *Data & Source of Truth*

* The source of truth for PNA data is available in:
** *Student Eligibility → Eligibility Reports → Eligibility Roster*
* To generate the correct dataset:
** Eligibility Type = *Paid*
** Reason = *Default*
* This must be used by developers and QA to validate PNA calculations and displayed values.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure:
** PrimeroEdge: [https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109335|https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109335|smart-link] 
* The details drawer must use this stored procedure as the data source for PNA values and school-level detail.

h3. *About This Metric*

* The drawer includes an *About this metric* section using existing system text.
* The text should be reused as-is from the current system content for Paid Not Applied.

----

h2. *Out of Scope*

* SchoolCafé implementation (separate story)

----

h2. *User Flow*

# A user logs into PrimeroEdge.
# The user navigates to *Insights*.
# The user lands on the Insights dashboard.
# The user filters the dashboard by timeframe and/or site.
# The user clicks either:
#* The *PNA KPI card*, or
#* The *PNA value* in the School Performance grid
# The *PNA Details Drawer* opens.
# The user views:
#* Drawer header with metric title and analysis date range
#* PNA Overview section
#* Details by School section
#* About this Metric section
# The user reviews school-level breakdowns and metric definition details.
# The user closes the drawer and returns to the dashboard.

----

h2. *Requirements*

h3. *1. Drawer Access & Header*

* Clicking the PNA KPI card must open the PNA Details Drawer.
* Clicking the PNA value in the School Performance grid must open the PNA Details Drawer.
* The drawer header must display:
** *Paid Not Applied*
** *Analysis for [date range]*
* The displayed date range must reflect the currently selected dashboard timeframe.

----

h3. *2. PNA Overview Section*

* The drawer must include a *PNA Overview* section.
* The overview section must display:
** *Actual Value*
** *Expected Value*
** *Difference*
* Values must use percentage formatting of *0.00%* where applicable.
* Difference must display both:
** Numeric difference
** Percentage difference, if supported by the existing drawer pattern
* The expected value comes from the benchmark configuration.

----

h3. *3. Details by School Section*

* The drawer must include a *Details by School* section.
* This section must display a grid with the following columns:
** School
** Students
** Paid
** PNA
* PNA must be formatted as *0.00%*.
* Students and Paid must be formatted as counts using existing whole-number formatting with commas.
* If the drawer is opened from the *PNA KPI card*, the grid must show the full filtered dataset.
* If the drawer is opened from a *School Performance grid row*, the grid must show only the selected school row.

----

h3. *4. About This Metric Section*

* The drawer must include an *About this Metric* section.
* This section must display the following text:

Paid Not Applied = (Count of students where Eligibility = Paid AND Reason = Default) ÷ Total relevant student population × 100

PNA represents the percentage of students who are identified as "Paid" status but do not actually have a meal application or direct certification on file. Essentially, it tracks how many students are defaulting to the "Paid" category simply because the district has no data to place them in the Free or Reduced-price categories.

----

h3. *5. Data Integration*

* The details drawer must use the *PrimeroEdge PNA stored procedure*.
* Data returned must support:
** Header date context
** Actual value
** Expected value
** Difference calculation
** School-level detail rows

----

h3. *6. Formatting & Consistency*

* The drawer layout must follow the same interaction and visual pattern as existing Insights details drawers.
* Numeric values must be formatted consistently with existing KPI detail patterns.
* PNA values must be displayed as percentages with *two decimal places*.

----

h3. *7. Performance Requirement*

* The details drawer must load without performance degradation.
* It must support large districts with many schools without excessive delay.

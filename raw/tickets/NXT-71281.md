# NXT-71281 — Insights - KPI - MPLH - Dashboard - SC

## Description
*I am a* district user
*that I want to* view Meals Per Labor Hour (MPLH) performance in the Insights dashboard using SchoolCafé data
*so that I* can monitor actual performance against expected benchmarks across sites and timeframes.

----

h2. *Notes*

* This story applies to *Insights – SchoolCafé data source*.
* MPLH (Meals Per Labor Hour) is part of the *front office KPIs* being integrated into Insights.
* This story covers the *Dashboard screen only*.
* This story is a *companion to the PrimeroEdge MPLH Dashboard story*.
* The *frontend behavior, UI, filtering, interactions, formatting, configurator behavior, grid behavior, and chart behavior must remain the same as the PrimeroEdge implementation*.
* Frontend developers should *not need to build new UI logic* for SchoolCafé.

h3. *Data & Source of Truth*

* The source of truth for MPLH data is available in:
** *Front Office Accountability Module → Labor Hours page*
** *MPLH Report*
* This must be used by developers and QA to validate MPLH calculations and displayed values.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure:
** SchoolCafé: [https://primeroedgenext.atlassian.net/browse/NXT-67454|https://primeroedgenext.atlassian.net/browse/NXT-67454|smart-link] 
* The dashboard must use the *SchoolCafé stored procedure* as the data source for MPLH values instead of the PrimeroEdge stored procedure.

----

h2. *Out of Scope*

* PrimeroEdge implementation (covered in separate story)
* Benchmark configuration UI changes
* MPLH detail drawer implementation
* Schoolie Update

----

h2. *Requirements*

h3. *1. Shared Frontend Implementation*

* The SchoolCafé implementation must reuse the *same frontend behavior and UI* as the PrimeroEdge dashboard implementation.
* No new or separate frontend workflow should be introduced for:
** KPI card
** Screen configurator
** School Performance grid
** Performance Trends chart
** Trend hover tooltip
** Filter interactions
** Click-to-open details drawer behavior

----

h3. *2. Backend Data Source Change*

* The backend must use the *SchoolCafé MPLH stored procedure* instead of the PrimeroEdge stored procedure.
* This data source change applies to:
** KPI card actual value
** KPI card expected comparison inputs
** School Performance grid values
** Performance Trends chart values
** Trend hover details
** Filtered dashboard results

----

h3. *3. Functional Consistency*

* All SchoolCafé MPLH dashboard behavior must remain functionally identical to the PrimeroEdge implementation, including:
** formatting
** filter response
** card visibility/sorting behavior
** grid placement
** chart dropdown placement
** click behavior
** performance expectations

----

h3. *4. Data Validation*

* SchoolCafé MPLH values must match:
** SchoolCafé stored procedure output
** Front Office Accountability Module → Labor Hours → MPLH Report

----

h3. *5. Performance Requirement*

* The SchoolCafé-backed implementation must perform at the same level as the PrimeroEdge implementation and must not introduce performance degradation.

## Acceptance Criteria
h3. *1. Shared Frontend Implementation*

* SchoolCafé MPLH dashboard uses the same UI and behavior as PrimeroEdge.
* No separate frontend implementation is introduced for SchoolCafé.

----

h3. *2. Backend Data Source Change*

* Dashboard uses the SchoolCafé MPLH stored procedure.
* KPI card values come from SchoolCafé data.
* Grid values come from SchoolCafé data.
* Trend chart values come from SchoolCafé data.
* Hover tooltip values come from SchoolCafé data.

----

h3. *3. Functional Consistency*

* Timeframe and site filters behave the same as PrimeroEdge.
* KPI card behavior matches PrimeroEdge.
* Grid behavior matches PrimeroEdge.
* Configurator behavior matches PrimeroEdge.
* Trend chart behavior matches PrimeroEdge.
* Click behavior matches PrimeroEdge.

----

h3. *4. Data Validation*

* MPLH values match SchoolCafé stored procedure output.
* MPLH values match the SchoolCafé source of truth report.

----

h3. *5. Performance Requirement*

* Dashboard performance is consistent with PrimeroEdge implementation.
* No performance degradation is introduced.

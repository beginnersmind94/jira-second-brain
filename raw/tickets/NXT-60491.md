# NXT-60491 — TH - WS - Custom Reports Async Data Flows

## Description
*As a* TeamHub user *I want* to be able to asynchronously update my Custom Report (CR) Distributions *So that* I can view my recently run reports

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* It is a backend only story
* The objective is to enable “offline mode” for CR Distributions (not starred CRs) so that users can use the app without having an internet connection
* This will achieve 2 things
** User won’t have to go over the internet to get their CRs when landing on the screen. They will come from a local store on their phone so it will speed up the load times on the screen
** If the user has no internet, they can still see their CRs 
* To achieve all this, you will ensure that:
** CR Distributed screen reads from local async storage
** CR Distributed screen updates read/unread status in local async storage
** CR Distributed screen deletes by writing to local async storage
** there is a sync agent that:
*** downloads distributed CRs
*** sends read/unread status updates
*** sends delete request
*** reconciles when their is a discrepancy 

h2. Flows

* #1, user logs in:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** App kicks off all CR Distributions reconciliation in the background without the user knowing it
** Start the CR sync job after the inbox and sent and before the Insights and Production
** App updates the local storage cache
** User clicks on Custom Reports and is redirected there
** User optionally views a distributed CR
*** App updates Read status in local store if it was unread
*** App stores this update into the sync queue
*** App manually initiates a sync update to the server to change the read status
*** App checks for any new updates from the server and downloads if there are
** User optionally deletes a distributed CR
*** App deletes in local store
*** App stores this delete update into the sync queue
*** App manually initiates a sync update to the server to delete the CR from the server
*** App checks for any new updates from the server and downloads if there are
* #2, Sync Schedule
** App maintains a schedule to check for updates
** App identified that the specified time has passed and initiates the sync job
** Sync job sends any unsent changes to the server and checks for all updates from the server
** Sync job reconciles any discrepancies

h2. Requirements

* Inbox screen
** Refresh button
*** On click:
**** Initiate the reconciliation process by immediately calling the sync agent to do a full sync
*** Indicate the sync status and if there is anything not synced
** Note: missing from Bolt code
* Login
** Upon successful login, initiate the reconciliation process by immediately calling the sync agent to do a full sync
* Logout
** Upon successful logout, initiate the reconciliation process by immediately calling the sync agent to do a full sync
* Schedule
** Run a delta reconciliation on a schedule per every n minutes
** Just look for changes since the last time
** choose a different number of minutes than from the other sync jobs like production and insights so that they don’t all try to run at the same time(e.g. 6 minutes for this one)

h4. *Data Requirements*

* Local Storage
** There should be local storage to save distributed CR in
** Should maintain state even when user logs out
* Message Queue
** There should be a message queue in local storage for the async job to use to determine what needs to be sent to the server
** CR screens should write to this queue any updates
* 
* APIs (at a minimum):
** Get All CRs
*** Called when doing a full reconciliation to get all the distributed CRs, their status and attachments
*** Send the user id and any other details needed
*** Should return a success/failure code
** Get CR Delta
*** Called when doing a partial reconciliation for distributed CRs, their status and attachments by the scheduled sync agent
*** Send the user id and last date/time stamp and any other details needed
*** Should return a success/failure code
** Post Update CR Status
*** Called when a user opens an unread distributed CR and it is saved to the queue
*** Post the message id and any other details needed
*** Have server save the read status update on the server and then return a success/failure code
** Post Delete CR 
*** Use one for both inbox and sent
*** Called when a user deletes a distributed CR and it is saved to the queue
*** Post the message id and any other details needed
*** Have server execute the delete and then return a success/failure code
** Log all errors to a sync error log that is visible

h4. *Performance & Usability*

* Updates should have no impact on user experience, they should not know it is happening and there should be no UX lagging

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Log all API to the *error log system/database* for further analysis

## Acceptance Criteria
h4. User Flow

* The Inbox screen includes a Refresh button.
** When tapped, it immediately triggers the sync agent to perform a full reconciliation using the "Get All CRs" API.
** The screen displays the current sync status and indicates if there are any unsynced changes.
* Upon successful login, the sync agent is automatically triggered to perform a full reconciliation.
* Upon successful logout, the sync agent is automatically triggered to perform a full reconciliation.
* A scheduled sync job runs every n minutes (e.g., every 6 minutes) to perform a delta reconciliation by checking for changes since the last sync.
** The scheduled sync uses the "Get CR Delta" API.

h4. Data Requirements

* Local storage is used to store distributed Customer Reports (CRs).
** The local storage retains state even after the user logs out.
* A message queue in local storage tracks all updates (new, status updates, deletes) that need to be sent to the server.
** Updates from CR screens are written to this queue.

h4. API Integration

* "Get All CRs" API:
** Called during a full reconciliation (login, refresh, logout).
** Sends the user ID and required details.
** Returns a success/failure code along with all distributed CRs, their statuses, and attachments.
* "Get CR Delta" API:
** Called during scheduled delta syncs.
** Sends the user ID and the last sync timestamp.
** Returns a success/failure code along with changes since the last sync.
* "Post Update CR Status" API:
** Called when a user opens an unread distributed CR.
** Posts the message ID and necessary details.
** The server saves the read status update and returns a success/failure code.
* "Post Delete CR" API:
** Called when a user deletes a distributed CR.
** Posts the message ID and required details.
** The server executes the deletion and returns a success/failure code.
* All API endpoints use HTTPS, versioned URLs (e.g., /api/v1/...), and implement rate limiting.

h4. Performance & Usability

* The sync process must occur in the background without impacting the user experience.
* Updates to local storage and message queue processing must not cause UI lag or disrupt user interactions.
* The screen remains responsive, with all tap interactions providing clear visual feedback.

h4. Error Handling

* All asynchronous calls (including API calls and local storage updates) use try/catch for error handling.
* Error boundaries are implemented to capture any unexpected render failures.
* Any errors are communicated using user-friendly messages via toast notifications, modals, or inline messages.
* All errors (from API calls or rendering) are logged to the central error log system/database for further analysis.

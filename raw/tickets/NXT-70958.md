# NXT-70958 — Insights - KPI - PNA - Configuration - PE

## Description
*I am a* district user *that I want to* configure benchmark values for Paid Not Applied (PNA) in Insights
*so that I* can evaluate whether schools are meeting expected eligibility participation thresholds.

----

h2. *Notes*

* This story applies to *Insights – PrimeroEdge data source*.
* PNA (Paid Not Applied) is part of the *front office KPIs* being integrated into Insights.
* This story covers the *Benchmark Configuration screen only*.

h3. *Data & Source of Truth*

* The source of truth for PNA data is available in:
** *Student Eligibility → Eligibility Reports → Eligibility Roster*
* To generate the correct dataset:
** Eligibility Type = *Paid*
** Reason = *Default*
* This must be used by developers and QA to validate PNA values.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure:
** PrimeroEdge: [https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109335|https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109335|smart-link] 
* The benchmark configuration must use this stored procedure for PNA values.

h3. *KPI Definition & Formatting*

* PNA is a *percentage-based KPI*.
* Format must be:
** *Whole number percentage (no decimals)*
** Example: *25%*
* Values must:
** Be *integers only (no decimals)*
** Be within the range *0 to 100*
* Calculation:
** *(Count of students where Eligibility = Paid AND Reason = Default) ÷ Total relevant student population × 100*

h3. *KPI Direction*

* This KPI is *Lower is Better*:
** Values closer to *0% indicate better performance*
** Higher values indicate worse performance
* Performance indicators, comparisons, and trends must respect this direction.

h3. *Performance Consideration*

* Benchmark configuration must support large districts with many sites and multiple years without performance degradation.

h2. *Out of Scope*

* SchoolCafé implementation (separate story)

h2. *User Flow*

# A user logs into PrimeroEdge.
# The user navigates to *Insights*.
# The user opens the *Benchmark Configuration drawer*.
# The user selects a timeframe (year/period).
# The user views PNA values for each school.
# The user may:
#* Manually enter PNA benchmark values
#* Use autofill to populate values
#* Use bulk update to adjust values
#* View history of changes
# The user saves the configuration.
# The system persists values and makes them available for dashboard comparison.

----

h2. *Requirements*

h3. *1. Grid Updates*

* Add *PNA* as a column in the benchmark configuration grid:
** Label: *PNA*
** Location: *last column*
** Format: *whole number percentage (no decimals)*
* Ensure PNA is included in:
** API responses
** Dataset used by the configuration screen

----

h3. *2. Year Selection Behavior*

* When the selected year changes:
** PNA benchmark values must update accordingly
** PNA actual values must refresh based on the selected year

----

h3. *3. Manual Entry*

* Users must be able to manually enter PNA values per school.
* Input must:
** Accept numeric values only
** *Not allow decimals*
** Represent whole number percentages (e.g., 25 = 25%)

----

h3. *4. Input Validation*

* PNA is a *required field* when saving.
* Validation rules:
** Must be numeric
** Must be an *integer (no decimals)*
** Must be between *0 and 100*
* If validation fails:
** User cannot save
** Validation message is displayed

----

h3. *5. Autofill Behavior*

* Autofill must populate PNA values using:
** Previous year’s actual value (rounded to whole number)
** OR current year’s value if previous year is not available
* Autofill must behave consistently with other KPIs.

----

h3. *6. Bulk Update Behavior*

* Add PNA field to bulk update popup:
** Label: *PNA*
** Format: *whole number percentage (no decimals)*
* Bulk update must:
** Apply changes to all *visible rows (respecting filters)*
** Support incremental updates (e.g., +1, -5)
* Updated values must reflect in the grid immediately after applying.

----

h3. *7. View History*

* Changes to PNA must be:
** Stored in backend
** Available in history log
* History must include:
** Add
** Edit
** Delete actions

----

h3. *8. Save Behavior*

* Clicking save must:
** Persist PNA values to database
** Associate values with selected year/period
* Updated values must:
** Be available for dashboard comparison

----

h3. *9. Data Integration*

* PNA values must come from:
** PrimeroEdge stored procedure
* Data must support:
** Actual values
** Benchmark values
** Year-based filtering

----

h3. *10. Formatting & Consistency*

* PNA must follow:
** Whole number percentage format (no decimals)
** Existing benchmark configuration UI patterns
* UI components must be reused (no new UI patterns introduced)

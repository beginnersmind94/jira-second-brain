# NXT-64759 — SC - MT - District + New & Edit Screens

## Description
As a Customer Support role user, I want to be able to create a state realm instead of a district so that we can sell our product to states.

*Requirements*

* On the District page, do the following:
* Fields:
* Add a field for organization level the same as we have in PE
** Single select drop down list
** Label: Org Level
** Required
** Values:
*** Same as in PE:
**** District
**** State
**** Area
**** Sub Area
**** County
** Default Value: District
** Location: Under District Name
* Add a field to select the parent
** Single select drop down list
** Label: Parent
** Optional
** Values:
*** List of all realms (District, state, etc.) that have an “active” status
** Default Value: Not Selected
** Location: Under District Description
* Change field names:
** District Name > Name
** District Description > Description
* Update District Code:
** Move to the right behind Parent
** If user select any level other than District, disable this field and insert N/A as the text
* Update District Time Zone:
** New name: Time Zone
** Move to after Address 2
* Move Skip Onboarding Experience down to where Implementation Complete is
* Move Implementation Complete over to the right



Data:

* Update all APIs that other modules will call to include Level and Parent as necessary

## Acceptance Criteria
h3. *1. Org Level Field*

* A new field must be added to the *District page*.
* *Label:* {{Org Level}}
* *Type:* Single-select dropdown list.
* *Required:* Yes.
* *Values:* Same as in PE – {{District}}, {{State}}, {{Area}}, {{Sub Area}}, {{County}}.
* *Default Value:* {{District}}.
* *Location:* Directly under the *District Name* field.

----

h3. *2. Parent Field*

* A new field must be added below the *District Description* field.
* *Label:* {{Parent}}.
* *Type:* Single-select dropdown list.
* *Optional:* Yes.
* *Values:* All realms (District, State, Area, Sub Area, County) that have an *Active* status.
* *Default Value:* Not Selected.
* *Location:* Under *Description* field.

----

h3. *3. Field Name Updates*

* {{District Name}} → Rename to {{Name}}.
* {{District Description}} → Rename to {{Description}}.

----

h3. *4. District Code Field Behavior*

* *Location:* Move to the right, positioned after *Parent*.
* *Behavior:*
** If *Org Level = District*, field remains *enabled*.
** If *Org Level = State*, {{Area}}, {{Sub Area}}, or {{County}}, field is *disabled* and shows {{N/A}} as text.

----

h3. *5. Time Zone and Layout Adjustments*

* Rename *District Time Zone* → {{Time Zone}}.
* Move *Time Zone* to appear *after Address 2* field.
* Move *Skip Onboarding Experience* to the position of *Implementation Complete*.
* Move *Implementation Complete* to the *right* side of the layout (same row as Skip Onboarding Experience).

----

h3. *6. Data and API Updates*

* Update all related APIs to include the new fields:
** {{Level}} (Org Level)
** {{Parent}}
* Ensure all dependent modules that call these APIs correctly process the new {{Level}} and {{Parent}} fields.
* Verify that {{State}} and other new Org Level values are supported across all integrations and UI components.

----

h3. *7. Validation*

* All required fields must be validated before save:
** {{Name}} (was District Name)
** {{Org Level}}
* If *Org Level = State*, ensure {{Parent}} can be blank.
* If *Org Level = District*, {{District Code}} must not be {{N/A}}.

----

h3. *8. System Performance*

* Ensure adding these fields does not introduce performance lag or API timeout.
* Form save and load operations must complete within normal thresholds (<3 seconds).

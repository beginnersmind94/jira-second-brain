# NXT-60019 — TH - WS - Inbox Async Message Flows

## Description
*As a* TeamHub user *I want* to be able to asynchronously update my inbox and sent messages *So that* I can use my app when have no connection

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* It is a backend only story
* The objective is to enable “offline mode” for inbox and sent messages so that users can use the app without having an internet connection
* This will achieve 2 things
** User won’t have to go over the internet to get their messages when landing on the screen. They will come from a local store on their phone so it will speed up the load times on the screen
** If the user has no internet, they can still see their messages
* To achieve all this, you will ensure that:
** the Inbox and Sent screens read/write to a local store
** there is a sync agent that:
*** sends composed messages
*** downloads inbox messages
*** downloads sent messages
*** sends read/unread status updates
*** downloads read/unread status updates
*** reconciles when their is a discrepancy 

h2. Flows

* #1, user logs in:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** App kicks off all inbox and sent message reconciliation in the background without the user knowing it
** App gets latest changes for Inbox
** App gets latest changes for Sent
** App updates the local storage cache
** User clicks on Inbox and is redirected there
** User optionally views an unread Inbox message
*** App updates status in local store
*** App stores this update into the sync queue
*** App manually initiates a sync update to the server to change the read status
*** App checks for any new updates from the server and downloads if there are
** User optionally deletes an Inbox  (or sent) message
*** App deletes in local store
*** App stores this delete update into the sync queue
*** App manually initiates a sync update to the server to delete the message from the server
*** App checks for any new updates from the server and downloads if there are
** User optionally composes a message
*** Note: Applies for new, reply and forward messages
*** App stores message in local store
*** App stores this new message as an update into the sync queue
*** App manually initiates a sync update to the server to store the message in the email queue on the server
**** Includes attachments 
*** Server sends the email on behalf of the app
*** Server updates the Sent items with the sent message
*** Note: Server sends the email, not the app and it doesn’t go into server sent items unless successfully sent
*** App checks for any new updates from the server and downloads if there are
* #2, Sync Schedule
** App maintains a schedule to check for updates
** App identified that the specified time has passed and initiates the sync job
** Sync job sends any unsent changes to the server and checks for all updates from the server
** Sync job reconciles any discrepancies

h2. Requirements

* Inbox screen
** Refresh button
*** On click:
**** Initiate the reconciliation process by immediately calling the sync agent to do a full sync
*** Indicate the sync status and if there is anything not synced
* Login
** Upon successful login, initiate the reconciliation process by immediately calling the sync agent to do a full sync
* Logout
** Upon successful logout, initiate the reconciliation process by immediately calling the sync agent to do a full sync
* Schedule
** Run a delta reconciliation on a schedule per every n minutes
** Just look for changes since the last time
** choose a different number of minutes than from the other sync jobs like production and insights so that they don’t all try to run at the same time(e.g. 4 minutes for this one, 5 for production, 9 for Insights, etc.)

h4. *Data Requirements*

* Local Storage
** There should be local storage to save messages in
** Should maintain state even when user logs out
* Message Queue
** There should be a message queue in local storage for the async job to use to determine what needs to be sent to the server
** Inbox and Sent screens should write to this queue any updates
* 
* APIs (at a minimum):
** Get Inbox Full
*** Called when doing a full reconciliation to get all the Inbox messages, their status and attachments
*** Send the user id and any other details needed
*** Should return a success/failure code
** Get Inbox Delta
*** Called when doing a partial reconciliation for Inbox messages, their status and attachments by the scheduled sync agent
*** Send the user id and last date/time stamp and any other details needed
*** Should return a success/failure code
** Get Sent Full
*** Called when doing a full reconciliation to get all the Sent messages, their status and attachments
*** Send the user id and any other details needed
*** Should return a success/failure code
** Get Sent Delta
*** Called when doing a partial reconciliation for Sent messages, their status and attachments by the scheduled sync agent
*** Send the user id and last date/time stamp and any other details needed
*** Should return a success/failure code
** Post Update Status
*** Called when a user opens an unread message and it is saved to the queue
*** Post the message id and any other details needed
*** Have server save the read status update on the server and then return a success/failure code
** Post Send Message
*** Called when a user composes a new, reply or forward message and hits send and it is saved to the queue
*** Post the message id and any other details needed
*** Have server execute the email send and save in Sent on the server and then return a success/failure code
** Post Delete Message
*** Use one for both inbox and sent
*** Called when a user deletes an inbox or sent message and it is saved to the queue
*** Post the message id and any other details needed
*** Have server execute the delete and then return a success/failure code
** Log all errors to a sync error log that is visible

h4. *Performance & Usability*

* Updates should have no impact on user experience, they should not know it is happening and there should be no UX lagging

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Log all API to the *error log system/database* for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *General Functionality*

* The system enables *offline access* for Inbox and Sent messages by using *local device storage*.
* Users can read, delete, and compose messages without internet access, and all actions will be queued and later synced.
* Upon reconnecting or at scheduled intervals, the *sync agent* reconciles updates between the local store and the server.
* All backend logic applies only to *SchoolCafé (SC) districts*. PrimeroEdge support will be handled in a separate story.

----

h4. *User Flows*

_User Login Flow_

* On successful login:
** The app immediately triggers the *full reconciliation sync*.
** Calls:
*** {{Get Inbox Full}}
*** {{Get Sent Full}}
** Updates the local cache with new or changed messages, read status, and attachments.

_User Reads a Message_

* When a user opens a message:
** The app updates the *read status* in local storage.
** The read status is added to the *local message queue*.
** The sync agent is immediately triggered to:
*** Call {{Post Update Status}}
*** Check for new messages via delta sync ({{Get Inbox Delta}}, {{Get Sent Delta}})

_User Deletes a Message_

* When a user deletes a message:
** The app removes the message from local storage.
** The deletion is added to the *local message queue*.
** The sync agent is immediately triggered to:
*** Call {{Post Delete Message}}
*** Check for updates via delta sync

_User Composes a Message_

* When a user composes and sends a new, reply, or forward message:
** The message is saved in *local storage*.
** The message is added to the *local message queue*.
** The sync agent is immediately triggered to:
*** Call {{Post Send Message}}
*** If the server send is successful, the message is stored in server-side Sent items and synced back

_Scheduled Sync_

* A background job runs every *n minutes* (e.g., every 4 minutes) to:
** Call delta APIs ({{Get Inbox Delta}}, {{Get Sent Delta}}) to check for new updates
** Send unsynced changes in the queue to the server
** Reconcile any discrepancies in data

_User Logout_

* On logout:
** The sync agent runs a *final full sync* before clearing user session
** Ensures updates are pushed before session ends

----

h4. *Inbox and Sent Screen Requirements*

* *Refresh button* on the Inbox screen:
** When tapped, manually triggers the full sync agent to:
*** Call {{Get Inbox Full}} and {{Get Sent Full}}
*** Display sync status and unsynced items if applicable
* All read, delete, and compose actions:
** Must write to *local async storage*
** Must push a sync entry to the *message queue*

----

h4. *Data Requirements*

* *Local Storage*:
** Stores Inbox and Sent messages, attachments, and metadata (e.g., read/unread, starred)
** Must *persist across logouts and app restarts*
* *Message Queue*:
** Stores all offline actions: read status changes, new messages, and deletions
** Sync agent reads from this queue to push updates to the server
* *APIs Required*:
** {{Get Inbox Full}}: Fetches complete inbox data
** {{Get Inbox Delta}}: Fetches only inbox changes since last sync
** {{Get Sent Full}}: Fetches complete sent messages
** {{Get Sent Delta}}: Fetches only sent message changes since last sync
** {{Post Update Status}}: Sends read status updates
** {{Post Send Message}}: Sends new, reply, or forwarded messages with attachments
** {{Post Delete Message}}: Sends message deletion updates
* All API responses must include a success/failure code and be *logged in the sync error log* if failed

----

h4. *Performance and Usability*

* All sync jobs must be *invisible to the user* and *not interrupt* UI responsiveness
* All screens (Inbox, Sent, Compose, Details) must remain usable even if the sync is in progress or offline
* Data updates must not cause screen lag, delays, or stuttering

----

h4. *Error Handling*

* All asynchronous storage and sync agent operations must use *try/catch* blocks
* All unexpected UI errors must be handled by *error boundaries*
* All errors (API failures, sync issues, local storage failures) must be:
** Logged to a centralized *error log system*
** Notified to the user only if critical, using toast or inline messaging

----

h4. *Security*

* All actions are only available to *authenticated users*
* Sync jobs, message reads, writes, and queue processing must only run in a secure authenticated session context

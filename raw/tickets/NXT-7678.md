# NXT-7678 — Direct Certification - Extensions - Actions

## Description
*User Story*

As a district admin I want a way to review and approve or dismiss the DC extensions generated by the system search.

*Mockup*

[https://www.figma.com/proto/yrkJeGfd3f9hWjqm0qvfir/Edge2.0_V1?node-id=172630%3A45&scaling=scale-down-width|https://www.figma.com/proto/yrkJeGfd3f9hWjqm0qvfir/Edge2.0_V1?node-id=172630%3A45&scaling=scale-down-width|smart-link] 

*Requirements*

These options should only be available for the *current academic year.*

* User should have options for what they want to do with each potential extension. These can be in-line grid options if we’re able to do that.
** Match
*** This would extend the eligibility from the DC student over to the matched students.
** Dismiss
***  This would mark the potential match as reviewed.
** Checkboxes
*** User can make selections, and then use a ‘Dismiss Selected’ or ‘Match Selected’ button to perform the same action (with the same comment) for all students who have been selected.
* Potential extensions that have been acted on should not come back up again when the user is performing a search.
** If Student A potentially extending to Student B has already been dismissed, then don’t show it again. 
*** It’s been acted on. The user can use the Reviewed option in another story to re-review this.
* Matching students should also place them in the same household.
* We need to track history / audit log for all of these.
** First Noticed
*** When was this potential extension first picked up by the system? Full timestamp.
*** What was the matching criteria at that time?
** Matched / Dismissed 
*** When was this potential extension acted on?
**** What user did it? 
**** What action did they take? 
**** When? Full timestamp.

## Acceptance Criteria
User can approve a match, or dismiss it, for the current academic year only.
We need to keep track of these - once acted on, the same extension shouldn't come up again as a potential extension.
Eligibility is assigned and updated appropriately for the student being matched.
Students are placed in a household together as matches are processed.
History tracking of actions taken, when the potential match was first logged, etc.

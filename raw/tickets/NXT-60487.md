# NXT-60487 — TH - WS - Custom Reports Viewer Screen

## Description
*As a* TeamHub user *I want* a screen that shows me the data of my Custom Report (CR) *So that* I can quickly view it without having to browse to it.

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the custom reports view screen on the home workspace page in SC
* Note: APIs should exist for this already

h2. User Flow

* User opens the application and is directed to login screen
* On login success, user is redirected to home screen
* User clicks Workspace and is redirected there
* User clicks on Custom Reports (CR) and is redirected there
* User is able to click on a CR and view the report with current data on this screen
* User is able to filter the data
* User is able to download the data

h2. Requirements

* Ensure the screen supports dark mode

_Header Section_

* Have a back button to Customer Reports
* Page Title: Report Viewer
* Download button
** on click: download the grid data to csv



_Body Section_

* Report Grid
** Data: API call for the associated CR
*** Data is dynamic, don’t hard code any columns, get columns from Jason file, do like in SC
** Filters
*** Include filters in each row in a filters row
** Lazy Loading
*** Do lazy loading as best as possible
*** There might be very large dataset so try to load the top 10 rows if you can



h4. *Data Requirements*

* APIs
** Call the API to get the data for the report
** API configuration should be stored with the CR
* There is no touching the async local storage on this screen

h4. *Performance & Usability*

* The screen must load within 2 seconds and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use try/catch in all asynchronous calls and use error boundaries for any unexpected render failures.
* Errors must be communicated using user-friendly messages:
** Use toast notifications, modals, or inline messages without disrupting the user experience
* Log any API or rendering errors to the error log system/database for further analysis



_Security_

* Same as Custom Reports screen

## Acceptance Criteria
h4. *Theme Support*

* The screen supports dark mode. All elements (backgrounds, text, icons, and buttons) must adjust for dark mode with optimal contrast and readability.

h4. *Header Section*

* A back button is provided that navigates the user to the Customer Reports screen.
* The page title displays "Report Viewer."
* A download button is present in the header.
** Tapping the download button triggers the download of the grid data in CSV format.

h4. *Body Section*

* The main component is a dynamic report grid.
* The grid retrieves its data from an API call associated with the specific Customer Report.
** The grid columns are dynamic. Column definitions and order are determined by a JSON configuration file, as implemented in SC.
* A filters row is included above the grid.
** Each column has a filter input to allow users to filter rows.
* The grid implements lazy loading.
** If the dataset is large, only the top 10 rows are loaded initially, with additional rows loaded as the user scrolls.

h4. *Data Requirements*

* The screen retrieves report data by calling the appropriate API.
** API configuration details are stored with the Customer Report.
* No data is read from or written to asynchronous local storage on this screen.

h4. *Performance & Usability*

* The screen must load within 2 seconds under normal operating conditions.
* The layout must be responsive and render correctly on mobile, tablet, desktop, as well as in compact and expanded views.
* All tap interactions (including back button, download button, filter inputs, and grid scrolling) must be responsive and provide clear visual feedback.

h4. *Error Handling*

* All asynchronous calls must use try/catch for error management.
* Error boundaries are implemented to catch and handle any unexpected rendering failures.
* User-friendly error messages are shown via toast notifications, modal dialogs, or inline messages, without disrupting the user experience.
* Any API or rendering errors must be logged to the error log system/database for further analysis.

h4. *Security*

* The security requirements for this screen are the same as for the Custom Reports screen.
** Only authenticated users with the appropriate permissions can access the Report Viewer screen.
** Data retrieval and operations are secured via HTTPS, and proper rate limiting is enforced on API endpoints.

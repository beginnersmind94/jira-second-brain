# NXT-70620 — FIN — Post Inventory Purchases (PURCH) by Valuation Category × Vendor (Level 2)

## Description
As a Child Nutrition Director, I want inventory purchases to post automatically to the ledger at *Valuation Category × Vendor* detail (per site, per closed inventory period), so that postings remain GL-manageable while still reconcilable to vendor spend.

h2. Revised Requirements (Purchase posting uses Purchase Report as source)

h3. R1 — Source of truth (no custom derivation)

For each *Site + Inventory Period*, Financial Oversight must ingest the *Purchase Report dataset* generated by the Inventory module.

* Financial Oversight must *not* calculate purchases by traversing receipt tables or re-deriving totals.
* The Purchase Report dataset is the authoritative source for: *Vendor*, *Valuation Category*, and *Purchase totals* for the period.

h3. R2 — Amount field to use (new finding)

When ingesting the Purchase Report dataset, the posting amount must use *Amount / Total Receipt Amount*, not *Market Value*. Purchases

h3. R3 — Level 2 grouping (as reported, not recomputed)

Financial Oversight must post purchases at *Valuation Category × Vendor* detail *exactly as represented in the Purchase Report dataset* for the site-period.

Output grain:

* One total per *Site + Period + Vendor + Valuation Category*

h3. R4 — Posting output

For each (Site, Period, Vendor, Valuation Category) total from the Purchase Report dataset, create one debit/credit pair:

||Debit||Credit||
|Valuation Category → Inventory Asset GL|Cash in Bank GL|

h3. R5 — Missing vendor handling

If the Purchase Report dataset returns a row with no vendor identifier, group it under *“Unknown Vendor”* and still post.

h3. R6 — Mapping gates

If any required mappings are missing (VC Asset GL, Cash in Bank), the posting job fails and *creates zero entries*.

## Acceptance Criteria
h3. AC1 — Amount basis is used (not Market Value)

*Given* Purchases report (or underlying data) shows Amount ≠ Market Value for a vendor in a period
*When* PURCH posts
*Then* the posted total equals *Total Receipt Amount / Amount*, not Market Value. Purchases

h3. AC2 — VC × Vendor aggregation produces correct number of entries

*Given* Site A, Period P has:

* Vendor V1: VC-Food $10,000; VC-NonFood $1,000
* Vendor V2: VC-Food $5,000
*When* posting runs
*Then* exactly 3 debit/credit pairs are created (V1/Food, V1/NonFood, V2/Food), and no receipt/item-level pairs exist.

h3. AC3 — Debit/Credit accounts are correct

*Given* any PURCH posting line for (Vendor, VC)
*Then* Debit = VC’s Inventory Asset GL
*And* Credit = configured Cash in Bank GL.

h3. AC4 — Missing VC mapping blocks posting and lists all missing

*Given* receipts include VC1 and VC2
*And* VC2 Asset mapping is missing
*When* posting runs
*Then* nothing posts
*And* the error lists VC2 as missing (and any other missing mappings).

h3. AC5 — Missing vendor uses “Unknown Vendor”

*Given* at least one receipt has vendor null
*When* posting runs
*Then* amounts from those receipts are included under Vendor = “Unknown Vendor”
*And* totals still tie to the period’s Total Receipt Amount.

h3. AC6 — No receipts means no entries (success)

*Given* a closed inventory period with zero receipts
*When* posting runs
*Then* zero PURCH entries are created
*And* job is marked successful with “0 entries created.”

h3. AC7 — Reopen/reclose causes reversal + repost

*Given* period P previously posted PURCH
*When* period P is reopened and closed again
*Then* reversal entries are created and new entries are posted
*And* reversal + repost succeed or fail together.

----

h2. Edge / empty / error states

* Do not create $0 entries for any Vendor×VC combination.
* Vendor name changes do not affect grouping; grouping is by vendor ID (if available).
* If item-to-VC mapping changes after posting, repost uses the current mapping only upon reclose (by design).

----

h2. Defaults

* Posting detail level: *Level 2 (VC × Vendor)*.
* Posting basis: *Amount / Total Receipt Amount* (not Market Value). 
* Posted Date: *period end date*.
* Credit GL: single configured *Cash in Bank*.

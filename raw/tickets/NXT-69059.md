# NXT-69059 — Create system accounts when Financials license is activated (Backend)

## Description
When a district activates the Financials license, the system automatically creates the required system accounts so the district can map them to GL accounts and automate ledger postings.

## Acceptance Criteria
* When the *Financials user is onboarded*, the system reads the district’s *Program Configuration*.
* For each Program row where *Configured = Yes*, the system creates *one* POS revenue System Account.
** The name must be the *Program display name* shown in Program Configuration, *without the code in parentheses*.
** Examples:
*** Program shows {{School Breakfast Program (SBP)}} → create System Account named {{School Breakfast Program}}
*** Program shows {{National School Lunch Program (NSLP)}} → create {{National School Lunch Program}}
*** Program shows {{After School Snack Program (ASP)}} → create {{After School Snack Program}}
* The system creates these tax System Accounts:
** {{Sales Tax (Cash)}}
** {{Sales Tax (Debit)}}
* For each Program/Service row where *Configured = Yes & Reimbursable = Yes*, the system creates *one* reimbursement revenue System Account.
** The name must be the *Reimbursement - Program display name* shown in Program Configuration, *without the code in parentheses*.
** Examples:
*** Program shows {{School Breakfast Program (SBP)}} → create System Account named {{Reimbursement - School Breakfast Program}}
*** Program shows {{National School Lunch Program (NSLP)}} → create {{Reimbursement - National School Lunch Program}}
*** Program shows {{After School Snack Program (ASP)}} → create {{Reimbursement - After School Snack Program}}
* Create two accounts with the base type as asset:
** {{Due from Federal Funds}}
** {{Cash in Bank}}
* Create these student a la carte system accounts:
** {{A La Carte - Student - Breakfast}}
** {{A La Carte - Student - Lunch}}
** {{A La Carte - Student - Snack}}
* Create these adult a la carte system accounts:
** {{A La Carte - Adult - Breakfast}}
** {{A La Carte - Adult - Lunch}}
** {{A La Carte - Adult - Snack}}
* Create these adult meal system accounts:
** {{Adult Meals - Breakfast}}
** {{Adult Meals - Lunch}}
** {{Adult Meals - Snack}}
* Create the following additional system accounts with the corresponding base types:
** {{Account Adjustments — Expenditure}}
** {{Bonus — Expenditure}}
** {{Prepaid Account — Liability}}
** {{Refunds — Liability}}
** {{Returned Checks — Liability}}
** {{Sales Tax Owed — Liability}}
** {{Additional Income — Revenue}}
** {{Bank Fees — Revenue}}
** {{Deposit Variances — Revenue}}
* Immediately after activation, all created System Accounts must appear in the System Accounts list for GL mapping.
* If users within a district enable licenses multiple times over the year, ensure that no duplicate system accounts are created.

Note: All meal sales revenue accounts have the base type ‘Revenue'

*Valuation Category System Account Creation*

_Data Source_

* Valuation categories are sourced from Item Management > Item Configuration > Valuation Categories

_Auto-Creation Logic_

* For each valuation category configured in Item Management, the system creates two system accounts:
** {{One Asset account (for inventory value on balance sheet)}}
** {{One Expense account (for cost of goods/usage)}}
* Account names match the Valuation Category name from Item Management

_Columns Sourced from Item Management (read-only)_

* {{Valuation Category (name)}}
* {{Valuation Method (e.g., FIFO, Last Received Price, Moving Average, Replacement Value, Standard Price)}}
* {{Data Source (e.g., Local, State Level, Food Distribution)}}

_Business Rules_

* Every valuation category must have both an Asset Account and an Expense Account mapped before it can be used in postings
* Changes to account mappings only affect future postings and do not retroactively change historical posted values
* Each Asset GL account can only be linked to one valuation category
* Each Expense GL account can only be linked to one valuation category
* Once a valuation category has any posted activity, its Asset and Expense GL accounts become permanently locked
* If a district needs to change GL accounts after posting, they must create a new valuation category; historical mappings remain frozen for audit integrity

_Validation on GL Account Deletion_

* If user tries to delete a GL account linked to a valuation category, block deletion and require reassignment first

_Warning Messages_

* At time of initial assignment:{{ "Once this valuation category has posted activity, these GL accounts will be permanently locked and cannot be changed."}}
* When attempting to edit a locked category: {{"This valuation category already has posted activity, so its GL accounts can't be changed. To use different accounts, create a new valuation category and move the items to it. Historical mappings are locked for audit integrity."}}

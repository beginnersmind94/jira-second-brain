# NXT-71413 — SC - MT - District - Realm Tab History

## Description
As a *Customer Support user*, I want to view a history of all changes made on the *Realm tab*, so that I can audit who made relationship changes, what was changed, and when.

----

h1. Background / Notes

The application already supports audit logging in other areas using a *View History drawer pattern*.

For the Realm tab:

* The audit history should be *global to the entire screen*, not tied to individual rows
* This differs from standard implementations where history is per-record
* This is intended for *corporate-level auditing of relationship changes*

We will:

* Reuse the *existing View History template*
* Use the *new Tailwind CSS-based version*, not the legacy version

----

h1. Current Navigation

*System > Districts > Realm tab*

----

h1. Technical / Implementation Notes

* Use existing audit logging infrastructure already implemented elsewhere in the application
* Use the *new Tailwind CSS View History drawer component*
* Do NOT use legacy (non-Tailwind) history UI
* Audit log should be triggered from *Save action on this screen*
* Log entries should capture:
** Add (new realm relationship)
** Edit (changes to existing relationships, if applicable)
** Delete (removal of relationships)
* Audit scope is *global to the Realm tab*, not per row
* No additional security required:
** Page is already restricted to Customer Support users

----

h1. Functional Requirements

h2. 1. Add “View History” button

* Add a *View History* button to the Realm tab UI
* Placement should follow existing patterns used elsewhere in the system

----

h2. 2. Open history drawer

* On click of *View History*, open a *drawer*
* Drawer must use the *new Tailwind View History component*

----

h2. 3. Global audit scope

* The audit history displayed must include *all changes across all districts/realms*
* Not limited to:
** a selected row
** a selected district
* This is a *screen-level audit log*

----

h2. 4. Track all Save-driven changes

Audit log must capture all changes triggered by the *Save button*, including:

* Adding a realm relationship
* Removing a realm relationship
* Modifying assigned realms (if applicable)

----

h2. 5. Audit log details

Each audit entry should include standard fields consistent with existing audit logs:

* User who made the change
* Timestamp
* Action type (Add / Edit / Delete)
* Entity affected (Realm relationship)
* Before value (if applicable)
* After value (if applicable)

----

h2. 6. Consistent audit format

* Audit entries must follow the same format and structure used in other parts of the system
* Ensure consistency for:
** field naming
** timestamps
** display format

----

h2. 7. No additional security required

* The Realm tab is already restricted to Customer Support users
* No additional permission checks needed for View History button or drawer

## Acceptance Criteria
h2. AC1: View History button is visible

Given the user is on the Realm tab,
then a *View History* button is displayed

----

h2. AC2: Drawer opens on click

Given the user clicks View History,
then a drawer opens using the *Tailwind View History template*

----

h2. AC3: Global audit history displayed

Given the drawer is open,
then it displays audit entries for *all realm relationship changes across all districts*

----

h2. AC4: Add actions logged

Given a user adds a realm relationship and clicks Save,
then an audit entry is created indicating an *Add action*

----

h2. AC5: Delete actions logged

Given a user removes a realm relationship and clicks Save,
then an audit entry is created indicating a *Delete action*

----

h2. AC6: Edit actions logged (if applicable)

Given a user modifies an existing relationship and clicks Save,
then an audit entry is created indicating an *Edit action*

----

h2. AC7: Audit entry includes required details

Given an audit entry is displayed,
then it includes:

* user
* timestamp
* action type
* affected realm(s)
* before/after values where applicable

----

h2. AC8: Consistent UI with existing audit logs

Given the drawer is displayed,
then it matches the styling and behavior of the *new Tailwind-based View History component*

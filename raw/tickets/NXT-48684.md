# NXT-48684 — OpenID - District Authentication - Dashboard

## Description
As a Customer Support user, I want to be able to setup a district’s OpenID authentication provider so that they can use SSO

*Notes*:

* Districts will be able to configure more than one authentication provider
* Currently we use SchoolCafe authentication, but in this story will be adding OpenID
* To do this, we will be updating the existing Security page
* Note that a districts authentication settings apply to users, but remember that if there are more than one auth types, then the user will have the option which to use (covered in the user story)
* These requirements apply to permanent accounts

*Flow:*

* User browses to System > Security page
* User chooses which auth types they want to enable (only 2 choices at the moment, will add more later)
* If the auth type is SchoolCafe (current method), it will show the existing settings on the page
* If the auth type is OpenID, then it will show the OpenID fields we need for Keycloak

## Acceptance Criteria
* Password Policy tab only shows if the district has SchoolCafe Authentication as a row in the grid
* Authentication tab added and is first
* Grid on Auth tab has no paging, filtering or sorting but does have Add and History buttons
* User is able to add a new authentication type but not able to add a new one with the same name that already exists
* Addition is logged in history log
* If all the auth types have been added, user can’t add anymore
* History button opens the history drawer
* When adding or editing and auth type, user can only select the Authentication Type and if it is Preferred or not
* User is able to select which Auth Type is Preferred
* User is not able to Edit a SchoolCafe Authentication row
* User is able to click edit on an Open ID auth row and it opens the Open ID Config drawer
* User must always have at least one authentication type so can’t delete if there is only one
* SchoolCafe Authentication is the default type for all districts and added to any existing districts when code is migrated
* If a user readds an auth type, the original settings are retained and used so that the user doesn’t have to redo them
* Email sent to users when there is a new enabled authentication method
* Email sent to users who haven’t configured a method yet

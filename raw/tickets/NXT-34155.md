# NXT-34155 — Team Hub - View or Update Profile

## Description
As a school admin, I want to view the profile information used by the app so that I can verify if configured correctly.

 

*Flow:*

·         Admin clicks on Settings button and flyout opens

·         Clicks on Profile and redirected to Profile Screen

·         Views profile info

 

*Background:*

·         This info exists in the Profile page of the SchoolCafé website under Workspace > Profile

·         We are copying this page for the app

 

*Assumptions:*

·         User has logged in

·         Settings Screen completed

·         You can consume the profile information with a profile API

·         This page will only work with 2.0 customers

*Requirements*:

·         Profile is a 2.0 specific functionality

·         Therefore, on screen load test the platform the user is using by looking at the Platform field in the settings screen

o   If is it SchoolCafé, then allow the screen to be viewed

o   If it is PrimeroEdge, then hide the screen from the user

·          

·         There are 2 views for this info, Read-Only and Edit, toggled by an Edit icon

·         Depending on the view, show the relevant fields

·         These requirements follow the way the web page is done, but if there is a better way to do this natively, let’s talk about it

·          

·         Fields for both views:

·         Edit – Toggle

o   If Yes, then show the Edit view

o   If No, then show the Read-Only view

o   Further requirements below in Edit view mode

·          

·         Fields for Read-Only View:

·         All these field are read only labels and text except profile picture which is read only image with no label

·         Show the following labels with their relative value next to their name like on the web page

o   Profile Picture (note: no label, just show the picture)

o   Name

o   Status

o   Email

o   Cell Phone

o   Work Phone

o   Roles

o   Access

·          

·         Groupings

o   If possible and aesthetically pleasing, group Roles and Access into a section labeled Roles & Permissions like on the web page, but can not do if it doesn’t look good

o   The same goes for the other fields with a group name of Contact Info

·          

·         Fields for Edit View:

·         Profile Picture

o   No label, just picture

o   Have a change button near it that works like a traditional profile image control where you can take a picture or upload a picture and have the usual cropping / edit functions native to cell phones

o   When finish selecting the image, automatically save it to the server; don’t wait for the save button on the rest of the fields

·         First Name – Textbox

o   Required

·         Middle Name – Textbox

o   Not Required

·         Last Name – Textbox

o   Required

·         Email – Label

o   Read-only text

·         Cell Phone – Textbox

o   Not Required

o   Validator: If you have an out of the box phone validator control you can use to ensure there are no invalid characters, please use it, but don’t want to force the user into a US style phone format only

o   If you can, when the user clicks in the box, auto show or populate their cell phone number they are using

·         Work Phone – Textbox

o   Required

o   Validator: If you have an out of the box phone validator control you can use to ensure there are no invalid characters, please use it, but don’t want to force the user into a US style phone format only

·         Roles – Label

o   Read-only text

·         Access – Label

o   Read-only text

·          

·         Cancel – Image button  

o   Use a traditional delete or trash icon

o   On click, the app deletes / forgets the changes, and toggles the screen back to Read-only mode

·         Save – Image button      

o   Use a traditional Save or Submit icon

o   On click, the app does the following:

§  Run validators on required fields

§  If validated, use the Profile API to push the changes to the server

§  Push First Name, Middle Name, Last Name, Cell Phone, Work Phone fields

·         Other fields are read-only

§  Wait for success message

§  Toggle the screen back to Read-only mode with changes staying in place

§  If you get an error message

·         Show it to the user

·         Allow them to hit OK

·         Close message box

·         Stay on the edit screen with the changes in place

·          

·         Groupings

o   If possible and aesthetically pleasing, group Roles and Access into a section labeled Roles & Permissions like on the web page, but cannot do if it doesn’t look good

o   The same goes for the other fields with a group name of Contact Info

·          

·         Security

o   If user not logged in, redirects the user to the login screen

User can’t see data for another district

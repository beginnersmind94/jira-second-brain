# NXT-60033 — TH - Sec - SC Auth - Profile

## Description
*As a* TeamHub user *I want* to be able to register my mobile App in SchoolCafé *So that* I can enable login on the app.

h3. Comments:

* We need a way to figure out what platform and district to use to login the user on TeamHub
* We have decided that the best way to do that is create a “handshake” using a mobile app code in SC and TeamHub
* There are several parts to this:
** TeamHub District Config screen where the user enters the code (another story)
** SC System Settings where the district decides if it will use TeamHub or not (another story)
** User Profile where the user gets the mobile app code to enter on TeamHub District Config screen (this story) 
** API for TeamHub District Config screen to call that matches the code from the mobile to the user profile (another story)

h3. Flow:

* User downloads TeamHub on mobile and goes to login screen (_TeamHub_)
* Login screen detects this is the first time and redirects to TeamHub District Config screen (_TeamHub_)
* TeamHub District Config screen has message telling user to go to their User Profile in SC to get their Mobile App Code (_TeamHub_)
* User logs into SC and goes to Profile page (_SchoolCafé_)
* SC detects the user has never gotten a mobile code before and shows the Get Code button (_SchoolCafé_)
* User clicks get code button and system generates a unique alphanumeric code and saves in the DB for that user ID (_SchoolCafé_)
* System replaces the button with the code for the user (_SchoolCafé_)
** Note: code will always be shown to user in the future. never need to generate a new code. It is not a time-based code like one-time passcode
* User goes back to TeamHub District Config screen and enters the code (_TeamHub_)
* TeamHub calls API in SC to match the code from the mobile to the user profile mobile app code (_TeamHub_)
* API returns success message and other key data (_SchoolCafé_)
* User goes to TeamHub login screen and logs in with their SC username/password (_TeamHub_)

h3. Requirements:

* Do the following in User Profile:
* On page load, check if the user has ever requested a mobile app code
** If no: Show the button
** If yes: Show the code
* Add a button on the page under the profile
** Label: Get Mobile App Code
** Display the button if no code exists, else hide it
** On click:
*** Use a random code generator to create a new unique code
*** If possible, would prefer to start the code with either a P or S so that we can just look at the code and know if it came from PrimeroEdge or SchoolCafé
*** Needs to be alphanumeric
*** Store the code in a table in the DB used to map mobile app codes to user IDs
*** It is not necessary to store this value in the user object or table, it won’t be used in SC user interface anywhere
*** The code generator must ensure each new code is unique across the system
**** If a collision is detected (rare, but possible), the generator should retry until a unique code is produced
*** See sample code generator below. 
**** My thoughts are to use only numbers and capital letters and exclude I and O since there will be many older users with poor eyesight. This would give 45,435,424 unique codes which is more than enough.
** After generating the code and saving it to the DB:
*** Hide the button
*** Show the label
* Add a label on the page under the profile
** Label: Mobile App Code
** Display the label and code it it exists
** Note: 
*** once a code is generated for a user, it is for them for the life of the account
*** we will use if for user/app tracking and reporting purposes, so don’t delete it
* 
* *Security*
** Regular user can generate their own code
** Customer Support, Technical Support roles can also generate the code for them



_Sample Code Generator:_

{noformat}const generateSchoolCafeCode = () => {
  // Allowed characters: digits and uppercase letters excluding I and O
  const allowedChars = "0123456789ABCDEFGHJKLMNPQRSTUVWXYZ";
  const codeLength = 6; // Total code length
  let code = "S"; // Hardcode 'S' as the first character

  // Generate 5 additional characters
  for (let i = 1; i < codeLength; i++) {
    const randomIndex = Math.floor(Math.random() * allowedChars.length);
    code += allowedChars[randomIndex];
  }
  
  return code;
};

// Example usage:
console.log(generateSchoolCafeCode());
{noformat}

h3. Explanation

* *Hardcoded Prefix:*
The code always starts with {{"S"}} because this generator is only for SchoolCafé.
* *Allowed Characters:*
The allowed character set includes digits ({{0-9}}) and uppercase letters, excluding lower-case letters as well as the letters {{I}} and {{O}} to avoid confusion.
* *Loop:*
A loop runs 5 times (since the first character is already set) to build the rest of the code randomly from the allowed characters

## Acceptance Criteria
h4. *On Page Load*

# *Check Existing Code*
#* The system checks whether the user already has a mobile app code stored in the database (mapping table).
#* If *no code* exists for the user:
#** Display the *"Get Mobile App Code"* button.
#** Hide or do not display the code label.
#* If *a code* exists:
#** Hide the *"Get Mobile App Code"* button.
#** Display the code label, showing the existing code.

h4. *Code Generation & Display*

# *Button Visibility*
#* The *"Get Mobile App Code"* button is only visible if no code exists for the user.
#* Once a code is generated, the button is hidden.
# *On Button Click*
#* A new unique code is generated using the random code generator.
#* The code:
#** *Must be alphanumeric* (letters and numbers).
#** *Should start with* either *"P"* or *"S"* (if possible), indicating the user’s platform (PrimeroEdge or SchoolCafé).
#* The newly generated code is stored in a dedicated *mapping table* (e.g., {{mobile_app_codes}}) linking the code to the user’s ID.
#* After storing:
#** The *"Get Mobile App Code"* button is hidden.
#** The *"Mobile App Code"* label is displayed, showing the newly generated code.
# *Code Persistence*
#* Once a code is generated for a user, it remains associated with their account indefinitely (i.e., “for the life of the account”).
#* The code is *not* deleted or changed in future sessions.
# *Label Display*
#* A label *"Mobile App Code"* appears under the user’s profile information whenever a code exists.
#* The label displays the actual code value.

h4. *Security & Permissions*

# *User Roles*
#* *Regular users* can generate their own code.
#* *Customer Support* and *Technical Support* roles can generate the code on behalf of any user.
#* No other roles should have the ability to generate or modify a user’s code unless explicitly granted.
# *Data Handling*
#* The code must *not* be stored in the main user table; it should be kept in a separate table for mapping and reporting.

h4. *Validation & Error Handling*

# *Unique Code Generation*
#* The code generator must ensure each new code is unique across the system.
#* If a collision is detected (rare, but possible), the generator should retry until a unique code is produced.
# *Failure Scenarios*
#* If the code generation or database save fails, an error message is displayed, and the user remains able to retry.
#* The button remains visible if the save fails, so the user can attempt code generation again.

h4. *Performance & Logging*

# *Response Time*
#* Generating and storing the code should complete within *2 seconds* under normal load.

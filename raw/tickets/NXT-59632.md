# NXT-59632 — WS - Benchmark Configuration

## Description
As a Workspace user, I want to configure KPI benchmarks so that I can set appropriate performance targets for my program.

h2. Comments

*Component*: KPI Benchmark Configuration

Enables:

* Benchmark setting
* Goal configuration
* Target adjustment
* School Year-specific targets
* District-specific targets
* School-specific targets
* Level-specific targets

h2. User Flow

# User logs into Workspace and goes to Insights page
# User opens benchmark config drawer by clicking on the button
# User can:
#* View current benchmarks
#* Adjust targets
#* Save settings

h2. Requirements

h3. Display Elements

* Filters:
** Note: Dallas was unable to include these filters in Bolt, but they are needed
* School Year
** Show current School year and next year
** Do not include historical school years
*** We don’t want them to be able to modify historical records
* Level:
** Use the list from the site selector
** All Schools (= District)
** Then list the site groups in alphabetical order
** Then list the individual schools in alphabetical order
* 
* List of KPI Cards
* Each KPI card has:
** KPI name
*** Read Only text
** Current benchmark
*** Read Only text
*** this is the value that is saved in the DB
*** note that it is for the selected school year in the filters
*** note that a user may update the textbox but it doesn’t affect the current benchmark unless they hit save
** Goal value
*** Textbox
*** Optional
*** this is a textbox that the user can change the benchmark
*** Default its value to the same as Current benchmark
** Performance range
*** slider linked to Goal value
*** Its range (min-max) should be determined by the type of value it is.
*** if a percentage, then 0-100%
*** if money (number), use this logic:
**** if there is a current value, set it at the midpoint, min = 0 and max = 2xmidpoint
**** if there is no current value:
***** see if there is a value in the textbox and if there is, set it to the midpoint
***** else set min = 0 and max = 100 until they put in a textbox value
** Description
*** read-only text, get value from API
** 
** See Bolt code for list of KPI cards and their descriptions
*** Pay attention to which are %, money or number
* Save button
** On click:
*** validate the fields
*** see what has changed
*** for anything that has changed, save the values to the database
*** post a success toast message
*** update the current values and sliders
*** stay on the same drawer, don’t close it as they will likely make changes to more than one page
* Cancel button
** On click, close the drawer

## Acceptance Criteria
h4. *User Permissions & Access Control*

* *Customer Support and Technical Support roles* can access and modify all KPI benchmarks.
* *District users* must:
** Have an active *Insights license*.
** Have *role permissions* to view and configure insights.

----

h4. *Filters*

* *School Year:*
** Displays only the *current school year and the next school year*.
** *Does not* include historical school years (modifying past benchmarks is not allowed).
* *Level:*
** Uses the *site selector list*.
** *Hierarchy order:*
**# *"All Schools" (District level)*
**# *Site Groups* (Alphabetically ordered)
**# *Individual Schools* (Alphabetically ordered)

----

h4. *List of KPI Cards*

Each KPI card displays:

* *KPI Name* → *Read-only* (from API).
* *Current Benchmark* → *Read-only* (retrieved from the API for the selected school year).
* *Goal Value Input* → Editable *textbox*:
** Users can modify this field.
** Defaults to the *Current Benchmark value*.
* *Performance Range Slider*:
** *Linked to Goal Value*.
** *Slider Range Logic:*
*** *Percentage-based KPI*: Min = 0%, Max = 100%.
*** *Money-based or number-based KPI*:
**** If a *current benchmark exists*, it is set as the *midpoint*, with:
***** *Min = 0*
***** *Max = 2x midpoint*
**** If *no benchmark exists*:
***** If *goal value is entered*, use it as the midpoint.
***** Otherwise, set *Min = 0, Max = 100* until user inputs a goal value.
** *Description:*
*** *Read-only text*, retrieved from the API.

----

h4. *Save Button Functionality*

* *On Click:*
** Validate all fields.
** Identify any *changed values*.
** Save *only modified values* to the database.
** Display a *success toast notification*.
** Update the *current benchmark values* and *performance range sliders*.
** Stay on the drawer (*do not close it*), allowing the user to continue modifying other benchmarks.

----

h4. *Cancel Button Functionality*

* *On Click:*
** Close the drawer without saving any changes.

----

h3. *Data & API Requirements*

h4. *APIs*

# *Retrieve School Year Data*
#* Returns *current school year and next school year* only.
# *Retrieve Level Data*
#* Returns *All Schools, Site Groups, and Sites* (same as the Dashboard).
# *Retrieve KPI Benchmark Data*
#* Returns:
#** *KPI Name*
#** *Description*
#** *Current Benchmark Value*
#** *Type* (Percentage, Money, Number)
# *Save KPI Benchmark Data*
#* Posts *updated values* back to the database.

h4. *Security & API Standards*

* All API communications *must occur over HTTPS*.
* API endpoints are *versioned* (e.g., {{/api/v1/...}}) to maintain compatibility.
* API requests *must be rate-limited* to prevent *abuse and denial-of-service attacks*.
* Ensure *data security* during retrieval and processing.

----

h3. *Performance & Usability*

* Ensure the page displays correctly in different layouts:
** *Full Width*
** *Compact*
** *Mobile Responsive*
** *Expanded*
* *Loading Speed*:
** The dashboard and configuration panel *must load within 2 seconds* under normal conditions.
** Interactions (such as opening panels and switching views) must be *smooth and responsive*.
* *Validation & Sanitization*:
** *Validate user inputs* to ensure correct values.
** *Sanitize text input fields* to prevent *injection attacks*.

----

h3. *Error Handling & Logging*

* *Use Error Boundaries* to capture UI-level issues.
* *Use Try/Catch Blocks* in all *asynchronous API calls*.
* *User-Friendly Error Messages*:
** Use *modal dialogs, toast notifications, or inline messages* to notify users of errors *without disrupting their experience*.
* *Database Logging*:
** Errors *must be logged* into the database for tracking and debugging.

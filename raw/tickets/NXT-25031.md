# NXT-25031 — Platform - BO - DM - Import users file for new customers

## Description
As a district admin or member of the Expert Care team, I want to bulk upload my site’s users so that I can speed up the process of onboarding as a new user.

*Background:*

This activity already exists in Classic and can be found here: [https://online.primeroedge.com/System/PrimeroManageUser.aspx|https://online.primeroedge.com/System/PrimeroManageUser.aspx] or System > Sites and Users > Users.

On that page, there is a button that says “Import User”.  We are mimicking that functionality and need to make the relevant updates in 2.0 to using Couchbase. 

It takes an excel file that is provided by us and is preformatted in a specific way.  That file is in the attachments.

*Existing Work:*

There was already some work done to import this data from Classic to 2.0.  You might be able to find those import scripts and start from there.  The source data will look a little different, but the insert scripts should be the same.

Also, the original code in classic to parse the excel file might be of use to you.

*General Flow:*

* User browses to [https://qa.perseusedge.com/system/Users|https://qa.perseusedge.com/system/Users|smart-link]  or System > Sites and Users > Users
* User selects button to import a file
* User selects an excel file in an upload file dialogue
** File is preformatted and provided by us
** Someone in customer care is presumed to have reviewed the file
* User selects upload
* System opens and parses the excel file
* System verifies required data is present
* System validates values, lengths
** Throws errors back to user and stops import if anything doesn’t validate
* System does any translations necessary for import
* System imports into Couchbase
* System closes upload dialogue box
* User then browses to report screen to verify the upload worked

*Requirements:*

*Users screen (System > Sites and Users > Users) updates:*

* Add a button (UX needed):
** Proposed text: Import Users
On click, open a file upload dialogue box (see below)
** After a successful upload, refresh the screen to include the imported records
*** If not successful, refresh is not required
** This button is only visible to those who have rights (described below)

*File upload screen (new):*

* UX needed - this screen should be consistent with the UX of other file upload components previously done
Fields:
* “Select file” upload box
** Standard file upload box
** UX should be same as in other places
** If can, filter to csv, xls, xlsx files
* Cancel button
** Upon click, cancel the upload process, close whatever is open and return to the Users page
* Message field
** Text field, read only, to show success and error messages
** Formatting should be consistent with elsewhere in 2.0
* Explanation text
** Read only text - “Allowed extensions: .xlsx,.xls,.csv.  Max Size is 100 mb.”
** Show near the upload box all the time
* Upload button
** This is the submit button
** Upon click, validate the file extension
*** If fails, show message: “File extension not accepted”
** Upon click, validate the size of the file
*** If fails, show message: “File too large”
** If validates, start the import routine (see below)
** If import routine is successful, show in the Message field: “Import successful”
** If import routine is not successful, show in the Message field: “Import failed” and then show all the error messages from the import routine
** Then allow user to close out the file upload screen and return to the Users screen or start the process again
* This page is only visible to those who have rights (described below)
 

*Import Routine*

* It is my opinion that in an effort to minimize complexity, that we should take an “all or nothing” approach to validation.  I think we should force the user to give us a clean file that completely validates before we try to do any data massaging or database work.  These requirements follow that approach.
See the attached import file for fields and requirements given to end users

* Validation
** First, validate the file
I strongly prefer that you do “cumulative validation” rather than “singular validation”.  What I mean by that is it is best to find all the validation errors at one time and tell them all at once, rather than finding one and stopping.  If they have 5 errors, tell them all five the first time, don’t tell them only about the first one, then keep them cycling through the process over and over.
For any validation error, append the error text to what will be returned and shown to the user
** Validate Required Fields:
*** Fields that are required:
**** Username/WindowsUserName
AuthenticationMode
First Name
Last Name
User Organization Level
UserStatus
StateLogin
DistrictLogin
Region
*** Error Message: “\[Field Name] is required on Row \[number]”
** Validate Length of fields
*** These values come from Classic db requirements.  If Couchbase needs different lengths, let me know.
User Key (ID) – 50 or empty
Username/WindowsUserName – 50
AuthenticationMode – 100
First Name – 50
Last Name – 50
Middle Initial – 50 or empty
Email – 50 or empty
Work Phone – 50 or empty
Cell Phone – 50 or empty
User Organization Level – 50
UserStatus – 7
StateLogin – 5
DistrictLogin – 5
Region – 200
Site – 200 or empty
List of roles for user in State realm – 200 or empty
List of roles for user in district realm – 200 or empty
Error Message: “\[Field Name] too long on Row \[number]”
If it says “or empty”, that means that it is not a required field, therefore if there is no value in the fields, skip it, but if there is a value than validate it
** Validate Username/WindowsUserName
*** If AuthenticationMode = “Windows”, validate:
value must contain a “\\” in the string
Error Message: “Username/WindowsUserName must be in Domain\\WindowsUsername format on row \[number] if using Windows authentication”
** Validate Username/WindowsUserName does not exist already
*** Use the value and compare to list of users already in the database for this site or state to confirm it is not a duplicate
Error Message: “Username/WindowsUserName in row \[number] already exists”
** Validate AuthenticationMode
*** Check the value against the NF_ADFSConfiguration table to ensure that it is a valid value
Error Message: “AuthenticationMode not recognized on row \[number]”
** Validate UserStatus
*** Value must be one of the following: TRUE, T, 1, Y, YES, FALSE, F, 0, N, NO,ACTIVE,INACTIVE (capitalization doesn’t matter”
Error Message: “UserStatus invalid on row \[number]”
** Validate StateLogin
*** Value must be one of the following: TRUE, T, 1, Y, YES, FALSE, F, 0, N, NO (capitalization doesn’t matter”
Error Message: “StateLogin invalid on row \[number]”
** Validate DistrictLogin
*** Value must be one of the following: TRUE, T, 1, Y, YES, FALSE, F, 0, N, NO (capitalization doesn’t matter”
Error Message: “DistrictLogin invalid on row \[number]”
** Validate Region
*** Check the value against the district / region in the database to ensure that it is a valid value
Error Message: “Region not recognized on row \[number]”
** Validate Sites
*** this field is not required, so skip if there is no value
If User Organization Level = “Site”, then validate:
*** Parse the string:
**** it uses “;” to separate a list of site codes (ex: 001;002;003)
may be only one site
*** Check the value of each site code in the list against the site codes in the database to ensure that it is a valid value
Error Message: “Site not recognized on row \[number]”
** Validate State Roles
*** this field is not required, so skip if there is no value
If User Organization Level = “State”, then validate:
*** Parse the string:
**** it uses “;” to separate a list of roles (ex: State Director;State Reviewer;Administrator)
may be only one role
*** Check the value of each role in the list against the roles in the database for this state to ensure that it is a valid value
Remember, you can get the state from the logged in user if needed
Error Message: “State roles not recognized on row \[number]”
** Validate District Roles
*** this field is not required, so skip if there is no value
If User Organization Level = “Site” or “District”, then validate:
*** Parse the string:
**** it uses “;” to separate a list of roles (ex: Cashier; Manager)
may be only one role
*** Check the value of each role in the list against the roles in the database for this site to ensure that it is a valid value
Remember, you can get the site from the logged in user if needed
Error Message: “Site roles not recognized on row \[number]”
** If the import file passes all these validations, then proceed to translation below
If it fails in even one validation, then do not proceed in the process, but stop and return the error message with all the errors appended to it to the user and show those errors in the message section
*  
* Translation
** Second, do any translation work you need to get it ready for Couchbase
** I don’t have any specific requirements for this as they will be determined by dev team
*  
* Import
** Finally, import the data
** In order to reach this phase, the import file must have completely validated
*** If not, stop here and return the validation errors to the user
** Perform an insert routine into the relevant database tables
** Verify that the data was inserted properly and return a success or failure
*** If can return the failure message, please do
*** If you can insert transactional (all or nothing) would be best
*  
* Security
** Only the following users can use the functionality described in this story
*** District users with Admin rights
*** Cybersoft employees with Admin rights
*** Members of the Export Care team

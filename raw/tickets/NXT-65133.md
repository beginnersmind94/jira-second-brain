# NXT-65133 — Add New Entry (UI)

## Description
As a School Finance Administrator, 

I want to create journal entries with flexible account selection and validation, 

so that I can efficiently record both template-based and manual transactions while ensuring financial accuracy.

Dev Notes

* Only manual entries are in scope for this story

## Acceptance Criteria
h2. UI Section:

* -*Navigation*: "Add New Entry" button on Ledger/Journal View pages navigates directly to the Add New Entry form -
* *Informational Guide*: Page displays info box with text: "💡 Quick Guide: Use Quick Entry mode by selecting a GL Entry Type to quickly populate accounts based on common transaction patterns, or enable Manual Entry mode to build fully custom entries from scratch. All entries must balance (debits = credits)."
* -*Site Selection*: Searchable Site dropdown (required with red asterisk) displays all sites in "Site Name - Code" format with type-to-filter functionality-
* *District Level Posting*: "🏛️ Post to District Level" checkbox below Site field with italic helper text "Use this for district-wide transactions that don't belong to a specific school site". When checked, Site field is disabled, set to "District Level", and the site is no longer required
* -Provide a field where user can select the Posted Date which cannot exceed today’s date -
* -System must auto-capture the actual timestamp when the user saves the entry (entered timestamp), which should populate the “Entered Date” column in the general ledger and journal display table.-
* -*Entry Table*: Multi-line entry table with columns (Account, Debit, Credit, Actions). Default shows one row with "+" button to add rows and "−" button to remove rows (last row cannot be removed)-
* -*Real-Time Validation*: System calculates total debits and credits in real-time as user types, displays balance status, and enables "Make Entry" button only when debits equal credits and both totals are greater than zero-
* *-GL Entry Type Dropdown-*-: Required dropdown populated with all configured GL Entry Types. When no type is selected, account dropdowns show placeholder "Select an entry type first..."-
** This will be dealt with once the system rules are configured and is non-functional for now
* *Quick Entry Mode (Default)*: -When GL Entry Type is selected, all account dropdowns filter to show only the template-defined accounts from System Rules for that specific entry type, providing guided repeatable entry creation-
** This will be dealt with once the system rules are configured and is non-functional for now
* *Manual Entry Checkbox*: "📝 Manual Entry (build from scratch)" checkbox below GL Entry Type field. When checked, GL Entry Type dropdown is disabled/grayed out and no longer required (red asterisk hidden)
** Default behavior for now is that the manual entry checkbox is selected and cannot be unselected
* *Manual Entry Mode*: When Manual Entry checkbox is checked, all account dropdowns populate with complete list of GL accounts from GL Accounts configuration page with searchable type-to-filter functionality
* *-Mode Switching-*-: Unchecking Manual Entry re-enables GL Entry Type dropdown (makes it required again) and account dropdowns revert to showing template-defined accounts based on selected entry type-
** This will be dealt with once the system rules are configured and is non-functional for now
* -*Account Dropdown Behavior*: All account dropdowns are searchable with type-to-filter functionality allowing users to type and find accounts quickly-
* -*Dynamic Row Accounts*: Newly added rows automatically inherit current filtering rules (Manual Entry shows all GL accounts)-
* -*Form Validation*: All required fields must be completed and debits must equal credits before "Make Entry" button is enabled-
* -*Balance Display*: System displays current debit total, credit total, and balance status message to user in real-time as amounts are entered-

h2. Backend Section:

* -*Transaction ID Generation*: When user clicks "Make Entry" button, system generates a unique transaction ID for the journal entry that can be used for tracking and reference-
* -*Description Recording*: System saves the Description/Reason entered by the user and associates it with all line items in the transaction-
* -*Debit Line Recording*: For each row with a debit amount, system creates a debit record in the general ledger with the selected account, debit amount, transaction ID, date, and description-
* -*Credit Line Recording*: For each row with a credit amount, system creates a credit record in the general ledger with the selected account, credit amount, transaction ID, date, and description-
* -*Multi-Line Transaction Linking*: All debit and credit lines entered in a single submission are linked together by the same transaction ID, allowing the complete entry to be viewed as one cohesive transaction-
* -*Site Code Assignment - Site Level*: When a specific site is selected (e.g., "Lincoln High School - 001"), system extracts the numeric site code (001) and embeds it in the account structure as the site segment (e.g., 1000-2000-3000-001-50000)-
* -*Site Code Assignment - District Level*: When "Post to District Level" checkbox is checked, system uses "000" as the site segment in the account structure (e.g., 1000-2000-3000-000-50000) instead of a specific site code-
* -*GL Account Mapping*: System records transactions using the GL account codes selected from the account dropdown (e.g., "1010 - Cash in Bank" records as account 1010 with appropriate site segment)-
* -*Entry Type Recording*: System marks all entries as "Manual Entry" type for reporting and audit purposes-
* -*Balance Verification*: System verifies one final time that total debits equal total credits before committing the transaction to the general ledger-
* -*Transaction Confirmation*: After successful recording, system displays confirmation message with the transaction ID and returns user to the Journal View or Ledger View showing the newly created entry-
* -*Audit Trail*: System records the username and timestamp for audit and compliance tracking-
* -*Reversibility*: Recorded transactions can be identified by their unique transaction ID for future reversal or adjustment if needed-

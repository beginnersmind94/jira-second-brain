# NXT-7681 — Direct Certification - Matching

## Description
*User Story*

As a district admin I want the system to take my student data, my direct cert (DC) file data, and make intelligent comparisons so that more students are matched.

*Mockup*

N/A

*Requirements*

This is the ‘brains’ of the DC matching process. 

* The system needs to take iterative passes comparing available DC File Data against Student Data.
** This largely depends on the data provided by the DC file. 
** If no ID numbers are provided, we shouldn’t bother with attempting to do ID number comparisons.
* The system needs to record if a match in data happens, and keep a hard copy of the data provided for both the student and from the DC file for future review.
** We’d want to see the student data as of the approval date if we ever look back at this, not the latest information.
** PrimeroEdge had an issue regarding this - whenever a user wanted to look back at this ‘scorecard,’ they were shown current student data, not student data as of the time of matching.
* The following passes should take place:
** ID Numbers
*** Student ID
*** Social Security Number
** Student Details
*** Names
**** Full Name
**** First Name
**** Middle Name
**** Last Name
**** *Fuzzy Name Matching as a last resort. (This is in another story)*
***** Algorithms:
****** Jaro Winkler
****** Levenshtein (Release 2)
****** Metaphone 2.0
*** Date of Birth
*** Gender
** Guardian Details
*** Full Name
*** First Name
*** Last Name
*** Contact Information
**** Street Address
***** We also need to be handle abbreviations properly in the middle of address. Like this example:
****** 123 *N* West Street from DC file, 123 *North* West Street from Student data.
****** NW = North West
**** Phone Number
* We need to keep track of how many points were allocated for each matching criteria, and a total value.
** We may end up with more than 100 “percentage points” but we’ll only want to show up to 100% in the UI later.
** Users will have access to change criteria values - we need to show a true account for how many points were allocated. 
** Example: A student was matched on 8/1/20 for 50 pts due to First Name + Last Name matching, If a user changes the value for First Name from 20% down to 10%, this match from 8/1/20 should still show 50 pts and not reduce.
* All matches that clear the Discard Percentage should be retained with copies of the information used to reach that percentage.
* If the Automatic Match Percentage is reached, eligibility should be extended to that student automatically.
** UNLESS a criteria set as must-match does not match.
** Example - region is configured that ‘date of birth’ must match for an automatic extension to happen. If date of birth does not match, then this would be potential match instead.
* Any other matches that are ‘good enough’ would be treated as Potential Matches.
** These should be made available for review in subsequent stories.

Automatic matches are happening if the percentage threshold is high enough.
Data is recorded for later review - both student data and file data.

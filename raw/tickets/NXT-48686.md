# NXT-48686 — OpenID - Login & Authorization Updates

## Description
As a SchoolCafe user, I want to be able to login to the site using my district’s OpenID authentication provider so that I don’t have to remember multiple passwords

*Notes*:

* We will be adjusting the flow of the login page to account for multiple authentication methods
* The authentication method is provided by the User Profile which in turn is controlled by the District profile

*Flow:*

* User browses to Login page
* User asked for username and hits Continue
* System checks the authentication type for the username
* If the auth type is SchoolCafe (current method), It will then hide the Continue button and show the Password fields (like currently does) where the user logs in like they normally do
* If the auth type is OpenID, then the Continue button redirects the user to their SSO provider, they authenticate there and are then redirected to Workspace

*Requirements*:

* These requirements apply to the login page
* Update the Login page to be in 2 screens
* Screen 1
** Show this when the user lands on the page
** Hide the following:
*** Password field
*** Remember Me
*** Forgot Password link
*** Sign In button
** Show new field
*** Continue button
**** On click:
**** Get the username (Email field) and look it up in the DB
**** If not found, do the same thing that you currently do
**** If found, then get the authentication types for the user account
**** If SchoolCafe Auth Type or Unknown for some reason, then show Screen 2
**** If OpenID is the only authentication type:
***** Get Keycloak Open ID configuration for User’s default district
***** Then redirect the user to the district’s OpenID provider 
****** Get URL from Keycloak’s Open ID configuration for that district
***** If OpenID information missing, show error message: Incorrect SSO configuration for your district, please contact your administrator for help.
**** Else, redirect to Screen 2
** If the user is unable to authenticate with SSO provider, they are redirected to the SSO Failure screen
*** Covered in another story
* Screen 2
** This is the screen as it currently is
** Add the following to it:
** If the user has OpenID as an option, then show them the SSO button:
** Button
*** Under Login button
*** Label: Login using SSO
*** Visible only if user has SSO as a valid authentication type
**** Else, don’t show
*** On click:
**** Redirect the user to their SSO login as you would from screen 1 if they only had SSO
**** Follow the SSO process from there
*** 
*** Note: we will add other buttons for the other authentication types when we get to those stories and they will work similarly to this one

## Acceptance Criteria
* New screen only shows Username and Continue button when a user lands on it
* If the user is using an OpenID auth provider only, they are redirected to that provider screen when hit continue
* If the user is using both as options, then they can choose between the 2 on screen 2
* If the user is using our SchoolCafe authentication, the normal login screen is used
* User is able to login to SchoolCafe via a 3rd party using Keycloak OpenID Connect broker
* User is shown error message for missing username
* User is shown error message for missing Keycloak OpenID config info

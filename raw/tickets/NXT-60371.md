# NXT-60371 — WS - Benchmark Configuration New Drawer

## Description
We need to update the Benchmark Config screen per Dallas' new design

h2. Comments

* Dallas did a new design for Benchmark config, but it is slightly incomplete
** It shows the grid, but is missing the actions
* This user story covers what is needed and what he did

h2. User Flow

# User logs into Workspace and goes to Insights page
# User opens benchmark config drawer by clicking on the button
# User can:
#* View current benchmarks/targets
#* Adjust targets
#* Save settings

h2. Requirements

h3. Display Elements

* Filters:
** Note: Dallas was unable to include these filters in Bolt, but they are needed
* School Year
** Show current School year and next year
** Do not include historical school years
*** We don’t want them to be able to modify historical records
** On selection change:
*** If possible, check if there are any pending changes in the textboxes and warn the user to save first if they want to retain them
*** Update the textboxes in the grid with the current values for that school year
** Default to current year
* 
* Grid of Sites (rows) and their related KPI values (columns)
** Data:
*** Rows show all the sites for the district
*** Columns show the relative KPI value
** Grid columns are editable textboxes
** Each textbox should default to the current benchmark/target
** Allow the user to save
** Textboxes are optional in the sense that the user can save without a value, but any textboxes without values will throw errors to the user on the dashboard
** 
* See Bolt code for list of KPI cards and their descriptions
** Pay attention to which are %, money or number
* Save button
** On click:
*** validate the fields
*** see what has changed
*** for anything that has changed, save the values to the database
*** post a success toast message
*** update the current values and sliders
*** stay on the same drawer, don’t close it as they will likely make changes to more than one page
* Cancel button
** On click, close the drawer

h3. Data Requirements

* APIs:
** School Year
*** Current school year and next year
** List of sites and KPI settings
*** Name, current value, type (%, number, money)
** Save KPI
*** Posts the updates back to the DB
* All communications between the drawer and APIs occur over HTTPS
* The system ensures that data is retrieved and processed securely
* The drawer interacts with versioned API endpoints (e.g., {{/api/v1/...}}) to ensure future compatibility and backward support
* API endpoints implement rate limiting to prevent abuse and safeguard against denial-of-service attacks

h3. Performance & Usability

* Ensure the page displays properly in these layout options: Full width, Compact, Mobile responsive, Expanded
* The dashboard should load within 2 seconds under normal operating conditions
* Interactions such as opening configuration panels and switching between views are smooth and responsive
* User inputs from selectors and textboxes are validated and sanitized to prevent injection attacks

h3. Security

* Customer Support and Technical Support roles can do anything
* District user must:
** District must have Insights license
** User must have role permission to edit insights

h3. Error handling

* Use Error Boundaries and Try/Catch in Asynchronous Code
* Use User-Friendly Error Messages and modal dialogs, toast notifications, or inline messages
* Log errors to the database so we can analyze issues later.

## Acceptance Criteria
h4. *Filters*

* The drawer includes a *School Year filter* with the following behavior:
** Displays *current school year* and *next school year* only.
** *Historical school years are excluded* from the list.
** Users cannot modify data from previous years.

----

h4. *Grid of Sites and KPI Values*

* The drawer displays a *grid* with:
** *Rows*: All sites in the district.
** *Columns*: KPI values, matching the list of KPI cards and descriptions from Bolt.
* Each cell in the grid is an *editable textbox*:
** Defaults to the *current benchmark or target value* for that site and KPI.
** Accepts user-entered input or can remain empty.
* *Textbox validation* rules:
** Users can *save with empty values*, but these will result in *errors on the dashboard*.
** Values must be validated and sanitized based on the *KPI type*:
*** Percent
*** Number
*** Money

----

h4. *Save Button Behavior*

* On clicking *Save*:
** The system *validates the fields* according to KPI types.
** Determines which fields have *changed*.
** Sends only the *changed values* to the backend using the Save KPI API.
** A *success toast message* is displayed after saving.
** The *current values and sliders* are updated based on the response.
** The drawer remains *open* to allow further changes.

----

h4. *Cancel Button Behavior*

* On clicking *Cancel*:
** The drawer closes without saving changes.

----

h4. *Data Requirements and API Integration*

* The drawer retrieves data using the following APIs:
** *School Year API*: Returns current and next school years only.
** *List of Sites and KPI Settings API*: Returns site list, KPI names, current values, and KPI types.
** *Save KPI API*: Accepts and stores the updated benchmark values.
* All API communication:
** Occurs over *HTTPS*.
** Uses *versioned endpoints* (e.g., {{/api/v1/...}}) to support compatibility and future changes.
** Includes *rate limiting* to prevent abuse or denial-of-service attacks.

----

h4. *Performance and Usability*

* The drawer and grid must:
** Load within *2 seconds* under normal conditions.
** Render responsively across:
*** Full width
*** Compact
*** Mobile
*** Expanded views
* User interactions, including typing and saving, must be:
** *Smooth and responsive*
** *Free of input lag*
* All user input is:
** *Validated* for appropriate type and format.
** *Sanitized* to prevent injection or formatting issues.

----

h4. *Security and Permissions*

* *Customer Support and Technical Support* roles have full access to edit all KPIs.
* *District users* can edit KPI targets only if:
** The district has an *active Insights license*.
** The user has the *role permission to edit insights*.

----

h4. *Error Handling*

* Use *try/catch* in all asynchronous logic.
* Implement *error boundaries* to prevent screen crashes from UI-level failures.
* Show *user-friendly error messages* via:
** Modal dialogs
** Toast notifications
** Inline field errors
* All caught errors (validation, API, rendering) must be *logged to the error database* for tracking and debugging.

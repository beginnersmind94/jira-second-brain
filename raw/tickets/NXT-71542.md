# NXT-71542 — SC - MT - District - Boundary Assignment

## Description
As a *District user, Customer Support, or Technical Support user*, I want to assign district roles to externally granted boundaries, so that users inherit boundary-based access through existing role assignments without needing a separate user assignment workflow.

----

h1. Background / Notes

We are introducing a *Boundary Assignment* tab on the *System > Districts* screen.

This tab allows districts to configure how *externally granted boundaries* map to their *local roles*.

h3. Key Design Principle

This screen does *not assign users directly to boundaries*.

Instead:

* Boundaries are assigned to *district roles*
* Users inherit access through their assigned roles via the *existing user management system*

h3. Important Clarification

This tab only shows:

{quote}*Boundaries granted to this district by other (parent/linked) realms*{quote}

It does *not show boundaries created by this district*.

Those are managed in a separate tab (*Boundary Definitions*).

----

h1. Current Navigation

*System > Districts > Boundary Assignment tab*

----

h1. Dependencies

* Boundary Definitions story
* District Boundaries story
* Central Boundaries story
* Roles & permissions system
* Realm relationship story

----

h1. Out of Scope

* Creating/editing roles
* Assigning users to roles
* Boundary propagation logic
* Authorization enforcement in modules

----

h1. Functional Requirements

h2. 1. Add Boundary Assignment tab

Add a new tab labeled *Boundary Assignment*

----

h2. 2. Restrict access

Access to this tab is controlled by the *District page resource permissions*.

h3. Requirements:

* Users must have *View permission* on the District page to see the tab
* Users must have *Edit permission* to:
** assign roles to boundaries
** modify assignments
* Delete permission is not applicable on this screen

h3. Note:

* This tab does *not define its own permission set*
* It inherits permissions from the *District page in the resource catalog*

----

h2. 3. Conditional tab visibility

* Tab is visible only if the district has at least one *linked realm relationship*
* If no linked relationships exist:
** tab is hidden

----

h2. 4. Load boundaries grid

Display all *boundaries granted to this district from linked realms*.

h3. Requirements:

* Only show boundaries where:
** the district has been granted access via a parent or linked entity
* Do not show:
** boundaries created by the district itself
* Include both:
** active boundaries
** inactive boundaries (if previously granted)

----

h2. 5. Grid structure

Columns:

* Granting Realm (name of district/parent that granted access)
* Boundary Name
* Status (Active / Inactive)
* Assigned Roles
* Actions

----

h2. 6. Status behavior

* If Status = *Inactive*
** the row is visually distinct (e.g., grayed out)

----

h2. 7. Assigned Roles column

* Allows selecting zero or more *district roles*
* Uses multi-select chips
* Roles come from the district’s role list
* If no roles are assigned, display placeholder text:
*“No roles assigned”*

----

h2. 8. Role assignment behavior

* Users can:
** add roles
** remove roles
** replace roles
* Role assignment is allowed for:
** active boundaries
** inactive boundaries
* It is valid for a boundary to have:
** zero roles assigned

----

h2. 9. Auto-save behavior

* Changes are automatically persisted when:
** roles are added
** roles are removed

h3. UX Requirements:

* Show a *saving indicator* while persisting changes
* Show a *success confirmation (e.g., toast or inline message)* after save completes

h3. Stability Requirement:

* Ensure rapid role changes do not cause conflicting or duplicate persistence operations
(e.g., debounce or queue updates)

----

h2. 10. No delete behavior

* Boundaries cannot be removed from this grid
* There are no delete actions on this screen
* This screen only manages role assignments

----

h2. 11. Row-level View History

* In the *Actions* column on the right side, add a *View History* button/icon
* On click:
** open the *new Tailwind history drawer*

h3. Scope:

* History is for that individual boundary assignment row

h3. Tracks:

* Assigned Roles changes only

----

h2. 12. Audit log update on persistence

* When role assignment changes are persisted to the database,
the system must also write the corresponding changes to the audit log

Audit entries must include:

* user making the change
* timestamp
* boundary assignment row affected
* before roles
* after roles

----

h2. 13. Empty state

If no boundaries are available:

* Display message:
*“No boundaries have been granted to this district.”*

----

h2. 14. Bulk Assign button

* Add a *Bulk Assign* button on the screen

----

h2. 15. Bulk Assign modal

On click of *Bulk Assign*:

* open a small modal popup

h3. Modal requirements:

* Title: *Bulk Assign Roles*
* Informational note:
*“Select which roles you would like to assign to all boundaries. Selected roles will be added to each boundary without removing existing roles.”*
* Include the same *role multi-select chip selector* used on the main page
* Allow selecting one or more roles
* Include:
** Cancel button
** Save button

----

h2. 16. Bulk Assign confirmation

When user clicks *Save* on the modal:

* Show confirmation dialog:
*“Are you sure you want to add the selected roles to all boundaries? Existing roles will not be removed.”*
* If user selects *No*:
** close modal
** make no changes
* If user selects *Yes*:
** apply selected roles using merge logic

----

h2. 17. Bulk Assign behavior (merge logic)

* Selected roles are *added* to each boundary
* Existing roles are *preserved*
* System must:
** prevent duplicate role assignments
** merge roles cleanly

----

h2. 18. Bulk Assign persistence and audit

* Persist updates to the database
* Create audit log entries for each affected boundary row

# NXT-71861 — Add Bulk Users Creation via File Upload for SchoolCafe Administrators

## Description
h2. Background

Currently, SchoolCafe Administrators must create users one at a time through the {{POST /api/User}} endpoint (UserController.AddUserAsync → UserManager.AddUserAsync). For districts onboarding many staff members simultaneously, this is time-consuming and error-prone. A bulk upload capability will allow administrators to create multiple users from a single file upload, significantly reducing onboarding time.

h2. Requirements

* [ ] New API endpoint {{POST /api/User/bulk}} that accepts a file upload (CSV/Excel) containing multiple user records
* [ ] Parse and validate the uploaded file — support CSV and XLSX formats
* [ ] Validate each row: required fields (first name, last name, email, role), data format, and business rules
* [ ] Reject rows with duplicate email addresses (both within the file and against existing users via {{IsUserEmailAvailableAsync}})
* [ ] Create validated users in SQL Server via UserRepository and provision Keycloak accounts for each user
* [ ] Return a detailed response with: total rows processed, successful creations count, failed rows with specific error messages per row
* [ ] Provide a downloadable template endpoint {{GET /api/User/bulk/template}} for the expected file format

h2. Technical Notes

Based on codebase analysis:

* *Affected area(s):*
** {{WebAPI/Controllers/UserController.cs}} — new bulk endpoint
** {{Components/User/UserManager.cs}} + {{IUserManager.cs}} — new {{BulkAddUsersAsync}} method
** {{Components/User/UserRepository.cs}} + {{IUserRepository.cs}} — new bulk insert method
** {{Contracts/Models/}} — new DTOs: {{BulkUserUploadRequest}}, {{BulkUserUploadResponse}}, {{BulkUserRowResult}}
* *Architecture pattern:* Controller → Manager → Repository (all async). Follows existing {{AddUserAsync}} flow but iterates over parsed rows with validation and error collection
* *Database:* SQL Server — User table inserts via ISqlDbManager parameterized queries. Consider transaction batching for performance
* *Dependencies:* Azure Blob Storage (file staging), Keycloak (user provisioning per row), existing NuGet packages sufficient (no new packages needed for CSV; may need EPPlus or ClosedXML for XLSX parsing)
* *Existing tests:* {{UserControllerTests}} (23 tests), {{UserManagerTest}} (13 tests) — extend with bulk creation scenarios
* *File upload:* Leverage existing {{IFormFile}} patterns from other controllers in the solution

h2. Acceptance Criteria

* [ ] GIVEN a SchoolCafe Administrator uploads a valid CSV/XLSX file with 5 user rows WHEN the bulk create endpoint is called THEN all 5 users are created in SQL Server and Keycloak, and a success response is returned with count=5
* [ ] GIVEN a file contains a row with an email that already exists in the system WHEN the bulk create endpoint is called THEN that row is rejected with error "Email already exists", remaining valid rows are still processed
* [ ] GIVEN a file contains two rows with the same email address WHEN the bulk create endpoint is called THEN the duplicate row is rejected with error "Duplicate email in file"
* [ ] GIVEN a file contains rows with missing required fields (first name, last name, email) WHEN the bulk create endpoint is called THEN those rows are rejected with specific validation errors per field
* [ ] GIVEN an empty file or unsupported format is uploaded WHEN the bulk create endpoint is called THEN a 400 Bad Request is returned with a descriptive message
* [ ] GIVEN a valid bulk upload completes WHEN the response is returned THEN it includes: total rows, successful count, failed count, and per-row error details for failures
* [ ] GIVEN the template endpoint is called WHEN a GET request is made to {{/api/User/bulk/template}} THEN a downloadable CSV/XLSX template with column headers is returned

h2. Out of Scope

* UI/frontend implementation for the bulk upload screen (separate ticket)
* Bulk update of existing users
* Role assignment beyond the role specified per row in the file
* Sending welcome/activation emails (can be a follow-up feature)
* File size limits beyond standard ASP.NET Core defaults

h2. Security Considerations

* Validate file content type and extension server-side — do not trust client headers alone
* Use parameterized SQL queries for all inserts — never construct raw SQL from file data
* Sanitize all user input fields parsed from the file (prevent XSS/injection in names, emails)
* Require authorization — only SchoolCafe Administrators should access the bulk endpoint
* Do not log PII (user emails, names) in plain text — follow existing Serilog patterns
* Limit file size to prevent DoS via large uploads

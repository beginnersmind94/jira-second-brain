# NXT-68638 — SC - MT - Roles & Permissions for Child Realms

## Description
As a Customer Support role user, I want to be able to control access to my children realms pages so that users don’t see pages they don’t need.

*Notes*

* This story covers when a central realm wants to control permissions for children realms
* It does not cover their own domain permissions (covered in another story)

*Requirements*

* On the System > Roles & Permissions page, do the following:
* On Children Districts tab, do the following:
* 
* Update the logic as follows:
* Check if the user is logged into a Central realm 
* If no:
** Hide this tab
* Else:
** Ensure the list of district roles are the regular ones for the Districts (not central realm roles)
** Ensure the resources in the Edit Role screen are trimmed to only those resources that have a District type from the resource catalogue
** Show the same grid and Add New functionality as on the other tab
*** Users can do the exact same thing as on the other page, but scoped differently
* 
* In addition to those fields, do the following:
* Add a field to let the central realm decide if it wants to manage the roles and permissions if its children
** Type: Toggle
** Label: Manage roles and permissions for all districts
** Default: False
** If false:
*** districts manage their own R&P
*** All action buttons disabled
** If true:
*** districts inherit the permissions from the central realm
*** Enable the + New button
*** Enable the buttons in the Actions column
* 
* Ensure changes are logged in History table



Child Realms update

* Add logic to the other tab:
** If a district is a child and its parent is managing its permissions, then:
** Disable them from adding or editing any permissions by disabling the buttons
** Show them a message: Roles and Permissions managed by Central realm
* 
* Add logic in the user token to pull parent roles and permissions if district is inheriting them

## Acceptance Criteria
h1. 

h2. *1. Page Context*

* Applies to *System > Roles & Permissions* page.
* Enhancements affect the *Children Districts* tab only.
* This story focuses on how a *Central realm* manages permissions for its *child realms* (Districts).
* Management of Central’s own permissions is covered separately.

----

h2. *2. Tab Visibility Logic*

* On page load, the system checks whether the current user is in a *Central realm*.
** *If not Central:*
*** The *Children Districts* tab is *hidden* entirely.
** *If Central:*
*** The *Children Districts* tab is visible and active.

----

h2. *3. Children Districts Tab Content*

* Display the same *grid layout and Add New functionality* as found on the *This Realm* tab, with all standard behaviors (paging, sorting, filtering, add/edit/delete).
* Logic and layout mirror the existing tab but are scoped to *child realms* rather than the current realm.

h3. Role and Resource Data

* When logged in as a *Central realm user*:
** Display the *regular district roles* (not Central realm roles).
** In the *Edit Role screen*, show only resources that have the {{District}} value in the *Realm Types* field from the Resource Catalogue.
** Resources tagged only as {{Central}} are excluded.

----

h2. *4. Manage Roles & Permissions Toggle*

h3. Field Details

* *Type:* Toggle
* *Label:* {{Manage roles and permissions for all districts}}
* *Default:* {{False}}
* *Location:* At the top of the *Children Districts* tab.

h3. Behavior

* *If False:*
** Each district manages its own Roles & Permissions independently.
** Disable all action buttons (Add New, Edit, Delete) on this tab.
** Show a message below the toggle:
*** _“Districts manage their own Roles & Permissions.”_
* *If True:*
** Central realm manages Roles & Permissions for all its child districts.
** Enable the *+ New* button and the *Actions* column buttons (Edit/Delete).
** Changes apply to *all child realms*.

----

h2. *5. History Logging*

* Log all changes in the *History table*, including:
** When the *Manage roles and permissions for all districts* toggle is changed.
** When roles or permissions are added, edited, or deleted via the Children Districts tab.
* Each log entry includes:
** User performing the action
** Timestamp
** Role or resource affected
** Change type (add, edit, delete, toggle update)

----

h2. *6. Child Realm Behavior*

* On the *This District / This Realm* tab for any *child realm*:
** If the parent Central realm is managing its permissions ({{Manage roles and permissions}} toggle = True):
*** Disable *Add*, *Edit*, and *Delete* buttons.
*** Show a clear message:
**** _“Roles and Permissions are managed by the Central realm.”_
** If the parent toggle = False:
*** Child realm regains control of its Roles & Permissions tab (standard district behavior).

----

h2. *7. User Token Integration*

* When a user logs into a *child district* whose permissions are centrally managed:
** The system checks the parent realm’s permissions during authentication.
** The user token includes:
*** Inherited roles and permissions from the Central realm.
** District-specific permissions are overridden by inherited ones.
** Token refresh or realm switch must also reapply inherited permissions dynamically.

----

h2. *8. Permissions*

* Only *Customer Support role users* can configure and toggle centralized management of Roles & Permissions.
* Central realm administrators can edit child permissions *only when the toggle is enabled*.
* Technical Support role may view all configurations in read-only mode.

----

h2. *9. Validation & Error Handling*

* If save or permission propagation fails:
** Show message: _“Unable to save changes. Please try again or contact support.”_
** Log all errors for support investigation.
* Prevent circular dependencies or propagation to non-child realms.

----

h2. *10. Performance & UX*

* Switching between tabs or toggling management on/off must occur within *3 seconds*.
* UI feedback (button states, messages) updates instantly after toggling.
* Design aligns with the current Roles & Permissions page layout and theme.

# NXT-70312 — FIN - Universal Bell Notifications for Data Changes & Pending Entries

## Description
As a Child Nutrition Director, I want to be notified via the bell icon when source data changes after posting or when entries couldn't post due to a closed period, so that I can take corrective action without manually monitoring the Manage Periods page.

h3. Outcome:

* Immediate Awareness: Users are alerted to data discrepancies the moment they log in, rather than discovering them during month-end reconciliation.
* Data Integrity: Eliminates the risk of "stale" data lingering in the system after a period is thought to be finalized.
* Workflow Efficiency: Removes the need for users to manually check the "Manage Periods" screen for updates.

## Acceptance Criteria
h3. 1. Scenario A: Source Data Changed After Posting

* When a POS or Inventory period is re-closed and the corresponding Financial period already has posted entries, the system fires a bell notification.
* This applies universally to all automated entry types from front office.
* This covers cases where entries already exist and need correction.

*Auto-adjust ON:*

* _"[Entry Type] at [Site] for [Period] was automatically adjusted. Original: $[Old Amount] → New: $[New Amount]. View details."_
* Tapping navigates to the journal showing the reversed transactions and the new entries.

*Auto-adjust OFF (default):*

* _"Source data changed for [Entry Type] at [Site] for [Period]. Original: $[Old Amount] → New: $[New Amount]. Review and adjust."_
* Tapping navigates to the Manage Periods page with the affected period expanded, showing the Data Changed warning badge, entry types, and amounts (posted vs. updated with delta).
* From there, the user can reopen the period if closed and trigger the repost.

h3. 2. Scenario B: Posting Blocked by Closed Financial Period

* When a posting is triggered but the destination Financial period is closed and no entries exist yet for that period/site/entry type, the entry is held as pending.
* Pending entries must retain: entry type, site, source period date range, and amounts.
* Pending entries must never be silently discarded.

*On blocked:*

* _"[Entry Type] posting for [Site] ([Source Period]) is pending — $[Amount]. Financial period [Period Name] is closed."_
* Tapping navigates to the Manage Periods page with the affected Financial period highlighted, showing pending entry count and amounts.

*On period reopen — Auto-adjust ON:*

* Pending entries post automatically.
* _"[X] pending entries ($[Total Amount]) for [Period Name] have been automatically posted. View details."_
* Tapping navigates to the journal showing the posted entries.

*On period reopen — Auto-adjust OFF (default):*

* _"[X] pending entries ($[Total Amount]) are ready for [Period Name]. Review and post."_
* Tapping navigates to the Manage Periods page with the affected period expanded, showing each pending entry with entry type, site, source period, and amounts.
* The user reviews and manually triggers the posting.

h3. 3. Notification Aggregation

* If multiple sites are affected in the same period for the same event (e.g., POS close triggers 15 sites), notifications should aggregate per period rather than per site.
* Example: _"12 entries pending across 8 sites for January 2026. $4,320.50 total."_
* Drill-down to individual sites happens on the landing page, not in the notification.

h3. 4. Reopen Period Confirmation

* If the notification navigates to a closed period and the user clicks "Reopen Period," the system must display a confirmation prompt before executing.

h3. 5. Notification Behavior

* All notifications route through the existing bell icon.
* Persistence, dismissal, and stacking behavior is governed by the existing notification infrastructure configuration. This story does not define or override that behavior.

h3. 6. Activity Log

* Activity log entries are created only when an actual reversal/repost or pending-to-posted action occurs.
* Notifications themselves do not generate activity log entries.
* Each activity log entry must include: entry type, site, period, amounts, and timestamp.

----

h2. Out of Scope

* Closed fiscal year prior period adjustments (separate story).
* Email or push notifications.
* Amount visibility on Manage Periods page (prerequisite story — must ship first).

h2. Dependencies

* FIN - Add Entry Amounts to Manage Periods Table (prerequisite — must be implemented before this story)
* FIN - Financial Period Management (dirty flag, Data Changed column, open/closed status)
* FIN - Posting Activity Log
* Existing bell icon notification infrastructure

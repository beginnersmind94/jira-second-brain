# NXT-60018 — TH - WS - Inbox Screen

## Description
*As a* TeamHub user *I want* a screen that provides quick access to my SchoolCafé Inbox and Sent features *So that* I can stay informed about my current tasks and notifications.

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the Inbox screens from SC
* This story covers the Inbox section, Sent will be covered in another story

h2. User Flow

* User opens the application and is directed to login screen
* On login success, user is redirected to home screen
* User clicks Workspace and is redirected there
* User clicks on Inbox and is redirected there
* User is able to view a list of inbox messages
* User is able to click on a message and see the details
* User is able to compose a new message
* User is able to reply and forward a message
* User is able to delete a message

h2. Requirements

* Ensure the screen supports dark mode

_Header Section_

* Have a back button to Workspace
* Page Title: Inbox



_Filters Section_

* Search textbox
** Filters messages 
** Filters against the user name, subject and body
* Inbox chip
** acts as a tab click
** If click it, show Inbox messages
** Show the count of unread messages in the chip
* Sent chip
** acts as a tab click
** If click it, show Sent messages
** Don’t show the count of unread messages in the chip as it doesn’t make much sense
* Sync
** Show last sync time
** have a Sync button
*** On click:
*** Run the asynchronous Inbox job to sync the mobile app to the server’s inbox (another story)



_Body Section_

* List of Inbox messages
** Be able to view a list of inbox messages
*** View: Avatar of sender, From, Date, Subject, Beginning of the body
*** Be able to star it (see below)
*** View a visual indicator of if it is not read or not (left side of card has a color)
*** Indicate if there is an attachment
** Be able to click on a message and view its details (another story)
** Be able to swipe the message to the left and be able to delete it (another story)
* Compose button
** when clicked, redirects the user to the compose inbox screen (another story)

h4. *Data Requirements*

* There should be no API calls on this screen
* All data should be pulled from the asynchronous storage on the phone
* The Delete and Star should write to the asynchronous storage on the phone
* There should be separate code (another story) that runs independently from this page that syncs the Inbox messages to the server and this screen should be agnostic to that.

h4. *Performance & Usability*

* The screen must load within *2 seconds* and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Errors must be communicated using *user-friendly messages*:
** Use *toast notifications, modals, or inline messages* without disrupting the user experience
* Log any API or rendering errors to the *error log system/database* for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *Display Elements*

h5. *Header Section*

* Includes a *back button* that navigates back to the Workspace screen.
* Displays the *page title* as "Inbox".

h5. *Filters Section*

* A *search textbox* allows the user to filter messages by:
** Sender name
** Subject
** Body text
* *Inbox chip*:
** Functions as a toggle/tab to display *Inbox* messages.
** Displays active styling when selected.
** Show the count of unread messages in the chip
* *Sent chip*:
** Functions as a toggle/tab to display *Sent* messages.
** Displays active styling when selected.
* *Sync section*:
** Shows the *last sync timestamp*.
** Includes a *Sync button* that, when clicked, triggers the background sync job to update the inbox (job handled in another story).

h5. *Body Section*

* Displays a scrollable list of inbox messages.
* Each message card shows:
** Avatar of the sender
** Sender name (From)
** Date
** Subject
** Beginning of the message body
** A *star icon* to mark/unmark as important (writes to async storage)
** A *visual unread indicator* (e.g., a colored bar on the left side)
** An *attachment icon*, if the message includes attachments
* Tapping a message opens the message detail view (covered in another story).
* Swiping a message to the left enables the user to delete it (covered in another story).
* A *Compose button* is fixed on screen:
** When tapped, navigates to the compose screen (covered in another story).

----

h4. *Data Requirements*

* All message data is *retrieved from the device's asynchronous local storage*.
* The screen *must not make any API calls directly*.
* Starring and deleting messages *must update local async storage*.
* The screen is agnostic to sync behavior; syncing is handled by a background process in another story.

----

h4. *Performance & Usability*

* The screen must:
** Load within *2 seconds* under normal operating conditions.
** Display correctly on all form factors:
*** Mobile
*** Tablet
*** Desktop
*** Compact and expanded views
* All tap and swipe actions must be responsive, with *clear visual feedback*.
* Layout and styling must support *dark mode*.

----

h4. *Security*

* Any *authenticated user* can access and interact with the Inbox screen.
* No additional role-based permissions are required.

----

h4. *Error Handling*

* All asynchronous calls must use *try/catch blocks* to handle errors.
* Implement *error boundaries* to gracefully capture UI-level rendering failures.
* Any errors encountered must:
** Be communicated using *user-friendly messages* via toast notifications, modals, or inline alerts.
** *Not block* the user from continuing other interactions.
** Be *logged to the error tracking system/database* for developer review.

# NXT-64761 — SC - MT - Roles & Permissions Update

## Description
As a Customer Support role user, I want to be able to control access to my realms pages so that users don’t see pages they don’t need.

*Notes*

* This story covers which resources (pages) Central Users can see in their own domain
* It does not cover what they can access in children domains (covered in another story)

*Requirements*

* On the System > Roles & Permissions page, do the following:
* Add 2 tabs
** This District
*** If the district is a Central District, change the name to This Realm
** Children Districts
*** The content for this tab will be covered in another story
*** Note: This is the tab for where a central district may optionally manage the permissions for children districts
* 
* All current content on the page is moved to the This District tab
* 
* Update the logic as follows:
* Check if the user is a Central realm user in their own district
* If no, make no changes
* Else:
** Ensure the list of district roles are the ones for the Central realm, not the District
*** Note: Central realm roles are covered in another story
** Ensure the resources in the Edit Role screen are trimmed to only those resources that have a Central type from the resource catalogue
*** Meaning, filter out the pages that are district specific
*

## Acceptance Criteria
h1. 

h2. *1. Page Structure Updates*

h3. Tabs

* On *System > Roles & Permissions*, add two tabs at the top of the page:
*# *This District*
*#* Displays current district roles and permissions (default view).
*#* If the current district is a *Central realm*, rename this tab to *This Realm*.
*# *Children Districts*
*#* Placeholder for managing child realm permissions (content covered in a future story).

h3. Tab Behavior

* The *This District / This Realm* tab is selected by default.
* All existing content currently on the Roles & Permissions page is moved under this first tab.
* *Children Districts* tab should be visible but inactive or empty for now, showing a message such as:
** _“This section will allow managing child district permissions in a future update.”_

----

h2. *2. Role and Resource Filtering Logic*

h3. Realm Detection

* On page load, the system checks whether the user is in a *Central realm* or *standard District realm*.
** If *not a Central realm*, no changes are applied — existing district behavior remains unchanged.
** If *Central realm*, update the logic below.

h3. Central Realm Logic

* Display only the *roles* that belong to the *Central realm* (not district-specific roles).
* When editing a role (Edit Role screen):
** Filter the *resources list* so that only resources from the *Resource Catalogue* tagged with *Central* in their {{Realm Types}} field are displayed.
** *District-only* resources (those without “Central” selected) must not appear.

h3. Resource Catalogue Integration

* The filtering uses the same {{Realm Types}} field added in the Resource Catalogue (covered in a previous story).
* Roles & Permissions dynamically queries this field to determine which resources to show.

----

h2. *3. Permissions Enforcement*

* Only users with the *Customer Support* role can access and modify realm-specific role permissions.
* Technical Support may have read-only access for review purposes.
* Central or District-level users:
** Cannot modify or create roles from this view unless explicitly granted edit permissions.
** Continue to see the tab but with disabled controls if permissions are insufficient.

----

h2. *4. Validation & Security*

* If a user without required permissions attempts to access restricted resources, display a standard message:
** _“You do not have permission to view or modify these roles.”_
* Ensure users from one realm cannot see or edit roles belonging to another realm.
* Prevent any attempt to modify {{Realm Type}} assignments directly from this page (these remain controlled via the Resource Catalogue).

----

h2. *5. UI/UX Requirements*

* Tabs and content must render within normal page load performance (<3 seconds).
* Role editing and filtering should remain responsive when switching between tabs.
* Follow the existing Roles & Permissions layout and styling for consistency.
* Use dynamic tab names ({{This District}} or {{This Realm}}) based on realm context to avoid confusion.

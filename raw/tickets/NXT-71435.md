# NXT-71435 — SC - MT - District - Boundary Definitions Tab

## Description
As a *Technical Support user*, I want to create and manage boundary definitions, so that boundaries can be used consistently across district and central boundary assignment workflows.

----

h1. Background / Notes

We are introducing a *Boundary Definitions* tab on the *System > Districts* screen.

This tab is used to manage the *master list of boundaries* available in the system.

Boundaries are:

* defined once at the system level
* reused across:
** District Boundaries tab
** Central Boundaries tab

Important rules:

* Boundaries are *global system entities*
* Boundaries are *not created at the district level*
* Only *Technical Support users* can create or manage boundaries
* Customer Support users can only assign boundaries, not create them

----

h1. Current Navigation

*System > Districts > Boundary Definitions tab*

----

h1. Technical / Implementation Notes

* Add new tab labeled *Boundary Definitions*
* Accessible only to *Technical Support role*
* Use existing UI patterns:
** grid listing
** drawer for create/edit
** row-level actions column (right-aligned)
** confirmation dialogs
* Boundary names must be unique
* Boundaries are stored as reusable system-level entities
* Use *soft delete model (Active / Inactive)*
* Drawer for *Add New* and *Edit* should be the same component
* History uses *row-level drawer pattern*
* District and Central Boundaries features depend on this story

----

h1. Dependencies

* *This story is a dependency for:*
** District Boundaries story
** Central Boundaries story

----

h1. Out of Scope

* Boundary assignment to districts
* Role mapping
* Parent-child propagation logic
* Enforcement of boundaries in modules
* Advanced categorization/grouping

----

h1. Functional Requirements

h2. 1. Add Boundary Definitions tab

Add a new tab labeled *Boundary Definitions*

----

h2. 2. Restrict access

* Accessible only to:
** Technical Support users
* Not visible to:
** Customer Support users
** District users

----

h2. 3. Display boundaries grid

Display a grid listing all boundaries with columns:

* Boundary Name
* Description (optional)
* Status (Active / Inactive)
* Actions (right-aligned)

----

h2. 4. Add “+ New Boundary” button

* Located top-right
* Visible only to Technical Support users

----

h2. 5. Create boundary

On clicking *+ New Boundary*:

* Open drawer
* Fields:
** Boundary Name (required)
** Description (optional)
** Status (default = Active)

----

h2. 6. Boundary name validation

* Boundary Name is required
* Must be unique across system

If validation fails:

* block save
* show standard error message

----

h2. 7. Save new boundary

* On Save:
** validate inputs
** show confirmation (optional)
** persist boundary

----

h2. 8. Edit boundary (row-level action)

* Add *Edit icon* in Actions column (right side)
* Clicking Edit:
** opens drawer (same as Create)
** allows editing:
*** Boundary Name
*** Description
*** Status (Active / Inactive)

Validation:

* Name must remain unique

----

h2. 9. Deactivate boundary (soft delete)

* Add *Delete/Trash icon* next to Edit in Actions column

On click:

* Show confirmation dialog:

Example message:
*“This action will deactivate the boundary. It will no longer be available for assignment but will remain in the system for historical purposes. Do you want to continue?”*

* If confirmed:
** set Status = Inactive

h3. Prevent deletion

* Boundaries are never hard deleted
* Only Active → Inactive transition is allowed

----

h2. 10. Reactivate boundary

* Performed via Edit drawer
* User can change:
** Status from Inactive → Active

----

h2. 11. Row-level View History

* Add *View History icon/button* in Actions column (after Delete)

On click:

* Open *View History drawer (Tailwind version)*

Scope:

* Displays history for *that individual boundary only*
* Not global across all boundaries

Tracks:

* creation
* edits
* status changes (active/inactive)

----

h2. 12. No auto-save

* Changes persist only after Save

----

h2. 13. Unsaved changes warning

* If user attempts to leave with unsaved changes:
** show warning

## Acceptance Criteria
h2. Access & Visibility

* Boundary Definitions tab exists
* Only Technical Support users can access it
* Customer Support and District users cannot see it

----

h2. Grid Display

* Grid displays:
** Boundary Name
** Description
** Status
** Actions column on right

----

h2. Create Boundary

* New Boundary button visible to Technical Support
* Clicking opens drawer
* User can enter name, description, status
* Name is required and unique
* Validation blocks duplicate names
* Valid save persists boundary

----

h2. Edit Boundary

* Edit icon appears in Actions column
* Clicking Edit opens drawer
* User can update name, description, status
* Validation ensures name uniqueness
* Save persists changes

----

h2. Deactivate Boundary

* Delete/Trash icon appears next to Edit
* Clicking shows confirmation dialog
* Confirmation sets status to Inactive
* Boundary is not deleted from system

----

h2. Reactivate Boundary

* User can edit boundary
* Changing status to Active reactivates boundary
* Reactivated boundary becomes available in assignment screens

----

h2. Row-Level History

* View History icon appears in Actions column
* Clicking opens history drawer
* Drawer uses Tailwind template
* Displays history for that specific boundary only

----

h2. Validation & UX

* Validation errors display using standard messaging
* Save only occurs after validation passes
* Unsaved changes warning appears when navigating away

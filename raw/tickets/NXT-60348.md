# NXT-60348 — TH - WS - Inbox Delete Functionality

## Description
*As a* TeamHub user *I want* to be able to delete my inbox and sent messages *So that* I can clean up my messages

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the Delete functionality from SC
* This story covers 3 scenarios of deleting an inbox/sent message:
** Swipe left on a card on the inbox screen
** Swipe left on a card on the sent screen
** Click Delete on the Message Details screen

h2. User Flows

* #1, Swipe left on a card on the inbox screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User swipes left on a message card and confirms Delete
** System updates the asynchronous inbox storage of the user’s deletion
* #2, Swipe left on a card on the Sent screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User clicks on the Sent chip
** User swipes left on a message card and confirms Delete
** System updates the asynchronous sent storage of the user’s deletion
* #3, Click Delete on the Message Details screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User sees a list of inbox messages (or sent messages)
** User clicks on a message and is redirected to the Message Details screen
** User clicks the Delete button and confirms Delete
** System updates the asynchronous inbox/sent storage of the user’s deletion
* Note: the only thing different is how the deletion is triggered, everything else works the same

h2. Requirements

* Inbox screen
** Add the swipe left to the cards
** When they do, show the Delete button
** Allow the user to click it
** If they do, ask them to confirm
** If they say no, do nothing 
** If they say yes, continue
** Send the deletion request to the asynchronous storage
** Trigger immediately the sync request to the server (will do it asynchronously)
** Show the user a confirmation message
** Remove the item from the list
* Sent screen
** Do the same things as on the Inbox screen, just for sent messages

* Message Details screen:
** Update the Delete button
** On click:
*** Ask them to confirm
*** If they say no, do nothing 
*** If they say yes, continue
*** Send the deletion request to the asynchronous storage
*** Trigger immediately the sync request to the server (will do it asynchronously)
*** Show the user a confirmation message
*** Remove the item from the list

h4. *Data Requirements*

* There should be no API calls on this screen
* All data should be pulled from the asynchronous storage on the phone
* The Delete button should write to the asynchronous storage on the phone, not connect to the external server
* There should be separate code (another story) that runs independently from this page that syncs the Inbox/sent messages to the server and this screen should be agnostic to that.

h4. *Performance & Usability*

* The screen must load within *2 seconds* and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Errors must be communicated using *user-friendly messages*:
** Use *toast notifications, modals, or inline messages* without disrupting the user experience
* Log any API or rendering errors to the *error log system/database* for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *Functional Requirements*

_Inbox and Sent Screens_

* Each message card supports swipe-left gesture.
* On swipe, display a Delete button on the card.
* Tapping Delete opens a confirmation modal:
** Message: "Are you sure you want to delete this message?"
** Buttons: Confirm / Cancel
* On confirmation:
** The message is removed from the corresponding local async storage.
** The sync job is triggered to push the deletion to the server.
** The message is removed from the visible list without reload.
** A success message (toast or inline) is displayed to the user.

_Message Details Screen_

* The Delete button triggers a confirmation modal with the same behavior as above.
* On confirmation:
** Message is deleted from async storage.
** Sync job is triggered.
** User sees a success message.
** User is returned to the Inbox or Sent screen with the message removed.

h4. *Data Requirements*

* All data is retrieved and updated using asynchronous storage on the device.
* There are no direct API calls on these screens.
* The sync to the server is handled by separate background logic (another story).
* The screen must be completely agnostic to the syncing mechanism.

----

h4. *Performance & Usability*

*** Mobile
*** Tablet
*** Desktop

** Visually responsive

# NXT-70777 — FIN — Payroll Import: Business Logic

## Description
As a district finance user, I want imported payroll journal entries to post to the ledger correctly, so that my Trial Balance, Balance Sheet, and P&L reflect accurate period activity and my books tie out for audit.

_Updated 2026-05-08. Major revision: entry boundary changed from one-entry-per-import to one-entry-per-PostedDate (each entry posts to exactly one period); line-level Description column removed (Financials 2.0's posting engine carries one description per entry, not per line); pre-upload now captures Entry Description as entry-header metadata; generated Entry Reference added for downstream GL Export; source file metadata persistence added for federal records retention. PostingDate renamed to PostedDate to match canonical Financials 2.0 field naming. Earlier 2026-05-07: added TL;DR section. Earlier 2026-05-06: added consolidated API Validations section reflecting the DE/BL contract established with Jonathan Smith._

## TL;DR

Expose two API ops to DE: **Validate** (run every check; write nothing) and **Commit** (run every check + write). _DE calls Validate at the Field Mapping validator gate; Commit at the user's final confirm._

**On every call, BL runs:**

* **Row structure:** required fields present (GLAccount, SiteCode, PostedDate, exactly one of Debit/Credit); PostedDate **parses**; Debit/Credit are decimals.
* **SiteCode exists.**
* Each `(GLAccount, SiteCode)` has a configured **sub-account** — **blocking**. Central Office is just another site.
* **GLAccount existence is non-blocking** — unknown accounts get written and flagged for review _(differs from Opening Balance, which blocks)_.
* **Derive** `(FY, Period)` per row from PostedDate in the district's timezone.
* **Closed period or unconfigured period** anywhere in the file = **reject the entire import** (all-or-nothing).
* **Balance:** absolute difference between Debits and Credits ≤ **$0.01** — file-wide.
* **Duplicate check per targeted** `(FY, Period)`: if a prior Payroll Import journal entry exists, return the period in the response. On Commit, the DE acknowledgment flag must specifically cover each detected period. _Authoritative duplicate check._

**Commit-only:** re-derive periods and re-verify closure.

**Response on pass:** rows, Debits, Credits, **per-period breakdown** `(FY, Period, row count, period Debits, period Credits)`, **per-PostedDate breakdown (entries to be created on Commit)**, unknown GLAccount list, duplicate periods (when present).
**Response on fail:** structured error list + error report URL.

**On Commit success:** create **one journal entry per unique PostedDate in the file**. Type **"Payroll Import"**. Each entry has one line per row sharing that PostedDate.

**Each entry's header carries:**

* PostedDate (drives the entry's FY/Period — single period per entry)
* Entry Description (verbatim from pre-upload, identical across all entries from this import)
* Generated Entry Reference: `{YYYYMMDD}-{shortId}`, ≤25 chars, unique per entry. Date-first ordering preserves date distinguishability after Oracle FBDI's 25-char REFERENCE4 prefix truncation.
* Source file metadata: original filename, file hash (SHA-256), uploaded by, uploaded at — replicated across every entry from one import.

**Each line carries:** GLAccount, SiteCode, Debit or Credit (zero on opposite side). No line-level description — engine doesn't carry it.

**Atomic write across all entries from this import.** Concurrent imports targeting overlapping periods → second to reach Commit is rejected as duplicate. Entries are **read-only** — correct via manual reversal/adjustment.

**Reports:** Lines = **period activity** on Trial Balance, Balance Sheet, P&L. Lines do **not** affect opening balances — those come only from the OB Import flow (NXT-71240).

**Abnormal balances are accepted** — we don't enforce normal balance direction by base type.

## Scope

BL layer for Payroll Import: API Validations (Validate + Commit), per-PostedDate journal entry creation, derived (FY, Period) per row, closed-period rejection (defense-in-depth, all-or-nothing), duplicate detection backstop, sub-account validation, unknown GLAccount handling, atomic ledger write, entry immutability, generated Entry Reference, source file metadata persistence, and downstream report contribution. The DE layer (companion **NXT-72160**) handles upload, mapping UI, validator gate (calls BL Validate), preview, saved mapping, results, and duplicate warning. This story receives validated data with separate Debit/Credit columns and each row's derived (FY, Period) after the user confirms.

## API Validations

BL exposes **Validate** (called by DE at the Field Mapping validator gate per **NXT-72160** §8.1) and **Commit** (called by DE on Confirm). Both run the same validation pass; Commit additionally writes. All checks below run on every call.

On any failure, the API returns a structured error list (row#, field, code, message) + error report download URL. DE surfaces. **No partial work.**

### 1) Row-level structural

1.1 **Required fields.** GLAccount, SiteCode, PostedDate, and exactly one of Debit/Credit are present on every row.

1.2 **PostedDate parses** as a valid date.

1.3 **Decimal numbers.** Debit and Credit are decimals on every row. (DE fail-fasts the first 10 rows per **NXT-72160** §7.2; BL re-checks every row.)

1.4 **Single-side rule.** Exactly one of Debit/Credit positive per row. Both populated, both blank/zero, or any negative = invalid.

### 2) Reference data

2.1 **SiteCode exists** in SchoolCafe Site Management.

2.2 **Sub-account configured** for each (GLAccount, SiteCode) per §4 — **blocking**. Central Office is just another site.

2.3 **GLAccount existence is non-blocking.** Unknown accounts accepted at validation; on Commit they are written + flagged for review per §5. _(Differs from Opening Balance, which blocks.)_

### 3) Period derivation (per row)

3.1 **Derive (FY, Period)** from each row's PostedDate in the district's timezone.

3.2 **Configured-period check.** If any row's PostedDate falls outside any configured Period (e.g., FY doesn't exist), reject the entire import.

3.3 **Closed-period check (all-or-nothing).** If any row's derived Period is closed, reject the entire import. Partial write of open-period rows is not permitted.

3.4 On Validate success, return the derived (FY, Period) for each row + per-period breakdown (row count, Debits, Credits) + **per-PostedDate breakdown (count + total per entry to be created on Commit)** for DE to render the Review summary per **NXT-72160** §3.2.

### 4) Aggregate

4.1 **Balanced.** |Sum Debits − Sum Credits| ≤ **$0.01** — payroll's natural rounding tolerance. File-wide, not per period — a payroll file is a single balanced unit even when rows span multiple PostedDates and periods. _(Differs from OB's exact $0.)_

### 5) Duplicate detection (per targeted period)

5.1 For each (FY, Period) the file targets, check whether a prior successful Payroll Import journal entry exists.

5.2 On Validate, return the list of periods where a duplicate is detected so DE can collect acknowledgment per **NXT-72160** §8.4.

5.3 On Commit, the duplicate acknowledgment flag must be in the payload and must specifically cover every period where BL detects a duplicate at write time. Missing or insufficient = reject. **Authoritative duplicate check** — prevents DE warning bypass.

### 6) Period status re-check (Commit only, defense-in-depth)

6.1 At Commit, re-derive (FY, Period) from each row's PostedDate and re-verify no row's Period closed since Validate. Any closed period → reject the entire import (all-or-nothing); user returned to validation.

### Response contract

**Success** — all checks pass:

* Status: `pass`
* Total rows
* Total Debits
* Total Credits
* Per-period breakdown — list of `(FY, Period, row count, period Debits, period Credits)`
* **Per-PostedDate breakdown — list of entries to be created on Commit (PostedDate, line count, entry Debits, entry Credits)**
* Unknown GLAccount list — codes flagged for review on Commit
* Duplicate periods (when detected) — DE acknowledgment required before Commit

**Failure** — any check fails:

* Status: `fail`
* Error list — row#, field, code, message
* Error report download URL (generated by BL; DE surfaces per **NXT-72160** §8.1)

## Requirements

### 1) Journal Entry Creation

1.1 On Commit (after all API Validations pass), create **one journal entry per unique PostedDate in the file**. Each entry contains one line per row sharing its PostedDate.

1.2 **Entry Header** carries:

* **Entry type:** "Payroll Import" — distinguishable from Opening Balance, manual entries, and other automated types
* **Source:** Import Monitor
* **PostedDate:** unique to this entry (drives FY/Period assignment)
* **(FY, Period):** derived from PostedDate; one entry posts to exactly one period
* **Entry Description:** verbatim from pre-upload, identical across all entries produced by this import
* **Entry Reference:** generated, format `{YYYYMMDD}-{shortId}`, ≤25 chars total. {YYYYMMDD} is the entry's PostedDate. {shortId} is a stable, compact identifier (e.g., base36 upload ID). Date-first ordering ensures Oracle FBDI's 25-char REFERENCE4 prefix truncation removes characters from {shortId} tail rather than the date — preserving entry distinguishability after truncation.
* **User / Timestamp:** confirming user + confirmation timestamp
* **Source file metadata:** original filename, file hash (SHA-256), uploaded by, uploaded at

1.3 Each line carries:

* **GLAccount**
* **SiteCode** — from the file; for district-wide entries, the Central Office site code
* **Debit** or **Credit** — positive in exactly one column; opposite side = zero

1.4 Each entry's lines all share the same PostedDate (and therefore the same FY, Period). A single entry never spans multiple periods.

1.5 DE enforces a fixed Debit/Credit format (separate columns; signed-amount transformed upstream per **NXT-72160** §2). Each validated row carries a positive value in exactly one column. Values map directly to the ledger line; opposite side stored as zero.

1.6 **Normal balance direction is not enforced.** Abnormal balances on individual lines are accepted (e.g., credit to an asset for a correcting entry). Correctness is enforced by the file-wide balance check in API Validations §4.

### 2) Closed-Period Rejection (defense-in-depth backstop, all-or-nothing)

2.1 The closed-period rule is enforced by API Validations §3.3 — see for authoritative behavior. The Commit-only re-check (API Validations §6) catches periods that close between Validate and Commit.

2.2 Prevents payroll from posting to a closed period even if DE's preview-time and confirm-time checks were bypassed (URL manipulation, concurrent close, DE bug).

2.3 On rejection, no entries or lines are written. Error identifies affected closed period(s) and row count for each, directing the user to either correct the file or post a prior-period adjustment via manual journal entry.

### 3) Duplicate Detection Backstop (per targeted period)

3.1 Duplicate detection is enforced by API Validations §5 — see for authoritative behavior.

3.2 The user's explicit acknowledgment of the DE-side duplicate warning (per **NXT-72160** §8.3) must reach BL with the validated data, and must cover all targeted periods where a duplicate was detected at preview time. Acknowledgment present + covering = BL writes. Acknowledgment absent or missing a current duplicate = BL rejects.

3.3 Backstop prevents bypass via URL manipulation, concurrent submissions, or client-side tampering. Authoritative duplicate check.

### 4) Sub-account Validation

4.1 Sub-account validation runs in API Validations §2.2 (authoritative rule). The sub-account routes site-level activity to a GL account.

4.2 Any line lacking a configured sub-account for its (GLAccount, SiteCode) → **entire import rejected** at the validator gate (DE never reaches Commit). Per §7 atomicity, no partial writes.

4.3 Central Office is just another site — same sub-account requirement as any other SiteCode. No special handling.

4.4 On rejection, the result surfaces offending rows (GLAccount, SiteCode, clear no-sub-account message). User fixes the file or configures the missing sub-account in Configuration > General Ledger, then re-uploads.

4.5 Validation runs in BL Validate (and re-runs at Commit for defense-in-depth). DE doesn't pre-validate sub-accounts — only that the SiteCode exists. Sub-account routing is a BL concern.

### 5) Unknown GLAccount Handling

5.1 Per API Validations §2.3, unknown GLAccounts are non-blocking at validation. On Commit, BL writes the line (payroll feeds legitimately reference newly-introduced accounts that need review but shouldn't halt month-end).

5.2 On write for an unknown GLAccount, BL persists a warning indicator on the account such that it appears in Configuration > General Ledger > GL Accounts with a clear visual flag for finance review.

5.3 Once a user reviews and confirms or updates the flagged account in Configuration, the warning indicator clears. The review workflow is not part of this story — only the warning persistence on write is.

5.4 An unknown GLAccount permitted under §5.1 still requires a configured sub-account per §4. If the account is unknown AND no sub-account exists for the (GLAccount, SiteCode), the import is rejected per §4 — sub-account check wins.

### 6) Entry Behavior

6.1 Payroll Import journal entries are **read-only** in the UI. Lines cannot be edited or deleted.

6.2 Corrections require a manual reversal or adjustment entry. The original Payroll Import entry remains intact for audit trail.

6.3 The "Payroll Import" entry type must be identifiable in the ledger so reports, filters, and audit views can distinguish payroll imports from manual entries, opening balance entries, and other automated types.

### 7) Atomic Ledger Write

7.1 The complete set of entries produced by an import (one entry per unique PostedDate, with all lines under each entry, across all targeted periods) is written **atomically**. All entries post or none.

7.2 Any system-level error during write rolls back all entries from this import. No partial writes — including no partial write of just one of multiple entries, or just the lines on one of multiple entries.

7.3 On rollback, DE is notified so the import result reflects the failure and the user is informed.

### 8) Source File Metadata Persistence

8.1 On Commit, BL persists with each entry: original filename, file hash (SHA-256), uploaded by (user identifier), uploaded at (timestamp).

8.2 The same source file metadata is replicated across every entry produced by a single import (since one file produces N entries).

8.3 Supports records retention for audit — auditor can trace any Payroll Import entry back to its source file.

### 9) Downstream Report Behavior

9.1 Payroll Import lines contribute to **period activity** on all standard reports: Trial Balance, Balance Sheet, P&L, and any period-based account activity views. Each line contributes to the period its parent entry's PostedDate assigns it to.

9.2 Lines from different entries (different PostedDates) contribute to their respective periods independently. A Payroll Import targeting two PostedDates produces two entries that contribute to (potentially) two different periods.

9.3 Lines are filterable and groupable by the "Payroll Import" entry type in reports that support entry type filtering.

9.4 Lines do **not** establish or modify opening balances. Opening balances come only from the Opening Balance Import flow (**NXT-71240** / **NXT-71249**).

## Acceptance Criteria
**AC1 — Journal Entries Created (One Per Unique PostedDate).** On Commit, BL creates one journal entry per unique PostedDate in the file, with entry type "Payroll Import". Each entry contains one line per row sharing its PostedDate. Each entry posts to exactly one (FY, Period).

**AC2 — Entry Header Carries Pre-Upload Entry Description.** Each entry's header carries Entry Description verbatim from the Pre-upload tab. The value is identical across every entry produced by a single import.

**AC3 — Generated Entry Reference Per Entry.** Each entry's header carries a generated Entry Reference of format `{YYYYMMDD}-{shortId}`, total length ≤25 characters. {YYYYMMDD} is the entry's PostedDate. {shortId} is a stable, compact identifier (e.g., base36 upload ID). Date-first ordering is required so Oracle FBDI's 25-char REFERENCE4 prefix truncation removes characters from {shortId} tail rather than the date.

**AC4 — Source File Metadata Persisted.** Each entry's header carries: original filename, file hash (SHA-256), uploaded by (user identifier), uploaded at (timestamp). Identical across all entries produced by a single import.

**AC5 — PostedDate is Per-Entry, Single Period.** Each entry's header carries a single PostedDate. All lines on the entry share that PostedDate. The (FY, Period) derived from PostedDate is identical across all lines on the entry. An entry never spans multiple periods.

**AC6 — Debit/Credit Amounts Mapped Directly.** Validated data carries separate Debit/Credit columns (signed-amount layouts transformed upstream by DE per NXT-72160 §2). Each line carries the row's Debit or Credit; opposite side = zero.

**AC7 — Normal Balance Not Enforced by Base Type.** An asset row with a credit balance (or any abnormal balance) is processed as-is. System does not reject or modify direction by base type.

**AC8 — Closed-Period Rejected (All-or-Nothing).** If any row's derived Period has been closed by the time BL runs (Validate or Commit), BL rejects the entire import — no entries are written. Error identifies affected closed period(s) and row count for each, directing the user to correct the file or post a prior-period adjustment via manual journal entry. *(Defense-in-depth — DE enforces the same rule upstream.)*

**AC9 — Duplicate Rejected Without DE Acknowledgment (per Targeted Period).** If a Payroll Import entry already exists for any (FY, Period) the file targets and the DE acknowledgment flag is absent or doesn't cover that period, BL rejects with an error listing affected period(s). No entries are written.

**AC10 — Duplicate Allowed With DE Acknowledgment.** If a Payroll Import entry exists for one or more targeted (FY, Period) and the DE acknowledgment flag covers each detected period, BL writes the new entries alongside the prior — both remain in the ledger for audit.

**AC11 — Unknown GLAccount Writes with Warning Flag.** If a line references a GLAccount not in the chart of accounts AND a sub-account exists for the (GLAccount, SiteCode), BL writes the line and persists a warning indicator on the account such that it appears in Configuration > General Ledger > GL Accounts with a clear visual flag. BL does not block. If the unknown account also lacks a sub-account, the import is rejected per AC18.

**AC12 — Atomic Ledger Write Across All Entries.** On any system error mid-write, the complete set of entries produced by this import (all entries, all lines) rolls back — no partial writes remain. DE is notified, user is informed.

**AC13 — Entries Are Read-Only.** UI edit or delete of a Payroll Import entry is blocked. Corrections require a manual reversal or adjustment.

**AC14 — Entry Type Identifiable for Reports and Filters.** When a report or audit view filters by entry type, Payroll Import entries can be isolated or excluded using the "Payroll Import" identifier — distinguishing them from manual entries, opening balance entries, and other automated types.

**AC15 — Lines Contribute to Period Activity (Not Opening Balance).** Once Payroll Import entries are written across one or more (FY, Period), Trial Balance, Balance Sheet, and P&L for any of those Periods show the lines as period activity. Opening Balance column is not affected — those come only from the Opening Balance Import flow (NXT-71240 / NXT-71249).

**AC16 — API Validate Operation Returns Pass or Fail.** BL Validate returns either pass (rows, Debits, Credits, per-period breakdown — FY, Period, row count, period Debits, period Credits — per-PostedDate breakdown of entries to be created on Commit, unknown GLAccount list, duplicate periods when detected) or fail (structured error list — row#, field, code, message — + error report download URL). No partial work.

**AC17 — Row-Level Structural Validation.** BL returns fail with affected rows for: missing required fields (GLAccount, SiteCode, PostedDate, exactly one of Debit/Credit), unparseable PostedDate, non-decimal Debit/Credit, both populated/both blank/both zero/negative.

**AC18 — Sub-Account Required for Every (GLAccount, SiteCode).** BL returns fail with offending rows when any (GLAccount, SiteCode) lacks a configured sub-account. Central Office is just another site. Applies regardless of whether the GLAccount is known or unknown.

**AC19 — Period Derivation and Configured-Period Check.** BL returns fail with offending rows when any row's PostedDate cannot be mapped to a configured (FY, Period) in the district's timezone (FY doesn't exist, date is unparseable, or date falls outside any configured Period).

**AC20 — Balance Check with $0.01 Tolerance.** BL returns fail when |Sum Debits − Sum Credits| > $0.01. Tolerance applies file-wide, not per period and not per entry — a payroll file is a single balanced unit even when rows span multiple PostedDates and periods.

**AC21 — Commit-Only Period Status Re-check.** At Commit, BL re-derives (FY, Period) per row and re-verifies period status. If any targeted period closed between Validate and Commit, BL rejects the entire import (all-or-nothing); user returned to validation.

**AC22 — Authoritative Duplicate Check at Commit (per Targeted Period).** At Commit, if a prior Payroll Import entry exists for any (FY, Period) the file targets and the payload doesn't include an acknowledgment covering each affected period, BL rejects. Prevents DE warning bypass via URL manipulation, concurrent submissions, or client-side tampering.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| All rows share single PostedDate | One journal entry created on Commit, all lines under it |
| Rows span multiple unique PostedDates (same period) | N entries created on Commit (one per unique PostedDate); all entries post to the same period |
| Rows span multiple PostedDates across multiple periods | N entries created on Commit; entries distributed across periods per their PostedDates |
| Any row's derived Period closed by the time BL runs | Entire import rejected (all-or-nothing); no entries written; user directed to correct file or post prior-period adjustment |
| Duplicate detected for one or more targeted (FY, Period); DE ack flag absent or insufficient | BL rejects as duplicate; affected period(s) listed; user directed to acknowledge or reverse prior entry |
| Duplicate detected; DE ack flag covers each detected period | New entries written alongside prior; both retained for audit |
| Unknown GLAccount; sub-account exists for (GLAccount, SiteCode) | Line written; warning indicator persisted on account for Configuration review; no rollback |
| Unknown GLAccount; sub-account does NOT exist | Entire import rejected per sub-account validation (AC18); unknown-account warning path does not apply |
| Abnormal balance on a line (e.g., credit to an asset) | Accepted; BL does not enforce normal balance direction by base type |
| System error mid-write | All entries from this import roll back; no partial write of any entry; DE notified; user notified |
| Two concurrent imports targeting overlapping (FY, Period) | Second to reach Commit is rejected as duplicate (per AC9 / AC22) |
| Row PostedDate outside any configured Period (e.g., FY not configured) | Entire import rejected per AC19; no partial write |
| File-wide \|Debits − Credits\| > $0.01 | Entire import rejected per AC20; no partial write. Tolerance applies file-wide, not per entry. |
| Validate-only call: any check fails | Structured error list + error report URL returned; no ledger write attempted |
| Commit: any check fails (incl. period status re-check) | Structured error list returned; no ledger write performed |

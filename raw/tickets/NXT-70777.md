# NXT-70777 — FIN — Payroll Import: Business Logic

## Description
As a district finance user, I want imported payroll journal entries to post correctly, so Trial Balance, Balance Sheet, and P&L reflect accurate period activity and tie out for audit.

## Scope

This story covers BL-specific Payroll Import behavior for the base Commit flow:

* what Commit does
* blocking validations
* exception rows for Import Monitor > Exceptions
* journal entry creation
* Payroll Import Batch ID assignment
* concurrent submission guard
* Posting Activity Log labels
* report contribution

Universal Import Monitor behavior is platform-owned and not repeated here.

Duplicate detection, overwrite selection, and reversal of previous Payroll Import batches are covered in a separate story.

## Concurrent Submission Lock

If another Payroll Import is already in progress for the same district, the second attempt is blocked. First job wins.

```
A Payroll Import is already in progress for this district. Wait for it to finish, then refresh Import Monitor before trying again.
```

No row-level exception. This is a job-level block.

## Commit

Commit runs on Save after DE staging passes.

When Commit starts, BL receives:

* Payroll Import rows from the file
* Entry Description from Pre-upload

Each row contains:

* GLAccount
* SiteCode
* PostedDate
* Debit
* Credit

BL runs all Payroll Import checks during Commit. There is no separate Validate step for this flow.

If any blocking validation fails, BL writes no journal entries and returns Mandatory exception rows for Import Monitor > Exceptions.

## Exception Rows

BL returns exception rows to Import Monitor. Import Monitor shows them under the Exceptions tab.

Each exception row includes:

* Column Name
* Type
* Description

Type values used by this story:

* Mandatory: blocks the full file; no journal entries are written

## Blocking Validations

Blocking validations reject the full import. No partial journal write is allowed.

### Row structure

Each row must include GLAccount, SiteCode, PostedDate, and exactly one positive decimal value in Debit or Credit. PostedDate must parse as a valid date. Debit/Credit must parse as decimals when populated.

Rows are invalid if both Debit and Credit are populated, both are blank, either populated value is zero, or either populated value is negative.

Exceptions are at the cell level. A row that violates multiple rules produces one exception per rule violated per cell. The GLAccount from the row identifies which row the exception belongs to.

**Decimal precision.** Amounts must have 2 or fewer decimal places. Do not round or truncate. If more than 2 decimal places, the row is blocked.

**Currency symbols.** Values containing currency symbols ($), commas, or other non-numeric formatting characters are treated as non-decimal. The file must use plain numeric values per the downloadable template (**NXT-72160** §2.1). No stripping or conversion is performed.

Exception row:

* Column Name: GLAccount, SiteCode, PostedDate, Debit, Credit, or Debit/Credit based on the failed field or rule
* Type: Mandatory
* Description uses one of these exact messages:

    * `GLAccount {GLAccount}: Required value is missing for {ColumnName}.`
    * `GLAccount {GLAccount}: PostedDate must be a valid date.`
    * `GLAccount {GLAccount}: {ColumnName} must be a valid decimal amount.`
    * `GLAccount {GLAccount}: {ColumnName} has more than 2 decimal places. Do not round — correct the source value.`
    * `GLAccount {GLAccount}: Enter either Debit or Credit, not both.`
    * `GLAccount {GLAccount}: Either Debit or Credit is required.`
    * `GLAccount {GLAccount}: {ColumnName} must be greater than 0.`


If GLAccount is blank or missing on the excepted row, use `Row {RowNumber}:` as fallback.

### SiteCode

SiteCode must exist in Site Management.

**Case-insensitive matching.** SiteCode matching ignores case and trims leading/trailing spaces. The posted entry uses the canonical SiteCode from Site Management.

**Must be active.** Inactive sites are blocked. Payroll is current period activity — posting to a deactivated site masks errors and creates invisible report lines.

Exception row:

* Column Name: SiteCode
* Type: Mandatory
* Description: `GLAccount {GLAccount}: SiteCode {SiteCode} is inactive.`

**Site type does not matter.** The system must not inspect or block based on site type (meal-program, accounting, Central Office, or any other). No exception.

**Blank SiteCode blocks the row.** No defaulting — including no default to Central Office.

Exception row:

* Column Name: SiteCode
* Type: Mandatory
* Description: `GLAccount {GLAccount}: SiteCode {SiteCode} does not exist in Site Management.`

### GLAccount

GLAccount must exist in the chart of accounts. Unknown GLAccount rejects the full import.

**Must be active.** If the GLAccount exists but is inactive, the row is blocked.

**Must be in a valid format.** If the GLAccount is malformed (does not match the expected account format), the row is blocked.

Exception rows:

* Column Name: GLAccount
* Type: Mandatory
* Description uses one of:

    * `GLAccount {GLAccount}: does not exist in the chart of accounts.`
    * `GLAccount {GLAccount}: is inactive.`
    * `GLAccount {GLAccount}: is not in a valid account format.`


### Sub-account

A configured sub-account must exist for every `(GLAccount, SiteCode)` pair where GLAccount exists in the chart of accounts. Central Office follows the same rule as any other site.

Exception row:

* Column Name: GLAccount / SiteCode
* Type: Mandatory
* Description: `GLAccount {GLAccount}: Sub-account is not configured for SiteCode {SiteCode}.`

### Period derivation

BL derives `(FY, Period)` from each row's PostedDate using the district's timezone. If any PostedDate falls outside configured fiscal periods, BL rejects the full import.

Exception row:

* Column Name: PostedDate
* Type: Mandatory
* Description: `GLAccount {GLAccount}: PostedDate {PostedDate} does not map to a configured fiscal period.`

### Closed period

If any derived period is closed, BL rejects the full import. Open-period rows are not partially written.

Exception row:

* Column Name: PostedDate
* Type: Mandatory
* Description: `GLAccount {GLAccount}: PostedDate {PostedDate} maps to closed FY {FY}, Period {Period}.`

### Per-entry balance

BL groups rows by PostedDate. Each PostedDate group must balance within $0.01.

Exception row:

* Column Name: PostedDate
* Type: Mandatory
* Description: `PostedDate {PostedDate}: Entries are out of balance. Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}.`

### File-wide balance

The full file must balance within $0.01. DE may catch file-wide imbalance before Commit; BL still validates balance during Commit and controls final pass/fail.

Exception row:

* Column Name: FileBalance
* Type: Mandatory
* Description: `File is out of balance. Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}.`

### Non-zero file

At least one Debit or Credit must be greater than zero in the file. If all amounts are zero, the file is blocked.

Exception row:

* Column Name: FileBalance
* Type: Mandatory
* Description: `File contains no non-zero amounts. At least one Debit or Credit must be greater than zero.`

### Monetary precision

Amounts must be processed using decimal-safe logic. Floating-point precision loss is not acceptable for ledger data. This applies to all arithmetic: balance checks, summation, and storage.

## What BL Writes on Success

When all checks pass, BL groups the file rows by PostedDate. Each unique PostedDate becomes one journal entry. A file with three distinct PostedDates produces three entries.

**Each entry includes:**

A header and one or more lines. The header records when and why. The lines record what posted where.

**Header fields (one per entry):**

* **Posted Date** — the PostedDate shared by all rows in this group
* **Transaction Number** — system-assigned; unique to this entry. Each PostedDate group gets its own Transaction Number. This is the identifier that ties all lines in the entry together.
* **Fiscal Year and Period** — derived from the Posted Date
* **Description** — the Entry Description the user typed on Pre-upload, applied to every entry in the import
* **Entered Date** — timestamp when BL wrote the entry
* **Entered By** — the authenticated user who ran the import

**Line fields (one line per file row in the group):**

* **Location** — from the row's SiteCode
* **Account Number** — from the row's GLAccount
* **Account Name** — looked up from the chart of accounts
* **Debit** — the row's Debit value, or zero if the row is a Credit
* **Credit** — the row's Credit value, or zero if the row is a Debit

Lines do not carry their own description. Description lives on the entry header only.

## Payroll Import Batch ID

Each successful Payroll Import Commit is assigned a system-generated Payroll Import Batch ID.

All journal entries and lines created by that Commit are linked to the same Payroll Import Batch ID.

This story only creates and stores the batch ID. Duplicate detection, overwrite selection, and reversal of prior Payroll Import batches are covered in a separate story.

## Complete Write or Rollback

BL must write all entries for the Commit successfully, or write none of them. If a system error occurs while entries are being written, BL rolls back all entries and lines from that import. No partial ledger write remains.

## Posting Activity Log

Payroll Import entries appear in the Posting Activity Log with the label:

* `Payroll Import` — original batch

Audit details per entry: Fiscal Year, Period, Transaction ID, Payroll Import Batch ID.

Additional labels for reversal/replacement of prior Payroll Import batches are defined in the separate overwrite story.

## Reports

Payroll Import lines contribute to period activity on Trial Balance, Balance Sheet, and P&L based on each entry's derived `(FY, Period)`.

Payroll Import does not create or modify opening balances.

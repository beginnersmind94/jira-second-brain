# NXT-70777 — FIN — Payroll Import: Business Logic

## Description
As a district finance user, I want imported payroll journal entries to post correctly, so Trial Balance, Balance Sheet, and P&L reflect accurate period activity and tie out for audit.

## Scope

This story covers BL-specific Payroll Import behavior for the base Commit flow:

* what Commit does
* blocking validations
* exception rows for Import Monitor > Exceptions
* journal entry creation
* Payroll Import Batch ID assignment
* report contribution

Universal Import Monitor behavior is platform-owned and not repeated here.

Duplicate detection, overwrite selection, and reversal of previous Payroll Import batches are covered in a separate story.

## Commit

Commit runs on Save after DE staging passes.

When Commit starts, BL receives:

* Payroll Import rows from the file
* Entry Description from Pre-upload

Each row contains:

* GLAccount
* SiteCode
* PostedDate
* Debit
* Credit

BL runs all Payroll Import checks during Commit. There is no separate Validate step for this flow.

If any blocking validation fails, BL writes no journal entries and returns Mandatory exception rows for Import Monitor > Exceptions.

## Exception Rows

BL returns exception rows to Import Monitor. Import Monitor shows them under the Exceptions tab.

Each exception row includes:

* Column Name
* Type
* Description

Type values used by this story:

* Mandatory: blocks the full file; no journal entries are written

## Blocking Validations

Blocking validations reject the full import. No partial journal write is allowed.

### Row structure

Each row must include GLAccount, SiteCode, PostedDate, and exactly one positive decimal value in Debit or Credit. PostedDate must parse as a valid date. Debit/Credit must parse as decimals when populated.

Rows are invalid if both Debit and Credit are populated, both are blank, either populated value is zero, or either populated value is negative.

A row that violates multiple rules produces one exception row per rule violated.

Exception row:

* Column Name: GLAccount, SiteCode, PostedDate, Debit, Credit, or Debit/Credit based on the failed field or rule
* Type: Mandatory
* Description uses one of these exact messages:

    * `Source row {RowNumber}: Required value is missing for {ColumnName}.`
    * `Source row {RowNumber}: PostedDate must be a valid date.`
    * `Source row {RowNumber}: {ColumnName} must be a valid decimal amount.`
    * `Source row {RowNumber}: Enter either Debit or Credit, not both.`
    * `Source row {RowNumber}: Either Debit or Credit is required.`
    * `Source row {RowNumber}: {ColumnName} must be greater than 0.`
    

### SiteCode

SiteCode must exist in Site Management.

Exception row:

* Column Name: SiteCode
* Type: Mandatory
* Description: `Source row {RowNumber}: SiteCode {SiteCode} does not exist in Site Management.`

### GLAccount

GLAccount must exist in the chart of accounts. Unknown GLAccount rejects the full import.

Exception row:

* Column Name: GLAccount
* Type: Mandatory
* Description: `Source row {RowNumber}: GLAccount {GLAccount} does not exist in the chart of accounts.`

### Sub-account

A configured sub-account must exist for every `(GLAccount, SiteCode)` pair where GLAccount exists in the chart of accounts. Central Office follows the same rule as any other site.

Exception row:

* Column Name: GLAccount / SiteCode
* Type: Mandatory
* Description: `Source row {RowNumber}: Sub-account is not configured for GLAccount {GLAccount} and SiteCode {SiteCode}.`

### Period derivation

BL derives `(FY, Period)` from each row's PostedDate using the district's timezone. If any PostedDate falls outside configured fiscal periods, BL rejects the full import.

Exception row:

* Column Name: PostedDate
* Type: Mandatory
* Description: `Source row {RowNumber}: PostedDate {PostedDate} does not map to a configured fiscal period.`

### Closed period

If any derived period is closed, BL rejects the full import. Open-period rows are not partially written.

Exception row:

* Column Name: PostedDate
* Type: Mandatory
* Description: `Source row {RowNumber}: PostedDate {PostedDate} maps to closed FY {FY}, Period {Period}.`

### Per-entry balance

BL groups rows by PostedDate. Each PostedDate group must balance within $0.01.

Exception row:

* Column Name: PostedDate
* Type: Mandatory
* Description: `Source rows {RowNumbers}: Entries for PostedDate {PostedDate} are out of balance. Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}.`

### File-wide balance

The full file must balance within $0.01. DE may catch file-wide imbalance before Commit; BL still validates balance during Commit and controls final pass/fail.

Exception row:

* Column Name: FileBalance
* Type: Mandatory
* Description: `File is out of balance. Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}.`

## Entry Creation

On full pass, BL creates one journal entry per unique PostedDate.

Each entry:

* contains only rows sharing that PostedDate
* posts to one derived `(FY, Period)`
* uses the same Entry Description from Pre-upload

## Entry Header

Each generated entry contains:

* Posted Date
* Transaction Number, system-assigned
* Description, from Pre-upload
* Entered Date, system-stamped at write time
* Entered By, from authenticated user

Transaction Number ties all lines in the entry together.

## Entry Lines

Each line contains:

* Location from SiteCode
* Account Number from GLAccount
* Account Name per existing GL account display behavior
* Debit or Credit from the row
* opposite side stored as zero

No line-level description is created.

## Payroll Import Batch ID

Each successful Payroll Import Commit is assigned a system-generated Payroll Import Batch ID.

All journal entries and lines created by that Commit are linked to the same Payroll Import Batch ID.

This story only creates and stores the batch ID. Duplicate detection, overwrite selection, and reversal of prior Payroll Import batches are covered in a separate story.

## Complete Write or Rollback

BL must write all entries for the Commit successfully, or write none of them. If a system error occurs while entries are being written, BL rolls back all entries and lines from that import. No partial ledger write remains.

## Reports

Payroll Import lines contribute to period activity on Trial Balance, Balance Sheet, and P&L based on each entry's derived `(FY, Period)`.

Payroll Import does not create or modify opening balances.

## Acceptance Criteria
## AC1 — Commit receives rows and Entry Description from DE

Given the DE staging check has passed
When Commit runs
Then BL receives Payroll Import rows and Entry Description from Pre-upload
And each row includes GLAccount, SiteCode, PostedDate, Debit, and Credit.

## AC2 — Blocking validation failure rejects the full import

Given any blocking validation fails during Commit
When Commit finishes
Then no journal entries or lines are written
And BL returns exception rows for Import Monitor
And those rows are shown under Import Monitor > Exceptions with Type = Mandatory.

## AC3 — Row structure is validated

Given Payroll Import rows are processed during Commit
When any row is missing GLAccount, SiteCode, or PostedDate
Or any row's PostedDate cannot be parsed
Or any row's Debit/Credit cannot be parsed as decimals when populated
Or any row has both Debit and Credit populated
Or any row has both Debit and Credit blank
Or any row's populated Debit/Credit value is zero or negative
Then BL rejects the full import
And returns one RowStructure exception row per rule violation for each affected row under Import Monitor > Exceptions
And a row that violates multiple rules produces multiple exception rows.

Exception row details:

* Column Name = GLAccount, SiteCode, PostedDate, Debit, Credit, or Debit/Credit based on the failed field/rule
* Type = Mandatory
* Description = one of the following exact messages:
  * `Source row {RowNumber}: Required value is missing for {ColumnName}.`
  * `Source row {RowNumber}: PostedDate must be a valid date.`
  * `Source row {RowNumber}: {ColumnName} must be a valid decimal amount.`
  * `Source row {RowNumber}: Enter either Debit or Credit, not both.`
  * `Source row {RowNumber}: Either Debit or Credit is required.`
  * `Source row {RowNumber}: {ColumnName} must be greater than 0.`

## AC4 — SiteCode must exist

Given any row contains a SiteCode that does not exist in Site Management
When Commit runs
Then BL rejects the full import
And returns a SiteNotFound exception row for each affected row under Import Monitor > Exceptions.

Exception row details:

* Column Name = SiteCode
* Type = Mandatory
* Description = `Source row {RowNumber}: SiteCode {SiteCode} does not exist in Site Management.`

## AC5 — Sub-account must exist for every GLAccount and SiteCode pair

Given any row's GLAccount exists in the chart of accounts
And its `(GLAccount, SiteCode)` pair does not have a configured sub-account
When Commit runs
Then BL rejects the full import
And returns a SubAccountMissing exception row for each affected row under Import Monitor > Exceptions
And Central Office follows the same sub-account rule as any other site.

Exception row details:

* Column Name = GLAccount / SiteCode
* Type = Mandatory
* Description = `Source row {RowNumber}: Sub-account is not configured for GLAccount {GLAccount} and SiteCode {SiteCode}.`

## AC6 — Unknown GLAccount is rejected

Given any row references a GLAccount that does not exist in the chart of accounts
When Commit runs
Then BL rejects the full import
And returns a GLAccountNotFound exception row for each affected row under Import Monitor > Exceptions.

Exception row details:

* Column Name = GLAccount
* Type = Mandatory
* Description = `Source row {RowNumber}: GLAccount {GLAccount} does not exist in the chart of accounts.`

## AC7 — PostedDate must map to a configured period

Given any row's PostedDate cannot be mapped to a configured `(FY, Period)` in the district's timezone
When Commit runs
Then BL rejects the full import
And returns an UnconfiguredPeriod exception row for each affected row under Import Monitor > Exceptions.

Exception row details:

* Column Name = PostedDate
* Type = Mandatory
* Description = `Source row {RowNumber}: PostedDate {PostedDate} does not map to a configured fiscal period.`

## AC8 — Closed periods are rejected

Given any row's PostedDate maps to a closed period
When Commit runs
Then BL rejects the full import
And returns a ClosedPeriod exception row for each affected row under Import Monitor > Exceptions
And no open-period rows are partially written.

Exception row details:

* Column Name = PostedDate
* Type = Mandatory
* Description = `Source row {RowNumber}: PostedDate {PostedDate} maps to closed FY {FY}, Period {Period}.`

## AC9 — Each PostedDate group must balance

Given rows are grouped by PostedDate
When total Debits and total Credits for any PostedDate group differ by more than $0.01
Then BL rejects the full import
And returns an EntryBalance exception row for each affected PostedDate group under Import Monitor > Exceptions.

Exception row details:

* Column Name = PostedDate
* Type = Mandatory
* Description = `Source rows {RowNumbers}: Entries for PostedDate {PostedDate} are out of balance. Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}.`

## AC10 — The full file must balance

Given all rows in the file are summed
When total Debits and total Credits differ by more than $0.01
Then BL rejects the full import
And returns a FileBalance exception row under Import Monitor > Exceptions.

Exception row details:

* Column Name = FileBalance
* Type = Mandatory
* Description = `File is out of balance. Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}.`

## AC11 — One journal entry is created per unique PostedDate

Given all blocking validations pass
When Commit writes the import
Then BL creates one journal entry for each unique PostedDate in the file
And each entry contains only rows sharing that PostedDate.

## AC12 — Each generated entry posts to one fiscal period

Given BL creates a journal entry for a PostedDate
When the entry is written
Then the entry posts to the `(FY, Period)` derived from that PostedDate
And the entry never spans multiple periods.

## AC13 — Transaction Number ties entry lines together

Given BL creates a journal entry
When the entry is written
Then the system assigns a Transaction Number
And every line in that entry shares the same Transaction Number.

## AC14 — Entry header carries Pre-upload Entry Description

Given the user entered Entry Description during Pre-upload
When BL creates journal entries
Then each generated entry header stores that Entry Description verbatim
And the value is identical across all entries created from the same import.

## AC15 — Entry header stores system-entered fields

Given BL creates a journal entry
When the entry is written
Then Entered Date is stamped at write time
And Entered By is stamped from the authenticated user.

## AC16 — Entry lines map uploaded values directly

Given a validated row is written as an entry line
When BL creates the line
Then SiteCode maps to Location
And GLAccount maps to Account Number
And Account Name follows existing GL account display behavior
And Debit or Credit is written from the row
And the opposite side is stored as zero.

## AC17 — No line-level description is created

Given a Payroll Import row is written as an entry line
When BL creates the line
Then no separate line-level description is stored
And Description remains an entry-header value only.

## AC18 — Payroll Import Batch ID is assigned

Given all blocking validations pass
When BL writes the Payroll Import
Then BL assigns a system-generated Payroll Import Batch ID to the Commit
And all journal entries and lines created by that Commit are linked to the same Payroll Import Batch ID.

## AC19 — Commit write is complete or rolled back

Given all blocking validations pass
And a system error occurs while entries or lines are being written
When Commit fails mid-write
Then all entries and lines from that import are rolled back
And no partial ledger write remains.

## AC20 — Payroll Import contributes to financial reports

Given Payroll Import entries have been written
When Trial Balance, Balance Sheet, or P&L is run for the affected period
Then the imported lines contribute to period activity for each entry's derived `(FY, Period)`.

## AC21 — Payroll Import does not affect opening balances

Given Payroll Import entries have been written
When opening balances are calculated or displayed
Then Payroll Import lines do not create or modify opening balances.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| All rows share single PostedDate | One journal entry created on Commit, all lines under it |
| Rows span multiple unique PostedDates (same period) | N entries created on Commit (one per unique PostedDate); all entries post to the same period |
| Rows span multiple PostedDates across multiple periods | N entries created on Commit; entries distributed across periods per their PostedDates |
| Any row's derived Period closed by the time BL runs | Entire import rejected (all-or-nothing); no entries written; user directed to correct file or post prior-period adjustment |
| Unknown GLAccount | Entire import rejected; no partial write |
| Abnormal balance on a line (e.g., credit to an asset) | Accepted; BL does not enforce normal balance direction by base type |
| System error mid-write | All entries from this import roll back; no partial write of any entry; DE and user notified |
| Row PostedDate outside any configured Period (e.g., FY not configured) | Entire import rejected; no partial write |
| File-wide \|Debits − Credits\| > $0.01 | Entire import rejected; no partial write |
| Commit: any check fails (incl. period status re-check) | Exception rows returned; no ledger write performed |

## Data Model Summary

| Attribute | Value |
| --- | --- |
| Entry scope | One entry per unique PostedDate in the file; N entries possible per import; each entry posts to exactly one period |
| PostedDate | Per-entry header field; unique across entries within an import; identical for all lines on an entry |
| Derived (FY, Period) | Per-entry; derived from header PostedDate; one entry = one period |
| Entry Description | Per-entry header; verbatim from Pre-upload; identical across all entries from one import; max 240 chars |
| Line description | Not stored — Financials 2.0 posting engine carries description at entry header level only |
| Amount storage | Debit or credit per line; opposite side = zero |
| Editable via UI | No — read-only, immutable |
| Correctable | Yes — via manual reversal or adjustment entry |
| Period constraints | Rejected if any row's derived Period is closed (all-or-nothing) |
| Sub-account constraints | Rejected if any (GLAccount, SiteCode) lacks a configured sub-account, where GLAccount exists in the chart of accounts (all-or-nothing) |
| Balance constraints | Each PostedDate group balances within $0.01; full file balances within $0.01 |
| Unknown account handling | Rejected if GLAccount is unknown |
| Complete Write or Rollback | All entries from one import write or none; no partial write of one of multiple entries |
| Contributes to | Period activity on Trial Balance, Balance Sheet, P&L (per each entry's derived period) |
| Affects opening balances | No — opening balances come only from Opening Balance Import (NXT-71240) |

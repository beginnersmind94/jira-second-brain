# NXT-70777 — FIN — Payroll Import: Business Logic

## Description
As a district finance user, I want imported payroll and month-end journal entries to post to the ledger correctly, so that my Trial Balance, Balance Sheet, and P&L reflect accurate period activity and my books tie out for audit.

_Updated 2026-05-04. Split from combined DE+BL story; Data Exchange now lives in companion story **NXT-72160**._

---

## Scope

This story covers the business logic layer for Payroll Import: journal entry creation, per-row PostingDate handling, derived (Fiscal Year, Period) assignment per row, closed-period rejection (defense-in-depth backstop, all-or-nothing), duplicate detection backstop across all targeted periods, sub-account validation, unknown GLAccount handling, atomic ledger write, entry immutability, and downstream report contribution. The data exchange layer (companion story **NXT-72160**) handles file upload, Import Type setup, template configuration (Amount Format + Column Mapping), row/import-level validation, period derivation from PostingDate, preview, saved template behavior, import results, and duplicate detection warning. This story receives validated data with each row carrying its derived (Fiscal Year, Period) assignment after the user confirms the import.

---

## Requirements

### 1) Journal Entry Creation

1.1 On receiving validated import data from the data exchange layer, the system must create **one journal entry** (header) containing **N lines** — one line per imported row. Header attributes:

* **Entry type:** "Payroll Import" (or equivalent dedicated type, distinguishable from Opening Balance and from manual journal entries)
* **Source:** Import Monitor (for audit)
* **User / Timestamp:** the confirming user and the confirmation timestamp
* **Targeted periods:** the set of (FY, Period) the import's rows post to (derived from PostingDates, may be one or more)

1.2 Each line carries:

* **GLAccount**
* **SiteCode** (the value from the file; for district-wide entries this is the site code designated as the Central Office in Site Management)
* **Description**
* **PostingDate** (from the row — payroll often contains mid-month accruals, reversals, and dated adjustments)
* **Derived Fiscal Year and Period** (derived from PostingDate by the data exchange layer per **NXT-72160** §3, passed through to BL on the line)
* **Debit** or **Credit** (per the Amount Format; opposite side stored as zero)
* **AccountName** (if mapped)
* **Comments** (if mapped)

1.3 PostingDate and the derived (FY, Period) are **per-line** attributes, not per-entry. A single Payroll Import entry may legitimately span multiple periods if the file contains rows with PostingDates in more than one open period. The data exchange layer has already verified that every row's PostingDate falls in an open, configured period.

1.4 The Amount Format conversion is:

* **Scheme A (Debit/Credit):** values map directly; opposite side stored as zero.
* **Scheme B (signed Amount):** positive values stored as debits, negative values stored as credits (absolute value); opposite side stored as zero.

1.5 The system does **not** enforce normal balance direction by base type. Abnormal balances on individual lines are accepted (e.g., a credit to an asset for a correcting entry). Batch correctness is enforced by the data exchange layer's $0.01-tolerance balance check before handoff.

### 2) Closed-Period Rejection (defense-in-depth backstop, all-or-nothing)

2.1 Before writing the journal entry, BL must re-verify the period status for every line. If **any** line's derived Period has been closed by the time the journal entry is written, the entire import is rejected. Partial writes of the open-period lines are not permitted.

2.2 This prevents payroll from posting to a closed period even if the data exchange layer's preview-time and confirm-time checks were somehow bypassed (URL manipulation, concurrent period close, or a bug in DE).

2.3 On rejection, no journal entry or lines are written. The user is returned an error identifying the affected closed period(s) and the row count for each, and directing them to either correct the file (remove or re-date the offending rows) or post a prior-period adjustment for those rows through the manual journal entry flow.

2.4 This is a backstop check — the data exchange layer (**NXT-72160**) applies the same all-or-nothing rule upstream at preview and at confirm. BL is the authoritative write layer and therefore the authoritative enforcement point.

### 3) Duplicate Detection Backstop (per targeted period)

3.1 Before writing the journal entry, BL must check for an existing Payroll Import journal entry for **each (FY, Period) the import targets**. A file that targets a single period checks one bucket; a file that spans multiple periods checks each one.

3.2 If a prior Payroll Import entry exists for any of the targeted (FY, Period) buckets, BL must reject the new import with an error identifying the affected period(s). BL does not silently overwrite, append, or merge.

3.3 The user's explicit acknowledgment of the DE-side duplicate warning (per **NXT-72160** §8.3) must be passed to BL along with the validated data. The acknowledgment must specifically cover all targeted periods where a duplicate was detected at DE preview time. If the acknowledgment flag is present and covers all currently-detected duplicates, BL allows the write to proceed. If the acknowledgment is absent or does not cover a duplicate that BL detects at write time, BL rejects.

3.4 This backstop prevents the DE warning from being bypassed by any means (URL manipulation, concurrent submissions, client-side tampering) and is the authoritative duplicate check.

### 4) Sub-account Validation

4.1 For each line, BL must verify that a sub-account exists for the (GLAccount, SiteCode) combination. The sub-account is the routing target for posting site-level activity to a given GL account.

4.2 If any line's (GLAccount, SiteCode) combination has no configured sub-account, the **entire import is rejected**. Per the atomicity rule (§7 of this story), partial writes are not allowed — a single missing sub-account blocks the whole import.

4.3 The Central Office is treated as just another site for this check. A row with the Central Office SiteCode requires a configured sub-account on the targeted GL account, the same as any other site. BL does not apply special handling based on whether the SiteCode happens to be the Central Office.

4.4 On rejection, the import result record must surface the specific offending row(s) — the GLAccount, SiteCode, and a clear message indicating that no sub-account exists for that combination. The user resolves by either correcting the file (changing the SiteCode or GLAccount on the offending rows) or configuring the missing sub-account in Configuration > General Ledger and re-uploading.

4.5 This check runs at write time. The data exchange layer (**NXT-72160**) does not pre-validate sub-account configuration — DE only validates that the SiteCode itself exists in Site Management. Sub-account routing is a BL concern.

### 5) Unknown GLAccount Handling

5.1 For each line, if the GLAccount does not exist in the chart of accounts, BL must still write the line to the ledger (payroll feeds legitimately reference newly-introduced accounts that need review but should not halt month-end processing — per **NXT-72160** §7.5).

5.2 When writing a line for an unknown GLAccount, BL must persist the warning indicator on the account record such that it appears in Configuration > General Ledger > GL Accounts with a clear visual flag for finance review.

5.3 Once a user reviews the flagged account and confirms or updates it in Configuration, the warning indicator is cleared. This review workflow is not part of this story — only the persistence of the warning flag on write is.

5.4 An unknown GLAccount that is permitted under §5.1 still requires a configured sub-account for the (GLAccount, SiteCode) combination per §4. If the account is unknown AND no sub-account exists for the (GLAccount, SiteCode) combination, the import is rejected per §4. The unknown-account warning path applies only when sub-account validation passes.

### 6) Entry Behavior

6.1 Payroll Import journal entries are **read-only** in the UI. Lines cannot be edited or individually deleted.

6.2 If a user needs to correct posted payroll entries, they create a manual reversal or adjustment entry. The original Payroll Import entry remains intact for audit trail.

6.3 The "Payroll Import" entry type must be identifiable in the ledger so that reports, filters, and audit views can distinguish payroll imports from manual journal entries, opening balance entries, and other automated entry types.

### 7) Atomic Ledger Write

7.1 The entire journal entry (header + all lines, across all targeted periods) must be written **atomically**. Either all lines post successfully, or no lines post.

7.2 If any system-level error occurs during the ledger write, the entire entry must roll back. No partial write is permitted, including no partial write of just the open-period lines or just the lines for one of multiple targeted periods.

7.3 On rollback, the data exchange layer must be notified so the import result record reflects the failure and the user is informed.

### 8) Downstream Report Behavior

8.1 Payroll Import journal entry lines contribute to **period activity** on all standard reports: Trial Balance, Balance Sheet, P&L (Revenue & Expenditure), and any period-based account activity views. Lines contribute to the period each line's PostingDate assigns it to.

8.2 Each line posts to its own PostingDate (per §1.3), so a line dated mid-period and a line dated end-of-period both contribute to the same Period's activity but may appear on different daily activity views. Lines from a single entry that span multiple periods contribute to each respective period's activity.

8.3 Payroll Import lines are filterable and groupable by the "Payroll Import" entry type in reports that support entry type filtering.

8.4 Payroll Import lines do **not** establish or modify opening balances. Opening balances are set only via the Opening Balance Import flow (**NXT-71240** / **NXT-71249**).

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| Any line's derived Period has been closed by the time BL runs | BL rejects entire import (all-or-nothing); no entry written; user directed to correct the file or post a prior-period adjustment for the affected rows |
| A Payroll Import entry already exists for any targeted (FY, Period), and DE acknowledgment is absent or does not cover that period | BL rejects entire import; no entry written; user directed to reverse the prior entry or re-run the import to acknowledge via DE |
| A Payroll Import entry already exists for a targeted (FY, Period), and DE acknowledgment covers it | BL writes the new entry alongside the prior one (both remain for audit) |
| Any line's (GLAccount, SiteCode) has no configured sub-account | BL rejects entire import (all-or-nothing); no entry written; offending rows surfaced in import result; user corrects the file or configures the missing sub-account |
| Line references an unknown GLAccount with a configured sub-account | Line written; warning indicator persisted on the account for Configuration review; no rollback |
| Line references an unknown GLAccount with no configured sub-account | BL rejects entire import per §4; user resolves sub-account configuration first |
| Abnormal balance on a line (e.g., credit to an asset) | Accepted; BL does not enforce normal balance direction by base type |
| System error mid-write (e.g., database failure, timeout) | Entire entry rolls back; no partial write; DE notified; user notified |
| Two concurrent imports targeting the same (FY, Period) | Second import to reach BL is rejected as a duplicate (per §3) |
| Line PostingDate falls in a period that does not exist (should not occur — DE blocks this) | If it somehow reaches BL, BL rejects the entire import and logs the DE validation gap for investigation |
| Line carries a derived (FY, Period) inconsistent with its PostingDate (should not occur — DE derives both) | If it somehow reaches BL, BL re-derives from PostingDate, treats PostingDate as authoritative, and logs the DE inconsistency for investigation |

## Acceptance Criteria
**AC1 — Journal Entry Created with Correct Structure** Given the data exchange layer passes validated import data after user confirmation, When the business logic layer processes the data, Then one journal entry is created with entry type "Payroll Import" containing one line per imported row, carrying the selected Fiscal Year and Period on the header.

**AC2 — PostingDate Stored Per Line** Given an import contains rows with different PostingDate values within the selected Period, When the journal entry is written, Then each line carries its own PostingDate from the file — not a single header-level date.

**AC3 — Scheme A Amounts Mapped Directly** Given the import uses Scheme A (Debit/Credit), When the journal entry lines are created, Then each line carries the debit or credit value from the file and zero on the opposite side.

**AC4 — Scheme B Amounts Converted Correctly** Given the import uses Scheme B (signed Amount), When the journal entry lines are created, Then positive values are stored as debits and negative values are stored as credits (absolute value), with zero on the opposite side.

**AC5 — Normal Balance Not Enforced by Base Type** Given an import contains a line with a credit to an asset or a debit to a liability (abnormal balance), When the business logic layer processes the import, Then the line is written as-is — the system does not reject or modify the balance direction based on account type.

**AC6 — Closed-Period Target Rejected** Given the target Period has been closed by the time the business logic layer runs, When BL receives the validated import, Then BL rejects the import with an error identifying the closed period and directing the user to reopen it through standard period management. No journal entry or lines are written. _(This is a defense-in-depth backstop — the data exchange layer enforces the same rule upstream at Period dropdown and at confirm.)_

**AC7 — Duplicate Rejected Without DE Acknowledgment** Given a Payroll Import journal entry already exists for the same Fiscal Year and Period and the DE duplicate-acknowledgment flag is absent, When BL receives the new import, Then BL rejects the import with an error directing the user to reverse the prior entry or re-run through DE with the duplicate warning acknowledged. No journal entry is written.

**AC8 — Duplicate Allowed With DE Acknowledgment** Given a Payroll Import journal entry already exists for the same Fiscal Year and Period and the DE duplicate-acknowledgment flag is present, When BL receives the new import, Then BL writes the new entry alongside the prior one — both remain in the ledger for audit trail.

**AC9 — Unknown GLAccount Writes with Warning Flag** Given an import contains a line with a GLAccount that does not exist in the chart of accounts, When BL processes the import, Then the line is written to the ledger and the account is persisted with a warning indicator such that it appears in Configuration > General Ledger > GL Accounts with a clear visual flag for finance review. BL does not block the import or the line.

**AC10 — Atomic Ledger Write** Given a system error occurs partway through writing the journal entry header and lines, When the error is encountered, Then the entire entry (header + all lines) rolls back — no partial write remains, DE is notified, and the user is informed the import failed.

**AC11 — Entry Is Read-Only** Given a Payroll Import journal entry exists in the ledger, When a user attempts to edit or delete it through the UI, Then the system blocks the action — the entry can only be corrected via a manual reversal or adjustment entry.

**AC12 — Entry Type Identifiable for Reports and Filters** Given Payroll Import entries coexist with manual journal entries, opening balance entries, and other automated entry types, When a report or audit view filters by entry type, Then Payroll Import entries can be isolated or excluded using the "Payroll Import" type identifier.

**AC13 — Lines Contribute to Period Activity (Not Opening Balance)** Given Payroll Import lines have been written for a Fiscal Year and Period, When the user runs a Trial Balance, Balance Sheet, or P&L for that Period, Then the lines appear as period activity. The Opening Balance column on these reports is not affected — opening balances come only from the Opening Balance Import flow (NXT-71240 / NXT-71249).

# NXT-70777 — FIN — Payroll Import: Business Logic

## Description
As a district finance user, I want imported payroll and month-end journal entries to post to the ledger correctly, so that my Trial Balance, Balance Sheet, and P&L reflect accurate period activity and my books tie out for audit.

_Updated 2026-05-07. Added TL;DR section. Updated 2026-05-06: added consolidated API Validations section reflecting the DE/BL contract established with Jonathan Smith — DE hands off the mapped file, BL runs all validations and returns pass/fail with totals + per-period breakdown or a structured error report._

---

## TL;DR

Expose two API ops to DE: **Validate** (run every check; write nothing) and **Commit** (run every check + write). _DE calls Validate at the Field Mapping validator gate; Commit at the user's final confirm._

**On every call, BL runs:**

* **Row structure:** required fields present (incl. `PostingDate`); PostingDate **parses**; Debit/Credit are decimals; _exactly one_ of Debit/Credit positive; Comments ≤ 50 chars.
* **SiteCode exists.**
* Each `(GLAccount, SiteCode)` has a configured **sub-account** — **blocking**. Central Office is just another site.
* **GLAccount existence is non-blocking** — unknown accounts get written and flagged for review _(differs from Opening Balance, which blocks)_.
* **Derive** `(FY, Period)` per row from PostingDate in the district's timezone.
* **Closed period or unconfigured period** anywhere in the file = **reject the entire import** (all-or-nothing).
* **Balance:** absolute difference between Debits and Credits ≤ **$0.01** — file-wide _(not per period; differs from OB's exact $0)_.
* **Duplicate check per targeted** `(FY, Period)`: if a prior Payroll Import journal entry exists, return the period in the response. On Commit, the DE acknowledgment flag must specifically cover each detected period. _Authoritative duplicate check._

**Commit-only:** re-derive periods and re-verify closure.

**Response on pass:** rows, Debits, Credits, **per-period breakdown** `(FY, Period, row count, period Debits, period Credits)`, unknown GLAccount list, duplicate periods (when present).
**Response on fail:** structured error list + error report URL.

**On Commit success:** create one journal entry, type **"Payroll Import"**, one line per row. **PostingDate is per-line, not per-entry** — _a single entry may legitimately span multiple periods_. Each line persists its own derived `(FY, Period)`. Each line stores Debit or Credit with **zero on the opposite side**. Entry is **read-only** — correct via manual reversal/adjustment.

**Unknown GLAccount path:** line writes; account flagged in Configuration > General Ledger for review. _But_ if `(GLAccount, SiteCode)` also has no sub-account, the import is rejected — **sub-account check wins**.

**Atomic write across all targeted periods.** Concurrent imports targeting overlapping periods → second to reach Commit is rejected as duplicate.

**Reports:** Lines = **period activity** on Trial Balance, Balance Sheet, P&L (per each line's derived period). Lines do **not** affect opening balances — those come only from the OB Import flow (NXT-71240).

**Abnormal balances are accepted** — we don't enforce normal balance direction by base type.

## Scope

This story covers the business logic layer for Payroll Import: API Validations (validate-only and commit modes), journal entry creation, per-row PostingDate handling, derived (Fiscal Year, Period) assignment per row, closed-period rejection (defense-in-depth backstop, all-or-nothing), duplicate detection backstop across all targeted periods, sub-account validation, unknown GLAccount handling, atomic ledger write, entry immutability, and downstream report contribution. The data exchange layer (companion story **NXT-72160**) handles file upload, Import Type setup, column mapping UI, validator gate (which calls the BL Validate API), preview, saved template behavior, import results, and duplicate detection warning. This story receives validated data with separate Debit and Credit columns and each row carrying its derived (Fiscal Year, Period) assignment after the user confirms the import.

## API Validations

The BL API exposes a **Validate** operation (called by DE on the Field Mapping tab per **NXT-72160** §8.1) and a **Commit** operation (called by DE on Confirm). Both run the same validation pass; Commit additionally writes the journal entry. All checks below run on every call.

If any check fails, the API returns failure with a structured error list (row number, field, error code, message) and a generated error report download URL. DE surfaces the result. **No partial work is performed.**

### 1) Row-level structural

1.1 **Required fields.** GLAccount, SiteCode, Description, and PostingDate are present on every row.

1.2 **PostingDate parses** as a valid date.

1.3 **Decimal numbers.** Debit and Credit are decimal numbers across all rows. (DE peeks at the first 10 rows for fail-fast detection per **NXT-72160** §7.2; BL re-checks every row.)

1.4 **Single-side rule.** Each row carries a positive value in **exactly one** of Debit or Credit. Rows with both populated, both blank, both zero, or any negative value are invalid.

1.5 **Comments length.** Comments ≤ 50 characters when provided.

### 2) Reference data

2.1 **SiteCode exists** as configured in SchoolCafe Site Management.

2.2 **Sub-account configured** for each (GLAccount, SiteCode) combination per §4 — **blocking**. Central Office is treated as just another site (no special handling).

2.3 **GLAccount existence is non-blocking.** Unknown accounts are accepted at validation; on Commit they are written to the ledger and a warning indicator is persisted on the account record for finance review per §5. _(This differs from Opening Balance, which blocks on unknown accounts.)_

### 3) Period derivation (per row)

3.1 **Derive (FY, Period)** from each row's PostingDate in the district's configured timezone.

3.2 **Configured-period check.** If any row's PostingDate falls outside any configured Period (e.g., the FY does not exist in the system), reject the entire import.

3.3 **Closed-period check (all-or-nothing).** If any row's derived Period is closed, reject the entire import. Partial write of the open-period rows is not permitted.

3.4 On Validate success, return the derived (FY, Period) for each row and the per-period breakdown (row count, Debits, Credits) so DE can render the summary on Review per **NXT-72160** §3.2.

### 4) Aggregate

4.1 **Balanced.** Absolute difference between Sum of Debits and Sum of Credits ≤ **$0.01** — tolerance permits payroll's natural rounding across many lines. The check applies to the file as a whole, not per period — a payroll file is a single balanced unit even when its rows span multiple periods. _(This differs from Opening Balance, which requires exact $0.)_

### 5) Duplicate detection (per targeted period)

5.1 For each (FY, Period) the file targets, check whether a prior successful Payroll Import journal entry exists.

5.2 On Validate, return the list of periods where a duplicate is detected so DE can collect the user's acknowledgment per **NXT-72160** §8.4.

5.3 On Commit, the duplicate acknowledgment flag must be present in the request payload and must specifically cover every period where BL detects a duplicate at write time. If acknowledgment is missing or does not cover a current duplicate, reject. **This is the authoritative duplicate check** — it prevents the DE-side warning from being bypassed.

### 6) Period status re-check (Commit only, defense-in-depth)

6.1 At Commit, re-derive (FY, Period) from each row's PostingDate and re-verify that no row's Period has been closed since Validate. If any period closed in the meantime, reject the entire import (all-or-nothing) and return the user to validation.

### Response contract

**Success** — all checks pass:

* Status: `pass`
* Total rows
* Total Debits
* Total Credits
* Per-period breakdown — list of `(FY, Period, row count, period Debits, period Credits)`
* Unknown GLAccount list — account codes flagged for finance review on Commit
* Duplicate periods (when detected) — DE acknowledgment required before Commit

**Failure** — any check fails:

* Status: `fail`
* Error list — row number, field, error code, message
* Error report download URL (generated by BL; DE surfaces the link per **NXT-72160** §8.1)

## Requirements

### 1) Journal Entry Creation

1.1 On Commit, after all validations pass per the API Validations section, the system must create **one journal entry** (header) containing **N lines** — one line per imported row. Header attributes:

* **Entry type:** "Payroll Import" (or equivalent dedicated type, distinguishable from Opening Balance and from manual journal entries)
* **Source:** Import Monitor (for audit)
* **User / Timestamp:** the confirming user and the confirmation timestamp
* **Targeted periods:** the set of (FY, Period) the import's rows post to (derived from PostingDates, may be one or more)

1.2 Each line carries:

* **GLAccount**
* **SiteCode** (the value from the file; for district-wide entries this is the site code designated as the Central Office, as configured in SchoolCafe)
* **Description**
* **PostingDate** (from the row — payroll often contains mid-month accruals, reversals, and dated adjustments)
* **Derived Fiscal Year and Period** (derived from PostingDate by the API Validations §3, persisted on the line)
* **Debit** or **Credit** (positive value in exactly one of the two columns, per the canonical contract; opposite side stored as zero)
* **AccountName** (if mapped)
* **Comments** (if mapped)

1.3 PostingDate and the derived (FY, Period) are **per-line** attributes, not per-entry. A single Payroll Import entry may legitimately span multiple periods if the file contains rows with PostingDates in more than one open period.

1.4 The data exchange layer enforces a fixed Debit/Credit amount format (separate Debit and Credit columns; signed-amount layouts are transformed by DE upstream of mapping per **NXT-72160** §2). Each validated row carries a positive value in exactly one of the two columns. The Debit and Credit values from the validated row map directly to the ledger entry line. The opposite side is stored as zero.

1.5 The system does **not** enforce normal balance direction by base type. Abnormal balances on individual lines are accepted (e.g., a credit to an asset for a correcting entry). Batch correctness is enforced by the API Validations §4 balance check.

### 2) Closed-Period Rejection (defense-in-depth backstop, all-or-nothing)

2.1 The closed-period rule is enforced by API Validations §3.3 — see that section for the authoritative behavior. The Commit-only re-check in API Validations §6 catches any period that closed between Validate and Commit.

2.2 This prevents payroll from posting to a closed period even if the data exchange layer's preview-time and confirm-time checks were somehow bypassed (URL manipulation, concurrent period close, or a bug in DE).

2.3 On rejection, no journal entry or lines are written. The user is returned an error identifying the affected closed period(s) and the row count for each, and directing them to either correct the file (remove or re-date the offending rows) or post a prior-period adjustment for those rows through the manual journal entry flow.

### 3) Duplicate Detection Backstop (per targeted period)

3.1 Duplicate detection is enforced by API Validations §5 — see that section for the authoritative behavior.

3.2 The user's explicit acknowledgment of the DE-side duplicate warning (per **NXT-72160** §8.3) must be passed to BL along with the validated data. The acknowledgment must specifically cover all targeted periods where a duplicate was detected at DE preview time. If the acknowledgment flag is present and covers all currently-detected duplicates, BL allows the write to proceed. If the acknowledgment is absent or does not cover a duplicate that BL detects at write time, BL rejects.

3.3 This backstop prevents the DE warning from being bypassed by any means (URL manipulation, concurrent submissions, client-side tampering) and is the authoritative duplicate check.

### 4) Sub-account Validation

4.1 Sub-account validation is part of the API Validations pass — see §2.2 of that section for the authoritative rule. The sub-account is the routing target for posting site-level activity to a given GL account.

4.2 If any line's (GLAccount, SiteCode) combination has no configured sub-account, the **entire import is rejected** at the validator gate (DE never reaches Commit). Per the atomicity rule (§7 of this story), partial writes are not allowed.

4.3 The Central Office is treated as just another site for this check. A row with the Central Office SiteCode requires a configured sub-account on the targeted GL account, the same as any other site. BL does not apply special handling based on whether the SiteCode happens to be the Central Office.

4.4 On rejection, the import result record must surface the specific offending row(s) — the GLAccount, SiteCode, and a clear message indicating that no sub-account exists for that combination. The user resolves by either correcting the file (changing the SiteCode or GLAccount on the offending rows) or configuring the missing sub-account in Configuration > General Ledger and re-uploading.

4.5 Sub-account validation runs in the BL Validate call (and again at Commit for defense-in-depth). The data exchange layer (**NXT-72160**) does not pre-validate sub-account configuration — DE only validates that the SiteCode itself exists as configured in SchoolCafe. Sub-account routing is a BL concern.

### 5) Unknown GLAccount Handling

5.1 Per API Validations §2.3, unknown GLAccounts are non-blocking at validation. On Commit, BL writes the line to the ledger (payroll feeds legitimately reference newly-introduced accounts that need review but should not halt month-end processing).

5.2 When writing a line for an unknown GLAccount, BL must persist the warning indicator on the account record such that it appears in Configuration > General Ledger > GL Accounts with a clear visual flag for finance review.

5.3 Once a user reviews the flagged account and confirms or updates it in Configuration, the warning indicator is cleared. This review workflow is not part of this story — only the persistence of the warning flag on write is.

5.4 An unknown GLAccount that is permitted under §5.1 still requires a configured sub-account for the (GLAccount, SiteCode) combination per §4. If the account is unknown AND no sub-account exists for the (GLAccount, SiteCode) combination, the import is rejected per §4. The unknown-account warning path applies only when sub-account validation passes.

### 6) Entry Behavior

6.1 Payroll Import journal entries are **read-only** in the UI. Lines cannot be edited or individually deleted.

6.2 If a user needs to correct posted payroll entries, they create a manual reversal or adjustment entry. The original Payroll Import entry remains intact for audit trail.

6.3 The "Payroll Import" entry type must be identifiable in the ledger so that reports, filters, and audit views can distinguish payroll imports from manual journal entries, opening balance entries, and other automated entry types.

### 7) Atomic Ledger Write

7.1 The entire journal entry (header + all lines, across all targeted periods) must be written **atomically**. Either all lines post successfully, or no lines post.

7.2 If any system-level error occurs during the ledger write, the entire entry must roll back. No partial write is permitted, including no partial write of just the open-period lines or just the lines for one of multiple targeted periods.

7.3 On rollback, the data exchange layer must be notified so the import result record reflects the failure and the user is informed.

### 8) Downstream Report Behavior

8.1 Payroll Import journal entry lines contribute to **period activity** on all standard reports: Trial Balance, Balance Sheet, P&L (Revenue & Expenditure), and any period-based account activity views. Lines contribute to the period each line's PostingDate assigns it to.

8.2 Each line posts to its own PostingDate (per §1.3), so a line dated mid-period and a line dated end-of-period both contribute to the same Period's activity but may appear on different daily activity views. Lines from a single entry that span multiple periods contribute to each respective period's activity.

8.3 Payroll Import lines are filterable and groupable by the "Payroll Import" entry type in reports that support entry type filtering.

8.4 Payroll Import lines do **not** establish or modify opening balances. Opening balances are set only via the Opening Balance Import flow (**NXT-71240** / **NXT-71249**).

## Acceptance Criteria
**AC1 — Journal Entry Created with Correct Structure** Given the data exchange layer passes validated import data after user confirmation, When the business logic layer processes the data, Then one journal entry is created with entry type "Payroll Import" containing one line per imported row, with each line carrying its own derived (Fiscal Year, Period) from PostingDate, and the entry header carrying the set of all targeted (FY, Period) combinations.

**AC2 — PostingDate Stored Per Line** Given an import contains rows with different PostingDate values, When the journal entry is written, Then each line carries its own PostingDate from the file — not a single header-level date. Lines from one entry may legitimately span multiple periods.

**AC3 — Debit/Credit Amounts Mapped Directly** Given the validated import data carries separate Debit and Credit columns (single canonical contract; signed-amount layouts are transformed by DE upstream of mapping per NXT-72160 §2), When the journal entry lines are created, Then each line carries the debit or credit value from the row and zero on the opposite side.

**AC4 — Normal Balance Not Enforced by Base Type** Given an import contains a line with a credit to an asset or a debit to a liability (abnormal balance), When the business logic layer processes the import, Then the line is written as-is — the system does not reject or modify the balance direction based on account type.

**AC5 — Closed-Period Rejected (All-or-Nothing)** Given any row's derived Period has been closed by the time BL runs (Validate or Commit), When BL processes the import, Then BL rejects the entire import — partial write of the open-period rows is not permitted — with an error identifying the affected closed period(s) and the row count for each, directing the user to either correct the file (remove or re-date the offending rows) or post a prior-period adjustment via manual journal entry. _(Defense-in-depth — DE enforces the same rule upstream at the validator gate and at confirm.)_

**AC6 — Duplicate Rejected Without DE Acknowledgment (per Targeted Period)** Given a Payroll Import journal entry already exists for any (FY, Period) the file targets and the DE duplicate-acknowledgment flag is absent or does not cover that period, When BL receives the new import, Then BL rejects with an error listing the affected period(s) and directing the user to acknowledge through DE or reverse the prior entry. No journal entry is written.

**AC7 — Duplicate Allowed With DE Acknowledgment** Given a Payroll Import journal entry already exists for one or more targeted (FY, Period) combinations and the DE duplicate-acknowledgment flag covers each detected period, When BL receives the new import, Then BL writes the new entry alongside the prior entries — both remain in the ledger for audit trail.

**AC8 — Unknown GLAccount Writes with Warning Flag** Given an import contains a line with a GLAccount that does not exist in the chart of accounts AND a sub-account exists for the (GLAccount, SiteCode) combination, When BL processes the import, Then the line is written to the ledger and the account is persisted with a warning indicator such that it appears in Configuration > General Ledger > GL Accounts with a clear visual flag for finance review. BL does not block the import or the line. If the unknown account also has no sub-account configured for the (GLAccount, SiteCode), the import is rejected per AC15.

**AC9 — Atomic Ledger Write** Given a system error occurs partway through writing the journal entry header and lines, When the error is encountered, Then the entire entry (header + all lines, across all targeted periods) rolls back — no partial write remains, DE is notified, and the user is informed the import failed.

**AC10 — Entry Is Read-Only** Given a Payroll Import journal entry exists in the ledger, When a user attempts to edit or delete it through the UI, Then the system blocks the action — the entry can only be corrected via a manual reversal or adjustment entry.

**AC11 — Entry Type Identifiable for Reports and Filters** Given Payroll Import entries coexist with manual journal entries, opening balance entries, and other automated entry types, When a report or audit view filters by entry type, Then Payroll Import entries can be isolated or excluded using the "Payroll Import" type identifier.

**AC12 — Lines Contribute to Period Activity (Not Opening Balance)** Given Payroll Import lines have been written across one or more (FY, Period) combinations, When the user runs a Trial Balance, Balance Sheet, or P&L for any of those Periods, Then the lines appear as period activity for that Period. The Opening Balance column on these reports is not affected — opening balances come only from the Opening Balance Import flow (NXT-71240 / NXT-71249).

**AC13 — API Validate Operation Returns Pass or Fail** Given DE calls the BL Validate API with a mapped file, When BL runs the validation pass, Then BL returns either pass with total rows, total Debits, total Credits, per-period breakdown (FY, Period, row count, period Debits, period Credits), unknown GLAccount list (account codes flagged for finance review on Commit), and duplicate periods (when detected) — or fail with a structured error list (row number, field, error code, message) and an error report download URL. No partial work is performed.

**AC14 — Row-Level Structural Validation** Given the file contains rows with missing required fields (GLAccount, SiteCode, Description, PostingDate), an unparseable PostingDate, non-decimal Debit or Credit values, both Debit and Credit populated on a row, both blank, both zero, any negative value, or Comments longer than 50 characters, When BL runs validation, Then BL returns fail with the affected rows listed.

**AC15 — Sub-Account Required for Every (GLAccount, SiteCode)** Given any line's (GLAccount, SiteCode) combination has no configured sub-account, When BL runs validation, Then BL returns fail with the offending rows surfaced. Central Office is treated as just another site for this check (no special handling). This rule applies regardless of whether the GLAccount is known or unknown in the chart of accounts.

**AC16 — Period Derivation and Configured-Period Check** Given any row's PostingDate cannot be mapped to a configured (FY, Period) in the district's configured timezone (e.g., the FY does not exist in the system, the date is unparseable, or the date falls outside any configured Period), When BL runs validation, Then BL returns fail with the offending rows listed.

**AC17 — Balance Check with $0.01 Tolerance** Given the absolute difference between Sum of Debits and Sum of Credits across all rows exceeds $0.01, When BL runs validation, Then BL returns fail. The tolerance applies to the file as a whole, not per period — a payroll file is a single balanced unit even when its rows span multiple periods.

**AC18 — Commit-Only Period Status Re-check** Given Validate succeeded but a period closes between Validate and Commit, When BL runs Commit, Then BL re-derives (FY, Period) from each row's PostingDate, re-verifies period status, and rejects the entire import (all-or-nothing) if any targeted period has closed since Validate, returning the user to validation.

**AC19 — Authoritative Duplicate Check (per Targeted Period)** Given a prior Payroll Import journal entry exists for any (FY, Period) the file targets and the request payload does not include a duplicate-acknowledgment that covers each affected period, When BL runs Commit, Then BL rejects the request. This prevents the DE-side warning from being bypassed via URL manipulation, concurrent submissions, or client-side tampering.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| Any row's derived Period closed by the time BL runs (Validate or Commit) | BL rejects the entire import (all-or-nothing); no entry written; user directed to correct the file or post a prior-period adjustment via manual journal entry |
| Duplicate detected for one or more targeted (FY, Period) combinations, DE ack flag absent or does not cover detected period(s) | BL rejects as duplicate; error lists affected period(s); user directed to acknowledge through DE or reverse the prior entry |
| Duplicate detected for one or more targeted (FY, Period), DE ack flag covers each detected period | BL writes new entry alongside prior; both retained for audit |
| Line references unknown GLAccount; sub-account exists for (GLAccount, SiteCode) | Line written; warning indicator persisted on the account for Configuration review; no rollback |
| Line references unknown GLAccount; sub-account does NOT exist for (GLAccount, SiteCode) | Entire import rejected per sub-account validation (AC15); unknown-account warning path does not apply |
| Abnormal balance on a line (e.g., credit to an asset) | Accepted; BL does not enforce normal balance direction by base type |
| System error mid-write | Entire entry rolls back; no partial write; DE notified; user notified |
| Two concurrent imports targeting overlapping (FY, Period) combinations | Second import to reach BL Commit is rejected as a duplicate (per AC6 / AC19) |
| Row PostingDate outside any configured Period (e.g., FY not configured) | BL rejects entire import per AC16; no partial write |
| Validate-only call: any check fails | BL returns structured error list + error report download URL; no ledger write attempted |
| Commit: any check fails (including period status re-check) | BL returns structured error list; no ledger write performed |

# NXT-70777 — FIN — Import Monitor: Journal Entry Lines Import (Payroll Use Case)

## Description
As a district finance user, I want to import journal entry lines into Financials via Import Monitor, so that recurring payroll and other month-end journal entries can be loaded efficiently without manual data entry.

----

h2. Requirements

h3. 1) Outcome

1.1 A district finance user can import journal entry lines into Financials via Import Monitor.

1.2 After a one-time template setup, recurring monthly imports require only upload → select fiscal year and period → preview → import.

1.3 The import flow supports month-end reconciliation by providing totals, errors, and import results, reducing risk of duplicate or missed entries.

h3. 2) Import Type

2.1 A new Import Monitor Import Type exists named *Payroll Import* with help text: _"Used for payroll and other month-end journal imports in Financials."_

2.2 Import Type supports the same upload formats as other Import Monitor types (XLSX/CSV per current platform behavior).

2.3 Import Type is available only to users with appropriate Financials permissions (same pattern as GL Accounts import access).

h3. 3) Import Template (one-time setup)

3.1 On the first import for this import type, the user must configure an *Import Template* consisting of:

* *Amount Format* — one of:
** _"My file has separate Debit and Credit columns"_ → Scheme A
** _"My file has a single Amount column (positive = debit, negative = credit)"_ → Scheme B
* *Column Mapping* — mapping of uploaded file columns to target fields (see Sections 5 and 6).

3.2 The Amount Format selection determines which amount target fields appear in the column mapping UI. Scheme A displays Debit and Credit as required target fields. Scheme B displays Amount as the required target field. The non-selected fields must not appear in the mapping UI.

3.3 The template is saved after the first successful configuration. The user can edit the template (amount format and column mapping) at any time.

3.4 Only one template exists per import type. Configuring a new template replaces the existing one.

h3. 4) Fiscal Year and Period Selection

4.1 After file upload, the user must select:

* *Fiscal Year* — must be a valid fiscal year in the system.
* *Period* — displayed as the *month name* corresponding to each period relative to the fiscal year start (e.g., if the fiscal year runs July 1 – June 30, the dropdown shows "July" through "June," not 1–12).

4.2 All rows in the imported file are assigned to the selected Fiscal Year and Period. These values are not mapped from the file.

4.3 The selected Period must be in an *open* status. If the period is closed, the system must block the import at this step with a message: _"[Month] for Fiscal Year [YYYY] is closed. Please open the period before importing."_

4.4 Fiscal Year and Period are selected fresh for each import and are *not* part of the saved template.

h3. 5) Import Flow

5.1 *First import (no template exists):* Upload → Amount Format Selection → Column Mapping → Save Template → Select Fiscal Year/Period → Preview → Confirm.

5.2 *Subsequent imports (template exists):* Upload → System displays a reminder of the saved amount format (e.g., _"Your template uses separate Debit and Credit columns."_) → Select Fiscal Year/Period → Preview → Confirm.

5.3 On subsequent imports, the saved column mapping and amount format are automatically applied. The user does not re-map columns or re-select the amount format unless they choose to edit the template.

h3. 6) Required Target Fields

The import mapping UI (during template setup or editing) must allow mapping uploaded columns to these target fields:

||Field||Notes||
|GLAccount|Account identifier per Financials rules|
|SiteCode|Site the entry applies to. Must reference a valid site in Site Management. For district-wide entries, use the site code designated as the district-level site in Site Management.|
|Description|Free text description for the journal entry line|
|PostingDate|Date the entry should post to the ledger. Must fall within the date range of the selected Period.|

*Amount fields (determined by the Amount Format in the template):*

||Amount Format||Target Fields Shown||Convention||
|Separate Debit and Credit columns|Debit, Credit|Separate positive values in each column|
|Single Amount column|Amount|Positive = debit, negative = credit|

h3. 7) Optional Target Fields

||Field||Notes||
|AccountName|Descriptive label for the account|
|Comments|External reference identifier; free text; max 50 characters|

h3. 8) Validation Rules (row-level)

8.1 GLAccount, SiteCode, Description, and PostingDate must be present for each row. Amount fields must be present per the template's amount format.

8.2 Amount validation must follow the template's amount format:

*If Scheme A (Debit/Credit):*

* Exactly one of Debit or Credit must be a positive number per row.
* Negative values are not allowed.

*If Scheme B (Amount):*

* Amount must be non-zero.
* Negative amounts are allowed (treated as credits by convention).

8.3 Zero values are not allowed in either scheme and will be flagged as invalid.

8.4 PostingDate must fall within the start and end dates of the selected Period for the selected Fiscal Year. Rows where PostingDate falls outside the period date range are flagged invalid with the error: _"Posting date does not fall within the selected period."_

8.5 GLAccount should be a valid account identifier per Financials rules. If the account is not recognized, the row is imported with a warning, and the account displays with a warning sign under GL Accounts in Configuration > General Ledger. This is not a blocking error.

8.6 SiteCode must reference a valid site in Site Management. Unknown site codes are flagged as invalid.

h3. 9) Validation Rules (import-level)

9.1 The system must calculate and display:

* Total rows
* Total debits and total credits (computed from the template's amount format)

9.2 Import must be blocked if the absolute difference between total debits and total credits exceeds *$0.01*. A difference of $0.01 or less is allowed to accommodate rounding across line items. Totals and any imbalance amount must be displayed to the user alongside the blocking message.

9.3 The import must be *atomic* — either all rows are imported successfully, or no rows are imported. Partial imports are not allowed. If any system-level error occurs during import processing, the entire import must roll back and the user must be notified that no data was written.

h3. 10) Preview and Confirm

10.1 The preview must display the selected Fiscal Year and Period, the amount format from the template, totals, and a sample of parsed rows.

10.2 The user must explicitly confirm to perform the import.

10.3 If any rows contain validation errors (excluding GLAccount warnings per 8.5), the confirm action must be disabled. All errors must be resolved by correcting and re-uploading the file before the import can proceed. The system must not allow the user to import only the valid rows.

h3. 11) Import Results and Auditability

11.1 Each import must produce an import result record consistent with Import Monitor patterns (status, counts, timestamps, user).

11.2 The user can view import results including:

* Success/failed status
* Selected Fiscal Year and Period
* Total debits and credits and record counts
* Row-level error messages for invalid records

h3. 12) Duplicate Risk (business requirement)

12.1 The system must detect and warn the user when an import appears to be a duplicate of a previous import for the same Fiscal Year and Period. The warning must occur before the user confirms the import.

12.2 If a potential duplicate is detected, the user must explicitly acknowledge the warning before the import can proceed. The system must not silently allow duplicate imports.

12.3 The specific detection mechanism is a technical design decision. The business requirement is: if a user accidentally uploads the same payroll file twice for the same period, the system must catch it and intervene before the second import writes to the ledger.

## Acceptance Criteria
*AC1 — Import Type Available* Given an authorized Finance user opens Import Monitor, When they view Import Types, Then Payroll Import is available.

*AC2 — First Import Requires Template Setup* Given no import template exists for Payroll Import, When the user uploads a file, Then they must select an amount format and configure column mapping before proceeding to Fiscal Year/Period selection.

*AC3 — Subsequent Imports Skip Template Setup* Given an import template already exists for Payroll Import, When the user uploads a file, Then the saved column mapping and amount format are automatically applied, the system displays a reminder of the amount format, and the user proceeds directly to Fiscal Year/Period selection.

*AC4 — Template Editable* Given a saved import template exists, When the user chooses to edit the template, Then they can modify the amount format and column mapping, and the updated template is used for future imports.

*AC5 — Period Displayed as Month Names* Given the fiscal year runs July 1 – June 30, When the user selects a Period, Then the dropdown displays "July" through "June" — not numeric values 1–12.

*AC6 — Fiscal Year and Period Selection Required* Given a user uploads a file for Payroll Import, When they proceed past file upload (and template reminder, if template exists), Then they must select a Fiscal Year and Period before reaching preview.

*AC7 — Closed Period Blocks Import* Given the user selects a Period that is in a closed status, When they attempt to proceed, Then the system blocks with a message identifying the closed period and does not allow the import to continue.

*AC8 — Required Fields Enforced* Given an upload contains rows missing GLAccount, SiteCode, Description, PostingDate, or the mapped amount field(s), When preview runs, Then those rows are flagged invalid and cannot be imported until corrected.

*AC9 — PostingDate Must Fall Within Selected Period* Given the selected Period spans October 1–31 and a row has PostingDate = July 15, When preview runs, Then that row is flagged invalid with an error indicating the posting date does not fall within the selected period.

*AC10 — Debit/Credit Rule Works (Scheme A)* Given Scheme A is used and a row has both Debit and Credit populated with > 0, When preview runs, Then the row is invalid with an error indicating only one side is allowed.

*AC11 — Signed Amount Rule Works (Scheme B)* Given Scheme B is used and a row has Amount = 0, When preview runs, Then the row is invalid with an error indicating zero is not allowed.

*AC12 — Batch Must Balance (within $0.01)* Given the absolute difference between total debit-side amounts and total credit-side amounts exceeds $0.01, When the user attempts import, Then import is blocked and totals are displayed with the imbalance amount.

*AC13 — Row Errors Block Entire Import* Given an upload contains one or more rows with validation errors (excluding GLAccount warnings), When the user views the preview, Then the confirm action is disabled and the system must not allow importing only the valid rows. All errors must be resolved by correcting and re-uploading the file.

*AC14 — Import Is Atomic* Given the user confirms an import and a system-level error occurs during processing, When the error is encountered, Then the entire import rolls back, no journal entry data is written, and the user is notified that the import failed.

*AC15 — Unknown GLAccount Imports with Warning* Given an upload contains a GLAccount that is not recognized in the chart of accounts, When the import completes, Then the row is imported successfully and the account displays with a warning sign under GL Accounts in Configuration > General Ledger.

*AC16 — SiteCode Validated Against Site Management* Given an upload contains a SiteCode that does not exist in Site Management, When preview runs, Then that row is flagged invalid with a clear error consistent with Import Monitor patterns.

*AC17 — Results Visible* Given an import completes, When the user opens import results, Then they can see status, the selected Fiscal Year and Period, record counts, totals, and row-level errors if any.

*AC18 — Duplicate Import Detected and Warned* Given a payroll import has already been completed for a Fiscal Year and Period, When the user attempts to import another file for the same Fiscal Year and Period, Then the system warns the user that a previous import exists for that period and requires explicit acknowledgment before proceeding.

h2. Edge / Empty / Error States

||Scenario||Behavior||
|No template configured and user attempts import|User is guided through template setup (amount format + column mapping) before proceeding|
|Empty file or header-only|Import blocked: _"No data rows found."_|
|Selected Period is closed|Import blocked at Fiscal Year/Period selection step with message identifying the closed period|
|Non-numeric amount fields|Row invalid with parsing error|
|Both Debit and Credit populated in same row (Scheme A)|Row invalid: _"Exactly one of Debit or Credit must have a value."_|
|Negative value in Debit or Credit column (Scheme A)|Row invalid: _"Debit and Credit values must be positive."_|
|PostingDate outside selected Period date range|Row invalid: _"Posting date does not fall within the selected period."_|
|Unknown GLAccount|Row imports with warning; account flagged in Configuration > General Ledger|
|Unknown SiteCode|Row invalid with clear error consistent with Import Monitor patterns|
|File contains mix of valid and invalid rows|Confirm action disabled; all errors must be resolved before import. No partial import allowed.|
|System error during import processing|Entire import rolls back; no data written; user notified of failure|
|Import attempted for a FY/Period that already has a completed import|System warns user of potential duplicate; requires explicit acknowledgment before proceeding|
|Uploaded file columns don't match saved template mapping|Preview flags unmapped/missing columns; user must fix file or edit template|

----

h2. Defaults

* *Fiscal Year and Period:* Selected at the import level after file upload. Period is displayed as the month name corresponding to the fiscal year calendar. These values are selected fresh for each import and are not part of the saved template.
* *Amount format reminder:* On subsequent imports, the system displays the saved amount format as a reminder (e.g., _"Your template uses separate Debit and Credit columns."_). The user does not re-select unless editing the template.
* For Scheme A, the unmapped side (Debit or Credit) is treated as 0 for totals.
* For district-wide entries, the user should use the site code designated as the district-level site in their Site Management configuration. The import UI guidance text should read: _"For district-wide entries, use your district-level site code as configured in Site Management."_
* Comments field: max 50 characters.
* Optional fields default to blank/null when not mapped.

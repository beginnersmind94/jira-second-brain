# NXT-71241 — Insights - KPI - MPLH - Configuration (SC)

## Description
*I am a* district user
*that I want to* configure benchmark values for Meals Per Labor Hour (MPLH) in Insights using SchoolCafé data
*so that I* can define target performance thresholds used in KPI calculations and comparisons.

----

h2. *Notes*

* This story applies to *Insights – SchoolCafé data source*.
* MPLH (Meals Per Labor Hour) is part of the *front office KPIs* being integrated into Insights.
* This story is a *companion to the PrimeroEdge MPLH Benchmark Configuration story*.
* The *UI, behavior, validation rules, and data persistence are identical* to the PrimeroEdge implementation.
* This implementation must *reuse the same UI components, API contracts, validation logic, and database tables*.

h3. *Data & Source of Truth*

* The *source of truth for MPLH data* is available in the *Front Office Accountability Module → Labor Hours page*.
* Users can click the *“MPLH Report”* button to view MPLH values:
** Per site
** Per period
* This must be used by *developers and QA to validate MPLH calculations and data accuracy*.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure for MPLH:
** SchoolCafé: [https://primeroedgenext.atlassian.net/browse/NXT-67454|https://primeroedgenext.atlassian.net/browse/NXT-67454|smart-link] 
* The system must *use the SchoolCafé stored procedure as the data source instead of PrimeroEdge*.

h3. *KPI Definition & Formatting*

* *MPLH is a rate-based KPI*.
* Format must be: *0.00 (two decimal places)*.
* Values must be *rounded to two decimal places*.
* Calculation:
** *MPLH = Equivalent Meals ÷ Actual Labor Hours*

----

h2. *Out of Scope*

* PrimeroEdge implementation (covered in separate story)
* Paid Not Applied (P&A) KPI
* Eligible Not Participating (EMP) KPI
* Any non-MPLH KPI configuration
* Changes to stored procedure logic (integration only)

----

h2. *User Flow*

* Same as *PrimeroEdge MPLH Benchmark Configuration story*

----

h2. *Requirements*

h3. *1. Shared Implementation Requirement*

* This implementation must:
** Reuse the *same UI components*
** Reuse the *same grid, drawer, and interaction patterns*
** Reuse the *same validation logic*
** Reuse the *same API structure and contracts*
** Reuse the *same database tables and persistence logic*
* No duplicate UI or logic should be created for SchoolCafé.

----

h3. *2. Data Source Swap (Primary Requirement)*

* The only functional difference from the PrimeroEdge implementation is:
** MPLH data must be sourced from the *SchoolCafé stored procedure* instead of PrimeroEdge
* This applies to:
** Autofill values
** Actual MPLH values displayed
** Any backend calculations

----

h3. *3. Consistency Requirement*

* All behavior must remain consistent with PrimeroEdge:
** Manual entry
** Autofill
** Bulk update
** View history
** Save behavior
** Validation rules
** Performance expectations

## Acceptance Criteria
h3. *1. Shared Implementation*

* SchoolCafé benchmark configuration uses the same UI and behavior as PrimeroEdge.
* No duplicate UI components are introduced.

----

h3. *2. Data Source Integration*

* System uses the SchoolCafé MPLH stored procedure.

h3. *4. Data Accuracy*

* MPLH values match:

** Front Office MPLH report

h3. *5. Performance*

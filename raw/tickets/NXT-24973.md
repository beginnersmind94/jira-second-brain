# NXT-24973 — Accountability - Transactions - Exceptions - Eligibility Variance - One-click resolution for exceptions with best outcome logic

## Description
As Sophie the Site Manager I am very busy and almost always make the same decision for serving exceptions. I want the system to make these decisions for me so I don’t have to.

*mockup*

[https://www.figma.com/proto/m3QGmo7Fjys7um36XDhwAY/Accountability-5.0?node-id=193%3A8761&scaling=scale-down-width&page-id=174%3A5157&starting-point-node-id=174%3A5158|https://www.figma.com/proto/m3QGmo7Fjys7um36XDhwAY/Accountability-5.0?node-id=193%3A8761&scaling=scale-down-width&page-id=174%3A5157&starting-point-node-id=174%3A5158|smart-link] 

*requirements*

* User has a ‘Resolve All' option on the Exceptions page for Eligibility Variances that will ask them how they want the system to resolve all variant meals.
* User can specify how this option will work for making a default selection for how to handle meals with these two sub-scenarios:
** *Eligibility Variances*
*** Options: Update to Current Eligibility, Keep as Serving Eligibility
* Use the following logic:
** Eligibility Variance
*** Use the selected option from the overlay.
* User can click the button, make their choices, and hit Save.
** User needs a confirmation prompt before these are added to a queue.
** If they confirm, add the decisions to a queue and begin processing in the background.
* Normal processing should also be added to the processing queue.
** Only 1 decision per transaction should be processed - update to the latest decision if someone else comes in and processes the same exception.
* User needs to be sent a message when the processing is complete.

## Acceptance Criteria
User has an option to bulk adjust the results in the Exceptions grid for Eligibility Variances.

User can select how they’d like the system to handle the exceptions. Confirmation is required by the user.

System will process the exceptions for the user.

* Any adjustments should be processed quickly.
* No potential for duplicate transactions as a result of 2 users initiating the task at the same time.

Normal processing on the page is also handled by the background process.

User is sent a message when the processing is complete.

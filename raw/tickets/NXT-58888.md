# NXT-58888 — FOH Dash - Claim Readiness - Site Readiness

## Description
As Dana the Director I want to see what sites are ready to go, and what sites need focused attention to be ready for claim generation.

*mockup*

!image-20250119-201159.png|width=1100,height=737,alt="image-20250119-201159.png"!

*requirements*

* User can see included Areas (see the Configuration story) and then data tied to what we want to monitor for each area (see below)
* User has a grid showing sites that are in the selected program for the selected timeframe / pos period. (check Program Configuration)
** Client-side filtering on all columns
** Pagination - default, 10 sites
** Filter to exclude ‘Ready’ sites
* User has links out to ‘Resolve’ each area → nav the user (new tab) to the respective page (see below) 
* -User can ‘Refresh’ the stats-
** -Same as the End of Day refresh button.-
** -User can see the last time data was refreshed for the selected Period (nice to have: for their region as well)-
** -User can see the current status of the background job-
** -User is given a message that the refresh may take a little while: Data will be refreshed shortly. This may take a few minutes.-
** -User is informed when the data has been refreshed: Data refresh completed.-
*** -Can we update the grid dynamically without a page load?-
*** -If we can’t we need to inform the user that they need to run their search again or press Apply for them.-
**** -Text: You will need to run your search again.-
** Refresh button is not necessary.



Links/page to take the user to for the Resolve actions:

* Sessions: Number of Sessions that are not in Reconciled/Deposited.
** Link takes them to the Sessions page. Load the Date ranges and session Status should not be in Reconciled/Deposited.
* Edit Checks: Number of unresolved Edit Checks per site.
** Link takes them to the Edit Checks page. Load the date range.
* Reverse Edit Checks: Number of low meal count instances per site.
** Link takes them to the Edit Checks with Reverse Edit Check enabled. Load the date range.
* Serving Exceptions: Number of duplicate meals & eligibility variances.
** Link takes them to the Transactions > Serving Exceptions page. Load the date range.
* Review Reasons: Number of transactions marked for review.
** Link takes them to the Transactions page with Review Reasons enabled. Load the date range.
* Applications: Number of unprocessed applications.
** Link takes them to the Applications page.

## Acceptance Criteria
* User can see the Site Readiness grid
** Data from search should load
** Client-side filtering
* User can only see included Areas
** for Serving Exceptions, show the number of duplicates and variances separately
** example: 0 | 6
* User sees stats for each site
* User can see Status for each site
** 0s across the board for each Area means the Status = Ready
** Othewise, Status = Not Ready
* User can filter to sites that are Status = Not Ready
* User has links to Resolve each section - new tab, page details in description

# NXT-48116 — Subscriptions - Backend Framework

## Description
As a SchoolCafe user, I would like to be able to be notified of certain events so that I can know that I need to take a given action.

*Notes:*

* This is the backend framework story to create the underlying features to be used by other modules
* The objective is to create the ability for system or teams to:
** Maintain a list of available events users can subscribe to (list of available events)
** Register an event subscription (save configuration, names, criteria, users, etc.)
** -Store the messages to send-
** Allow teams to check if there is a subscription matching their event
** Allow teams to notify the framework that an event needs to be distributed
** Distribute the message per the event to the registered users
** Record the execution of the event distribution (history, audit)
** Provide a list of subscriptions per user (including if in role)
** Provide a list of users per subscription

*Requirements:*

Events List

* Create a table to maintain a list of events that users can subscribe to
* Teams will add (register) an event subscription to this list in the backend (there is no front end to this)
* The table needs the following columns at the minimum
** User Friendly event subscription name
*** Ex: Student File Import
*** Ensure can have spaces in the name
*** This field will be seen by users as a Subscription Type
** Description
*** Optional short description of the event subscription
** Module
** Section (Event)
*** Corresponds to the Section list in User Profile
*** Values: Sites & Users, Management, Configuration, Integration and more later
** Criteria
*** This is a list
*** Most will use Additions, Changes, Deletions, but it can be anything
** Usual ID, date created, etc. columns
* Create and API end point to show this list in the UI on the Subscriptions page



Subscriptions List

* Create a table for modules to record that a user has created a subscription to the event
** Note: the Event list above is just a list of events and this list is where the user is registering they want to subscribe to an event
** Data will be inserted into this list by modules from the front end
** Whoever wants to use events will have to write a front end interface that allows them to push subscriptions to this list
** At a minimum, this list needs the following:
*** ID
*** Event ID (other table id)
*** Users List
*** Roles list
*** criteria
**** JSON? XML?
* Create an endpoint for the modules to get the joined event and subscription data by event id 
** To be used by the module code to see if there are subscription events for their given activity
* Create an endpoint to get a list of all the joined events and subscriptions 
** To be used by the dashboard grid
* Create an endpoint to get the joined event and subscription data by user 
** To be used by the my subscriptions
** Include the ones in which a user is in a role linked to the subscription
* Create an endpoint to get a list of all the users for a subscription 
** To be used by the event page and used by the distribution framework



Subscription Distributions

* Be able to log when a subscription is distributed
** Try to leverage existing framework
* Be able to produce a list of what event subscriptions where distributed to whom and when
** This will be used by the monitoring page



Subscription History Log

* Be able to record the audit / history log for any add/edit/delete operations per subscription 
** This will be used by the dashboard page

## Acceptance Criteria
* Devs rom other modules can register their subscriptions
* Runtime code can add, edit and delete subscriptions
* Framework allows to define which users to send subscriptions to and what criteria

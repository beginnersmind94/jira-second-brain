# NXT-67061 — Ignore, Old (SC - MT - Central - Link Users to Districts)

## Description
As a Customer Support role user, I want to be able to view data related to a Central realm so that I can manage user access.

*Notes*

* This story covers managing user access to an Org Level and assumes the Central Screen story has been completed
* A Realm is a generic term for District, state, area, etc.
* Note: 
** Users for a central realm can be managed in the Users screen like normal
** This story manages giving a central realm user access to children realms

*User Flow*

* District is created
* User is added via users screen
* User is assigned to roles of the Central realm that have the permissions they want for children realms access
** In other words: if Central realm admin wants a user to be able to view Back Office Reports, they setup a role to have access to Back Office Reports and adds the user to it
** When the user logs into any child realm, they will have these same permissions
* In the Central page, the user is assigned to a child realm
* 
* Later:
* The user logs in and browses to Central
* The user clicks the View button on the district they are given rights to
* System redirects the user to that district
* User can see pages they were granted access to via the Central roles and permissions

*Requirements*

* On the System > Central page, do the following:
* Add Users tab to the screen if not already there
* 
* Filters 
** Allow the user to type in a name of a child realm and filter the users grid to only those with access to that child realm
* 
* Add a button:
** Label: + New
** Permissions: User must have page add permission, else it is disabled
** On click:
*** Allow the user to select a user in the current realm and add it to the grid
*** Cannot add a user that is already added
* Add a grid to show the users of the current realm
** Data: All users that have been added
** Grid should have paging, sorting and filtering
** Columns:
*** Name 
**** name of user
*** Realms
**** Count of children realms the user has been added to
**** On hover over, show the full list of realms the user has access to
*** Actions
* Actions column:
* Edit button
** On click:
*** Validate user has edit rights to the screen
**** If not, show message: You do not have permission
*** Allow the user to add or remove children realms from their list
**** Preferably do this via inline editing on the row
* Delete button
** On each row in the grid, allow the user to remove a user from the list
** Use the usual trash can icon
** Permissions: User must have page delete permission, else it is disabled
** On click:
*** Show confirmation button
*** If Yes, unlink the user from the child realms and remove from the grid
*** Show a green success message if successful
* 
* If there are no users in the grid:
** Show a message: No users added.
* 
* Ensure that when adding or removing a District to the user, that:
** Districts list in Add/Edit/View User and Profile are updated to show all the districts correctly
** Sites list in the same screen are updated to show all the sites correctly for all the correct districts

## Acceptance Criteria
h2. Tabs

* Ensure *Users tab* exists on the System > Central page.
* If it does not exist, create the tab.
* The tab is accessible to users with Central page permissions.

----

h2. Filters

* Add a text filter above the users grid:
** Allows typing the name of a child realm.
** Filters the grid to only display users who have access to that child realm.
** Filter operates dynamically and clears properly when the input is removed.

----

h2. + New Button

* A button labeled *+ New* appears above the grid.
* *Permissions:*
** Enabled only if the user has *Add* permission for the page.
** Disabled otherwise.
* *On Click:*
** Opens a selector to choose a user from the current realm.
** User cannot add a duplicate (already added) user.
** On confirmation, the user is added to the grid.
** Show a *green success message* after successful addition.

----

h2. Grid

* Displays all users added to the current realm.
* Supports *paging*, *sorting*, and *filtering*.
* *Columns:*
*# *Name* – Displays user’s full name.
*# *Realms* –
*#* Displays the count of child realms the user has been granted access to.
*#* On hover, shows a tooltip listing all realm names.
*# *Actions* – Contains buttons for editing or removing a user.

----

h2. Actions Column

h3. Edit Button

* *Visible:* For all rows.
* *Enabled:* Only if the user has *Edit* permission.
* *Behavior:*
** On click, validate permissions.
** If no permission, show message: “You do not have permission.”
** If permitted, allow inline editing to *add or remove child realms* assigned to the user.
** Save updates immediately after inline edit confirmation.
** Show success message upon successful update.

h3. Delete Button

* *Visible:* For all rows.
* *Enabled:* Only if the user has *Delete* permission for the page.
* *Behavior:*
** On click, show confirmation prompt.
** If confirmed:
*** Unlink the user from all child realms they were assigned.
*** Remove the user from the grid.
*** Display a *green success message* confirming the removal.
** If canceled: No action taken.

----

h2. Empty State

* If there are *no users in the grid*, display message:
** “No users added.”

----

h2. Data Integrity & Relationships

* Ensure users added are unique (cannot add duplicates).
* When unlinking or deleting, remove relationships cleanly between user and child realms.
* Ensure child realm access inheritance is maintained correctly through Central roles and permissions.

----

h2. Performance & UX

* Grid and inline editing respond within standard performance thresholds (<3 seconds).
* Filtering and hover tooltips perform smoothly.
* All success and error messages are styled consistently with the system’s notification standards.

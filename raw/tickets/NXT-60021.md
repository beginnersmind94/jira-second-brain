# NXT-60021 — TH - WS - Custom Reports Screen

## Description
*As a* TeamHub user *I want* a screen that provides quick access to my Custom Reports (CR) starred and distributed features *So that* I can stay informed about my current operations.

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the custom reports card on the home workspace page in SC
* This story covers both the starred and distributed tabs
** CR Report viewer will be another screen
* Note: APIs should exist for this already

h2. User Flow

* User opens the application and is directed to login screen
* On login success, user is redirected to home screen
* User clicks Workspace and is redirected there
* User clicks on Custom Reports (CR) and is redirected there
* User is able to view a list of starred CRs
* User is able to click on a CR and view the report with current data
* User is able to view a list of recently distributed CRs
* User is able to download a CR
* User is able to delete a distributed CR

## Acceptance Criteria
h4. *Theme Support*

* The screen supports dark mode, with all colors and text adjusted for optimal readability in dark mode.

h4. *Header Section*

* A back button is provided that navigates the user to the Workspace screen.
* The page title is displayed as "Custom Reports."

h4. *Filters Section*

* A search textbox is present that filters custom reports (CRs) based on title and description.
* Two filter chips are available:
** Starred chip: Acts as a tab. When tapped, it displays only the starred custom reports.
** Distributed chip: Acts as a tab. When tapped, it displays only the custom reports distributed to the user.
* The Distributed chip shows the count of unread custom reports in a similar manner as for Inbox messages.
* A sync section displays the last sync time and includes a Sync button.
** Tapping the Sync button triggers the asynchronous sync job to update custom reports from the server (sync logic is handled in another story).

h4. *Starred Section*

* Displays a list of custom reports that are starred by the user.
* Each custom report is presented in a card format showing:
** An icon
** The title
** The module name
** Last viewed timestamp
** A View button which, when tapped, navigates the user to view the custom report (view functionality is handled in another story).

h4. *Distributed Section*

* Displays a list of custom reports that have been distributed to the user.
* Each card in the distributed section shows:
** An icon
** The title
** The module name
** The distributed date
** A Download button that, when tapped, opens the document view (files are pre-downloaded via sync).
** The file size is also displayed.
* When the Download button is tapped:
** The action behaves as a document view rather than a file download.
* A Delete option is available:
** When tapped, the delete action is saved to the local sync queue.
** The sync job is triggered immediately to initiate deletion on the server.
** The deleted custom report is removed from the displayed list.

h4. *Data Requirements*

* The screen retrieves all data from the device's asynchronous local storage; no direct API calls are made.
* Changes such as Delete and Read/Unread status updates are written to the local storage.
* A separate process (another story) handles the synchronization of custom reports with the server, so the screen remains agnostic to sync operations.

h4. *Performance & Usability*

* The screen must load within 2 seconds under normal operating conditions.
* The layout must be responsive and render correctly on Mobile, Tablet, Desktop, as well as in Compact and Expanded views.
* All tap interactions and other user actions must provide clear visual feedback and be highly responsive.

h4. *Error Handling*

* All asynchronous operations use try/catch blocks for error management.
* Error boundaries are implemented to catch any unexpected render failures.
* Errors are communicated to the user via user-friendly toast notifications, modal dialogs, or inline messages without disrupting the user experience.
* All API or rendering errors are logged to the error log system/database for further analysis.

h4. *Security*

* The user must have a Custom Reports district license; if the license is missing, the Custom Reports screen should not appear in the menu.
* The user must possess the "View Custom Reports" permission as defined by their role.
* If the user lacks the necessary permissions:
** The screen displays an error message: "You do not have permissions to view custom reports. Contact your administrator if you want to be granted permission."
** The user is not allowed to interact with the Custom Reports screen.

## Description
*As a*: TeamHub user *I want to*: reset my forgotten password *So that*: I can regain access to my account

*Comments*:

* Screen: Forgot Password screen ({{app/(auth)/forgot-password.tsx}})
* Handles password reset flow dependent upon platform and district

*User Flow*:

* User opens the application and is directed to login screen
* User clicks on Forgot Password link and is redirected to this screen
* System checks the state tool to see if use has saved District and Password
** If not, app tells user to select their platform and district first and gives them link to that screen
* User enters email address
* System checks if it has an email address for that user
* If yes:
* System sends the user a verification code
* App shows the user the verification code screen
* User enters verification code
* System gets the password rules for the district on that platform
* User sets new password and system validates using the rules
* If good, System sends the new password to the server to be saved
* User is redirected to login

## Acceptance Criteria
h4. *General Flow*

* On screen load, the application retrieves Platform, District, and Email values from the asynchronous state management datastore.
* If the Platform and District values are missing, the Forgot Password screen:
** *Does not* show the email input.
** Displays a message: _"First select your district on the district configuration screen."_
** Provides a button that navigates the user to the District Configuration screen.
* If the Platform and District values are present, the screen displays the Forgot Password functionality as described in the following views.

----

* Ensure the screen supports dark mode from the app settings

h4. *View Logic*

* *Only one view is shown at any time.*
* *View 1: Initial Prompt (Fallback)*
** *Message:* _"Enter your email address and we’ll send you a code to reset your password."_
* *View 2: Email Entry and Code Generation*
** *Email Field:*
*** Displayed as a textbox.
*** Pre-populated with the user’s saved email address from the datastore.
*** Marked as required and validated to be in email format.
** *Send Reset Code Button:*
*** Enabled only when a valid email is present.
*** On click:
**** Validates the email field.
**** Generates a random 6-digit code (with two randomly chosen digits repeating twice).
**** Stores the generated code for later validation.
**** Triggers an API call to send the reset code via email.
**** Transitions to *View 3*.
* *View 3: Code Verification*
** *Code Verification Input:*
*** Displayed as a textbox.
*** Marked as required.
** *Submit Button:*
*** Enabled only when a code is entered.
*** On click:
**** Validates the entered code against the stored generated code.
**** If valid, transitions to *View 4*.
* *View 4: New Password Entry*
** *New Password Field:*
*** Displayed as a password textbox.
*** Required.
*** Validated using the district’s password schema.
** *Confirm Password Field:*
*** Displayed as a password textbox.
*** Required.
*** Must match the New Password field.
** *Save Button:*
*** Enabled only when both password fields are valid and match.
*** On click:
**** Validates the password fields.
**** Initiates an API call to update the password on the server for the specified district.
**** Displays a progress indicator (spinning wheel) while waiting for confirmation.
**** On successful update, redirects the user to the login screen.

----

h4. *Back-End Requirements*

* *Data Retrieval:*
** Ensure the API can check the asynchronous datastore for the Platform, District, and Email values.
* *Code Generator:*
** Implement a routine that generates a random 6-digit code containing two randomly chosen digits that each appear twice.
* *Code Sender:*
** Provide an API endpoint (in SC) that accepts parameters (recipient email, subject, from, message text) and sends the email.
** Secure the endpoint with token authorization/handshake to ensure only the app can use it.
* *Update Password:*
** Create API endpoints in both SC and PE that:
*** Accept the Platform, District, user ID, and new password.
*** Encrypt the password and validate the request.
*** Log the update action in the server’s history.
*** Return a validation response confirming the password update.
* *Password Schema:*
** Expose an API endpoint that returns the district’s password schema (by district and platform) to be used for client-side password validation.

----

h4. *Security & Usability*

* The Forgot Password screen is accessible to all users, with no security trimming applied.
* Provide clear validation messages and progress indicators during all asynchronous operations (e.g., email sending, password updating).

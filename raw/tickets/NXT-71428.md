# NXT-71428 — SC - MT - District - Central Boundaries Tab

## Description
As a *Customer Support or Technical Support user*, I want to configure boundaries and associated roles at a parent level and apply them to one or more child districts, so that access policies can be distributed efficiently and consistently.

----

h1. Background / Notes

The *Central Boundaries* tab allows parent realms (e.g., state, county, management company, central realm) to:

* define which boundaries apply to children
* define which roles (with permissions) are propagated with each boundary
* select which children receive those boundaries
* control whether boundaries are optional or mandatory
* control whether propagated roles and permissions are optional or mandatory

Roles and permissions are treated as a *single unit*:

* when propagated, both the *role object and its associated permissions* are applied together

This tab is the *parent-level configuration counterpart* to the District Boundaries tab.

UI should remain *visually consistent* with the District Boundaries tab, but reflect a *bulk / parent-driven workflow*.

----

h1. Current Navigation

*System > Districts > Central Boundaries tab*

----

h1. Technical / Implementation Notes

* Add new tab labeled *Central Boundaries*
* UI should mirror *District Boundaries* wherever practical
* Use existing UI patterns:
** type-ahead Select Realm
** grid layout
** multi-select dropdown (NOT chips) for districts
** confirmation dialogs
** inline validation
** standard failure messaging
* Select Realm must include only realms with *at least one linked child realm*
* Grid must include:
** Enabled
** Boundary Name
** Boundary Requirement
** Role & Permission Requirement
** Linked Roles
** Target Districts
* *Linked Roles* must use parent realm’s role list
* Roles include permissions and propagate together
* *Target Districts uses multi-select dropdown*
** first option: *All*
** default: *All*
* Save requires validation + confirmation
* Audit log handled separately
* Boundary creation handled in *Boundary Definitions tab*

h1. Functional Requirements

h2. 1. Add Central Boundaries tab

Add a tab labeled *Central Boundaries*

----

h2. 2. Restrict access

Accessible only to:

* Customer Support
* Technical Support

District users cannot access this tab

----

h2. 3. Select Realm field

* Type-ahead searchable dropdown
* Filters as user types
* Only shows realms that have *at least one linked child realm*

Helper text:
*“List is restricted to realms that are linked to at least one child realm.”*

----

h2. 4. Add View History button

* Visible on screen
* Behavior out of scope

----

h2. 5. Load boundaries grid

Once a parent realm is selected, display all boundaries in a grid

----

h2. 6. Grid structure

Columns:

* Enabled
* Boundary Name
* Boundary Requirement
* Role & Permission Requirement
* Linked Roles
* Target Districts

UI guidance:

* Linked Roles = wide column
* Target Districts = wide column

----

h2. 7. Boundary Requirement behavior

User must select:

* Optional
* Mandatory

Behavior:

* *Mandatory*
** boundary is applied to children
** children cannot disable it
* *Optional*
** boundary is applied
** children may enable/disable

----

h2. 8. Role & Permission Requirement behavior

User must select:

* Optional
* Mandatory

Behavior:

* *Mandatory*
** roles (with permissions) are propagated
** children cannot modify or remove them
* *Optional*
** roles are propagated
** children may modify or replace

----

h2. 9. Linked Roles selection

* User selects one or more roles from the *parent realm’s role list*
* Roles include associated permissions
* Users can:
** add roles
** remove roles
** update selections before save

----

h2. 10. Target Districts selection

* Uses *multi-select dropdown (NOT chips)*
* First option: *All*
* Default: *All*

Behavior:

* If *All selected* → applies to all current linked children
* If *All deselected* → user selects one or more districts

----

h2. 11. Enable/disable boundary

* Controlled by checkbox
* If unchecked:
** all other controls in row are disabled

----

h2. 12. Disable row controls when unchecked

When boundary is not enabled:

* Boundary Requirement disabled
* Role & Permission Requirement disabled
* Linked Roles disabled
* Target Districts disabled

----

h2. 13. Save Changes

* Button located bottom-right
* On click:
** perform validation
** show confirmation dialog
** if confirmed → persist changes

----

h2. 14. Validation rules

For each enabled boundary:

* Boundary Requirement is required
* Role & Permission Requirement is required
* At least one role must be selected
* At least one district must be selected (unless All is selected)

On failure:

* block save
* show standard failure message
* highlight invalid row(s)

Example inline messages:

* *“Boundary requirement is required.”*
* *“Role & Permission requirement is required.”*
* *“At least one role is required.”*
* *“At least one target district is required.”*

----

h2. 15. Cancel / Reset

* Requires confirmation
* Reverts to last saved state

----

h2. 16. No auto-save

Changes persist only after Save + confirmation

----

h2. 17. Unsaved changes warning

* Triggered when navigating away

----

h2. 18. Bulk selection

* Select All / Deselect All for boundaries
* Does not override row-level validation

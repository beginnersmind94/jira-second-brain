# NXT-60345 — TH - WS - Inbox Compose Screen

## Description
*As a* TeamHub user *I want* a screen to compose inbox messages *So that* I can communicate with my team members

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the Inbox Compose screens from SC
* This story covers 3 scenarios of composing an inbox message:
** Add button from inbox screen
** Reply or Forward buttons from Message Details screen
** Inbox Message button from the Team screen

h2. User Flows

* #1, Compose button from Inbox Screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User clicks the Compose button and is redirected to this screen
** System loads a blank screen (no presets)
* #2, Reply or Forward buttons from the Details screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User sees a list of inbox messages (or sent messages)
** User clicks on a message and is redirected to the Message Details screen
** User clicks the Reply or Forward button and is redirected to this screen
** System loads the body of a the screen like an email body
** System adds any attachments
** If Reply:
*** System prepends RE: in front of the subject line and includes the original subject line
*** System add the Sender to the To field
** If Forward:
*** System prepends FWD: in front of the subject line and includes the original subject line
*** System leaves the To field blank
* #3, Send Message button on the Team screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Team and is redirected there
** User sees a list of Team members
** User clicks the Send Message button and is redirected to this screen
** System loads all blank fields except the To field and prepopulates the To with the name of the team member

h2. Requirements

* Ensure the screen supports dark mode

_Header Section_

* Have a back button to Inbox
* Page Title: New Message

_Body Section_

* Usual Email fields:
* To
** List from Team Members screen
** Multi-select
** Do not allow external email addresses or ad hoc email addresses
** Ensure can’t select same team member twice
* CC and Bcc
** If can be done, do them, if not, we can live without. Be consistent with Send messages from SC compose screen
* Subject
** Textbox
** Required
** Max Length same as Subject field in SC Compose screen
** Sanitize data entry
** Default blank if coming from Compose on Inbox screen or Send Message from Team screen
** Default RE: or FWD: if coming from Details screen
* Message
** Multi-line Textbox
** Required
** Max Length same as Body field in SC Compose screen
** Sanitize data entry
** Default blank if coming from Compose on Inbox screen or Send Message from Team screen
** Default with a couple of line breaks at the top and then the original body if coming from Details screen
* Attachment
** Use the Expo attachment functionality
** User can attach files, photos, etc. using mobile’s device functionality
** Default with no attachment if coming from Compose on Inbox screen or Send Message from Team screen
** Default with attachment if the original has it if coming from Details screen
* Send button
** On click:
** Validate the required fields (To, Subject, Body) have data
*** If no, indicate to the user what is missing
*** If yes, continue
** Store the message in the asynchronous data store
** Manually trigger in code for the Inbox/Sent aynch job to run immediately
*** Note: rather than sending the email directly to the server, it saves it to the local storage and then tells the sync agent to run
** Show the user a success message
** Redirect the user to the Inbox screen

h4. *Data Requirements*

* There should be no API calls on this screen
* All data should be pulled from the asynchronous storage on the phone
* The Send button should write to the asynchronous storage on the phone, not connect to the external server
* There should be separate code (another story) that runs independently from this page that syncs the Inbox/sent messages to the server and this screen should be agnostic to that.

h4. *Performance & Usability*

* The screen must load within *2 seconds* and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Errors must be communicated using *user-friendly messages*:
** Use *toast notifications, modals, or inline messages* without disrupting the user experience
* Log any API or rendering errors to the *error log system/database* for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *General Requirements*

* The screen supports dark mode, with proper contrast and readability for all UI elements.
* The screen loads within 2 seconds and renders correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions are responsive and provide clear visual feedback.
* The screen is accessible to any authenticated user.
* No API calls are made on this screen; all data is handled via the device's asynchronous storage.

----

h3. *User Flow Scenarios*

h4. *Scenario 1: Compose from Inbox Screen*

* Tapping the Compose button in the Inbox screen redirects the user to the Compose Message screen.
* All fields load blank by default:
** To: blank
** Subject: blank
** Message: blank
** Attachment: none

h4. *Scenario 2: Reply or Forward from Message Details*

* Tapping Reply or Forward in the Message Details screen opens the Compose Message screen.
* The screen auto-fills:
** *To field*:
*** Reply: prefilled with the sender of the original message.
*** Forward: left blank.
** *Subject field*:
*** Reply: "RE: [original subject]"
*** Forward: "FWD: [original subject]"
** *Message field*:
*** Pre-filled with line breaks at the top, followed by the original message body.
** *Attachment*:
*** Included if the original message had attachments.

h4. *Scenario 3: Send Message from Team Screen*

* Tapping Send Message from a team member in the Team screen opens the Compose Message screen.
* The screen prepopulates the To field with the selected team member.
* All other fields are blank by default.

----

h3. *Display Elements*

h4. *Header Section*

* Includes a Back button that navigates to the Inbox screen.
* Displays the page title: "New Message".

h4. *Body Section*

* *To Field*:
** Allows multi-select from the list of team members.
** Does not allow external or ad hoc email addresses.
** Does not allow the same team member to be selected more than once.
* *CC and Bcc Fields*:
** If supported in the app (consistent with SC Compose screen), they are included.
** If not supported, they are omitted entirely.
* *Subject Field*:
** Required field.
** Rendered as a single-line textbox.
** Max length matches the SC Compose screen subject limit.
** Data entry must be sanitized to prevent injection or formatting issues.
** Defaults:
*** Blank in scenarios 1 and 3.
*** Pre-filled with "RE:" or "FWD:" in scenario 2.
* *Message Field*:
** Required field.
** Rendered as a multi-line textbox.
** Max length matches the SC Compose screen body limit.
** Data entry must be sanitized.
** Defaults:
*** Blank in scenarios 1 and 3.
*** Pre-filled with line breaks and the original message body in scenario 2.
* *Attachment Field*:
** Uses Expo’s native attachment functionality.
** Allows attaching photos, files, etc. using the mobile device.
** Defaults:
*** No attachment in scenarios 1 and 3.
*** Carries over attachments from the original message in scenario 2.

----

h3. *Send Button Behavior*

* When tapped:
** Validates that To, Subject, and Message fields are populated.
** If validation fails:
*** Show user-friendly error messages indicating missing fields.
** If validation succeeds:
*** Save the message to the asynchronous storage (not sent to server).
*** Trigger the sync job manually in code to queue the message for sending (sync logic is handled in a separate story).
*** Display a success toast notification to the user.
*** Redirect the user back to the Inbox screen.

----

h3. *Data Handling*

* All message data (To, Subject, Body, Attachments) is stored in local async storage.
* This screen does not interact with external servers or APIs.
* Syncing to the server is handled by a background agent not included in this story.

----

h3. *Error Handling*

* Use try/catch blocks around all asynchronous storage interactions.
* Implement error boundaries to catch rendering issues.
* Display user-friendly error messages via:
** Toasts
** Modals
** Inline messages
* Log all caught errors to the error logging database/system for review.

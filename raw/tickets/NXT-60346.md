# NXT-60346 — TH - WS - Inbox Details Screen

## Description
*As a* TeamHub user *I want* a screen to view the full inbox messages *So that* I can see what my team members sent me

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the Inbox Details screens from SC
* This story covers 2 scenarios of viewing an inbox message:
** User clicks card on the inbox screen
** User clicks card on the sent screen

h2. User Flows

* #1, Inbox screen
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User clicks the message card and is redirected to this screen
** System loads the fields by default
* #2, Sent screen:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** User clicks Workspace and is redirected there
** User clicks on Inbox and is redirected there
** User clicks the Sent chip and the screen shows sent items
** User clicks the message card and is redirected to this screen
** System loads the fields by default

h2. Requirements

* Ensure the screen supports dark mode

_Header Section_

* Have a back button to Inbox or Sent tabs depending on where they came from
* Page Title: the subject of the message
* Star button
** If clicked: Star or Unstar the message accordingly

_Body Section_

* Usual Email fields:
* Avatar
** If it is an Inbox message, Avatar of user sent From 
** If it is a Sent message:
*** Check if it has one recipient
*** If yes, show their Avatar
*** If no, show a default person outline avatar
* From
** Show only if it is an Inbox message
** Show Name and email of the sender
* To
** Show only if it is a Sent message
** Show the list of users the message was sent to
* CC and Bcc
** If can be done, do them, if not, we can live without. Be consistent with Send messages from SC compose screen
* Date/Time
** Use smart date time like Today, Yesterday, 1 week ago
** rules are in the code
* Body
** Show full message body
* Attachment
** Show list of attachments, with their name and size
** Be able to download/view the attachment
*** Note: in reality, it is not a download, but a view since it would have already downloaded as part of the sync job
* Reply button
** On click:
** Redirect the user to the Compose screen and load its values automatically
*** Note: a separate story covers how to load the values on that screen
* Forward button
** On click:
** Redirect the user to the Compose screen and load its values automatically
*** Note: a separate story covers how to load the values on that screen
* Delete
** On click:
*** Do nothing at the moment
*** A separate story covers the delete functionality
* Note: If the bold code has Archive button, ignore it. We aren’t doing that

h4. *Data Requirements*

* There should be no API calls on this screen
* All data should be pulled from the asynchronous storage on the phone
* There should be separate code (another story) that runs independently from this page that syncs the Inbox/sent messages to the server and this screen should be agnostic to that.

h4. *Performance & Usability*

* The screen must load within *2 seconds* and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Errors must be communicated using *user-friendly messages*:
** Use *toast notifications, modals, or inline messages* without disrupting the user experience
* Log any API or rendering errors to the *error log system/database* for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *User Flows*

_Inbox Screen Flow_

* When the user logs in and navigates to Workspace > Inbox, they can tap a message card.
* Tapping the message redirects them to the Message Details screen.
* The screen loads the message fields from local asynchronous storage.

_Sent Screen Flow_

* When the user logs in and navigates to Workspace > Inbox, they can tap the Sent chip to view sent messages.
* Tapping a sent message card redirects them to the Message Details screen.
* The screen loads the message fields from local asynchronous storage.

h4. *Display Elements*

_Dark Mode Support_

* The entire screen supports dark mode, with text, icons, and background colors adjusting appropriately for visibility.

_Header Section_

* Displays a back button that returns the user to either the Inbox or Sent tab, based on where they came from.
* The page title is the subject of the message.
* Includes a star icon that can be tapped to toggle the starred status of the message.
** Tapping the icon updates the local async storage.

_Body Section_

* Displays an avatar:
** If the message is from Inbox, shows the avatar of the sender.
** If the message is from Sent:
*** If one recipient: show their avatar.
*** If multiple recipients: show a default person-outline avatar.
* From field:
** Visible only for Inbox messages.
** Displays the sender’s name and email.
* To field:
** Visible only for Sent messages.
** Displays the list of users the message was sent to.
* CC and Bcc:
** Included only if feasible and consistent with the SC Compose screen.
* Date/Time:
** Displayed using smart formatting (e.g., Today, Yesterday, 1 week ago) based on the predefined rules in code.
* Message body:
** Displays the full text of the message.
* Attachments:
** Each attachment is listed by name and file size.
** Tapping the attachment opens it for viewing.
** Files are already downloaded via the sync process, so no download logic is needed here.

h4. *Interaction Buttons*

* _Reply_:
** Tapping the Reply button redirects the user to the Compose screen.
** A separate story defines how data is loaded on that screen.
* _Forward_:
** Tapping the Forward button redirects the user to the Compose screen.
** A separate story defines how data is loaded on that screen.
* _Delete_:
** Currently inactive.
** The button may appear, but has no functionality.
** The Archive button, if present in the component library, must be ignored.

h4. *Data Requirements*

* All data displayed on this screen must be pulled from the phone’s asynchronous storage.
* No API calls are made on this screen.
* The sync process that updates Inbox and Sent messages runs independently and is handled in a separate story.

h4. *Performance & Usability*

* The screen must load in under 2 seconds under normal operating conditions.
* It must display correctly and responsively across:
** Mobile
** Tablet
** Desktop
** Compact and expanded views
* All tap and navigation interactions must be smooth and visually responsive.

h4. *Error Handling*

* All asynchronous data access must use try/catch blocks.
* Error boundaries must be implemented to catch rendering errors.
* Any user-facing errors must be presented with non-intrusive, friendly messaging using:
** Toast notifications
** Modal dialogs
** Inline alerts
* All exceptions must be logged to the central error logging system for monitoring and debugging.

h4. *Security*

* The screen is accessible to any authenticated user.
* No special permissions are required beyond successful authentication.

# NXT-70295 — FIN - Automated GL Posting from Inventory Period Close

## Description
As a Child Nutrition Director, I want the system to automatically generate balanced general ledger entries for all inventory withdrawals (including waste/spoilage), manual inventory adjustments, inter-site transfers, and internal distributions when an inventory period closes, so that my food and supply costs appear in Financials without manual journal entries and are ready for export to the district ERP.

!image-20260311-132224.png|width=818,alt="image-20260311-132224.png"!

h2. Business Context

* SchoolCafe 2.0 inventory's primary period output is Valuation + Usage reporting by *Valuation Category* (called "Valuation Group" in PrimeroEdge).
* "Commodity/USDA foods" is *not* a system flag in SC 2.0. Districts define commodity classification via Valuation Categories. 
* Receipt valuation already reflects discounts and charges in the final valuation amount. Fee breakdowns (delivery, processing) exist in underlying tables but are not surfaced as separate reportable columns today. *Financials treats the final valuation amount as the source of truth; it does not decompose it.*
* Waste/spoilage recorded via withdrawal reason codes is *included in withdrawal postings*. Production-record waste (Insights) uses fair market value estimates and *will not reconcile* to withdrawal-based postings. This is a *documented product limitation*, not a bug.

🔄 *Data source clarification:* All dollar amounts for inventory postings come from the inventory module's valuation calculations as reflected in the Valuation/Usage reports. Financials does not independently calculate inventory values; it consumes what inventory provides.

h2. Scope

h3. In Scope

* Period-close batch postings for 5 entry types: PURCH, WD, ADDINV, INVTRFRFR, INVTRFRTO.
* System-defined entry types only (no user-created rules for inventory).
* Per-site ledger entries for PURCH, WD, ADDINV, INVTRFRFR, INVTRFRTO.
* Validation gates before posting: period state, mapping completeness, balancing.
* Reposting with reversal when a previously-posted period is reopened and re-closed.

h2. Requirements

h3. R1. Trigger and Timing

🔄 *R1.1 — Trigger mechanism:* Posting is initiated automatically by the system when an inventory period transitions to *Closed* status for a site. There is no manual "Run Posting" button as of now.

🔄 *R1.2 — User notification:* When posting completes (success or failure), the system logs the result in the *Posting Activity Log*. If posting fails, the failure reason appears in the log entry. The name of the job is ‘Inventory Posting.’

*R1.3 — Posting date:* The posting date for all period-close entry types equals the *last day* of the inventory period (e.g., period 2026-01-01 to 2026-01-31 → posting date = 2026-01-31).

*R1.4 — No partial posting:* If any entry type within a posting batch fails validation, the *entire batch* for that site-period is rejected. No entries are created. The Posting Activity Log records the failure with the specific reason.

h3. R3. Account Determination

*R3.1 — Mapping source:* GL accounts for inventory postings are determined by the *Valuation Category → GL Account mappings* configured on the Account Mapping page in Financials.

🔄 *R3.2 — Required mappings per entry type:* Each Valuation Category used by any transaction in the posting scope must have Asset GL Account AND Expense GL Account.

h3. R4. Entry Aggregation and Per-Site Behavior

🔄 *R4.1 — Aggregation level:* The system creates *one ledger entry per entry type per Valuation Category per site per period.* For example, if Site A has withdrawals of items in VC "Purchased Food" totaling $2,000 and VC "USDA Foods" totaling $800, posting creates two WD entries for Site A: one for $2,000 and one for $800. Each entry is independently balanced (debits = credits).

*R4.2 — Per-site entry types:* PURCH, WD, ADDINV, INVTRFRFR, INVTRFRTO each create entries scoped to a single site.

h3. R5. Balancing and Audit Trail

*R5.1 — Balance rule:* Every created ledger entry must have total debits = total credits. If a computed entry would be unbalanced (e.g., due to rounding), the system must reject the entire posting batch and log the error.

🔄 *R5.2 — Audit trail — what the user sees:* When a user views a posted inventory entry in the Journal or Ledger view and opens the entry detail (via row click or detail action), the detail view displays:

* The *entry type* (e.g., "WD — Inventory Withdrawals").
* The *Valuation Category* the entry relates to.
* The *posting date* and *site*.

h3. R6. Journal and Ledger Visibility

*R6.1 — Appearance:* Posted inventory entries appear in both the Journal and Ledger views alongside other entry types (POS entries, manual entries, etc.).

🔄 *R6.2 — Identification:* Each inventory-generated entry must display:

* *Source module:* "Inventory" (so users can distinguish it from POS or manual entries).
* *Entry type code and name:* e.g., "WD — Withdrawals"
* *Site*
* *Valuation Category*

*R6.3 — Filtering:* Entries are filterable by site, fiscal year, financial period, GL account, and entry type. 

h3. R7. Dirty Flag and Reposting

*R7.1 — Dirty detection:* If a previously-posted inventory period is reopened in the inventory module, the corresponding posting batch in Periods is updated to display this.

*R7.3 — Repost trigger:* When the inventory period is re-closed, the system automatically initiates a repost for that site-period if the user has the setting in Periods on, otherwise no action needed.

*R7.4 — Repost mechanics:*

# System generates *reversal entries* for every entry in the original posting batch. Each reversal entry has:
#* Posting date = *the same posting date as the original entry* (last day of the inventory period), NOT today's date. 🔄
#* All amounts with reversed signs (debits become credits, credits become debits).
#* A reference to the original entry it reverses.
# System generates a *new posting batch* from the updated inventory data, following all the same rules (R1–R6).
# The original entries are *never edited or deleted*. They remain in the ledger with their original values.

----

h2. Entry Types — Detailed Specifications

h3. ET1 — Purchases (PURCH)

||Attribute||Value||
|*Posting date*|Last day of inventory period.|
|*Debit*|Valuation Category → Asset GL Account|

|*Aggregation*|One entry per Valuation Category per site per period.|
|*Note*|The credit side (Cash in Bank) is a single configured GL account, not derived from the Valuation Category mapping. This is the only entry type where debit and credit draw from different mapping sources.|

|*Aggregation*|One entry per Valuation Category per site per period.|

h3. ET3 — Add to Inventory (ADDINV)

||Attribute||Value||
|*Posting date*|Last day of inventory period.|
|*Debit*|Valuation Category → Asset GL Account|
|*Credit*|Valuation Category → Expense GL Account|
|*Value Source*|Add to Inventory in Usage Report|
|*Aggregation*|One entry per Valuation Category per site per period.|

h3. ET4 — Transfer Out (INVTRFRFR)

||Attribute||Value||
|*Posting date*|Last day of inventory period.|
|*Debit*|Valuation Category → Expense GL Account|
|*Credit*|Valuation Category → Asset GL Account|
|*Value source*|Transfers In from Usage Report|
|*Aggregation*|One entry per Valuation Category per site per period.|

## Acceptance Criteria
h1. Acceptance Criteria — Inventory Period-Close GL Postings

*Module:* Financial Oversight · SchoolCafe 2.0

----

h2. Trigger & Timing

*Scenario: Automatic posting on period close*

* *Given* an inventory period for a site has transactions in scope
* *When* the inventory period transitions to Closed status
* *Then* the system automatically initiates a posting job named "Inventory Posting" for that site-period with no manual trigger required

*Scenario: Posting date equals last day of period*

* *Given* an inventory period with date range 2026-01-01 to 2026-01-31
* *When* the posting job generates ledger entries
* *Then* every entry in the batch has a posting date of 2026-01-31

*Scenario: Successful posting logged*

* *Given* a posting job completes without errors
* *When* the user views the Posting Activity Log
* *Then* a log entry confirms successful completion for that site-period

*Scenario: Failed posting logged with reason*

* *Given* a posting job fails validation
* *When* the user views the Posting Activity Log
* *Then* a log entry shows the specific failure reason for that site-period

*Scenario: No partial posting on validation failure*

* *Given* a posting batch where one or more entry types fail validation
* *When* the system attempts to post
* *Then* zero entries are created for the entire site-period batch

----

h2. Account Determination

*Scenario: GL accounts resolved from Valuation Category mapping*

* *Given* a Valuation Category has an Asset GL Account and Expense GL Account configured on the Account Mapping page
* *When* the posting job generates entries for transactions in that Valuation Category
* *Then* the system uses those mapped GL accounts for the debit and credit sides per the entry type rules

*Scenario: Missing mapping blocks all posting*

* *Given* any Valuation Category present in the posting scope is missing either its Asset GL Account or Expense GL Account mapping
* *When* the inventory period transitions to Closed status
* *Then* no posting job runs for any entry type in that site-period — including entry types that do not use the Valuation Category with the missing mapping

----

h2. Entry Aggregation & Per-Site Behavior

*Scenario: One entry per type per Valuation Category per site per period*

* *Given* Site A has withdrawals in VC "Purchased Food" totaling $2,000 and VC "USDA Foods" totaling $800
* *When* the posting job runs for Site A's period
* *Then* the system creates exactly two WD entries for Site A — one for $2,000 (Purchased Food) and one for $800 (USDA Foods)

*Scenario: Entries are scoped to a single site*

* *Given* transactions exist across multiple sites in the same period
* *When* the posting job runs
* *Then* each generated entry (PURCH, WD, ADDINV, INVTRFRFR, INVTRFRTO) is scoped to exactly one site

*Scenario: Each entry is independently balanced*

* *Given* a posting job generates multiple entries
* *When* any single entry is examined
* *Then* total debits = total credits on that entry

----

h2. Balancing

*Scenario: Unbalanced entry rejects entire batch*

* *Given* a computed entry would result in debits ≠ credits (e.g., due to rounding)
* *When* the system validates the batch
* *Then* the entire posting batch for that site-period is rejected and the error is logged in the Posting Activity Log

----

h2. Audit Trail

*Scenario: Entry detail shows source traceability*

* *Given* a posted inventory entry exists in the Journal or Ledger view
* *When* the user opens the entry detail
* *Then* the detail displays:
** Entry type (e.g., "WD — Inventory Withdrawals")
** Valuation Category
** List of source transaction IDs from the inventory module that were aggregated into the entry
** Posting date
** Site

----

h2. Journal & Ledger Visibility

*Scenario: Inventory entries appear alongside other entry types*

* *Given* inventory posting has completed for a site-period
* *When* the user views the Journal or Ledger
* *Then* the posted inventory entries appear alongside POS and manual entries

*Scenario: Inventory entries are identifiable*

* *Given* a posted inventory entry exists
* *When* the user views it in the Journal or Ledger
* *Then* it displays source module ("Inventory"), entry type code and name (e.g., "WD — Withdrawals"), site, and Valuation Category

*Scenario: Entries are filterable*

* *Given* posted entries exist in the ledger
* *When* the user applies filters
* *Then* entries are filterable by site, fiscal year, financial period, GL account, and entry type

----

h2. Dirty Flag & Reposting

*Scenario: Reopened period flags the posting batch*

* *Given* an inventory period was previously posted
* *When* the period is reopened in the inventory module
* *Then* the corresponding posting batch in the Periods view reflects this state

*Scenario: Repost triggered on re-close with setting enabled*

* *Given* a previously-posted inventory period has been reopened and the user has the repost setting enabled in Periods
* *When* the period is re-closed
* *Then* the system automatically initiates a repost for that site-period

*Scenario: Repost not triggered when setting is off*

* *Given* a previously-posted inventory period has been reopened and the user has the repost setting disabled in Periods
* *When* the period is re-closed
* *Then* no repost is initiated

*Scenario: Reversal entries use original posting date*

* *Given* a repost is initiated for a site-period
* *When* the system generates reversal entries
* *Then* each reversal entry's posting date equals the original entry's posting date (last day of the inventory period), not today's date

*Scenario: Reversal entries invert amounts*

* *Given* a repost is initiated for a site-period
* *When* the system generates reversal entries
* *Then* all amounts are the exact inverse of the originals (debits become credits, credits become debits) and each reversal references the original entry it reverses

*Scenario: New batch follows original rules*

* *Given* reversal entries have been generated
* *When* the system generates the new posting batch from updated inventory data
* *Then* the new batch follows all the same posting rules (trigger, account determination, aggregation, balancing, audit trail)

*Scenario: Original entries are immutable*

* *Given* a repost has occurred
* *When* the user views the ledger
* *Then* the original entries remain unmodified and undeleted with their original values

----

h2. Entry Types

h3. PURCH — Purchases

*Scenario: Purchase posting*

* *Given* a site-period has purchases recorded in the Usage Report
* *When* the posting job runs
* *Then* for each Valuation Category with purchases:
** Debit → Valuation Category's Asset GL Account
** Credit → Cash in Bank GL Account (single configured account, not derived from Valuation Category mapping)
** Amount = Purchases column value from Usage Report
** One entry per Valuation Category per site per period

h3. WD — Inventory Withdrawals

*Scenario: Withdrawal posting*

* *Given* a site-period has usage recorded in the Usage Report
* *When* the posting job runs
* *Then* for each Valuation Category with usage:
** Debit → Valuation Category's Expense GL Account
** Credit → Valuation Category's Asset GL Account
** Amount = Usage value from Usage Report
** One entry per Valuation Category per site per period

h3. ADDINV — Add to Inventory

*Scenario: Add-to-inventory posting*

* *Given* a site-period has add-to-inventory transactions in the Usage Report
* *When* the posting job runs
* *Then* for each Valuation Category with add-to-inventory amounts:
** Debit → Valuation Category's Asset GL Account
** Credit → Valuation Category's Expense GL Account
** Amount = Add to Inventory value from Usage Report
** One entry per Valuation Category per site per period

h3. INVTRFRFR — Transfer Out

*Scenario: Transfer-out posting*

* *Given* a site-period has transfer-out transactions in the Usage Report
* *When* the posting job runs
* *Then* for each Valuation Category with transfer-out amounts:
** Debit → Valuation Category's Expense GL Account
** Credit → Valuation Category's Asset GL Account
** Amount = Transfers In value from Usage Report
** One entry per Valuation Category per site per period

h3. INVTRFRTO — Transfer In

*Scenario: Transfer-in posting*

* *Given* a site has received items from another site during the period per the Usage Report
* *When* the posting job runs
* *Then* for each Valuation Category with transfer-in amounts:
** Debit → Valuation Category's Asset GL Account
** Credit → Valuation Category's Expense GL Account
** Amount = Transfers In value from Usage Report
** One entry per Valuation Category per site per period

# NXT-2897 — SSO - ADFS (SAML)

## Description
As a Customer Support user, I want to be able to setup a district’s ADFS authentication provider so that they can use SSO

*Notes*:

* This story is for SAML (ADFS) in Keycloak
* This feature will be done exactly the same way as OpenID and the requirements will reference this

*Flow:*

* User browses to System > Security page
* User clicks the ADFS auth type Add new button
* User is redirected to ADFS Add New drawer and completes it
* User saves the config to Keycloak
* User tests the config
* User returns and optionally sets ADFS as the default SSO provider
* User optionally views History
* User optionally sets ADFS as their own SSO provider
* User optionally sets ADFS as someone else’s SSO provider if they have rights

*Requirements*:

*Security Screen*

* These requirements apply to the System > Security page > Authentication tab:
* 
* Add ADFS Provider to the + New drop down button
** On click for ADFS Provider:
*** Open the Add New ADFS Provider
*** Note: they can have more than one ADFS provider, so allow them to add multiple of this type
* 
* On the grid, 
** Authentication Type column 
*** Ensure it includes ADFS
** Description
*** For ADFS: Default the text with: Login using district’s SSO ADFS provider
** Can mark preferred and Active/Inactive the same as OpenID
** Actions
** Edit
*** On click for ADFS row, open the Edit ADFS drawer
** Delete
*** Handle exactly the same was as for OpenID



*Add New Drawer*

* Create an Add New ADFS provider drawer

* Ensure it works the same as for OpenID
** All the same functionality

* Ensure you clone the existing ADFS configuration if it exists

* Note: we are managing Enabled and Preferred from the grid on the Authentication dashboard, so not showing them here. If you do show them, then show them read-only

** Want to force them to intentionally enable on the grid because there is a lot of business logic to this event and they might want time to test and configure
* Set Preferred = False

*Edit Drawer*

* Do the same as for OpenID, but using the ADFS fields



*Add New User page*

* On Edit user screen, ensure that ADFS work just like OpenID for this screen



*Edit User page*

{color:#0747a6}*Technical Details*{color}

This is almost similar to OpenId Connect that we implemented. The only change will be configurations and way to interact with Keycloak token



Attached the sample code [^ADFSSampleCode.txt]   to generate and validate the ADFS Token. Once the token is validated we need to call our KeycloakEndpoints using service token to fetch the token for the username  that was retrieved from ADFS token (using impersonate access)

## Acceptance Criteria
* Admin is able to add ADFS as an authentication method for their district
* Admin is able to assign ADFS as the preferred method
* Admin is able to enable / disable ADFS method
* Admin is able to delete ADFS method if there are other rows in the grid
* Users are able to set ADFS in their profile if it is a valid option
* Admins are able to set ADFS for a user if it a valid option
* Changes to ADFS auth show up in History page
* Passwords are handled the same was as for OpenID
* If user is ADFS method, they redirect to their provider when click next on login screen
* User is redirected back to our website with valid auth token after logging in

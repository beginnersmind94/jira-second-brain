# NXT-71249 — FIN — Opening Balance Import: Data Exchange

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions before any period transactions are recorded.

---

## TL;DR

Build a new **Opening Balance Import** type in Import Monitor — standard 4-tab flow. _BL owns validation and writes; DE owns UI and orchestration._

**Flow:**

* **Pre-upload:** _Fiscal Year_ dropdown + **Description** input (10–50 chars) on the left; **Download Template** link + help text on the right (matches production Import Setup layout). Eligibility query fires on FY selection; restatement gate triggers here if BL flags the FY.
* **File Upload:** `xlsx / xls / csv / txt`, **single sheet only** (reject multi-sheet at upload). **FILE DATA PREVIEW** button shows raw first rows.
* **Field Mapping:** Map 4 required targets (`GLAccount`, `SiteCode`, `Debit`, `Credit`) + 1 optional (`AccountName` — if mapped, must match the AccountName recorded for the GLAccount in CoA). Column mapping only — no validation here.
* **Review:** FY, Description, totals, effective date (from BL). User confirms → Save calls BL Commit.

**Restatement gate** fires on Pre-upload when the eligibility query flags the FY as having a **closed period** or a **prior OB import**. Gate requires the user to type the confirmation phrase `Yes, I would like to overwrite the previous import.` — Continue disabled until the typed value matches the phrase. The **Description entered on Pre-upload doubles as the restatement reason** — no separate reason field. Description + confirmation timestamp pass to BL on Commit.

**At Commit**, BL runs every business rule (per NXT-71240) in one atomic pass. If anything fails, the entire file is rejected; no ledger write. Failures surface in **Import Monitor → Exceptions tab**. If eligibility state changed between FY selection and Commit, BL blocks Commit via a modal on Review; the user re-satisfies the gate.

**Atomic.** All rows write or none. _No partial imports, ever._

**Saved mapping** auto-persists on success — auto-loads next time. Every import still goes through Commit under the same rules. **FY and Description are not persisted** — entered fresh per import.

**Roles with edit access:** Director, Central Office, SchoolCafe Admin. Customer/Technical Support always full.

### 3) Pre-upload Tab (Fiscal Year, Description, Eligibility)

_JAS: can be in the pre-upload screen after select the import type. needs to be validated by Dev Mgr. on how to do all this. Need to see this in a UX. what fields are needed? fiscal year selector, restatement textbox? there seems to be a lot of business logic on this._ Resolved by RM: FY selector + Description input live on the Pre-upload tab (left column). Eligibility query fires on FY selection. Restatement gate runs on Pre-upload when the query returns Restatement required. Right-side panel contains the Download Template link and help text per AC23/AC24.

3.1 On Pre-upload, after Import Type selection, a **Fiscal Year** dropdown lists all FYs. User picks one before uploading.

3.2 Selector does no business logic — just lists FYs.

3.3 On FY selection, DE calls an **eligibility query** to BL. _This is distinct from validation — it returns FY state at selection time, not a file processing pass._ Returns one of:

* **Eligible** — proceed.
* **Restatement required** — closed periods exist for this FY, a prior OB import exists, or both.
* **Not eligible** — prior FY has open periods, or CoA lacks A/L/E accounts.

3.4 On **Not eligible**, Next is blocked with a message and deep link (Period Management — _Financials > Configuration > Periods_ — or Chart of Accounts, depending on the cause).

3.5 On **Restatement required**, a confirmation gate appears on Pre-upload before progress. The gate:

* States which condition(s) triggered the restatement (closed periods, prior import, or both). Each trigger shows its own message:

    * Closed periods: `The selected fiscal year has one or more closed periods. To post opening balances, reopen the affected periods first.`
    * Prior OB import: `An Opening Balance Import already exists for this fiscal year. Continuing will reverse the existing entries and replace them with this import.`
    * Both triggers: both messages above are shown.

* When closed periods is a trigger: provides two links — `Reopen periods in Period Management` (_Financials > Configuration > Periods_) and `Post a manual adjustment instead (no period reopening required)` (_Financials > General Ledger > Add New Entry > Manual Entry_).
* For prior imports: existing OB entries will be hard-reversed and replaced on Commit (see **NXT-71240** §5).
* Requires the user to type the confirmation phrase `Yes, I would like to overwrite the previous import.` Continue is disabled until the typed value matches the phrase.
* Inline note (always shown): _"The Description you entered above will be saved as the restatement reason for audit. Edit it if needed before continuing."_ — no separate reason field.

3.6 Description and confirmation timestamp persist to the audit record and hand off to BL for ledger entry audit metadata (§10; **NXT-71240** §1.1).

3.7 All rows assigned to the selected FY (not mapped from file).

3.8 One active OB import per FY. Replacement acknowledgment goes through the restatement gate.

3.9 FY and Description selected/entered fresh per import. Not part of the persisted mapping.

3.10 At Commit, BL re-checks eligibility. If state changed since FY selection (period closed, another OB import landed), Commit is blocked. A blocking modal appears on the Review tab: `Eligibility changed since you selected the fiscal year. {Trigger}. Re-confirm the fiscal year to continue.` `{Trigger}` is one of: `A period was closed.` / `A prior Opening Balance Import was created.` / `A period was closed and a prior Opening Balance Import was created.` When the user dismisses the modal, the wizard returns to Pre-upload with the restatement gate visible. File, mapping, Description, and FY selection persist; the user must re-type the confirmation phrase to proceed.

3.11 **Description (Pre-upload).** Required text input on Pre-upload, **minimum 10 characters, maximum 50 characters**. One description for the entire import — the OB import produces a single ledger entry with many lines (per **NXT-71240** §1), and the description applies to that entry. Next disabled until Description is within the 10–50 character range.

3.12 Description is **not** a file column. If a user's file contains a column named Description, it is ignored — Description is a single wizard-entered value.

### 4) File Contract

_JAS: this would need to be on the pre upload tab after selecting the import type._ Resolved by RM: Single canonical contract — separate Debit and Credit columns required. Users conform their file to the downloadable template (§4.3). No signed-amount support and **no transformation in DE**. Files that don't conform fail per row-level validation (NXT-71240 §1).

4.1 **Single canonical format:** separate **Debit** and **Credit** columns. Each row has a positive value in exactly one of the two. Row-level rejection microcopy (BL Commit):

* Non-numeric (first 10 rows fail-fast): `Source row {RowNumber}: Debit or Credit contains non-numeric characters. Use the Download Template for the correct format.`
* Both populated: `Source row {RowNumber}: Both Debit and Credit have values. Each row must have exactly one of Debit or Credit.`
* Both blank or zero: `Source row {RowNumber}: Debit and Credit are both blank or zero. Each row must have a positive value in exactly one column.`
* Negative value: `Source row {RowNumber}: Negative value in Debit or Credit. Values must be positive.`

4.2 No user-facing amount format selector. No transformation logic in DE. Field Mapping always shows Debit and Credit as canonical target fields.

4.3 **Template available.** A **Download Template** link in the Pre-upload right-side help panel (per AC23) provides an Excel file (`.xlsx`) with the canonical headers — GLAccount, SiteCode, Debit, Credit, AccountName — and no data rows. Users conform their file to this template before upload.

### 7) Validation

_All validation runs at BL Commit per **NXT-71240** §1–§7 (row-level rules, GL account checks, site checks, sub-account, uniqueness, file-level integrity, restatement enforcement). NXT-71240 is the single source of truth; row-level rules are not duplicated here to prevent drift. The QA-facing summary lives in this story's Acceptance Criteria (AC6–AC11, AC22, AC26)._

No data validation runs in DE. DE only collects the FY, Description, file, mapping, and restatement metadata, then hands the package to BL Commit on Save (§8, §12).

### 8) Validation Architecture

BL runs everything in one atomic Commit pass — see **NXT-71240** §0.6 + §1–§7.

8.1 No data validation in the wizard. The wizard is pure setup (Pre-upload + File Upload + Field Mapping + Review).

8.2 On Save, DE hands the mapped file, selected FY, Description, and restatement metadata (when applicable) to BL's Commit. BL runs all row-level and import-level checks in one pass.

8.3 **Atomic — all rows or none.** Any failed check rejects the entire file; no ledger write. Unbalanced file rejection: `Total debits ({TotalDebits}) do not equal total credits ({TotalCredits}). Difference: {Variance}. File rejected — debits must equal credits.` (currency values format as `$1,234.56`.)

8.4 Failures surface in **Import Monitor → Exceptions tab** per AC13. Row-level failures list every failing row with its error message. File-level failures show a single error message from the relevant AC. Commit-time eligibility changes (per AC17) surface as a blocking modal on Review, not in the Exceptions tab.

8.5 BL re-checks eligibility at Commit (§3.10). If state changed since FY selection, Commit is blocked and the user re-satisfies the gate.

### 12) Handoff to Business Logic

12.1 On Save, DE passes to BL's Commit:

* The mapped file (rows with canonical Debit/Credit columns)
* Selected fiscal year
* Entry description (10–50 chars, user-entered on Pre-upload)
* Restatement metadata when applicable (confirmation timestamp) — _Description doubles as restatement reason; not sent separately._

BL: runs all business rules in one atomic pass, assigns the effective date, creates the ledger entry and lines, persists the metadata, handles hard reversal of any previous import, makes the data available to downstream reports.

12.2 On failure, BL returns structured error details (rows + reasons). DE surfaces in Import Monitor → Exceptions tab per AC13.

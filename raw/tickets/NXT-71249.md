# NXT-71249 — FIN — Opening Balance Import: Data Exchange

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-07. Compressed §1–§12 for readability (\~30% trim, no semantic changes). Earlier 2026-05-07: added TL;DR section._

---

## TL;DR

Build a new **Opening Balance Import** type in Import Monitor — standard 4-tab flow. _BL owns validation and writes; DE owns UI and orchestration._

**Flow:**

* **Pre-upload:** _Fiscal Year_ dropdown — that's it. No business logic on the selector.
* **File Upload:** `xlsx / xls / csv / txt`, **single sheet only** (reject multi-sheet at upload). **FILE DATA PREVIEW** button shows raw first rows.
* **Field Mapping:** Map 5 required targets (`GLAccount`, `SiteCode`, `Debit`, `Credit`, `Description`) + 1 optional (`AccountName`). **Validate** button calls BL API. Next disabled until pass. On fail, surface BL's error + downloadable error report link.
* **Review:** FY, totals, effective date (from BL). User confirms.

**Restatement gate** fires on Field Mapping when BL flags the FY as having a **closed period** or a **prior OB import**. UI must require:

1. User **types the FY identifier** into a confirmation field — Continue disabled until exact match.
2. **Reason** free-text, ≥ 10 chars.

Reason + confirmation timestamp pass to BL on confirm.

**At confirm**, BL re-checks. If anything changed since Validate, block confirm and send the user back to re-satisfy the gate.

**Atomic.** All rows write or none. _No partial imports, ever._

**Saved mapping** auto-persists on success — auto-loads next time, but every import _still goes through Validate_. **FY is not persisted** — selected fresh per import. Add a **Validate** action on each row in import history.

**Roles with edit access:** Director, Central Office, SchoolCafe Admin. Customer/Technical Support always full.

**What NOT to build:**

* An amount format selector — _single canonical Debit/Credit contract_ (DE transforms signed-amount files upstream of mapping).
* Conditional fields — same 5 required targets every time.
* Any balancing or business validation in DE — BL owns it.

Import Monitor pipeline for Opening Balance Import: import type setup, file upload, prerequisites, fiscal year selection, column mapping, validation, preview, saved mapping, results. On confirm, validated data hands off to BL (companion **NXT-71240**) for ledger entry creation.

## Acceptance Criteria
**AC1 — Import Type Available.** When a Finance user views Import Types in Import Monitor, Opening Balance Import appears regardless of prerequisite state.

**AC2 — Pre-upload Tab: Fiscal Year Selector.** When the user selects Opening Balance Import, the Pre-upload tab shows a Fiscal Year dropdown listing all FYs. The user must pick one before uploading.

**AC3 — Validator Gate.** When the user clicks Validate on Field Mapping, BL runs all row-level (§7) and import-level (§8) checks. Next stays disabled until pass. On success: confirmation + row count.

**AC4 — Prerequisite Check (Chart of Accounts).** If the chart of accounts has no Asset, Liability, or Equity account, validation fails with a blocking message linking to Chart of Accounts. Review is unreachable until satisfied.

**AC5 — Restatement Gate Triggered.** When the selected FY has a closed period and/or a prior OB import, validation surfaces a confirmation gate requiring (a) typed FY identifier (exact match) and (b) Reason for restatement (≥10 chars). Continue disabled until both satisfied. Gate names the trigger(s) and links to the manual adjustment entry flow.

**AC6 — Restatement Audit Metadata Persisted.** When the gate is satisfied and the import confirmed, reason + timestamp persist on the import audit record and hand off to BL for entry audit metadata (NXT-71240 §1.1).

**AC7 — Required Fields Mapped.** On Field Mapping, GLAccount, SiteCode, Debit, Credit, and Description must be mapped. Blank Description is accepted at validation; auto-filled with _"Opening Balance Import"_ at import time.

**AC8 — Decimal Format Fail-Fast.** If any of the first 10 rows has non-numeric characters in Debit or Credit, validation fail-fasts with a wrong-file error. Review not reached.

**AC9 — Debit/Credit Row Validity.** Each row must have exactly one of Debit or Credit positive. Invalid: both populated, both blank/zero, or any negative.

**AC10 — Account Type Restriction Enforced.** Rows referencing Revenue or Expenditure GLAccounts are flagged invalid: _"Opening balances cannot be set for Revenue or Expenditure accounts."_

**AC11 — Unknown GLAccount or SiteCode Blocked.** Rows with a GLAccount not in the chart of accounts, or a SiteCode that doesn't match a SchoolCafe site, are flagged invalid (Import Monitor error pattern).

**AC12 — Duplicate GLAccount + SiteCode Blocked.** Multiple rows with the same (GLAccount, SiteCode) are flagged invalid: _"Duplicate entry for account [GLAccount] at site [SiteCode]. Each account may only appear once per site."_ User must consolidate to a single net balance per (account, site) before re-uploading.

**AC13 — Batch Must Balance (Strict Equality).** If total debits ≠ total credits by any amount, BL returns failure. DE surfaces BL's message as-is. No variance permitted.

**AC14 — Atomic Import.** On any system error during import processing, the entire import rolls back; no data is written; user is notified.

**AC15 — Downloadable Error Report on Validation Failure.** On validation failure with row-level errors, a download link appears for the BL-generated error report (rows + reasons).

**AC16 — FILE DATA PREVIEW on File Upload.** On File Upload, FILE DATA PREVIEW opens a collapsible panel with the first few parsed rows + all detected columns — raw data, no mapping. Field Mapping shows only the column mapping UI + Validate.

**AC17 — Review is Final Confirm Only.** Review shows the selected FY, totals (rows, debits, credits), and the auto-assigned effective date from BL. No validation errors here — caught upstream at the validator gate. User must explicitly confirm.

**AC18 — Mapping Auto-Loads but Re-Validates.** After a prior OB import, the column mapping auto-loads on Field Mapping. File still passes through the validator gate under the same §7 and §8 rules. FY is selected fresh per import — not persisted.

**AC19 — Validate Action on Import History Row.** Each prior import in history exposes a Validate action that re-runs validation against the original file under the same rules.

**AC20 — Confirm-Time Re-Verification of FY Eligibility.** If a period closes or another OB import lands on the FY between gate-satisfied and confirm, BL blocks confirm with an explanation. User must re-satisfy the gate (re-type confirmation, re-enter reason).

**AC21 — Results View Structure.** Import Results: Summary at top — success/failed, selected FY, total rows, total debits/credits, restatement reason + confirmation timestamp (when applicable), timestamp/user. Per-row processing details below, per Import Monitor pattern. No row-level validation errors here — caught at the validator gate (§8.1) upstream.

**AC22 — Multi-Sheet File Rejected at Upload.** Multi-sheet files are rejected at upload with an error directing the user to consolidate and re-upload. No sheet is processed.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| No Asset/Liability/Equity accounts in chart of accounts | Validation fails; blocking message + link to Chart of Accounts |
| Empty file or header-only | Validation fails: _"No data rows found."_ |
| Multi-sheet file uploaded | Rejected at upload; user directed to consolidate to a single sheet |
| Non-numeric characters in Debit or Credit (any of first 10 rows) | Fail-fast: rejects entire import; wrong-file error |
| Both Debit and Credit populated in same row | Row invalid: _"Exactly one of Debit or Credit must have a value."_ |
| Both Debit and Credit blank or zero in same row | Row invalid: _"Exactly one of Debit or Credit must have a positive value."_ |
| Negative value in Debit or Credit | Row invalid: _"Debit and Credit values must be positive."_ |
| Unknown GLAccount | Row invalid (Import Monitor error pattern) |
| Unknown SiteCode | Row invalid (Import Monitor error pattern) |
| Revenue or Expenditure GLAccount | Row invalid; explicit account-type error |
| Duplicate GLAccount + SiteCode in same file | Rows flagged invalid; user must consolidate before re-uploading |
| Selected FY has closed periods | Restatement gate triggered: typed FY confirmation + reason (≥10 chars). User must reopen affected periods through Period Management before posting. |
| Prior OB import exists for selected FY | Restatement gate triggered: typed FY confirmation + reason (≥10 chars). Existing entries hard-reversed and replaced on confirm (per NXT-71240 §5). |
| Period closed OR new prior import created between gate-satisfied and confirm | Confirm blocked; user re-satisfies the restatement gate |
| File contains mix of valid and invalid rows | Validation fails with downloadable error report; no partial import; user corrects and re-uploads |
| Total debits ≠ total credits | Validation fails; blocking message from BL surfaced as-is |
| System error during import processing | Entire import rolls back; no data written; user notified |

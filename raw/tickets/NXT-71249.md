# NXT-71249 — FIN — Opening Balance Import: Data Exchange

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-04._

---

## Scope

This story covers the Import Monitor pipeline for the Opening Balance Import type: import type setup, file upload, prerequisites check, amount format selection, column mapping, validation, preview, saved mapping, and import results. On successful confirmation, validated data is handed off to the business logic layer (companion story **NXT-71240**) for ledger entry creation.

---

## Requirements

### 1) Import Type

1.1 A new Import Monitor Import Type exists named **Opening Balance Import** with help text: _"Used to establish beginning account balances for Assets, Liabilities, and Equity at fiscal year start or system go-live."_

1.2 Import Type supports the following upload formats, consistent with other Import Monitor types: **xlsx, xls, csv**.

1.3 Import Type is available only to users with appropriate Financials permissions.

1.4 Files must contain a single sheet. Multi-sheet files are rejected at upload with a clear error directing the user to consolidate to a single sheet and re-upload. The system does not attempt to process any sheet from a multi-sheet file.

### 2) Prerequisites

2.1 The Opening Balance Import type is always visible in Import Monitor for authorized users. However, after file upload, the system must check that at least one Asset, Liability, or Equity account exists in the chart of accounts before proceeding to Amount Format Selection.

2.2 If the prerequisite is not met, the system must display a blocking message after file upload explaining what is missing and how to resolve it, with a link to Chart of Accounts configuration.

2.3 Example blocking message: _"Opening Balance Import requires a chart of accounts with at least one Asset, Liability, or Equity account. Please import your chart of accounts first."_ With a link to Configuration > General Ledger > GL Accounts.

### 3) Fiscal Year Selection

3.1 After file upload and prerequisite check, the user must select a **Fiscal Year** from a dropdown. The dropdown lists only fiscal years in which **no period has been closed**. Fiscal years with any closed period are excluded because establishing or changing an opening balance after close would retroactively invalidate already-closed financial reports.

3.2 All rows in the imported file are assigned to the selected Fiscal Year. This value is not mapped from the file.

3.3 Each fiscal year may have only one active opening balance import. If an import already exists for the selected fiscal year, the system must warn the user that proceeding will replace it (see Section 8.3).

3.4 Fiscal Year is **not** persisted as part of saved mapping templates. It must be selected fresh for each import.

3.5 On import confirmation, the system must re-verify that no period in the selected fiscal year has been closed since dropdown load. If any period has been closed in the interim, the confirm action must be blocked with an error directing the user to reopen the affected period(s) through standard period management before retrying. _(The business logic layer applies the same check as a backstop — see companion story **NXT-71240**.)_

### 4) Amount Format Selection

4.1 After Fiscal Year selection and before column mapping, the user must select an **Amount Format** from two options:

* _"My file has separate Debit and Credit columns"_ → Scheme A
* _"My file has a single Amount column (positive = debit, negative = credit)"_ → Scheme B

4.2 The selection determines which amount target fields appear in the column mapping UI. Scheme A displays Debit and Credit as required target fields. Scheme B displays Amount as the required target field. The non-selected fields must not appear in the mapping UI.

4.3 The Amount Format selection is persisted as part of saved mapping templates.

### 5) Required Target Fields

The import mapping UI must allow mapping uploaded columns to these target fields:

| Field | Notes |
| --- | --- |
| GLAccount | Must reference an Asset, Liability, or Equity account |
| SiteCode | Site the balance applies to. For district-wide entries, use the site code designated as the Central Office in Site Management. |
| Comments | Free text reference field; max 50 characters. If blank, the system auto-fills with _"Opening Balance Import"_. |

**Amount fields (determined by the Amount Format selection in Section 4):**

| Amount Format | Target Fields Shown | Convention |
| --- | --- | --- |
| Separate Debit and Credit columns | Debit, Credit | Separate positive values in each column |
| Single Amount column | Amount | Positive = debit, negative = credit |

### 6) Optional Target Fields

| Field | Notes |
| --- | --- |
| AccountName | Descriptive label for the account |

### 7) Validation Rules (row-level)

7.1 GLAccount, SiteCode, and Comments must be present in the column mapping. Amount fields must be present per the selected amount format. If a row's Comments value is blank, the system auto-fills with _"Opening Balance Import"_.

7.2 Amount validation must follow the selected amount format:

**If Scheme A (Debit/Credit):**

* Exactly one of Debit or Credit must be a positive number per row.
* Negative values are not allowed.

**If Scheme B (Amount):**

* Amount must be non-zero.
* Negative amounts are allowed (treated as credits by convention).

7.3 Zero values are not allowed in either scheme and will be flagged as invalid.

7.4 GLAccount must correspond to an Asset, Liability, or Equity account in the chart of accounts. Rows referencing Revenue or Expenditure accounts are invalid and blocked with the error: _"Opening balances cannot be set for Revenue or Expenditure accounts."_

7.5 GLAccount must exist in the chart of accounts. Unknown accounts are flagged as invalid with a clear error consistent with Import Monitor patterns.

7.6 SiteCode must reference a valid site in Site Management. Unknown site codes are flagged as invalid.

7.7 Each GLAccount + SiteCode combination must appear only once in the import file. Duplicate combinations are flagged as invalid with the error: _"Duplicate entry for account \[GLAccount\] at site \[SiteCode\]. Each account may only appear once per site."_ The user must consolidate to a single net balance per account per site in their file before uploading.

### 8) Validation Rules (import-level)

8.1 The system must calculate and display:

* Total rows
* Total debit-side amounts and total credit-side amounts (computed from the selected amount format)

8.2 Import must be blocked if total debits ≠ total credits. No variance is permitted — the batch must balance exactly. Totals must be displayed to the user alongside the blocking message.

8.3 If an opening balance import already exists for the selected fiscal year, the system must warn the user that proceeding will replace the existing opening balances and require explicit confirmation before proceeding. _(The reversal and re-creation of ledger entries is handled by the business logic layer — see companion story **NXT-71240**.)_

8.4 The import must be **atomic** — either all rows are imported successfully, or no rows are imported. Partial imports are not allowed. If any system-level error occurs during import processing, the entire import must roll back and the user must be notified that no data was written.

### 9) Preview and Confirm

9.1 After file upload, prerequisite check, Fiscal Year selection, amount format selection, and column mapping, the user must see a preview showing the selected Fiscal Year, totals, the auto-assigned effective date (provided by the business logic layer), a sample of parsed rows, and any validation errors before confirming.

9.2 The user must explicitly confirm to perform the import.

9.3 If any rows contain validation errors, the confirm action must be disabled. All row-level errors must be resolved (by correcting and re-uploading the file) before the import can proceed. The system must not allow the user to import only the valid rows.

9.4 The preview must clearly indicate which accounts will receive opening balances and the direction (debit/credit) of each.

### 10) Saved Mapping Behavior

10.1 The user can save a column mapping as a named template for the Opening Balance Import type. The saved template includes the selected amount format.

10.2 Future imports using the same template automatically apply the saved amount format and column mapping and proceed directly to preview.

10.3 The user can edit or update the saved mapping and amount format at any time.

### 11) Import Results and Auditability

11.1 Each import must produce an import result record consistent with Import Monitor patterns (status, counts, timestamps, user).

11.2 The user can view import results including:

* Success/failed status
* Total debits and credits
* Record counts
* Row-level error messages for invalid records

11.3 The import record must be visible in audit history so administrators can see when opening balances were set or replaced and by whom.

### 12) Handoff to Business Logic

12.1 On successful confirmation, the data exchange layer passes the validated import data to the business logic layer (**NXT-71240**), which is responsible for: assigning the effective date, creating the ledger entry and lines, handling hard reversal of any previous import, and making the data available to downstream reports.

## Acceptance Criteria
**AC1 — Import Type Available** Given an authorized Finance user opens Import Monitor, When they view Import Types, Then Opening Balance Import is available regardless of whether prerequisites are met.

**AC2 — Prerequisite Check Blocks Import When Not Met** Given the chart of accounts contains no Asset, Liability, or Equity accounts, When the user uploads a file for Opening Balance Import, Then the system displays a blocking message with a link to Chart of Accounts configuration, and does not proceed to Fiscal Year Selection.

**AC3 — Fiscal Year Selection Required** Given a user uploads a file for Opening Balance Import and prerequisites are met, When they proceed past file upload, Then they must select a Fiscal Year before reaching Amount Format Selection or column mapping.

**AC4 — Amount Format Selection Drives Mapping UI** Given a user selects "My file has separate Debit and Credit columns," When the column mapping UI loads, Then Debit and Credit appear as required target fields and Amount does not appear. Given a user selects "My file has a single Amount column," When the column mapping UI loads, Then Amount appears as the required target field and Debit/Credit do not appear.

**AC5 — Account Type Restriction Enforced** Given an upload contains rows referencing Revenue or Expenditure GLAccounts, When preview runs, Then those rows are flagged invalid with an error stating opening balances cannot be set for those account types.

**AC6 — Required Fields Enforced** Given an upload contains rows missing GLAccount, SiteCode, or the mapped amount field(s), When preview runs, Then those rows are flagged invalid and cannot be imported until corrected. Comments must be mapped but blank values are accepted and auto-filled with the default placeholder.

**AC7 — Zero Amount Blocked (Scheme B) / Both Debit and Credit Populated (Scheme A)** Given an upload using Scheme B contains rows with Amount = 0, or an upload using Scheme A contains rows where both Debit and Credit are populated or both are zero, When preview runs, Then those rows are flagged invalid with an error indicating the amount entry is not valid.

**AC8 — Batch Must Balance (strict equality)** Given total debit-side amounts ≠ total credit-side amounts by any amount, When the user attempts import, Then import is blocked and totals are displayed with the imbalance amount.

**AC9 — Row Errors Block Entire Import** Given an upload contains one or more rows with validation errors, When the user views the preview, Then the confirm action is disabled and the system must not allow importing only the valid rows. All errors must be resolved by correcting and re-uploading the file.

**AC10 — Import Is Atomic** Given the user confirms an import and a system-level error occurs during processing, When the error is encountered, Then the entire import rolls back, no opening balance data is written, and the user is notified that the import failed.

**AC11 — Duplicate GLAccount + SiteCode Blocked** Given an upload contains more than one row for the same GLAccount and SiteCode combination, When preview runs, Then those rows are flagged invalid with an error identifying the duplicate, and the import cannot proceed until the user consolidates to one row per account per site.

**AC12 — Existing Import Replacement Warning** Given an opening balance import already exists for the selected fiscal year, When the user attempts to confirm a new import, Then the system warns that existing opening balances will be replaced and requires explicit confirmation before proceeding. _(Hard reversal of previous entries is handled by the business logic layer.)_

**AC13 — Saved Mapping Does Not Persist Fiscal Year** Given a saved mapping template exists, When the user uploads a new file with the same layout, Then the amount format and column mapping are auto-applied, but the user must still select a Fiscal Year before proceeding to preview.

**AC14 — Results Visible** Given an import completes, When the user opens import results, Then they can see status, the selected Fiscal Year, record counts, totals, and row-level errors if any.

**AC15 — SiteCode Validated Against Site Management** Given an upload contains a SiteCode that does not exist in Site Management, When preview runs, Then that row is flagged invalid with a clear error consistent with Import Monitor patterns.

**AC16 — Closed-Period Fiscal Years Excluded from Dropdown** Given one or more fiscal years in the system have any period in a closed state, When the user reaches Fiscal Year Selection, Then those fiscal years are not available in the dropdown.

**AC17 — Closed-Period FY Blocked at Confirm** Given the user selected a fiscal year at dropdown load, and a period in that fiscal year has since been closed, When the user confirms the import, Then the system blocks the import with an error directing the user to reopen the affected period(s) through standard period management before retrying. _(The business logic layer enforces the same rule as a backstop.)_

## Edge / Empty / Error States

| Scenario | Behavior |
| No Asset/Liability/Equity accounts in chart of accounts | Import blocked after file upload with message and link to GL Accounts configuration |
| Empty file or header-only | Import blocked: _"No data rows found."_ |
| Non-numeric Amount field | Row invalid with parsing error |
| Both Debit and Credit populated in same row (Scheme A) | Row invalid: _"Exactly one of Debit or Credit must have a value."_ |
| Both Debit and Credit are zero in same row (Scheme A) | Row invalid: _"Exactly one of Debit or Credit must have a positive value."_ |
| Negative value in Debit or Credit column (Scheme A) | Row invalid: _"Debit and Credit values must be positive."_ |
| Unknown GLAccount | Row invalid with clear error consistent with Import Monitor patterns |
| Unknown SiteCode | Row invalid with clear error consistent with Import Monitor patterns |
| Multiple FiscalYear values in one file | Import blocked; opening balance import must target a single fiscal year |
| FY with a closed period is selected (detected at confirm) | Import blocked with error directing user to reopen the affected period(s) through period management before retrying |
| Revenue or Expenditure GLAccount | Row invalid with explicit account type error |
| Duplicate GLAccount + SiteCode in same file | Rows flagged invalid: _"Duplicate entry for account [GLAccount] at site [SiteCode]."_ User must consolidate to one row per account per site. |
| Re-import for a fiscal year that already has opening balances | System warns user, requires explicit confirmation before proceeding |
| File contains mix of valid and invalid rows | Confirm action disabled; all errors must be resolved before import. No partial import allowed. |
| System error during import processing | Entire import rolls back; no data written; user notified of failure |

## Defaults

- **Fiscal Year:** Selected at the import level after file upload. Not mapped from the file. Not saved as part of mapping templates. Selected fresh for each import.
- **Amount format selection:** The user selects their amount format after Fiscal Year selection and before column mapping. The import UI must surface the appropriate guidance text based on the selection. For Scheme B: _"Positive values = debit, negative values = credit."_ For Scheme A: _"Enter a positive value in exactly one column per row."_
- For district-wide entries, the user should use the site code designated as the Central Office in their Site Management configuration. The import UI guidance text should read: _"For district-wide entries, use the site code assigned to your Central Office in Site Management."_
- **Comments default:** Comments is a required field in the column mapping. If a row's value is blank, the system auto-fills with _"Opening Balance Import"_. Every ledger entry must carry a description.
- AccountName defaults to blank/null when not mapped.

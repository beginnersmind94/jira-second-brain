# NXT-71249 — FIN — Opening Balance Import: Data Exchange

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-04._

---

## Scope

This story covers the Import Monitor pipeline for the Opening Balance Import type: import type setup, file upload, prerequisites check, fiscal year selection, column mapping, validation, preview, saved mapping, and import results. On successful confirmation, validated data is handed off to the business logic layer (companion story **NXT-71240**) for ledger entry creation.

3.6 All rows are assigned to the selected FY. Not mapped from the file.

3.7 Each FY may have only one active opening balance import. The Restatement required gate handles replacement acknowledgment.

3.8 FY is selected fresh per import. Not part of saved templates (which only persist column mapping).

3.9 At final confirm, BL is re-checked. If eligibility has changed since the gate was satisfied, confirm is blocked and the user re-satisfies the gate.

## Acceptance Criteria
**AC1 — Import Type Available.** Given an authorized Finance user opens Import Monitor, When they view Import Types, Then Opening Balance Import is available regardless of whether prerequisites are met.

**AC2 — Pre-upload Tab: Fiscal Year Selector.** Given the user selects Opening Balance Import, When the Pre-upload tab loads, Then a Fiscal Year dropdown lists all FYs in the system. The user must pick an FY before uploading the file.

**AC3 — Validator Gate.** Given the user uploads a file on the File Upload tab, When they click Validate, Then the system runs all row-level (§7) and import-level (§8) checks. The Next button to proceed to Field Mapping is disabled until validation passes. On success, the user sees a confirmation message with totals.

**AC4 — Prerequisite Check (Chart of Accounts).** Given the chart of accounts contains no Asset, Liability, or Equity accounts, When validation runs, Then validation fails with a blocking message linking to Chart of Accounts configuration. Field Mapping cannot be reached until the prerequisite is satisfied.

**AC5 — Restatement Gate Triggered.** Given the selected FY has at least one closed period OR a prior opening balance import exists for it (or both), When validation runs, Then a confirmation gate appears requiring (a) the user to type the FY identifier exactly into a confirmation field, and (b) a free-text Reason for restatement of at least 10 characters. The Continue action is disabled until both are satisfied. The gate states which condition(s) triggered the restatement and surfaces a routing hint to the manual adjustment entry flow for users who only need to correct a single account balance.

**AC6 — Restatement Audit Metadata Persisted.** Given the user satisfies the restatement gate, When the import is confirmed, Then the reason text and confirmation timestamp are persisted on the import audit record and handed off to BL for persistence on the ledger entry's audit metadata (per NXT-71240 §1.1).

**AC7 — Required Fields Mapped.** Given an upload reaches Field Mapping, When the user views the mapping UI, Then GLAccount, SiteCode, Debit, Credit, and Description must be mapped. Blank Description values are accepted at validation time and auto-filled with _"Opening Balance Import"_ during import.

**AC8 — Decimal Format Fail-Fast.** Given an upload contains non-numeric characters in the Debit or Credit columns in any of the first 10 rows, When validation runs, Then validation fails fast with an error indicating the user likely uploaded the wrong file. Field Mapping is not reached.

**AC9 — Debit/Credit Row Validity.** Given a row in the uploaded file, When validation runs, Then exactly one of Debit or Credit must contain a positive number. The following are flagged invalid: both Debit and Credit populated, both blank or zero, or any negative value in either column.

**AC10 — Account Type Restriction Enforced.** Given an upload contains rows referencing Revenue or Expenditure GLAccounts, When validation runs, Then those rows are flagged invalid with the error: _"Opening balances cannot be set for Revenue or Expenditure accounts."_

**AC11 — Unknown GLAccount or SiteCode Blocked.** Given an upload contains a GLAccount that does not exist in the chart of accounts, OR a SiteCode that does not reference a valid site as configured in SchoolCafe, When validation runs, Then those rows are flagged invalid with a clear error consistent with Import Monitor patterns.

**AC12 — Duplicate GLAccount + SiteCode Blocked.** Given an upload contains more than one row for the same GLAccount + SiteCode combination, When validation runs, Then those rows are flagged invalid with the error: _"Duplicate entry for account [GLAccount] at site [SiteCode]. Each account may only appear once per site."_ The user must consolidate to a single net balance per account per site before re-uploading.

**AC13 — Batch Must Balance (Strict Equality).** Given total debits ≠ total credits by any amount, When validation runs, Then validation fails. Totals (total rows, total debits, total credits) and the imbalance amount are displayed alongside the blocking message. No variance is permitted.

**AC14 — Atomic Import.** Given the user confirms an import and a system-level error occurs during processing, When the error is encountered, Then the entire import rolls back, no opening balance data is written, and the user is notified that the import failed.

**AC15 — Downloadable Error Report on Validation Failure.** Given validation fails with one or more row-level errors, When the user views the failure message, Then a downloadable error report is available listing the affected rows and reasons so the user can correct the file and re-upload.

**AC16 — Sample Rows on Field Mapping.** Given validation has passed and the user reaches the Field Mapping tab, When the screen loads, Then the first few parsed rows are displayed alongside the column mapping UI, showing GLAccount, SiteCode, Debit, Credit, and Description as parsed — including which side (debit or credit) each row will post to.

**AC17 — Review is Final Confirm Only.** Given the user reaches the Review tab, When the screen loads, Then they see the selected Fiscal Year, totals (rows, total debits, total credits), and the auto-assigned effective date provided by BL. No validation errors appear at Review (caught upstream at the validator gate). The user must explicitly confirm to perform the import.

**AC18 — Saved Mapping Auto-Applies but Re-Validates.** Given a saved mapping template exists, When the user uploads a new file using that template, Then the column mapping is auto-applied, but the file must still pass through the validator gate using the same row-level (§7) and import-level (§8) rules as the original import. Fiscal Year is not persisted on saved templates and must be selected fresh for each import.

**AC19 — Validate Action on Import History Row.** Given a prior import exists in Import Monitor history, When the user views the history view, Then a Validate action is available on each row to re-run validation against the original file using the same rules.

**AC20 — Confirm-Time Re-Verification of FY Eligibility.** Given the user satisfied the restatement gate at validation time, and a period in the selected FY has been closed OR another opening balance import has been completed for the FY since then, When the user attempts to confirm, Then confirm is blocked with an error explaining what changed. The user must re-satisfy the restatement gate (re-type confirmation, re-enter reason).

**AC21 — Results View Structure.** Given an import completes, When the user opens import results, Then a Summary section at the top of the view shows: success/failed status, selected Fiscal Year, total rows, total debits and credits, restatement reason and confirmation timestamp (when applicable), and timestamp/user. Per-row processing details appear below the summary, consistent with Import Monitor patterns. Row-level validation errors do not appear here — those are caught at the validator gate (§8.1) before the import runs.

**AC22 — Multi-Sheet File Rejected at Upload.** Given an upload contains more than one sheet, When the file is uploaded, Then the file is rejected at upload with a clear error directing the user to consolidate to a single sheet and re-upload. The system does not attempt to process any sheet.

| Unknown SiteCode | Row invalid with clear error consistent with Import Monitor patterns |

## Defaults

- **Description default:** Description is a required mapped field. If a row's value is blank, the system auto-fills with _"Opening Balance Import"_ at import time (not validation time). Every ledger entry must carry a description.
- **AccountName:** Optional target field. Defaults to blank/null when not mapped.
- **District-wide entries:** For district-wide entries, the user should use the site code designated as the Central Office in their Site Management configuration. Import UI guidance text: _"For district-wide entries, use the site code assigned to your Central Office in Site Management."_
- **Effective date:** Auto-assigned by BL on the resulting ledger entry — not selected by the user, not in the file (per NXT-71240).

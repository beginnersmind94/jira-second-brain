# NXT-60117 — WS - Revenue Details

## Description
As a Workspace user, I want to view detailed KPI information so that I can understand performance metrics in depth.

h2. Comments

*Component*: KPI Details Panel

The panel shows:

* Name
* Date 
* Summary KPIs
* Detailed metric analysis
* Historical performance
* Contributing factors (optionally)
* School breakdowns (optionally)
* Action recommendations (optionally)

h2. User Flow

# User logs into Workspace and goes to Insights page
# User clicks on KPI card
# Panel opens

h2. Requirements

h3. Display Elements

_Header Section:_

* Icon
* KPI name
** Text of the name of the KPI
* Time period
** Date from the dashboard
* Close button



_Summary KPIs Section:_

* Shows the summary KPIs from the dashboard
* Section Name: Revenue Overview
* Fields:
* Actual Revenue
** Same as dashboard card
* Expected Revenue
** Same as dashboard card
* Difference
** Calculate: Expected - Actual



_Analysis Section:_

* Meals Section
** Grid
** Columns:
*** Category
*** Revenue
** Summary row at the bottom
** No paging, sorting, filtering
* 
* A La Carte Section:
** Grid
** Columns:
*** Category
*** Revenue
** Summary row at the bottom
** No paging, sorting, filtering



_Summary Section_

* About This Metric:
** Explanation text explaining the metric. Get the text from Bolt

h3. Interactions

* Action Items:
** None, read-only

h3. Data Requirements

* APIs:
* Summary section
** API for historical and actual
** Use same as dashboard
* Meals section
** New API for sum of revenue by category in the table
* A La Carte
** New API for sum of revenue by category in the table
* All communications between the drawer and APIs occur over HTTPS
* The system ensures that data is retrieved and processed securely and API endpoints implement rate limiting
* The drawer interacts with versioned API endpoints (e.g., {{/api/v1/...}}) to ensure future compatibility and backward support

h3. Performance & Usability

* Ensure the drawer displays properly in these layout options: Full width, Compact, Mobile responsive, Expanded
* The drawer should load within 2 seconds under normal operating conditions and opening the drawer is smooth and responsive

h3. Security

* Inherits dashboard page’s view permissions, no need to set permissions on the cards directly

h3. Error handling

* Use Error Boundaries and Try/Catch in Asynchronous Code and log errors to the database
* Show User-Friendly Error Messages in modal dialogs, toast notifications, or inline messages without disrupting the entire experience

## Acceptance Criteria
h4. *Display Elements*

h5. *Header Section*

* Displays an icon representing the KPI.
* Displays the KPI name as a text label.
* Displays the time period (retrieved from the dashboard).
* Includes a close button that allows the user to exit the drawer.

h5. *Summary KPIs Section (Revenue Overview)*

* Displays Actual Revenue (same value as the dashboard KPI card).
* Displays Expected Revenue (same value as the dashboard KPI card).
* Displays Difference (calculated as Expected - Actual Revenue).

h5. *Analysis Section*

* *Meals Section*
** Displays a grid with the following columns:
*** Category
*** Revenue
** Includes a summary row at the bottom.
** No paging, sorting, or filtering is applied.
* *A La Carte Section*
** Displays a grid with the following columns:
*** Category
*** Revenue
** Includes a summary row at the bottom.
** No paging, sorting, or filtering is applied.

h5. *Summary Section*

* Displays an "About This Metric" section that explains the KPI.
* The text is retrieved from Bolt.

----

h3. *Interactions*

* The KPI details drawer is read-only (no action items).
* The close button exits the drawer.

----

h3. *Data Requirements*

* *APIs*
** Summary Section: Uses the same API as the dashboard for retrieving historical and actual revenue.
** Meals Section: Uses a new API to retrieve the sum of revenue by category.
** A La Carte Section: Uses a new API to retrieve the sum of revenue by category.
* *Security & API Standards*
** All API communications must occur over HTTPS.
** The system ensures secure data retrieval and processing.
** API endpoints must implement rate limiting to prevent abuse.
** The drawer interacts with versioned API endpoints (e.g., {{/api/v1/...}}) to ensure future compatibility and backward support.

----

h3. *Performance & Usability*

* Ensure the drawer displays correctly in the following layouts: Full width, Compact, Mobile responsive, Expanded
* The drawer must load within 2 seconds under normal operating conditions.
* Opening the drawer must be smooth and responsive.

----

h3. *Security*

* The KPI details drawer inherits the dashboard’s view permissions.
* No additional permissions need to be set on individual KPI cards.

----

h3. *Error Handling*

* Error Boundaries must be implemented to capture UI failures.
* Try/Catch blocks must be used for all asynchronous API calls.
* User-Friendly Error Messages
** Errors must be displayed using modal dialogs, toast notifications, or inline messages.
* Database Logging
** Errors must be logged into the database for tracking and troubleshooting.

# NXT-71543 — SC - MT - District - Cross-Realm Access Resolution

## Description
As a *user from a linked parent realm*, I want the system to resolve my granted access when I enter a child district, so that I receive the correct child-district roles and permissions based on the boundaries I qualify for and the child’s role mappings.

----

h1. Background / Notes

The platform already supports:

* Users belonging to a single *home realm*
* Users accessing multiple *child districts*
* A *district redirect / switch flow*

What does not yet exist is a complete authorization model for:

* determining which *boundaries* a parent user qualifies for in a child district
* resolving which *child-local roles* correspond to those boundaries
* granting the correct *child permissions* during district entry

This story defines the *runtime propagation and resolution logic*.

----

h2. Key Design Principle

* Users remain members of their *home realm only*
* No child-local user accounts are created
* Access is resolved *at runtime when entering a child district*
* Authorization is based on:
** parent role → boundary eligibility
** child boundary → child role mapping

----

h1. User Flow / System Flow

h2. User Flow

# User logs in to their *home realm*
# User navigates to *System > Districts > General*
# User sees a restricted list of districts they can access
# User clicks the *redirect/access* button for a child district
# System initiates child-district entry flow
# System resolves access using:
#* parent roles
#* boundary grants
#* child role mappings
# System builds a child-context session/token
# User is redirected into the child district homepage
# User can only see/do what is allowed by resolved child roles

----

h2. System Flow

# Receive request to enter child district
# Identify:
#* user id
#* home realm id
#* target child realm id
# Validate parent-child relationship
# Retrieve user’s parent-role memberships
# Resolve boundaries granted from child to parent
# Determine which boundaries the user qualifies for
# Resolve child roles linked to those boundaries
# Union roles and permissions
# Build child-context session/token
# Redirect user into child district

----

h3. Child Side:

----

h2. Runtime Access Flow

# Parent user logs into home realm
# User selects child district
# System begins access resolution
# System retrieves:
#* parent roles
#* linked boundaries
# System determines:
#* which boundaries the user qualifies for
# System retrieves:
#* child roles mapped to those boundaries
# System resolves:
#* effective child roles
#* effective permissions
# System builds child-context session/token
# User enters child district
# User operates using *child-local permissions*

----

h1. Technical / Implementation Notes

* Do not create child user accounts
* Resolve access at runtime only
* Use existing:
** parent role mappings
** child role mappings
* Ignore inactive data:
** boundaries
** roles
** relationships
* Ensure efficient queries or caching
* Ensure consistency with token/session lifecycle
* Audit resolution if supported

h1. Functional Requirements

h2. 1. Trigger resolution on district entry

When a user enters a child district, the system must perform access resolution before granting access.

----

h2. 2. Identify user and realms

System must identify:

* user id
* home realm
* target child district

----

h2. 3. Validate relationship

Access allowed only if:

* parent and child are linked
* relationship is active

----

h2. 4. Retrieve parent roles

System retrieves:

* active parent-role memberships

----

h2. 5. Resolve qualified boundaries

System determines:

* which boundaries are granted
* which boundaries the user qualifies for

Based on:

* parent roles
* boundary mappings

----

h2. 6. Filter active records

Only consider:

* active boundaries
* active mappings
* active roles
* active relationships

----

h2. 7. Resolve child roles

System retrieves:

* child-local roles linked to qualified boundaries

----

h2. 8. Union roles

* Combine all roles
* Remove duplicates

----

h2. 9. Resolve permissions

* Gather permissions from child roles

----

h2. 10. Build child-context session/token

Token must include:

* user identity
* home realm
* acting child realm
* effective child roles
* permissions

----

h2. 11. No child user creation

System must not:

* create child user accounts
* persist user-role assignments in child

----

h2. 12. Require valid access path

Access granted only if:

* at least one boundary qualifies
* at least one child role exists

----

h2. 13. Support “not ready” state

If:

* boundary exists
* but no roles/mappings exist

Then:

* access is not granted

----

h2. 14. Respect correct parent-child pairing

If multiple parents exist:

* only use mappings for the user’s home realm

----

h2. 15. Re-resolve on each entry

Access must be recalculated:

* every time user enters child district

----

h2. 16. Session refresh behavior

System must define:

* when changes take effect
(next login, token refresh, etc.)

----

h2. 17. Exclude inactive data

Ignore:

* inactive boundaries
* inactive roles
* inactive mappings

----

h2. 18. Align with district access list

If user:

* no longer qualifies

Then:

* block entry OR remove district from list

----

h2. 19. Audit access resolution

Log:

* user
* parent realm
* child realm
* boundaries
* roles
* outcome

----

h2. 20. Future extensibility

Design must support:

* future role inheritance
* future propagation enhancements

----

h1. Runtime Resolution Rule

When a parent user enters a child district:

# Retrieve parent roles
# Resolve qualified boundaries
# Resolve child roles mapped to those boundaries
# Union roles and permissions
# Build child-context session

h1. 

h2. Relationship Validation

----

h2. Boundary Qualification

* Inactive data ignored

----

* Child roles resolved from boundaries
* Multiple boundaries → union of roles

----

* No child user account required

----

h2. Access Control

* Missing mappings → no access

----

h2. Freshness

* Access reflects latest configuration on entry

----

h2. Audit

* Access resolution events logged

# NXT-65123 — Fin - Journal view (UI)

## Description
As a user,
When I toggle to journal view, I want to view journal entries by site, date, and account details,
So that I can track financial transactions and verify posting accuracy for audits and reconciliation.

*Dev Notes:*

* This sprint covers MANUAL entries only
* System-generated entries (automated postings from POS/Inventory modules) will be implemented later

## Acceptance Criteria
*Page Header:*

* Page title displays "View Journal"

*Help/Info Section:*

* Collapsible info panel at top with header "What does this page show? (Click to expand)"
* When expanded, shows two sections side-by-side:

*Left section - "What is a Journal?"*

* Header with pencil emoji: "📝 What is a Journal?"
* Text: "Think of a journal like a daily diary of all money activity. It shows EVERY transaction that happened on specific dates, across ALL accounts and schools, in the order they occurred."
* Example text in italics: "Example: 'On June 15th: Lincoln High had lunch sales, Kennedy Middle got food delivery, District Office paid utilities' - all mixed together chronologically."

*Right section - "Journal vs Ledger - Quick Difference"*

* Header with pencil emoji: "📝 Journal = Daily Diary"
* Text: "ALL transactions by date: 'What happened June 15th across the entire district?'"
* Header with bar chart emoji: "📊 Ledger = Account History"
* Text: "ONE account over time: 'How did Student Prepaid Accounts change this month?'"

*Quick Tip:*

* Light bulb icon with text: "💡 Quick tip: Use filters to narrow down to specific dates or schools. Click any journal entry to see the detailed breakdown."

*Filter Options:*

* Searchable dropdown labeled "Location" with options:
** "-- ALL --" (default, shows all transactions)
** "District" (shows only district-level transactions)
** Individual sites (e.g., "001 Lincoln High School", "002 Kennedy Middle School")
*** Enable multi-select
** User can type to search/filter locations
* Date range picker with two fields:
** "From (Posted Date)" date field (format: MM/DD/YYYY)
** "To (Posted Date)" date field (format: MM/DD/YYYY)
** This refers to the post date
* Users can filter to see only reversed transactions (displays reversed and reversal pair)
* Purple "Apply" button to execute filters

*Paginated Journal List:*

* Journal entries are displayed with pagination controls at bottom (Previous / Page X of Y / Next)
* Each page loads a limited number of journal entries of 10 per page
* Pagination maintains the current filter state across pages
* User can navigate between pages without resetting filters

*Journal List Display:*

* Table with columns (left to right):
** Expandable arrow (▶)
** Posted Date (format: MM/DD/YYYY)
** Transaction ID (blue hyperlink, numeric format like "10001")
** Location (text, e.g., "001 Lincoln High School" or "District")
** Description (text - this comes from the user-entered description when creating manual entry)
** Entered Date (format: MM/DD/YYYY)
** User Name (text)
** Total Amount (currency format with $ and commas, e.g., "$450.00")
** Reverse icon (↻ symbol)

*Expandable Row:*

* Clicking arrow or anywhere on row expands to show line-level details
* Expanded section has light purple/lavender background
* Shows table with columns:
** Location
** Account Code (alphanumeric, e.g., "A-4100.1547")
** Account Description (text, e.g., "Purchased Food" - this comes from the GL account master data, NOT user-entered description)
** Debit (currency format, shows "-" if no value)
** Credit (currency format, shows "-" if no value)

*Reversal Display:*

* Original transactions that have been reversed show yellow badge with text "REVERSED" next to description
* Reversal transactions show yellow badge with text "REVERSED" next to description
* Reversal transaction descriptions are prefixed with "REVERSAL: " followed by user-entered reason

*Data Accuracy:*

* Sum of debit amounts in expanded view equals sum of credit amounts
* Total in summary row matches the net of debits/credits

*Reverse Function:*

* Clicking reverse icon (↻) opens side drawer from right
* Side drawer title: "Reverse Transaction"
* Side drawer shows original transaction details:
** Transaction ID
** Posted Date
** Location
** Description
** Total Amount
** Debit/Credit breakdown
* Side drawer has required text field labeled "Reason for reversal" (cannot be empty)
* Side drawer has two buttons: "Cancel" (gray) and "Reverse Transaction" (purple/primary)
* After clicking "Reverse Transaction", side drawer closes and page refreshes showing both original (now with REVERSED badge) and new reversal transaction

*Sorting & Filter Behavior*

* *Default sort*
** Grid is *sorted by Posted Date in descending order (newest first)* by default.
** Applied on initial load and whenever filters are cleared.
* *Sortable columns*
** User can change sort by clicking any sortable column header (e.g., Posted Date, Entered Date, Transaction ID, Username).
** Sorting persists when navigating between pages until the user changes it or clears filters.
* *Searchable / filterable fields*
** The grid supports column-level filters for:
*** *Entered Date* (date picker / range)
*** *Posted Date* (date picker / range)
*** *Transaction ID* (exact or “starts with” text search)
*** *Username / Entered By* (text search, case-insensitive, “contains” match)
** Filters can be combined; results reflect the *intersection* of all active filters.
** When filters change, the current sort order (e.g., Posted Date ↓) is preserved unless explicitly changed by the user.

*Permissions & Visibility:*

* Only users with appropriate permissions can see the reverse icon (↻)
* Users without reverse permission do not see the icon at all
* User roles determine visibility of financial data (e.g., district-wide vs. location-specific access based on assigned locations)

*Empty State Handling:*

* If no journal entries match the filters, display message: "No results found"
* Message appears centered in the table area where results would normally display

*Action Button:*

* Purple "Generate Report" button in bottom right corner (for exporting journal data)
** Functionality of generate report button is out of scope for this story

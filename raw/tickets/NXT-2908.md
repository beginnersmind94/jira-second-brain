# NXT-2908 — Authentication - Set Login Context

## Description
As a user I want to ability to choose my school district from a list before I sign in to ensure I’m able to use my specific account for access.

Beyond Username and Password, we need the ability to set the initial school district. This allows us to retain “duplicate” usernames for various school districts. Meaning, johnsmith can be a username across several districts and the only way to authenticate that user to a district is to first select the school district. The benefit of requiring context selection before the username/passwod is that we can apply custom branding. From a design perspective, we may ask the user (browser) before the login controls even appear “Select your school district”. If the district is selected, it loads any branding (if available), but then loads the username and password fields for the user to sign in. It’s a very small workflow of Context > Login Controls > Authentication and navigate to Workspace.

Note: we previously had the idea of context within Classic in the form of the Realm selection. 

Consider SchoolCafe for a similar idea (just the district context selection). SchoolCafe also allows users to get to their specific district using a particular URL, i.e. [schoolcafe.com/PrimeroISD|http://schoolcafe.com/PrimeroISD] would immediately take the user to the Primero ISD context. Video example: [https://www.loom.com/share/f059469772ee467e8be596ca396371e8|https://www.loom.com/share/f059469772ee467e8be596ca396371e8]

## Acceptance Criteria
1. User is required to set the context before authentication occurs on username and password
2. Once Context is set on a browser instance, it should retain unless it is changed (this avoids the constant need to set it each time)
3. User should be able to modify the context
4. Need the ability to have a "Central" entry for context that allows users with access to the central realm

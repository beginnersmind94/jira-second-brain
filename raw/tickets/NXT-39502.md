# NXT-39502 — Adult & Service Employee Import – Handling Optional Exceptions

## Description
As a SchoolCafé admin, I want to be able to import the record if an optional field doesn’t validate so that I can get started with using the system.

*Flow:*

* Open SchoolCafé and browse to Settings > Imports & Exports > Imports
* Find a Adult Import you want to use
* Click on Import for that row
* Follow the import process
* When the import routine comes across a row that fails because an optional field causes it to fail, update the field, log it as an exception and keep on importing the rest of the records
* Notify the user when the import finishes

 

*Notes:*

* This story corresponds to how we handle exceptions of optional fields when importing in Student Import process
* We want to do the exact same thing for Adult Import and for the Service Employee Import

 

*Requirements*:

* When click on Import button for a Adult Import:
* When running, if there is a record that fails because of an optional field, do the following:
** Currently it causes the import to fail – don’t do this
** If it fails because the text is too long, trim it to its max length
** Else if it fails because of any other reason, empty the field
** Record if you trim or remove text in the history log for that record
** Continue processing the remaining records
* 
* For each field, put a required length validator per the database schema
** Probably the same as is for Student, but verify
* 
* When viewing the History log from Import Monitor, ensure that these exceptions show in the Exception tab as Optional
* 
* For PhoneNumber, validate like this:
** Do exactly as Student Import
** Strip out anything that is not a number, including spaces, (, ), -
** Trim to only 10 digits
* If the field had more than 10 digits after stripping everything out, Log as an Optional exception like you do for Student Import
** Treat it the way we do exceptions for optional fields by marking as Optional exception and still importing the row
** This exception will show up in the exceptions table
* If the field has less than 10 digits after trimming
** Then make it blank as it is a malformed phone number
** Treat it the way we do exceptions for optional fields by marking as Optional exception and still importing the row
** This exception will show up in the exceptions table
* 
* For the Email field:
* Do like for Student Import
* Put validators on the import routine that do the following:
** Require a @ character
** Require a . character
* If either the @ or . are missing:
** Make the field empty since they do not have a valid email address format
** Log as an Optional exception so the user can know we modified their data
** Treat it the way we do exceptions for optional fields by marking as Optional exception and still importing the row
** This exception will show up in the exceptions table

## Acceptance Criteria
* When importing, if there is a record that fails because of an optional field, the import routine repairs the field and continues
* Log imports that are trimmed or emptied are marked as exceptions and visible in the Exceptions tab

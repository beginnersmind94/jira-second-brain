# NXT-72753 — Payroll Import Duplicate Detection and Overwrite Previous Import

## Description
## Description

As a district finance user, I want the system to detect when a Payroll Import may duplicate a previous Payroll Import, so that I can choose whether to keep the previous import or overwrite it without losing the audit trail.

## Scope

This story covers Payroll Import duplicate detection and overwrite behavior after the base Payroll Import Commit flow exists.

This story includes:

* duplicate detection by PostedDate
* Payroll Import Batch ID lookup
* duplicate exception rows in Import Monitor > Exceptions
* user selection for Overwrite previous Payroll Import
* overwrite behavior at the batch level
* reversal of prior Payroll Import batches
* posting the new Payroll Import batch
* complete write or rollback when overwrite is selected

This story does not cover base Payroll Import row validation, period validation, balance validation, journal entry creation, or report contribution. Those are covered in the base Payroll Import Commit story.

## Duplicate Detection

Each successful Payroll Import is stored with a system-generated Payroll Import Batch ID. All journal entries and lines created by that import are linked to the same batch ID.

Duplicate detection is triggered by PostedDate, not only `(FY, Period)`, because a district may import multiple payroll files within the same fiscal period.

For each source row in the new file, BL checks whether prior successful Payroll Import entries exist with the same PostedDate.

If a matching PostedDate is found, BL identifies the prior Payroll Import Batch ID that created the existing entries.

If matching prior Payroll Import entries exist and no overwrite selection is provided, BL rejects the import and returns one Mandatory DuplicateImport exception row for each source row whose PostedDate matched a prior Payroll Import batch.

Exception output when overwrite selection is missing:

* Column Name: PostedDate
* Type: Mandatory
* Description: `Source row {RowNumber}: Payroll Import Batch {BatchId} already contains entries for PostedDate {PostedDate}. Select whether to overwrite the previous Payroll Import before continuing.`

## Overwrite Previous Payroll Import Selection

When duplicate PostedDate rows are detected, the user must choose whether to overwrite the previous Payroll Import.

Overwrite previous Payroll Import behavior:

* No: keep existing Payroll Import batches and post the new batch alongside them.
* Yes: reverse all journal entries and lines linked to each matched prior Payroll Import Batch ID, then post the new batch.

## Batch-Level Overwrite Rule

Overwrite is batch-level.

BL must not reverse only the entries or rows with the duplicated PostedDate.

If a prior batch is selected for overwrite, every transaction linked to that Payroll Import Batch ID is reversed.

Overwrite never deletes prior ledger entries. It creates reversal entries for all transactions linked to the matched prior Payroll Import Batch ID, then posts the new entries under a new Payroll Import Batch ID.

The reversal entries and new entries must save together. If any part fails, BL rolls back the whole Commit.

## Acceptance Criteria
## AC1 — Duplicate detection checks PostedDate against prior Payroll Import entries

Given a Payroll Import file is submitted
When BL checks for duplicate Payroll Import activity
Then BL checks each source row’s PostedDate against prior successful Payroll Import entries
And duplicate detection is not based only on `(FY, Period)`.

## AC2 — Prior Payroll Import Batch ID is identified for matching PostedDate

Given a source row’s PostedDate matches prior successful Payroll Import entries
When BL detects the match
Then BL identifies the Payroll Import Batch ID that created the prior entries.

## AC3 — Duplicate rows are shown in Import Monitor when overwrite selection is missing

Given a source row’s PostedDate matches prior successful Payroll Import entries
And no Overwrite previous Payroll Import selection is provided
When BL returns the Commit response
Then BL rejects the import
And no new journal entries are written
And one DuplicateImport exception row is shown under Import Monitor > Exceptions for each source row whose PostedDate matched a prior Payroll Import batch.

Exception row details:

- Column Name = PostedDate

- Type = Mandatory

- Description = `Source row {RowNumber}: Payroll Import Batch {BatchId} already contains entries for PostedDate {PostedDate}. Select whether to overwrite the previous Payroll Import before continuing.`

## AC4 — Overwrite previous Payroll Import = No keeps existing batch

Given duplicate PostedDate rows were detected
And the user selects Overwrite previous Payroll Import = No
When Commit runs
Then BL keeps the existing Payroll Import batch unchanged
And posts the new Payroll Import as a new batch if all base Commit validations pass.

## AC5 — Overwrite previous Payroll Import = Yes reverses matched prior batches

Given duplicate PostedDate rows were detected
And the user selects Overwrite previous Payroll Import = Yes
When Commit runs
Then BL identifies each matched prior Payroll Import Batch ID
And creates reversal entries for all journal entries and lines linked to each matched prior Payroll Import Batch ID
And posts the new Payroll Import as a new batch if all base Commit validations pass.

## AC6 — Overwrite is batch-level, not row-level

Given a matched prior Payroll Import Batch ID contains entries for multiple PostedDates
When Overwrite previous Payroll Import = Yes
Then BL reverses every transaction linked to that Payroll Import Batch ID
And BL does not reverse only the rows or entries with the duplicated PostedDate.

## AC7 — Overwrite does not delete prior ledger entries

Given Overwrite previous Payroll Import = Yes
When BL processes the overwrite
Then prior journal entries are not deleted
And prior journal entries are not edited
And reversal entries are created to offset the prior Payroll Import batch.

## AC8 — Reversal entries and new entries save together

Given Overwrite previous Payroll Import = Yes
When BL writes reversal entries and the new Payroll Import entries
Then the reversal entries and new entries must all save successfully
And if any part fails, BL rolls back the whole Commit
And no partial reversal or partial new Payroll Import remains.

## AC9 — New import receives a new Payroll Import Batch ID

Given the duplicate flow completes successfully
When BL posts the new Payroll Import
Then the new Payroll Import receives a new system-generated Payroll Import Batch ID
And all new journal entries and lines are linked to that new Batch ID.

## AC10 — Multiple duplicate PostedDates can map to multiple prior batches

Given the new file contains PostedDates that match more than one prior Payroll Import Batch ID
And the user selects Overwrite previous Payroll Import = Yes
When Commit runs
Then BL reverses all matched prior Payroll Import batches
And posts the new Payroll Import as one new batch if all base Commit validations pass.

## AC11 — Duplicate handling does not bypass base Commit validations

Given the user selects either Yes or No for Overwrite previous Payroll Import
When Commit runs
Then BL still runs all base Payroll Import Commit validations
And any Mandatory base validation failure blocks the Commit.

## Acceptance Criteria
h2. UI View: 

* *Reversal Action:* Users can click the reverse icon (↻) on any transaction row in the Journal or Ledger view.
* *Reversal Drawer:* Clicking the icon opens a side drawer displaying concise original transaction details (Transaction ID, Description, Amount).
* *Reason for Reversal:*
** The drawer includes a required text area for the "Reason for Reversal".
** The field is pre-filled with  *"REVERSAL OF #[Original_ID]: "* (e.g., _"REVERSAL OF #10004: Duplicate entry"_) which cannot be erased.
** Users must enter a reason after this prefix to proceed.
** In the journal and ledger export, the Description field will be prefixed with *"REVERSED BY #[Reversal_ID]: "* (e.g., _"REVERSED BY #10050: Equipment Purchase"_) for the original transaction. However, the description remains unchanged in the UI.
* *Resulting Reversal Transaction:*
* A new transaction is created immediately following the original transaction.
* Its description is set to the text entered in the reason field (e.g., "REVERSAL: Duplicate entry").
* It displays a clickable badge: *"REVERSAL OF #XXXXX"*. Clicking this badge opens the details drawer for the _original_ transaction.
* *Original Transaction Update:*
** The original transaction's description remains unchanged.
** It displays a clickable badge: *"REVERSED BY #XXXXX"*. Clicking this badge opens the details drawer for the _reversal_ transaction.
** *Visibility:* Both the original and the reversal transactions remain visible in the journal/ledger.
** *Restriction:* Reversing a transaction that is already a reversal is not allowed (implied by logic, though explicit enforcement logic might need verification if not already present).

h2. Backend View:

* System knows which transaction reversed another transaction (queryable from either direction)
* System knows if a transaction has been reversed
* Reversal metadata is stored: reason (description), user, timestamp

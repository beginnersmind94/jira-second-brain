# NXT-60126 — WS - Meals Details

## Description
As a Workspace user, I want to view detailed KPI information so that I can understand performance metrics in depth.

h2. Comments

*Component*: KPI Details Panel

The panel shows:

* Name
* Date 
* Summary KPIs
* Detailed metric analysis
* Historical performance
* 

h2. User Flow

# User logs into Workspace and goes to Insights page
# User clicks on KPI card
# Panel opens

h2. Requirements

h3. Display Elements

_Header Section:_

* Icon
* KPI name
** Text of the name of the KPI
* Time period
** Date from the dashboard
* Close button



_Summary KPIs Section:_

* Note: This section is missing from Bolt but is to be included
** Use the same APIs as from the dashboard
* Shows the summary KPIs from the dashboard
* Section Name: Overview
* Fields:
* Actual 
** Same as dashboard card
* Expected 
** Same as dashboard card

 

_Meal Service Section:_

* Label: Meal Service Overview
** Grid
** Columns:
*** Meal Type
*** Free
*** Paid
*** Reduced
*** Total
** Total row at the bottom
** No paging, sorting, filtering



_Summary Section_

* About This Metric:
** Explanation text explaining the metric. Get the text from Bolt

h3. Interactions

* Action Items:
** None, read-only

h3. Data Requirements

* APIs:
* Summary section
** API for historical and actual
** Use same as dashboard
* Meal Service Section
** New API for rows and columns and totals 
* All communications between the drawer and APIs occur over HTTPS
* The system ensures that data is retrieved and processed securely and API endpoints implement rate limiting
* The drawer interacts with versioned API endpoints (e.g., {{/api/v1/...}}) to ensure future compatibility and backward support

h3. Performance & Usability

* Ensure the drawer displays properly in these layout options: Full width, Compact, Mobile responsive, Expanded
* The drawer should load within 2 seconds under normal operating conditions and opening the drawer is smooth and responsive

h3. Security

* Inherits dashboard page’s view permissions, no need to set permissions on the cards directly

h3. Error handling

* Use Error Boundaries and Try/Catch in Asynchronous Code and log errors to the database
* Show User-Friendly Error Messages in modal dialogs, toast notifications, or inline messages without disrupting the entire experience

## Acceptance Criteria
h5. *Header Section*

* Displays an icon representing the KPI.
* Displays the KPI name as a text label.
* Displays the time period retrieved from the dashboard.
* Includes a close button that allows the user to exit the drawer.

h5. *Summary KPIs Section (Overview)*

* This section is missing from Bolt but must be included.
* Uses the same APIs as the dashboard to display summary KPI values.
* The section title is *"Overview"*.
* Displays the following fields:
** *Actual* – Same value as the dashboard KPI card.
** *Expected* – Same value as the dashboard KPI card.

h5. *Meal Service Section*

* Displays a section labeled *"Meal Service Overview"*.
* Contains a grid with the following columns:
** *Meal Type*
** *Free*
** *Paid*
** *Reduced*
** *Total*
* Includes a total row at the bottom that sums each column.
* No paging, sorting, or filtering is applied.

h5. *Summary Section*

* Displays an *"About This Metric"* section explaining the KPI.
* The explanation text is retrieved from Bolt.

----

h3. *Interactions*

* The KPI details drawer is read-only and does not contain any action items.
* The close button exits the drawer.

----

h3. *Data Requirements*

h5. *APIs*

* *Summary Section*
** Uses an existing API to retrieve historical and actual KPI values, the same as the dashboard.
* *Meal Service Section*
** A new API retrieves rows, columns, and totals for meal service data.

h5. *Security & API Standards*

* All API communications must occur over HTTPS.
* The system ensures secure data retrieval and processing.
* API endpoints implement rate limiting to prevent abuse.
* The drawer interacts with versioned API endpoints (e.g., {{/api/v1/...}}) to ensure future compatibility and backward support.

----

h3. *Performance & Usability*

* Ensure the drawer displays correctly in the following layouts:
** Full width
** Compact
** Mobile responsive
** Expanded
* The drawer must load within 2 seconds under normal operating conditions.
* Opening the drawer must be smooth and responsive.

----

h3. *Security*

* The KPI details drawer inherits the dashboard’s view permissions.
* No additional permissions need to be set on individual KPI cards.

----

h3. *Error Handling*

* Implement error boundaries to capture UI failures.
* Use try/catch blocks for all asynchronous API calls.
* Display user-friendly error messages using modal dialogs, toast notifications, or inline messages.
* Log errors to the database for tracking and troubleshooting.

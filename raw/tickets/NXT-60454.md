# NXT-60454 — TH - WS - Tasks Async Message Flows

## Description
*As a* TeamHub user *I want* to be able to asynchronously update my Tasks *So that* I can use my app when have no connection

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* It is a backend only story
* The objective is to enable “offline mode” for tasks so that users can use the app without having an internet connection
* It should work exactly the same way you did Inbox offline mode but for tasks
* This will achieve 2 things
** User won’t have to go over the internet to get their tasks when landing on the screen. They will come from a local store on their phone so it will speed up the load times on the screen
** If the user has no internet, they can still see their tasks
* To achieve all this, you will ensure that:
** the Tasks Add New and Edit screen reads/writes to a local store
** there is a sync agent that:
*** sends when there is a new task
*** sends updates of existing tasks
*** downloads existing tasks
*** reconciles when their is a discrepancy 

h2. Flows

* #1, Login:
** User opens the application and is directed to login screen
** On login success, user is redirected to home screen
** App kicks off all Tasks reconciliation in the background without the user knowing it
** Tasks can run after inbox, sent, CR, Insights and Production, but before team
** App gets latest changes for Tasks
** App updates the local storage cache
* #2, Sync Schedule
** App maintains a schedule to check for updates
** App identified that the specified time has passed and initiates the sync job
** Sync job sends any unsent changes to the server and checks for all updates from the server
** Sync job reconciles any discrepancies
* #3, New task
** User browses to Tasks
** User creates a new task
*** App stores message in local store
*** App stores this new task as an update into the sync queue
*** App initiates a sync update to the server to store the message in the task on the server
*** Server saves task on server and sends success/failure message back
*** App checks for any new updates from the server and downloads if there are
* #4, Edit task
** User browses to Tasks
** User clicks on an existing task and makes a change to it
*** App stores message in local store
*** App stores this new task as an update into the sync queue
*** App initiates a sync update to the server to store the message in the task on the server
*** Server saves update to existing task on server and sends success/failure message back
*** App checks for any new updates from the server and downloads if there are

h2. Requirements

* Tasks screen
** Refresh button
*** On click:
**** Initiate the reconciliation process by immediately calling the sync agent to do a full sync
*** Indicate the sync status and if there is anything not synced
* Login
** Upon successful login, initiate the reconciliation process by immediately calling the sync agent to do a full sync
* Logout
** Upon successful logout, initiate the reconciliation process by immediately calling the sync agent to do a full sync
* Schedule
** Run a delta reconciliation on a schedule per every n minutes
** Just look for changes since the last time
** Choose a different number of minutes than from the other sync jobs like production and insights so that they don’t all try to run at the same time(e.g. 4 minutes for this one, 5 for production, 9 for Insights, etc.)

h4. *Data Requirements*

* Local Storage
** There should be local storage to save messages in
** Should maintain state even when user logs out
* Message Queue
** There should be a message queue in local storage for the async job to use to determine what needs to be sent to the server
** New and Edit Task screens should write to this queue any updates
* 
* APIs (at a minimum):
** Get Tasks Full
*** Called when doing a full reconciliation to get all the tasks
*** Send the user id and any other details needed
*** Should return a success/failure code
** Get Tasks Delta
*** Called when doing a partial reconciliation for tasks by the scheduled sync agent
*** Send the user id and last date/time stamp and any other details needed
*** Should return a success/failure code
** Post New Task
*** Called when a user assigns a new task and hits save and it is saved to the queue
*** Post the user id and task details needed
*** Have server save the task on the server and then return a success/failure code
** Post Update Task
*** Called when a user updates a task and hits save and it is saved to the queue
*** Post the task id, user id and task details needed
*** Have server save the task on the server and then return a success/failure code
** Log all errors to a sync error log that is visible

h4. *Performance & Usability*

* Updates should have no impact on user experience, they should not know it is happening and there should be no UX lagging

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Log all API to the *error log system/database* for further analysis

## Acceptance Criteria
h4. *Trigger Points for Sync*

*Tasks Screen - Refresh Button*

* The screen includes a Refresh button.
* On tap:
** Triggers a full sync reconciliation using the {{Get Tasks Full}} API.
** Displays sync status on the screen (e.g., "Syncing...", "Last synced at X", or "Pending changes not yet synced").
** If errors occur during sync, they are logged and user is notified via toast or inline message.

*Login Event*

* On successful user login:
** Automatically triggers the full reconciliation process using the {{Get Tasks Full}} API.

*Logout Event*

* On successful logout:
** Automatically triggers the full reconciliation process to ensure any pending changes are synced before session ends.

*Scheduled Sync*

* A background process runs a delta sync every n minutes (e.g., every 4 minutes).
** {{Get Tasks Delta}} API is used to fetch only changes since the last sync.
** The interval is staggered from other scheduled syncs (e.g., 5 mins for production, 9 mins for insights) to avoid conflicts.
** Sync includes sending queued task changes and retrieving updates from the server.

----

h4. *Data Requirements*

*Local Storage*

* Local async storage maintains all task data persistently, even after logout.
* Includes task records and metadata (e.g., due date, status, description).
* Changes from the user (new tasks or edits) are saved immediately to local storage.

*Message Queue*

* A local task sync queue tracks all task-related updates:
** New tasks (from New Task screen)
** Edited tasks (from Edit Task screen)
* Sync agent reads from this queue and posts updates to the server.

----

h4. *APIs and Sync Logic*

*Get Tasks Full*

* Called during full syncs (on login, logout, refresh).
* Returns a complete set of tasks for the authenticated user.
* Server response includes a success/failure status and the updated task list.

*Get Tasks Delta*

* Called during scheduled syncs.
* Takes in user ID and timestamp of last sync.
* Returns only updated/changed tasks since that time.
* Server responds with success/failure and the delta set.

*Post New Task*

* Called when a new task is saved and added to the sync queue.
* Posts task data and user ID to server.
* Server stores the task and returns success/failure.

*Post Update Task*

* Called when a task is edited and added to the sync queue.
* Posts task ID, user ID, and updated task data to server.
* Server processes the update and returns success/failure.

*Error Logging*

* All failed sync attempts and API errors are logged to a visible sync error log.
* Log entries include timestamp, API name, payload (non-sensitive), and error message.

----

h4. *Performance & Usability*

* Sync operations must not block or interrupt the user interface.
* The user should be able to interact with the app while sync runs in the background.
* The app must not stutter, freeze, or slow down due to syncing.
* Sync status is visible but non-intrusive.

----

h4. *Error Handling*

* All asynchronous calls use try/catch to handle errors gracefully.
* UI errors are caught using error boundaries to prevent crashes.
* All API failures are:
** Logged to the central error log system.
** Reported to the user only when necessary, using toast, modal, or inline messages.

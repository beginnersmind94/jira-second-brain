# NXT-66797 — GL Account Re-upload and Replacement Logic

## Description
As a district finance administrator, I want to be able to upload a new Chart of Accounts file that replaces my currently active account structure, so that I can correct validation errors or update my accounts while maintaining a complete audit trail of all files I've actually implemented.

## Acceptance Criteria
* *When user uploads a new file* (either from the "Fix and re-upload now" option or initiating a fresh upload):
** System processes the new file through the same validation workflow
** User sees the same decision point: continue with this upload or fix and re-upload
** *If user chooses to proceed* (whether validation passed completely OR had errors but they chose "Continue with current import"), this becomes the new active Chart of Accounts and replaces the previous active one
** *If user chooses to fix and re-upload again*, this file is discarded (not saved)
* *Only files that users choose to "proceed with" are saved* (regardless of validation status) with:
** Upload timestamps
** Username of uploader
** Validation results from when they chose to proceed
** The accounts that were actually imported (including flagged ones)
* *If user has changed the account code structure configuration* between uploads:
** System validates the new file against the current/updated account code structure
* *Previous active accounts are replaced* when a new file is implemented

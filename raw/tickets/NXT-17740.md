# NXT-17740 — UI - Scheduled Notifications 

## Description
External communication to parents and other entities needs to occur when certain conditions within the various modules are met.  We would like a mechanism that detects when these conditions occur and when they do, will send the communication related to the condition.

We would like for the various modules to be able to own and manage the definition of these conditions as well as the communication templates that are sent.  These condition definitions and templates should be able to changed at runtime by the various modules.

## Acceptance Criteria
* Modules shall have the ability to list, add, update, delete, enable, and disable these execution of these external notification jobs. (This should be  current state)
* The end-user shall have the ability to create a scheduled notification (by selecting “Scheduled” from the Notification dropdown)
** When a “Scheduled” notification type is selected, the user will be given the ability to:
*** Define a calendar window (between a Start Date and an End Date) within which the scheduled job will be active
*** Define a frequency (Day of week with multiple days allowed) at which the module’s notification mechanism will be called within the active window
*** Define a start time for the time of day(s) at which the module’s notification mechanism will be called for each enabled day-of-week
* The module team has the ability to define a triggering condition that, when met, would inform the platform that the notification can be sent (registration / plug-n-play)
** The code/logic for the condition evaluation will be owned by the module team. The platform will provide the ability for the module team to “register” this code/logic and the platform will call this code as part of the trigger condition evaluation.

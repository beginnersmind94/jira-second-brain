# NXT-71360 — Insights - KPI - ENP - Dashboard - SC

## Description
*I am a* district user *that I want to* view Eligible Not Participating (ENP) performance in the Insights dashboard using SchoolCafé data
*so that I* can monitor actual performance against expected benchmarks across sites and timeframes.

----

h2. *Notes*

* This story applies to *Insights – SchoolCafé data source*.
* ENP (Eligible Not Participating) is part of the *front office KPIs* being integrated into Insights.
* This story covers the *Dashboard screen only*.
* This story is a *companion to the PrimeroEdge ENP Dashboard story*.
* The *frontend behavior, UI, filtering, interactions, formatting, configurator behavior, grid behavior, and chart behavior must remain the same as the PrimeroEdge implementation*.
* Frontend developers should *not need to build new UI logic* for SchoolCafé.
* This implementation must reuse the same:
** KPI card behavior
** filter behavior
** configurator behavior
** grid behavior
** click behavior
** trend chart behavior
** tooltip behavior
** user-specific saved layout behavior

h3. *Data & Source of Truth*

* The source of truth for ENP data is available in:
** *Accountability → Reports → Participation*
** Click the *Non-Participation* tab
** Select *Free and Reduced* in the *Eligibility* field
* This must be used by developers and QA to validate ENP values.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure:
** SchoolCafé: [https://primeroedgenext.atlassian.net/browse/NXT-67456|https://primeroedgenext.atlassian.net/browse/NXT-67456|smart-link] 
* The dashboard must use the *SchoolCafé stored procedure* as the data source instead of the PrimeroEdge stored procedure.

h3. *KPI Definition & Formatting*

* ENP is a *percentage-based KPI*.
* Format must be: *0.00% (two decimal places)*.
* Values must be:
** Rounded to two decimal places
** Between *0 and 100*
* The dashboard must display:
** Actual value
** Expected value (*ENP Goal from benchmark configuration*)

h3. *KPI Direction*

* This KPI is *Lower is Better*:
** Values closer to *0% indicate better performance*
** Higher values indicate worse performance
* Performance indicators, comparisons, and trends must respect this direction.

h3. *Filtering Behavior*

* All ENP components must respond to *Timeframe and Site filters*, consistent with the PrimeroEdge implementation.

h3. *Interaction Behavior*

* Clicking on ENP values (card or grid) must open an *ENP details drawer* (covered in a separate story), consistent with the PrimeroEdge implementation.

----

h2. *Out of Scope*

* PrimeroEdge implementation (covered in separate story)

----

h2. *User Flow*

* Same as *PrimeroEdge ENP Dashboard story*, except data is sourced from SchoolCafé.

----

h2. *Requirements*

h3. *1. Shared Frontend Implementation*

* The SchoolCafé implementation must reuse the *same frontend behavior and UI* as the PrimeroEdge dashboard implementation.
* No new or separate frontend workflow should be introduced for:
** KPI card
** Screen configurator
** School Performance grid
** Performance Trends chart
** Trend hover tooltip
** Filter interactions
** Click-to-open details drawer behavior

----

h3. *2. Backend Data Source Change*

* The backend must use the *SchoolCafé ENP stored procedure* instead of the PrimeroEdge stored procedure.
* This data source change applies to:
** KPI card actual value
** KPI card expected comparison inputs
** School Performance grid values
** Performance Trends chart values
** Trend hover details
** Filtered dashboard results

----

h3. *3. Functional Consistency*

* All SchoolCafé ENP dashboard behavior must remain functionally identical to the PrimeroEdge implementation

----

h3. *4. Trend Hover Behavior*

* The SchoolCafé implementation must use the same hover tooltip structure as PrimeroEdge for ENP

----

h3. *5. Data Validation*

* SchoolCafé ENP values must match:
** SchoolCafé stored procedure output
** SchoolCafé Non-Participation report

----

h3. *6. Performance Requirement*

* The SchoolCafé-backed implementation must perform at the same level as the PrimeroEdge implementation and must not introduce performance degradation.

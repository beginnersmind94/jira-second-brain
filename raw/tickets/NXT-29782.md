# NXT-29782 — IM - Onboarding Items - POS Information 

## Description
IM - Onboarding Items - POS Information

As  an Item Management Super User / Administrator, I need to be able to onboard Items POS Information using an excel file in order to speed the process of bringing customers working with SchoolCafe’s Backoffice.

## Acceptance Criteria
*Entry Point:*

* System → Import Dashboard → Import Set Up
The user should able to onboard the data based on the system templates defined by the system Item template.

*The Items details data onboarding process shall follow the following procedure/steps*

*Step 1 - Pre-Upload (Platform Team will manage this)*

* Select Import Type
** Options
*** *POS Items*
** Select Data Source
*** By Default, set to Local
Out of Scope - other data sources other than local
The user shall be prompted (Would you like to download a template?) at this step either to download a template or not

 *Step 2 - File Upload (Platform Team will manage this)*

* If user selects Yes, the system will download a template (excel format) and the user will use the template to fill the data
If the No, then template then the user shall be prompted to upload a file
Once file is uploaded the system shall provide the user with visual confirmation of success or failure. Any empty file gets filtered at this stage
The user shall be prompted to perform Step 3 - Field Mapping

*Step 3 - Field Mapping (Platform Team will manage this)*

* Fields shall be mapped by selecting fields from source file (loaded template) and dragging and dropping to the mapped area
All mandatory or optional fields can be mapped at this stage
* Once fields are mapped the user shall be provided with:
** an option to preview the file data as mapped;
an option to return to the previous step
an option to continue to next step (Step 4 - Review)
* If user selects the previous option the system shall remove the uploaded file and move to Step 2
if the user selects continue then the system shall move to step 4

 

*Notes:*

* All fields mark in red are *MANDATOTY*
Mapping should like

*Step 4 – Review (Platform Team will manage this)*

* The system shall display:
**  map fields between Source and Target (Source - SchoolCafe)
** Current File Name (Data File use)
*** User shall be able to rename the file
* The system shall provide the user with an option
**  to enable (Activate) the file mapping;
 an option to edit the mapping;
 and an option to save the mapping
* If user selects to save the mapping then the user shall receive visual confirmation of success or failure of the saving process and ask the user for acknowledgement
* If the mapping is successful then the user shall be directed to the Import dashboard where the user shall have the option:
** to disable the mapping
** to import the data according to mapping;
*** If user selects Import then the system shall use created mapping for import process
**** The user shall load data file (drag and drop area or selecting from specified file location)
Once file is loaded the system shall display data as mapped
**** Then user shall provide the user with option
***** Cancel the import process
Move the previous page
***** Complete the import file process
****** User shall be presented with information about file import and be presented with options
******* to go to import dashboard
To go to Monitoring page where the user can view the progress of file import and view log file

*Business Rules:*

* *Base Rules*
** The templates shall provide option for INSERT and UPDATE operation ONLY , the users are required to update the records via UI only
All fields mark in red are *MANDATORY*
** If field mark as mandatory is not provided then ignore record and continue with next
*** Capture the Names of the Fields that are not provided
**  If the mandatory fields are found with data type mismatch, then ignore record and continue with next
*** Capture the Names of the Fields that are not matched with Data Type
** If mandatory field is based on the values of configurations and did not match, ignore record otherwise
*** Capture the Names of the Fields that are not matched with configuration
** If Item ID is not defined, create new item code
*** Capture the Item Code created for each row, this would be download by the user for verification (Excel Download option)
** Item Code provided and not conflicting with existing codes then create new item code. If not
*** Capture the Codes as “Duplicate Item ID”
** Fields Name between brackets represents value of the field name. for example for Item ID: \[Item ID] it will mean Item Code \[POS-17927]



* *Field Level Rules*
** Default POS Item Status = Active once it is imported
** Data Source:
*** Default is Local, the data for this is editable via UI once imported.
No duplicate Item ID or Item Description (Name) per data source allowed (In case if duplicate ID exists, it should be treated as Update)
No duplicate \[English Button Name] is allowed if DM Reimbursable flag is defined (Yes)
Spanish Button Name (Optional), if the value defined the duplication is not allowed, if DM Reimbursable flag is defined (Yes)
POS Item Category should be based on configuration Accountability>Settings>Configuration->POS Item Category
Meal type field shall be a valid one. Validate against the configuration
Grade groups should be valid grade groups (validate against grade group configuration -Systems Menu?)
*** Allow Sale Field - If allow sale = “Yes” then price fields for each grade group / Meal Type needs to be populated (should be treated as Mandatory)
**** Capture the Log – If allow sale = “Yes” and if price values are not provided and proceed with next record
Zeros are accepted as value if explicitly stated
*** If DM Reimbursable is set to “Yes” then validate that \[English Button Name] and price fields are populated
** -Only one template can be active at a time-
-POS Item Category (Menu Item Category will updated to include flag to enable to selection of a Menu Item Category as a POS Item Category )-
-Columns highlighted in yellow are added as needed according to school district programs (optional)-
** POS Item price cannot be a negative value
*** POS Item price can be 0 or left blank

 

*Error Log Rules*

* One record in the excel template may encounter multiple logs, all the errors against a row should be captured
If multiple logs are captured for a record, then all the log messages should be reflected as separated by semicolon (Ex: \[Field Name] –Mandatory Field; \[Field Name] – Configuration Mismatch
Each Log should be shown a new line so that user can fix them easily

* *Mandatory Fields*
** Item Description (Mandatory Field)
English POS Button Name (Mandatory Field), in case of DM Reimbursable flag = Yes
Is Meal (Mandatory)
If mandatory field is missing / not provided then log the import with a message " \[Field Name] - Mandatory Field” is missing and proceed with next record. For example, Mandatory Field: \[Item Name] is missing
** If mandatory field is based on a configuration (Ex: Dropdown value) apply required validation and continue with next field
*** If the validation is passed (Ex: Configuration based dropdown) use the value for import and continue with next field
If the validation is failed then log error on transaction file with error message "\[Field Name] – Configuration Mismatch"

* *Non-Mandatory Fields*
** If value for a field is not provided (blank) continue with next field unless there is a conditional validation needed (pls refer the attached excel sheet with all the messages)
** If value for a field is provided and requires validation then apply required validation and continue with next field
*** If the validation is passed (Ex: Configuration based dropdown) use the value for import and continue with next field
If the validation is failed then log error on transaction file with error message "\[Field Name] - Configuration Mismatch "
*** In case of Data type mismatch, If the validation is failed then log error on transaction file with error message "\[Field Name] - Data Type Mismatch"

* *Duplicate or Already Exists in DB*
** If duplicate Item IDs are found then capture the log message as “\[Item Code]  - Duplicate” and proceed with next record
If duplicates Item Names, are found flag them as duplicate in log transaction file with message “\[Item Name] - Duplicate” and proceed with next record
If duplicate \[English Button Name] are found them log them in transaction log file and with message “\[English Button Name] – Duplicate”

 *Data Type Mis-match*

* Applicability – Both Mandatory Fields & Optional Fields
If the data type mismatch occurs (Ex: Varchar Vs Number) then capture the log message as "\[Field Name] - Data Type Mismatch"

* *Set All Price Flag*

* If Set to Yes then then Generic Price field must be populated
** If set to Yes then take the price value provided and apply it to fields Price 1 thru Price (n)
If not, then capture a log message as “\[POS Price] – Missing Value”
* If set to Yes and any Price(n) field has a provided value then later value will overwrite the POS Price for that specific field
* If Set to No and If values for fields Price(n) are not provided then it will be assigned a blank
** Blank cannot be reflected as Zero, this should be captured as exception if flag is Allow Sale
Zero value will reflect as Zero
*  
* *Meal Type Description*
** If provided, it must exist and be valid, that is it should define as part of a meal District program
If provided and it does not exist log error in transaction file with message as “\[Meal Type] – Configuration Mismatch"

* -If Meal Type is not a valid one then log error in transaction file with message “Invalid Meal Type assigned to: Item Code / Item Name “ and proceed with next record-
 
If POS Item Category is not a valid one then log error in transaction file with message “ \[POS Item Category] – Configuration Mismatch “ and proceed with next record
 
-If Grade Group is not a valid one then log error in transaction file with message “Invalid Grade Group assigned to: Item Code / Item Name “ and proceed with next record-
* 
If allow sale = “Yes” and Price(s) are missing then flag them and log error in transaction log file with message “ \[Field] - Missing Price Details” , and proceed with next record



General Rule:

All the white spaces - Start and End of the fields should be trimmed

All the exception messages are available in the attached excel sheet, and it should be captured based on exception conditions as mentioned

Once the import is completed, the user should be able to view the log as per UX design (attached below). The summary should list the total number of records in Excel, how many are processed, how many are created, how many are updated, how many are exceptions, Start Date/Time, End Date/Time and Status (as Completed)

Excel - user should be able to download Excel with all the original data + Import Status (Success/Fail) + exception messages associated with that record

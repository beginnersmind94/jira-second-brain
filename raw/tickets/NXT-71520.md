# NXT-71520 — K12 - Ins - Date Selector Update

## Description
*I am a* district user
*and I want to* use the enhanced date range selector on the K12 Insights screen
*so that I can* view KPI cards for the exact time period I need using the same improved filtering experience as the Insights dashboard.

----

h2. *Notes*

* This story applies to the *K12 Insights mobile app*.
* This story covers the *date range selector update only*.
* The current mobile date selector should be updated to match the enhanced behavior implemented on the *Insights dashboard*.
* The mobile app is written in *React*, so the updated date selector behavior may be portable to mobile.
* Developers should review the *Insights dashboard code* to identify the date selector enhancements that were made and replicate that behavior on mobile.

h3. *Preset Ranges*

* Preset date range values must match those available on the *Insights dashboard*.
* Examples may include:
** Today
** This Week
** This Month
** Year to Date
** Custom Range
* The available presets and behavior must remain consistent with the dashboard implementation.

----

h2. *Out of Scope*

* Site filter updates
* KPI card additions or updates
* Changes to card configuration drawer behavior outside of date range interaction

----

h2. *User Flow*

# A user logs into the K12 Insights mobile app.
# The user navigates to the *Insights* screen from the menu.
# The user opens the *Date Range filter*.
# The user selects a date range using the updated selector.
# The user applies the filter.
# The Insights screen refreshes and displays KPI cards for the selected date range.
# The user may optionally adjust the site filter or configure visible cards.

----

h2. *Requirements*

h3. *1. Replace Existing Date Range Selector*

* Replace or enhance the current mobile date range selector to align with the updated *Insights dashboard* date selector.
* The mobile implementation should reuse the same behavior and interaction patterns as closely as possible.

----

h3. *2. Reuse Dashboard Enhancements*

* The mobile app must incorporate the same enhancements recently added to the Insights dashboard date selector.
* Developers must review the Insights dashboard implementation and replicate the relevant behavior in the mobile app.

----

h3. *3. API Reuse*

* The mobile app should reuse the same APIs as the Insights dashboard date selector where possible.
* No unnecessary new API should be introduced if the existing dashboard APIs support the mobile use case.

----

h3. *4. Insights Screen Integration*

* After applying a new date range:
** KPI cards must refresh based on the selected date range.
* The date range selector must work in combination with:
** Site filter

----

h3. *5. Selection Persistence*

* The selected date range should persist:
** During the user session
** When navigating away and returning to the Insights screen (if consistent with existing app behavior)

----

h3. *6. UI/UX Consistency*

* The mobile date selector should align with the Insights dashboard behavior as closely as possible within mobile design constraints.
* Users must be able to clearly understand:
** the selected date range
** when the date range has changed
** what data scope is currently applied

----

h3. *7. Cross-Platform Support*

* The updated date range selector must function correctly for:
** SchoolCafé
** PrimeroEdge

----

h3. *8. Performance Requirement*

* The date range selector must perform efficiently on mobile devices.
* Applying date range changes and reloading KPI cards must not introduce noticeable delay.

## Acceptance Criteria
h3. *1. Replace Existing Date Range Selector*

* The mobile date range selector is updated to match the enhanced Insights dashboard behavior.
* Old behavior is replaced where applicable.

----

h3. *2. Reuse Dashboard Enhancements*

* Mobile implementation reflects the date selector enhancements from the Insights dashboard.
* Developer has reviewed and replicated relevant dashboard date selector changes.

----

* No unnecessary duplicate API implementation is introduced.

----

h3. *4. Insights Screen Integration*

* KPI cards update after the user applies a new date range.

----

h3. *5. Selection Persistence*

* Selected date range persists during session/navigation (if applicable).

----

h3. *6. UI/UX Consistency*

* Selected date range is clearly displayed.
* Preset ranges match the Insights dashboard.
* Date filter behavior is consistent with the Insights dashboard pattern.

----

h3. *7. Cross-Platform Support*

* Works correctly in SchoolCafé context.
* Works correctly in PrimeroEdge context.

----

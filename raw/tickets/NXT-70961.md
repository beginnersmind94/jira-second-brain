# NXT-70961 —  Insights - KPI - MPLH - Configuration (PE)

## Description
*I am a* district user
*that I want to* configure benchmark values for Meals Per Labor Hour (MPLH) in Insights
*so that I* can define target performance thresholds used in KPI calculations and comparisons.

----

h2. *Notes*

* This story applies to *Insights – PrimeroEdge data source*.
* MPLH (Meals Per Labor Hour) is part of the *front office KPIs* being integrated into Insights.
* Insights dashboards support *district-level and site-level KPI configuration across configurable timeframes*.
* Benchmark configuration is accessed via a *drawer from the Insights dashboard*.

h3. *Data & Source of Truth*

* The *source of truth for MPLH data* is available in:
** *POS → Management → Meals Per Labor Hour*
* This page provides MPLH values:
** Per site
** Per time frame
* This must be used by *developers and QA to validate MPLH calculations and data accuracy*.

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure for MPLH:
** PrimeroEdge: [https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109334|https://dev.azure.com/Cybersoft-Technologies-Inc/PrimeroEdge%20Classic/_workitems/edit/109334|smart-link] 
* The Insights team must *integrate the existing stored procedure into the Insights product*.

h3. *KPI Definition & Formatting*

* *MPLH is a rate-based KPI*.
* Format must be: *0.00 (two decimal places)*.
* Values must be *rounded to two decimal places*.
* Calculation:
** *MPLH = Equivalent Meals ÷ Actual Labor Hours*

h3. *Configuration Behavior*

* Users can configure benchmarks for:
** Historical time frames
** Current time frame
** One future time frame
* Benchmark values are stored per *site (school) and time frame*.
* Validation occurs at the *UI level when saving benchmark values*.
* Autofill uses:
** Previous year actual value if available
** Otherwise, current year actual value
* Bulk updates apply *incremental changes across all visible rows (sites)*, respecting filters.
* History tracking must capture all benchmark changes.
* Changing the selected *year/time frame must update both benchmark values and actual values displayed in the grid*.
* Updated benchmark values must *immediately impact KPI comparison logic on the Insights dashboard*.

h3. *QA Note*

* QA must validate MPLH values against:
** PrimeroEdge MPLH screen (POS → Management → Meals Per Labor Hour)
** Stored procedure outputs
* QA must test:
** Manual entry
** Autofill behavior
** Bulk updates
** History tracking
** Save and reload consistency
** Large districts with many sites
** Mixed scenarios (manual + autofill + bulk update combined)

----

h2. *Out of Scope*

* SchoolCafé implementation (separate story)
* Paid Not Applied (P&A) KPI
* Eligible Not Participating (EMP) KPI
* Any non-MPLH KPI configuration
* Changes to stored procedure logic (integration only)

----

h2. *User Flow*

# A user logs into PrimeroEdge.
# The user navigates to *Insights*.
# The user lands on the *Insights dashboard*.
# The user selects the *MPLH KPI* and clicks *Benchmark Configuration*.
# The Benchmark Configuration drawer opens.
# The user selects a *time frame (year)*.
# The system displays a grid of *sites (schools)* with editable MPLH fields.
# The user may:
#* Manually enter MPLH values per site
#* Use *Autofill* to populate values
#* Use *Bulk Update* to increment values across sites
#* Click *View History* to review past changes
# The user edits values directly in the grid.
# The user clicks *Save Changes*.
# The system validates and persists the benchmark values.
# Updated benchmarks are used in MPLH KPI calculations and reflected on the dashboard.

----

h2. *Requirements*

h3. *1. Benchmark Configuration Access*

* Users must be able to access Benchmark Configuration from the Insights dashboard.
* The configuration screen must open as a *drawer UI*.
* The drawer must be specific to the *MPLH KPI*.

----

h3. *2. Time Frame Selection*

* Users must be able to select a time frame.
* Supported frames include:
** Historical time frames
** Current time frame
** One future time frame
* Changing the selected time frame must:
** Update displayed benchmark values
** Update actual MPLH values used for autofill/reference
* Benchmark values are stored per *site and selected frame*.

----

h3. *3. Grid Display (Site-Level Configuration)*

* The system must display all sites (schools) in a grid.
* Each row represents a site.
* A column for *MLPH* must be added:
** Label: *MLPH*
** Location: *last column in grid*
** Type: *rate (0.00 format)*
* Grid must be populated via API/dataset including MPLH values.

----

h3. *4. Manual Entry (Edit Behavior)*

* Users must be able to manually enter or update MPLH values per site.
* Fields must be editable directly within the grid.
* Values must follow:
** Numeric input only
** Maximum 2 decimal places
** Minimum value: *0.00*
** Maximum value: *100.00*
* Values must be rounded to 2 decimal places.
* Changes must persist upon save.

----

h3. *5. Autofill Functionality*

* Users can click *Autofill* to populate MPLH values.
* Autofill must:
** Use *previous year actual value* if available
** Otherwise use *current year actual value*
* Autofill applies to all visible rows in the grid.
* Autofill must update the MPLH textboxes immediately.

----

h3. *6. Bulk Update Functionality*

* Users can click *Bulk Update*.
* A modal/dialog must open allowing entry of an increment value.
* The modal must include:
** Field: *MLPH*
** Location: *last field in modal*
** Format: *same as grid (0.00)*
* Bulk update must apply to:
** *All rows currently visible in the grid (respecting filters)*
* Applying changes must update all applicable grid values.

----

h3. *7. View History*

* Users can click *View History*.
* The system must display a history of benchmark changes.
* History must include MPLH changes for:
** Add
** Edit
** Delete
* History includes:
** Previous values
** Updated values
** Timestamp
** User who made the change

----

h3. *8. Save Behavior*

* Users must be able to click *Save Changes*.
* MPLH is a *required field*:
** Users cannot save if any required value is missing or invalid
* System must validate all entries before saving.
* MPLH values must be saved to the database.
* Updated values must be reflected:
** On subsequent loads
** On the Insights dashboard after closing the drawer

----

h3. *9. Data Integration (Stored Procedure & API)*

* The system must integrate the existing *PrimeroEdge MPLH stored procedure*.
* MPLH must be included in:
** Backend API responses
** Dataset used to populate the grid
* Stored procedure must support:
** Autofill values
** Actual value comparisons

----

h3. *10. Performance Requirement*

* The benchmark configuration grid must load and save without performance degradation, even for large districts.

# NXT-70340 — FIN — Prepaid Liability Posting Rules for All Payment Types

## Description
As a Child Nutrition Director, I want all prepayment-related transactions to automatically post to the general ledger on Period Close, so that my Prepaid Liability balance is accurate and reconcilable before exporting to the district ERP.

*Current State:* All prepayment transactions post to a single "Prepaid Liability" account but the posting rules are not defined for all payment types.

*Solution:* Define the debit and credit accounts for every payment type that touches Prepaid Liability. One GL account. One story. Each rule maps directly to a transaction type already captured by POS.

Financials is a consumer of POS data. It does not create, modify, or enrich transaction data. It reads what POS provides and posts accordingly.

{quote}*Note on data points:* All data points referenced in this story already exist in the account mapping list. No new data points are required. One-time or rare events (balance imports, balance overrides, intersite transfers) do not warrant dedicated data points — districts handle those as manual journal entries using the GL accounts they already have in their ERP.{quote}

## Acceptance Criteria
h1. AC1 — Posting Setting: Distributed vs. Centralized

This story respects the "Prepaid Account, Bonus & Refund Posting" setting configured in Financials › Configuration › Settings › Posting Settings. The setting has two mutually exclusive options:

||Option||Behavior||
|*Distributed (Enrollment Site) — Default*|Entries post per enrollment site. Each site with a non-zero net amount for the period gets its own entry.|
|*Centralized (District Office)*|All transactions across the district are combined into one entry posted to the Central Office site. Central Office site code is pulled from Site Configuration; if none exists, falls back to a default code of all zeros matching the district's site code length.|

*Scope:* This setting controls posting for Prepaid Account (AC2), Bonuses (AC3), and Refunds (AC4). All other automated payment types in this story (AC5–AC8) follow the same setting for consistency.

*Switching:* When a district toggles between Distributed and Centralized, the change applies to future postings only. Previously posted entries are not restated.

Throughout this story, *[Site]* in entry descriptions and aggregation rules means: the enrollment site (if Distributed) or the Central Office (if Centralized).

h1. AC2 — Pre Payment (POS and Online Deposits)

*Entry Type Name:* Pre Payment

Parents load money onto student accounts either at the register (cash/check) or online (web/app). Both post to the same single Prepaid Account — there is no split into separate POS and Online buckets at the GL level.

The system generates two separate entries per site per period — one for POS deposits and one for Online deposits. The distinct entry descriptions give the district visibility into how much came from each source through the ledger, without requiring separate GL accounts.

*POS Deposits (Cash/Check):*

||Field||Value||
|Debit|Cash in Bank|
|Credit|Prepaid Account|
|Amount|Sum of all POS prepayment transactions for the site and period|
|Entry Description in Posting Activity Log|POS Pre Payments for [Closed POS Period]|

{quote}Description Example: {{POS Pre Payments for Period Start Date - Period End Date}}{quote}

*Online Deposits:*

||Field||Value||
|Debit|Cash in Bank|
|Credit|Prepaid Account|
|Amount|Sum of all online prepayment transactions for the site and period|
|Entry Description in Posting Activity Log|Online Pre Payments for [Closed POS Period]|

{quote}Description Example: {{Online Pre Payments for Period Start Date - Period End Date}}{quote}

*Why one account, two entries:* The district's GL has one Prepaid Account line — simple to map, simple to export. But the CN Director or finance officer can filter the ledger by entry description at any time to see exactly how much came from the register vs. online. The reconciliation workflow is preserved: POS entries match to the bank deposit slip, Online entries match to the processor settlement report.

h1. AC4 — Refunds (Automatic | Source: POS Period Close)

*Entry Type Name:* Refunds

Records the financial impact when a student's prepaid balance is refunded to the parent/guardian. Since the actual cash disbursement is handled by the district's ERP (Oracle, Tyler Munis, etc.), this entry records the net effect in a single step rather than creating an intermediate refund liability.

||Field||Value||
|Debit|Prepaid Account|
|Credit|Cash in Bank|
|Amount|Sum of all refund transactions for the site and period|
|Entry Description|Refunds for [Closed POS Period]|

{quote}Description Example: {{Refunds for Period Start Date - Period End Date}}{quote}

⚠️ *Data Point Removal:* The legacy "Refunds (Liability)" system account/data point should be removed from the Financials 2.0 chart of accounts. It is no longer needed since we are not using a two-step refund posting model. This means one fewer account for districts to map during GL configuration and one fewer manual ledger entry type to maintain.

h1. AC8 — Account Transfer

*Entry Type Names:* Account Transfer Out, Account Transfer In

Money moves between student accounts — commonly principal account to student (covering meal costs) or parent transferring between siblings at different schools. Prepaid Account as a whole doesn't change at the district level, but in Distributed mode, site-level balances shift when the students are enrolled at different sites.

*Important for developers:* The principal account is tied to the serving site, not the enrollment site. Financials posts at the enrollment site level. This means when a principal covers a meal for a visiting student (enrolled elsewhere), it is always a cross-site transfer — even though the POS transaction happened at a single physical location.

*New Entry Types:*

||Entry Type Name||Purpose||Source||
|Account Transfer Out|Records prepaid funds leaving a site when a student enrolled elsewhere receives a transfer (e.g., principal covers a visiting student's meal, parent transfers between siblings)|POS Close|
|Account Transfer In|Records prepaid funds arriving at a site when a student enrolled there receives a transfer from another site|POS Close|

*Rules Drawer Updates:* These two entry types must appear in the business rules drawers of all data points involved in the transactions:

* Due From Other Sites — Account Transfer Out
* Due To Other Sites — Account Transfer In
* Prepaid Account — Account Transfer Out, Account Transfer In

*New Data Points Required:*

||Data Point||Base Type||Description||Source||Type||
|Due From Other Sites|Asset|Receivable from another site when prepaid funds transfer out to a student enrolled elsewhere|POS Close|Automatic|
|Due To Other Sites|Liability|Payable to another site when prepaid funds transfer in from a student enrolled elsewhere|POS Close|Automatic|

*Centralized mode:* No GL entry. All Prepaid Account posts to one Central Office account, so transfers between students have no GL impact regardless of enrollment site.

*Distributed mode, same enrollment site:* No GL entry. Prepaid Account at that site is unchanged.

h3. *Distributed mode, different enrollment sites:*

*Site losing funds (source student's enrollment site):*

||Field||Value||
|Debit|Due From Other Sites|
|Credit|Prepaid Account|
|Entry Description in Posting Activity Log|Account Transfer Out for [Closed POS Period]|

{quote}Description Example: {{Account Transfer Out for Period Start Date - Period End Date}}{quote}

*Site gaining funds (destination student's enrollment site):*

||Field||Value||
|Debit|Prepaid Account|
|Credit|Due To Other Sites|
|Entry Description in Posting Activity Log|Account Transfer In for [Closed POS Period]|

{quote}Description Example: {{Account Transfer In for Period Start Date - Period End Date}}{quote}

||Field||Value||
|Amount|Sum of all transfer transactions between the site pair for the period|

Each entry is self-balancing at the site level. At the district level, Due From Other Sites and Due To Other Sites net to zero.

{quote}*Note:* These entries only generate when the source and destination students are enrolled at different sites. Same-site transfers (e.g., principal account to student at the same school) produce no GL entry.{quote}

h2. Note: Transaction Types With No Automated Posting

The following transaction types affect Prepaid Account but have no system-generated entries. They are documented here for completeness — there is nothing for QA to test.

*Balance Import (BALIMPORT):* One-time migration event. POS handles student-level imports. Financials picks up the site-level total at period close and it flows into Prepaid Account. The offsetting entry is the district's responsibility — they post it in their ERP using whatever clearing or suspense account they already have.

*Balance Override (OLDBALOVERIDE):* Migration cleanup when the initial import file was incorrect. POS reverts and reprocesses. Financials picks up the net difference at the site level at period close. Same as balance import — the offsetting entry is on the district.

h2. Aggregation Rules (Applies to All)

*Distributed mode:* One entry per enrollment site, per period, per payment type. Only sites with a non-zero net amount for the period generate an entry.

*Centralized mode:* One entry per district (posted to Central Office), per period, per payment type. All site-level transactions are summed into a single district-level entry.

----

h2. Validation Rules (Applies to All)

All new data points are created by the system and require mapping. Existing rules for validation apply.

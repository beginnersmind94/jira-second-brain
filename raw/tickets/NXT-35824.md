# NXT-35824 — Add Sub Items to Resources - API

## Description
As a SchoolCafé admin, I want to be able to set permissions to items other than pages and treat in a hierarchical way so that I can more easily manage permissions for my site.

*Flow:*

* Open SchoolCafé and browse to Settings > Roles & Permissions
* Click on a role
* On the right side, see General permissions for that role
* View resources
* View sub resources such as tabs and sub-pages (new) (This story)
* Optionally edit permissions
* Optionally start the R&P stepper (New) (Another story)
* Optionally, export the permissions (New) (Another story)

 

*Notes:*

* I am breaking up the updates into several stories to simplify dev work, this is the first one that lays down the foundation for the rest
* We will leverage existing functionality in the way we are doing CRUD security, and will reapply it to things other than base pages such as sub-pages and tabs

 

*Requirements*:

* Values for the Resource Type field
** Right now, only have Page
** Add the following:
*** Sub-Page
*** Tab
*** Function
*** Area
* ·          
* Rearrange the list of resources according to the attached excel sheet
** We will put them in order of the menu on the left
** We will create the concept of hierarchy or nesting and reflect this in the table
*** The example UI shows what this can look like
*** See Accounts resource
* ·          
* We will give you default permissions for the standard roles
** Need you to add default permissions to the new client setup scripts if they are not already there, or update them to the new values
** See excel sheet for default values
* ·          
* Hiding for Inheritance:
** Sometimes a page is the result of a button click of its parent page
** Therefore, Page 2 permissions = Page 1 button click permissions
** Users don’t want to have to duplicate permission editing
** They want Page 2 to inherit Page 1 action button permissions and to be hidden from user
** To do this, we will need to be able to set visibility in Resources grid to a True/False for every item in backend (no UI for this is needed)
*** Default is True
*** If false, don’t show that item or any of its children in the grid
*** If true, show it like normal
*** The developer writing in the security will need to know to apply the button click’s permissions to the sub page
* ·          
* We want to be able to grey out / disable a checkbox if it is not applicable to a screen
** Applicability will be defined in the attached excel sheet
** If not applicable, disable
** Else, enable like usual
** Default is enabled

## Acceptance Criteria
* The list of resources is in order of the menu on the left
* UI shows nested hierarchy or nesting and reflect this in the table like attachment
* User can view & edit sub resources such as tabs and sub-pages in the resources grid
* Values for the Resource Type field include: Sub-Page, Tab, Function
* ·          
* Default permissions for the standard roles are applied when setting up a new client
* ·          
* We can set visibility in Resources grid to a True/False for every item in backend and it is reflected in the grid
* ·          
* We can grey out / disable a checkbox if it is not applicable to a screen per the attached excel sheet:
** If not applicable, disable
** Else, enable like usual
** Default is enabled

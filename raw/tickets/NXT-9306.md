# NXT-9306 — Integration - SIS Import - Settings/Conditions

## Description
As a central office administrator I want to set certain conditions for how my student enrollment interacts with student information imports.

Important for Pilot:

* Withdrawal Threshold - Setting takes into consideration a given percentage amount of the entire active student population. User sets a percentage amount (Ex - 10%) - if the file does not contain students who are currently enrolled (active) and that total exceeds the setting percentage (ex, > 10%), then the file should move to a “Cancelled” status before being processed for the reason of “Exceeds withdrawal threshold of \[setting amount]%”. If reason cannot be mentioned just yet, the Cancel action should at least take pace.
** 1 Setting is needed:
*** Student Information Import - Withdrawal Threshold Percentage
**** Numeric entry for with percentage (ex - 10%)
***** Would display as \[  10  ]%
* Manually Created Accounts - Setting takes into consideration how the customer wants the SIS import to interact with any accounts that have been manually created by users. Often times …
** there is a certain portion of the student population that is not tracked in the SIS. The child nutrition department wants to track these accounts as they are serving them within the cafeteria and need these as reference for audits, etc. Customers will create these accounts and want to retain them as enrolled (Active) accounts and to not be withdrawn if they are not on the SIS import file (which will be the case).
** there is a lag in the individual school registrars for adding students who have recently enrolled. PK students often lag behind for weeks at a time during the start of school. To better track the student record, customers will often manually create student records with an ID that they know is going to be given to that student at the time when they start becoming present within the SIS file. In these cases, the child nutrition department will prefer that IF an SIS file sends a student record with that same ID, that we update the information for that matched ID number. With this option we should be careful. Having just a match on the ID can be an issue, so we want to check on adding a second validator such as the first 3 letters of the last name before confirming the match and updating.
** 2 settings are needed:
*** Manually Created Accounts - Account Information
**** Option: Update account information for matching students (default)
**** Option: Retain manually entered account information
*** Manually Created Accounts - Retain Enrollment Status
**** Option: Yes (default)
**** Option: No
** Important note to discuss - An account may be manually created, but IF a student match is found, we should change the flag to no longer be manual. OR, we should have a separate flag for manual_creation (Y/N). We only want the second setting to apply to those students who are truly manual AND unmatched to a student record from the SIS.
*** Decision - We will update the IsManual flag if the SIS import matches a manual student.

The quickest way for the pilot would be to utilize existing setting within the system for withdrawal threshold, add 2 new settings for manual accounts, and then hook into the current student import integration job.

The ideal way would be to create a new configuration section within the Integration job for “Conditions” that would allow us to put these similar “settings”, but they would be specific to just that  job/data set definition. This would prevent a lot of settings within our normal Settings page (it could get really cluttered as we add imports/exports). This approach also allows the customer to configure as they are setting up the integration job without having to then know to navigate to settings. I would add a “Conditions” step/section in the last step before the save action. Ask the user how they want to set the withdrawal threshold and how they want to handle manually created accounts. Once they answer, they can then save the job. They can always edit these conditions later in the integration edit capability.

# NXT-49317 — OTP - Login Updates

## Description
As a Customer Support user, I want to be able to setup a district’s One-Time Passcode authentication so that they can use a code sent to them to complete login

*Notes*:

* This story is for One-Time Passcode login screens and login process
* One-Time Passcode configuration covered in other stories
* This story assumes that coding for district and user configuration screens have been completed

*Flow (Happy Path):*

* User browses to Login page
* User enters their email address and clicks continue
* System sends them a one-time passcode
* User enters the passcode on Login screen 2
* System authenticates them and lets them in

*Requirements*:

* On Login screen, do these changes:
** There are 4 scenarios with different flows covered individually below:
** If the only method is One-Time Passcode 
** If the user has both SchoolCafe and One-Time Passcode methods
** If the user has both SSO and One-Time Passcode 
** If the user has SchoolCafe, SSO and One-Time Passcode 





*Only One-Time Passcode Process*

* In this scenario, there is no other way for the user to login
* If the only method is One-Time Passcode :
* When the user clicks Continue on the first screen:
* Check their authentication methods
** In this case, it is only One-Time Passcode , so no point in showing all the other auth stuff on screen 2
* Get the preferred One-Time Passcode method
* If they have configured their preferred One-Time Passcode method:
** Follow the Authentication via One-Time Passcode process below
* If they have NOT set their preferred One-Time Passcode method:
** Follow the process in The Unconfigured One-Time Passcode only process for Login screen below
* One-Time Passcode Screen 2
* In this scenario, there is no other way for the user to login, so do not show buttons for any other method
** Follow the Authentication via One-Time Passcode process below





*One-Time Passcode & SchoolCafe Process*

** Therefore, by default the user must use SchoolCafe authentication

*** Put a tooltip over it: Configure in your profile

**** Enable the button
**** Change tooltip to: Login via one-time password
**** On click, redirect to One-Time Passcode screen 2
** User authenticates via normal SchoolCafe process and moves on
* In  scenario #2
** There are 2 ways the district allows the user to login: SchoolCafe and One-Time Passcode 
** The user has configured One-Time Passcode as their preferred method
** Therefore, the process will try to authenticate the user via One-Time Passcode authentication
* When the user clicks Continue on the first screen:
* Check their authentication methods
** In this case, it is SchoolCafe and One-Time Passcode 
* Get the preferred method which is One-Time Passcode 
* User is redirected the the One-Time Passcode Screen 2

** Follow the Authentication via One-Time Passcode process below

*The Unconfigured One-Time Passcode only process for Login screen*

* In this scenario, the user has arrived at the login screen and their district wants them to use One-Time Passcode but the user has not configured One-Time Passcode in their user profile
** This might happen for a new user
** Might happen if District changes user method to One-Time Passcode only
* Therefore, we are forced with a choice:
** We can make district admin set it up for him
** Or try to leverage existing user profile information to initiate the process
*** This flow will try to let the user to configure their own info
* We will look at the user profile and see if there is an email
** There is supposed to be an email, it is required field
** But there is no guarantee it is is well formed or the one the user actually uses
* The following happens on screen 2:
* If there is an email, then do the following:
** Show the user this message:
*** You have not configured your two-factor authentication methods. We have sent you a one time code to the email in your profile: [insert email address masked like this: ***@mydistrict.com]. 
** Send the user the One-Time Passcode email with the one time code like you normally would do for One-Time Passcode emails
* If the user gets the email, they can proceed:
** Allow the user to enter the code on the 2nd screen like normal
** If they authenticate:
*** If the district allows Email One-Time Passcode :
**** Save One-Time Passcode email authentication for them in the DB and set it as preferred using the email address
*** If the district does not allow Email One-Time Passcode , then do not save it to the DB
*** Note: we need to force them to configure at least one One-Time Passcode method, so we must force them to do that and not allow them to go anywhere else until they do
*** Redirect the user to the Authentication tab on User Profile
**** Show them a message: You must configure your two-factor authentication
**** Allow them to go through the normal process of setting it up
**** Don’t allow them to do anything else in the site until they have configured it
**** If you saved the email as their preferred method, then that technically qualifies as having configured one, so they can browse away
** If they are unable to authenticate:
*** At this point the user is blocked from logging in and needs assistance from a district admin to enable their authentication method on the Edit User screen because their only method is One-Time Passcode and it is not configured and we cannot trust an unauthenticated user for setting it up.
*** Show them this message: Your account is not configured to use two-factor authentication and your district does not allow any other methods. Therefore, contact your system administrator and ask them to edit your user account to enable you to login with two-factor authentication.

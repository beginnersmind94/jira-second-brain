# NXT-12394 — Accounts - Account Details - Return Check (Multiple Students)

## Description
*User Story*

As an administrator, I need to be able to process a returned check in the system that covers the scenario where multiple accounts are associated with the payment so that there is a record that can be tracked for future auditing.

*Mock-up*

[https://www.figma.com/proto/m3QGmo7Fjys7um36XDhwAY/Accountability-5.0?page-id=711%3A16224&node-id=711%3A16225&viewport=352%2C48%2C0.36&scaling=scale-down-width&starting-point-node-id=711%3A16225|https://www.figma.com/proto/m3QGmo7Fjys7um36XDhwAY/Accountability-5.0?page-id=711%3A16224&node-id=711%3A16225&viewport=352%2C48%2C0.36&scaling=scale-down-width&starting-point-node-id=711%3A16225|smart-link] 

*Requirements*

* In the Account Details page of Accounts, where there is a ‘More Actions’ drop-down menu, when a user selects the menu, show an option for *Return Check*.
* When that button is selected, open an action slide out panel should appear where the user can process a returned check.
* Details:
** Check field
*** Displays the checks currently applied to the student account.
*** Displayed as “#\[Check Number] for $\[Amount]”
*** UPDATE (2/18): Since we cannot display the total payment amount field here, let’s remove the payment amount from the Check drop down options. Instead of “Check 1234 for $40.00” (for example), display each option as just “Check #\[Check Number]. I.e “Check #1234, Check #9876, etc.
** When multiple accounts are applied to the check payment, we should display the list of students beneath some information text: “This payment was applied to this account and others. Payment and bonuses applied to those accounts will be deducted as well, but Return Check fees will be only be incurred on this account.”
** Data to show and update based on the check selected.
*** List of accounts and their related payment amounts as it relates to the selected check
*** Total Check Amount
**** Display the amount of the check payment
*** Previous Balance
**** Display the primary account’s current balance
*** Payment Amount Applied
**** Based on the amount of the payment that was applied to the primary account.
*** Bonus Amount Applied
**** Based on the bonus amount that was applied to the primary account (if any).
*** Return Check Fee
**** The Return Check Fee is the sum of BKCHKFEE and ISDCHKFEE in system settings. It should only be deducted from the primary account.
** New Balance
*** Sum of Previous Balance - (Payment Amount Applied + Bonus Amount Applied + Return Check Fee) for the primary account
** Comments
*** A comment is required for this transaction.
** “Do not accept subsequent checks on this account (and any other accounts associated with this payment)” checkbox
*** When checked, after the user selects apply, update *both* the primary account and related household accounts to not be able to accept future check payments in their account Restrictions
**** Note: the text for this checkbox should be dynamic, based on a multi-account payment when being selected from the list of checks. If multiple accounts are associated, only then should add the “(and any other accounts associated with this payment)” text. 
**** Otherwise, if can continue to say, “Do not accept subsequent checks for this account.”
* Once the requirements above are met, the user should be allow to select the Save button to complete the Returned Check, and the balances for each student should be updated appropriately.

Error handling:

* IF Comments IS NULL
** Display “A comment is required.”
* IF the selected check matches multiple patrons
** Display the message from above: “This payment was applied to this account and others. Payment and bonuses applied to those accounts will be deducted as well, but Return Check fees will be only be incurred on this account.”
* When ‘Save’ is selected, multiple un-editable *transactions* should then be entered into the patron’s account, corresponding to each fee (check amount, bonus amount (if > 0.00), return check fee (if > 0.00)).
* A *session* is also created if an open session for that site and user is not open, and a single transaction is entered into that session.
* If an *OPEN session* exists for that site and user, the transaction is added to the open session.
* This story is complete when a user can navigate to the Return Check menu from the Student Info page and perform a check return for that account and related accounts, with the option to prevent future check payments for all of them, with balance/check amount/bonus amount/check fees display changes and error handling where expected, and a transaction and session being created

## Acceptance Criteria
The user can arrive from the 'More Actions' section of Student Info page

The user sees the fields and data above, matching the mock-up

Process:

* The user is able to see all checks applied to the student account for the current academic year. User is able to select the check from the dropdown list.
* After selection, if there are other related students who received money from the check, they are listed in a table with their individual details
* Additionally, the return check information is listed: Total Check Amount, Previous Balance, Payment Amount Applied, Bonus Amount Applied, and Return Check Fee as they apply to the primary (selected) account
* After selection, if multiple accounts are associated, the information box with populated information should appear.
* The Return Check Fee is the sum of BKCHKFEE and ISDCHKFEE in system settings and is applied to the account that the return check action is performed
* A comment must be entered
* Transaction is entered to reverse the Payment Amount and Bonus on any accounts associated with the original payment. These are to be entered as negative CREDIT entries.
* Transaction is entered to record the fees on the account that the return check action is performed if the Fees are not 0 transaction type is Check Return Fee. Fees should be entered as negative CREDIT entries. Note: Other students will not incur a fee (only balance adjustments).
* The transaction is added to the session to reverse the check transaction type is Return Check. It should be entered as a negative CREDIT transaction.
* The transaction date is the current date/time.
* User can back out of the transaction by selecting 'Close' and/or 'Cancel' to return to Account Info. A prompt should warn the user before exiting to avoid accidental actions.
* The Do not accept check flag is set to false for the account (we should require the user to make a decision to not accept by checking).

# NXT-60494 — TH - IN - Insights Screen

## Description
*As a* TeamHub user *I want* a screen that provides quick access to my Insights module *So that* I can stay informed about my current operations.

h2. Comments

* This story is for {color:#ff5630}-SchoolCafé (SC) districts-{color}
** PrimeroEdge districts {color:#ff5630}-screen stories will come in another story-{color}
* Basically, we want to mimic the cards on the workspace Insights page
* Note: APIs should exist for this already from workspace 

h2. User Flow

* User opens the application and is directed to login screen
* On login success, user is redirected to home screen
* User clicks Insights and is redirected here
* User is able to view a list of KPI cards
* User is able to filter the card data per site or all sites
* User is able to filter the data by date range
* User is able to select which cards to show / hide
* User is able to set the order of the cards

h2. Requirements

h3. _Home Screen:_

* Upon logging in:
** Check if the district has the Insights license
** If no, stop
** If yes, continue
** Make a call to the Workspace Insights API for the district using the “All Sites - today” default value
** Store the values in the local storage for the Insights screen to use
** Do the call as one of the first things called when the user logs in
* Schedule
** Call the API again on a schedule per every n minutes
** Don’t need a delta
** Choose a different number of minutes than from the other sync jobs like production and insights so that they don’t all try to run at the same time(e.g. 11 minutes for this one)

h3. _Insights Screen:_

_Header Section_

* Have a back button to Workspace
* Page Title: Insights



_Filters Section_

* Sites

** Multi-select

* End Date
** Show only for Custom Dates
** Date Picker (no time)

** If shown, default to Today
* Settings button

_Widget Drawer_

* Title: Customize Cards

_Card Section_

## Acceptance Criteria
*User Flow and Pre-Sync*

* Upon login, the system checks if the district has an Insights license.
** If the district does not have an Insights license, the Insights screen is not accessible and the process stops.
** If the district does have the Insights license, the system calls the Workspace Insights API with the default parameter "All Sites - today."
* The results from the API call are stored in local storage for use by the Insights screen.
* This API call is one of the first actions executed upon login.
* A scheduled job runs every n minutes (e.g., 11 minutes) to call the API again for a full sync (no delta reconciliation), ensuring the data is up-to-date.

*Insights Screen - Header*

* The screen supports dark mode, with all UI elements adjusted for optimal readability.
* The header includes a back button that navigates the user back to the Workspace screen.
* The header displays the page title "Insights."
* The header includes a Sync section that shows the last sync time and a Sync button.
** Tapping the Sync button manually triggers the asynchronous sync job to refresh data.

*Insights Screen - Filters Section*

* The Filters section includes a dropdown list for Sites:
** Supports multi-select.
** Defaults to "All" (which includes all district sites).
** The list of sites is populated from local storage (loaded during the initial login sync).
* The Filters section includes a dropdown list for Date:
** Single select with options: Today, Yesterday, This Week, Last Week, This Month, Last Month, This Year, Last Year, All, Custom.
** Defaults to "Today."
** If "Custom" is selected, additional fields appear:
*** Start Date: Date picker (no time); defaults to 7 days prior; must be less than or equal to End Date.
*** End Date: Date picker (no time); defaults to Today; must be greater than or equal to Start Date.
* The Filters section includes a Settings button that opens a Widget Drawer.
* The Filters section includes an Apply button:
** Tapping Apply calls the Workspace API with the selected filters to update the card values.

*Widget Drawer*

* The Widget Drawer opens when the Settings button is tapped.
* It displays the title "Customize Cards."
* It lists all KPI cards with a Show/Hide toggle for each card.
** When enabled, the card is shown on the Insights screen; when disabled, it is hidden.
* Users can set the display order of the cards.
* Changes made in the Widget Drawer do not close the drawer automatically to allow multiple adjustments.

*Insights Screen - Card Section*

* The Card Section displays KPI cards grouped by category.
* Each card shows:
** A label (KPI name).
** The current value.
** The target (benchmark) value.
** The delta between current and target values.
** A visual indicator (green if above target, red if below target).
* On initial login, the card values are pulled from local storage using the "All Sites - today" default set.
* When users update filters and apply changes, the Workspace API is called to retrieve updated card values, which then update the local storage and the display.

*Data Requirements and API Integration*

* The Sites list is not fetched via an API on the Insights screen; it is loaded from local storage during login.
* Card values are stored in local storage on the initial login sync.
* API calls are made when:
** The user logs in (full sync, default "All Sites - today").
** The user updates filters (calls Workspace API to update card values).
** A scheduled full sync runs every 11 minutes.
* All API calls are made over HTTPS, using versioned endpoints (e.g., /api/v1/...) and enforce rate limiting.
* There are no direct API calls from the Insights screen for site data; all data is pulled from local storage.

*Performance & Usability*

* The Insights screen must load within 2 seconds under normal operating conditions.
* The layout must be responsive and render correctly on Mobile, Tablet, Desktop, Compact, and Expanded views.
* All tap and interactive elements (back button, sync button, filters, settings, apply) must be responsive and provide clear visual feedback.
* Lazy loading or efficient data handling should be implemented to ensure smooth user experience.

*Error Handling*

* All asynchronous calls (API calls, local storage operations) must be wrapped in try/catch blocks.
* Error boundaries must be implemented to catch unexpected render failures.
* User-friendly error messages are displayed via toast notifications, modal dialogs, or inline messages without disrupting the user experience.
* All errors (API or rendering) are logged to the central error log system/database for further analysis.

*Security*

* The Insights screen is accessible only to authenticated users.
* The district must have an active Insights license for the screen to be available.
* The user must have the "View Insights" permission assigned through their role.
* If the user lacks the required license or permission:
** The screen should show an error message: "You do not have permissions to view Insights. Contact your administrator for access."
** The Insights screen should not be accessible in the menu for these users.

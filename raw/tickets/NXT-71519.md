# NXT-71519 — K12 - Ins - Sites Update

## Description
*I am a* district user
*and I want to* filter the Insights screen by one or more sites, site types, or all sites
*so that I can* view KPI cards for the locations most relevant to me.

----

h2. *Notes*

* This story applies to the *K12 Insights mobile app*.
* The mobile app Insights screen displays:
** KPI cards
** Site filter
** Date range filter
** Configure Cards drawer
* This story covers the *site filter update only*.
* The current mobile site filter is *single select* and must be replaced.
* The new site filter already exists on the *Insights dashboard* and should be reused/adapted for mobile.
* The mobile app is written in *React*, and the filter component should be portable.
* The mobile app must use the *new multi-select API* provided by the Insights team.
* This functionality must work for both:
** *SchoolCafé*
** *PrimeroEdge*

h3. *Site Types*

* Site types are *district-dependent* and must not be hardcoded.
* The filter must display *all available site types for the logged-in district*.
* Examples include (but are not limited to):
** Elementary Schools
** Middle Schools
** High Schools

h3. *Default Behavior*

* Default selection must be:
** *All Sites*

----

h2. *Out of Scope*

* Date range filter updates
* KPI card additions or updates
* Grid, chart, or benchmark configuration functionality
* Changes to card configuration drawer behavior outside of site filter interaction

----

h2. *User Flow*

# A user logs into the K12 Insights mobile app.
# The user navigates to the *Insights* screen from the menu.
# The user opens the *Site filter*.
# The user may choose:
#* All sites
#* One or more site types
#* One site
#* Multiple individual sites
# The user applies the filter.
# The Insights screen refreshes and displays KPI cards for the selected site scope.
# The user may optionally adjust the date range or configure visible cards.

----

h2. *Requirements*

h3. *1. Replace Existing Site Filter*

* Replace the current *single-select site filter* with the new *multi-select site filter* from the Insights dashboard.
* The mobile implementation should reuse the same behavior and interaction patterns as closely as possible.

----

h3. *2. Default Selection*

* The site filter must default to:
** *All Sites*
* When no manual selection is made, all KPI data must reflect all sites.

----

h3. *3. Multi-Select Support*

* The site filter must support:
** Selecting *All Sites*
** Selecting one site
** Selecting multiple sites
* All selected sites must be applied together when the user taps *Apply*.

----

h3. *4. Site Type Support*

* The site filter must support filtering by *site type*.
* Site types must:
** Be dynamically loaded based on the logged-in district
** Not be hardcoded
* Selecting a site type must:
** Include all sites associated with that type

----

h3. *5. API Update*

* The mobile app must use the *new multi-select API*.
* The existing *single-select API* must no longer be used.
* The API must support:
** All sites
** Site types
** Multiple individual sites

----

h3. *6. Insights Screen Integration*

* After applying the site filter:
** KPI cards must refresh based on selected sites
* The site filter must work in combination with:
** Date range filter

----

h3. *7. Selection Persistence*

* The selected site filter should persist:
** During the user session
** When navigating away and returning to the Insights screen (if consistent with existing app behavior)

----

h3. *8. UI/UX Clarity*

* The filter must clearly indicate:
** When *All Sites* is selected
** When specific sites are selected
** When a site type is selected
* Users must be able to easily understand their current filter context.

----

h3. *9. Cross-Platform Support*

* The site filter must function correctly for:
** SchoolCafé
** PrimeroEdge

----

h3. *10. Performance Requirement*

* The site filter must perform efficiently on mobile devices.
* Applying filters and loading KPI cards must not introduce noticeable delay.

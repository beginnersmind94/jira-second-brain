# NXT-69933 — FIN - Post Reimbursements Due on Claim Creation (with Period Close Fallback)

## Description
*As a* district finance manager, *I want* reimbursements due to post to the general ledger when I create a claim, or automatically when the POS period closes if I don't, *so that* expected federal/state revenue is always reflected in my financial statements.

## Acceptance Criteria
h2. How Entries Are Structured

All postings post at the site level with two ledger entries: 

|| ||Debit||Credit||
|Federal|Due from Federal Funds (Asset)|Reimbursement Program Revenue – Federal|
|State|Due from State Funds (Asset)|Reimbursement Program Revenue – State|

Postings are created at the site level, per program, per funding source (State/Federal), per meal type. For each eligible funding source, create one debit/credit pair (Federal and/or State).

Entry description: "[Program Name] Reimbursement Claim" (or "Accrual" or "Reversal").

----

h2. When Posting Is Allowed

Two conditions, checked per site:

# *All POS periods for that site within the claim month are closed.* Claims are always monthly. If POS periods are weekly, all weeks in that month must be closed for that site. Entries are made for sites as their POS periods are closed, referencing their site-level sub accounts in the ledger/journal. 
# *The financial period containing the last day of the claim month is open.* This is a job-level gate. If it's closed, nothing posts for any site.
# All Reimbursement-related data points are mapped, along with the other active data points that require mapping. 

*The “Posted Date” is always the last day of the claim month.*

----

h2. Two Posting Paths

*Path 1 — User files a claim.* The user clicks "Create Claim" or "Regenerate Claim" at any time (claims are not gated by POS period status). The system processes each site: if entries already exist, reverse then repost; if not, just post. 

*Path 2 — No claim exists.* When a site's last POS period for the month closes and no claim exists, the system posts an estimated accrual (meal counts × configured rates) for that site. This fires per site, not district-wide. If the user eventually files a claim, the reversal logic cleans up the accrual automatically.

----

h2. How Reversals Work

Whenever a posting job finds existing entries for the same site + program + funding source + claim period, it reverses them before posting new ones.

* The lookup does *not* filter by meal type — it matches on site + program + funding source + claim period to catch all prior entries.
* The reversal uses the original's snapshotted GL accounts. The new posting uses the current GL configuration.
* The reversal and new posting succeed or fail together.

----

h2. What Shows in the Activity Log

||Label||When||Triggered By||
|Reimbursement Claim|User filed or regenerated a claim|System|
|Reimbursement Accrual|Site's POS periods closed, no claim exists|System|
|Reimbursement Reversal|Prior entries reversed before reposting|System|

On success: entries created, total amount, sites involved, "View in Journal →" link. On failure: the specific reason, identifying which site, program, or period caused it. 

Update on {{2026-02-25}}: After discussion with Hussain, for now reversal will fall within the first two labels instead of being its separate type. 

----

h2. Validation

Checked before any entries are created:

||Check||Scope||Result||
|Financial period a claim belongs to is closed and no claim or accrual has posted|Entire job|Notify user in “Periods” 
(Out of Scope for this story)|
|GL mapping missing|Entire job|The job fails and the user is notified this is because data points need to be mapped 
(Out of Scope for this story)|
|No state-level reimbursement|That program|No entry for “Due from State Funds” for that program (Skip, not fail)|

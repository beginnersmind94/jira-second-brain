# NXT-73629 — FIN — Custom Reports Pre-built Templates: Oracle FBDI + Tyler Munis

## Description
As a district finance user, I want a pre-configured Custom Report for my ERP, so that I can generate the file (Munis) or the ledger data (Oracle) without configuring columns from scratch.

---

## TL;DR

Depends on NXT-73628 (Financial Ledger Entries data set). Two pre-configured saved reports on top of it, shared with the district by implementation at onboarding:

* **Munis** — a complete, loadable Import Journals file. Four columns, all mapped, no constants. Ships as-is.
* **Oracle FBDI** — a **data-only** report. It outputs the ledger columns the district pastes into Oracle's own FBDI Journal Import spreadsheet, which already holds the fixed values (STATUS, Ledger ID, source, category, currency, actual flag) and supplies DATE_CREATED / CREATED_BY. The district pastes, generates the CSV, and loads it — a few minutes per period.

**Why Oracle is data-only:** Custom Reports cannot output a constant-value column for a district user (only a Customer Support–authored Dynamic Query can). Rather than hand-build SQL per district, we hand Oracle districts clean ledger data and let Oracle's standard FBDI spreadsheet stamp the constants — which is how Oracle finance teams already do it.

**Note — Custom Reports does not have a "start from template" wizard.** These are saved reports, pre-configured and shared. The district can run, clone, reschedule, or set up FTP using standard Custom Reports tools.

---

## Scope

**In scope:** Two pre-configured saved reports (Munis Import Journals; Oracle FBDI data-only), created and shared by implementation at onboarding. A one-page "load to Oracle via FBDI" instruction for district finance.

**Out of scope:** Changes to the Custom Reports framework. A constant-value column for district users (does not exist — see Future option). Turnkey/automated Oracle delivery. Upload to Oracle UCM or Munis SFTP.

---

## Requirements

### 1) Delivery mechanism

1.1 Both reports are created by implementation at district onboarding using the standard Custom Reports creation wizard, built on the Financial Ledger Entries data set.

1.2 Reports are shared with the appropriate district roles via the Sharing tab (not PII-gated — financial ledger data is not student PII).

1.3 The district user receives a share notification and can run, clone, or configure distribution/scheduling on the shared report.

### 2) Munis — Import Journals report (complete file)

| Column (Alt Name) | Source |
| --- | --- |
| Account | Account Number (full GL account string as stored) |
| Debit | Debit (2 decimals, no symbols, no separators; blank on credit lines) |
| Credit | Credit (2 decimals, no symbols, no separators; blank on debit lines) |
| Description | Description |

2.1 No constants. Batch and journal header values are entered in the Munis UI at import time. This file is loadable as produced.

### 3) Oracle FBDI — data-only report

Outputs only the ledger columns the district pastes into Oracle's FBDI Journal Import spreadsheet. Column order is set in the report (Source Table drag order) and preserved on export, so it lines up with the spreadsheet's ledger columns.

| Column (Alt Name) | Source | Pastes into Oracle column |
| --- | --- | --- |
| Posted Date | Posted Date (YYYY/MM/DD) | ACCOUNTING_DATE |
| Segment 1 … Segment N | Segment 1 … Segment N per the district's Account Code Structure | SEGMENT1 … SEGMENTn |
| Debit | Debit (2 decimals, no symbols, no separators; blank on credit lines) | ENTERED_DR |
| Credit | Credit (2 decimals, no symbols, no separators; blank on debit lines) | ENTERED_CR |
| Transaction Number | Transaction Number | REFERENCE4 |
| Description | Description | REFERENCE5 |

3.1 The report contains no constant or header columns. STATUS, LEDGER_ID, USER_JE_SOURCE_NAME, USER_JE_CATEGORY_NAME, CURRENCY_CODE, ACTUAL_FLAG, DATE_CREATED, and CREATED_BY are all supplied in Oracle's FBDI spreadsheet — the district fills the constants once (fill-down) and the spreadsheet supplies DATE_CREATED / CREATED_BY.

3.2 Posted Date already comes through as YYYY/MM/DD from the data set (NXT-73628) — the format Oracle requires — so the paste lands in the right format with no formatting work by the district. One-time check at the first district's setup: confirm Excel doesn't re-interpret it when pasted into the FBDI spreadsheet (starting from YYYY/MM/DD, the worst case is it stays unchanged).

3.3 Segments map positionally — Segment 1 → SEGMENT1, and so on. This assumes the district's Account Code Structure segment order matches their Oracle chart-of-accounts segment order; confirm at onboarding (a mismatch produces a valid-but-wrong file).

### 4) Amount format (both reports)

4.1 Two decimals, no currency symbols, no thousands separators. Example: `1234.56`, not `$1,234.56`.

### 5) Scheduling and FTP

5.1 Implementation sets up scheduling and FTP at onboarding if the district requests it; the district can change it later using standard Custom Reports tools.

5.2 Munis can be scheduled and delivered end to end. Oracle's scheduled/FTP delivery produces the data file; the district still does the paste-and-load in Oracle's spreadsheet each period.

## Edge / Empty / Error States

| Case | Behavior |
| --- | --- |
| Selected period has no journal entries | File contains column headers only |
| District's Account Code Structure has N segments | Oracle report outputs exactly N segment columns (from the data set) |
| District modifies a pre-configured report | Changes persist; original not restored — advise district to clone before modifying |

---

## Defaults

* Both reports: Contains PII = No.
* Munis: complete loadable file, no constants.
* Oracle: data-only, no constants; constants live in Oracle's FBDI spreadsheet.
* Both: all columns pre-wired to Financial Ledger Entries data set fields.

---

## Future option (not now — build only on demand)

When Oracle self-service demand outgrows the manual paste step, upgrade Oracle to a turnkey scheduled file via a Customer Support–authored **Dynamic Query** report: raw SQL bakes in the constants, the YYYY/MM/DD date, 2-decimal amounts, and the district's segment split → a complete loadable file delivered on schedule. This is **dev-cheap because the Financial Ledger Entries data set (NXT-73628) does not change** — the upgrade is an additive report, and Munis and the data-only Oracle report stay exactly as they are; nothing built now is thrown away. Resolve at that time: (1) does a Dynamic Query report still support Sharing / Scheduling / FTP; (2) does it build on the registered data set or query the ledger directly. A separate backlog item should track the longer-term alternative — a self-service constant-value column in Custom Reports — once demand justifies platform work.

## Acceptance Criteria
**AC1 — Pre-configured reports are visible to the district**

**Given **implementation has created and shared the Munis and Oracle FBDI reports with the district,
**When **a district user opens Custom Reports,
**Then **both reports appear in their dashboard and can be run, cloned, or edited.

**AC2 — Oracle FBDI data report outputs ledger columns only**

**Given **the Oracle FBDI data report runs for a selected period,
**When **the file is generated,
**Then **it contains only these columns, in order — Posted Date, Segment 1…N, Debit, Credit, Transaction Number, Description — and no constant or header columns (no STATUS, LEDGER_ID, USER_JE_SOURCE_NAME, USER_JE_CATEGORY_NAME, CURRENCY_CODE, ACTUAL_FLAG, DATE_CREATED, or CREATED_BY). Those are supplied by the district in Oracle's FBDI Journal Import spreadsheet.

**AC3 — Amount formatting**

**Given **a debit line of $1,234.56,
**When **either report generates the file,
**Then **the amount appears as 1234.56 — no comma, no currency symbol. Credit lines show the credit column populated and the debit column blank.

**AC4 — Munis report produces a complete four-column file**

**Given **the Munis report runs for a selected period,
**When **the file is generated,
**Then **it contains exactly four columns in order — Account, Debit, Credit, Description — with amounts formatted per AC3 and no constant columns (Munis header values are entered in the Munis UI at import).

**AC5 — District can schedule and set FTP delivery**

**Given **a pre-configured report shared with the district,
**When **the district user opens it and configures the Scheduling and Distribution tabs,
**Then **they can set a recurring schedule and FTP destination using standard Custom Reports tools, and the file is delivered on schedule.

**AC6 — Cloning preserves configuration**

**Given **a district user clones a pre-configured report,
**When **they open the clone,
**Then **all columns, Alt Names, and data source mapping are preserved in the clone.

# NXT-71351 — Insights - KPI - PNA - Configuration - SC

## Description
*I am a* district user *that I want to* configure benchmark values for Paid Not Applied (PNA) in Insights using SchoolCafé data
*so that I* can evaluate whether schools are meeting expected eligibility participation thresholds.

----

h2. *Notes*

* This story applies to *Insights – SchoolCafé data source*.
* PNA (Paid Not Applied) is part of the *front office KPIs* being integrated into Insights.
* This story covers the *Benchmark Configuration screen only*.
* This story is a *companion to the PrimeroEdge PNA Benchmark Configuration story*.
* The *frontend behavior, UI, validation rules, formatting, grid behavior, autofill, bulk update, history, and save behavior must remain the same as the PrimeroEdge implementation*.
* Frontend developers should *not need to build new UI logic* for SchoolCafé.
* This implementation must reuse the same:
** benchmark configuration grid
** validation logic
** autofill behavior
** bulk update behavior
** history tracking
** save behavior

----

h3. *Data & Source of Truth*

* The source of truth for PNA data is available in:
** *SchoolCafé → Eligibility Module → Reports → Roster Report*
* To generate the correct dataset:
** Eligibility = *Paid*
** Basis = *Default*
** Click *Apply*
* This must be used by developers and QA to validate PNA values.

----

h3. *Stored Procedure Integration*

* The front office team has already created the stored procedure:
** SchoolCafé: [https://primeroedgenext.atlassian.net/browse/NXT-67455|https://primeroedgenext.atlassian.net/browse/NXT-67455|smart-link] 
* The system must use the *SchoolCafé stored procedure* as the data source instead of the PrimeroEdge stored procedure.

----

h3. *KPI Definition & Formatting*

* PNA is a *percentage-based KPI*.
* Format must be:
** *0.00% (two decimal places)*
* Values must:
** Be *rounded to two decimal places*
** Be within the range *0 to 100*
* Calculation:
** *(Count of students where Eligibility = Paid AND Basis = Default) ÷ Total relevant student population × 100*

----

* PrimeroEdge implementation (covered in separate story)

----

h2. *User Flow*

* Same as *PrimeroEdge PNA Benchmark Configuration story*.

----

h2. *Requirements*

h3. *1. Shared Frontend Implementation*

* The SchoolCafé implementation must reuse the *same frontend behavior and UI* as the PrimeroEdge benchmark configuration.
* No new or separate frontend workflow should be introduced for:
** grid behavior
** manual entry
** validation
** autofill
** bulk update
** history
** save behavior

----

h3. *2. Backend Data Source Change*

* The backend must use the *SchoolCafé PNA stored procedure* instead of the PrimeroEdge stored procedure.
* This applies to:
** actual PNA values
** autofill values
** year-based values
** dataset population

----

h3. *3. Functional Consistency*

* All SchoolCafé PNA benchmark configuration behavior must remain identical to the PrimeroEdge implementation, including:
** percentage formatting
** validation rules (0–100, 2 decimals)
** required field behavior
** bulk update logic
** autofill logic

----

h3. *4. Data Validation*

* SchoolCafé PNA values must match:
** SchoolCafé stored procedure output
** SchoolCafé Eligibility Roster Report

----

h3. *5. Performance Requirement*

* The SchoolCafé-backed implementation must perform at the same level as the PrimeroEdge implementation and must not introduce performance degradation.

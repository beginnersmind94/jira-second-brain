# NXT-71500 — SC - PBIRD - Activity Report – Add New

## Description
As a user viewing the Activity Report, I want to initiate the creation of a distribution configuration when one does not exist so that I can schedule and distribute this report.

----

h2. *Description*

When a user initiates distribution from the Activity Report page and no existing distribution configuration exists, the system will prompt the user to create a new configuration.

If the user confirms, they will be redirected to the *Power BI Reports module* and the shared *Add New Power BI report configuration screen* will be launched, preconfigured for the Activity Report.

This story focuses only on the *initial validation and redirection into the Add New flow*.

----

h2. *User Flow*

# User navigates to:
#* *Accountability → Reports → Activity Report*
# User configures filters and generates the report
# User clicks *ellipsis (⋮)* → *Distribute*
# System checks for an existing distribution configuration
# No configuration exists
# System displays confirmation message
# User selects:
#* *Yes* → Redirect to Add New configuration flow
#* *No* → Stay on Activity Report page

----

h2. *Out of Scope*

* Behavior when a distribution configuration already exists _(covered in separate story)_
* Distribution, scheduling, or execution logic
* Power BI Reports dashboard functionality beyond redirect

----

h2. *Requirements*

h3. *Configuration Check*

* When user clicks *Distribute*:
** System checks if a distribution configuration exists for the report
* If no configuration exists:
** Display message:

{quote}*"The report currently does not have a distribution configuration. Would you like to create one?"*{quote}

----

h3. *User Decision*

* If user selects *Yes*:
** Redirect to Power BI Reports module
** Automatically launch Add New configuration screen
* If user selects *No*:
** Close dialog
** Remain on Activity Report page

----

h3. *Cross-Module Navigation*

* The Add New screen resides in the *Power BI Reports module*, not within the Activity Report screen
* User must be redirected to:
** *Reports → Power BI Reports*
* The Add New screen should automatically open upon navigation

----

h3. *Add New Screen Usage (Critical)*

* Use the *existing Add New screen within the Power BI Reports module*
* This screen is shared across *all Power BI report types* (e.g., Activity Report, Revenue Report, etc.)
* *Do NOT create separate Add New screens per report type*
** Example of what should NOT happen:
*** Activity Report Add New screen
*** Revenue Report Add New screen
*** etc.
* The system must pass necessary parameters so the screen initializes for:
** Activity Report configuration

----

h3. *Shared Components Clarification*

* Each report type has its own Add New screen:
** Managed View Reports → Managed View Add New
** Custom Reports → Custom Reports Add New
** Power BI Reports → Power BI Add New
* Within these flows, the following tabs/components are shared:
** Distribution
** Scheduling
** Review
** (Sharing may be optional)

----

h3. *Stepper Flow*

* Use the standard stepper pattern:
** Selection
** Configuration
** Distribution
** Scheduling
** Review
* *Sharing tab:*
** May be skipped or hidden
** Dev team may remove or leave as-is based on effort

----

h3. *Selection Screen*

* Use existing Power BI Selection screen (aligned with current patterns)
* Default values:
** *Name*:
*** Default: {{Activity Report {MM-DD-YY}}}
*** Example: Activity Report 03-25-26
*** Dev may refine naming logic if needed
** *Owner*:
*** Default to logged-in user
** *Module*:
*** Default to *Accountability*
** *Contains PII*:
*** Default to *No*
* Required field validation and flow follow existing report patterns

----

h3. *Configuration Screen (Power BI Specific)*

* This is the only screen that differs from other report types
* Display Power BI configuration details:
** Prefer formatted/structured display
** Raw JSON may be used if necessary
* If a configuration object (e.g., JSON) exists:
** Display it in a readable format
** Improve usability compared to current Managed View implementation if feasible

----

h3. *Distribution Screen*

* Use existing Distribution component with no changes
* *Dev Note:*
** Validate whether *Threshold* is supported for Power BI reports
** Confirm feasibility with Dev Manager

----

h3. *Scheduling & Review*

* Use existing shared components with no changes
* Behavior consistent with other report types

----

h3. *Save Behavior*

* On Save:
** Persist the Power BI report configuration using the *Power BI reporting framework and database structures*
* The process should follow the *same pattern and flow* as Managed View Reports:
** Validation
** Save
** Confirmation
* *Important Clarification:*
** Do NOT save Power BI reports in Managed View report tables
** Power BI reports must be stored in their *own appropriate data structures*

----

h3. *Post-Save Navigation*

* After successful save:
** Redirect user to *Power BI Reports dashboard*
** Newly created report is visible in the list

----

h3. *Report Identification*

* Power BI reports must be referenced using a *unique identifier from Power BI Report Server*
* Do NOT rely on:
** Report name alone
* The system should use:
** Report ID or path from Report Server
* This ensures:
** Proper mapping across environments
** No conflicts with duplicate report names

----

h3. *Audit Logging*

* When a new Power BI report configuration is created:
** The action must be recorded in the *audit log*
* Audit entry must include:
** User
** Timestamp
** Action type (Create)
* The record must be viewable in:
** *History drawer*
* Behavior must align with Managed View Reports

----

h3. *Naming Behavior*

* Default naming convention:
** {{Activity Report {MM-DD-YY}}}
* If multiple reports are created with the same name:
** System must follow the same *naming conflict resolution logic* as Managed View Reports
* Developers may:
** Reuse or extend existing naming logic

## Acceptance Criteria
* Given a user selects Distribute and no configuration exists, when the system evaluates the report, then a confirmation message is displayed
* Given the confirmation message is displayed, when the user selects Yes, then the system redirects to the Power BI Reports module and opens the Add New configuration screen
* Given the user selects No, when the dialog closes, then the user remains on the Activity Report page
* Given the Add New screen is opened, when it loads, then it is preconfigured for the Activity Report with default values populated
* Given the user completes and saves the configuration, when the process finishes, then the user is redirected to the Power BI Reports dashboard and the new report is visible

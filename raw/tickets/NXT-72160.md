# NXT-72160 — FIN — Payroll Import: Data Exchange

## Description
As a district finance user, I want to import journal entry lines into Financials via Import Monitor, so that recurring payroll and other month-end journal entries can be loaded efficiently without manual data entry.

_Updated 2026-05-04. Created 2026-04-21 as the Data Exchange companion to **NXT-70777** (Business Logic)._

---

## Scope

This story covers the Import Monitor pipeline for the Payroll Import type: import type setup, file upload, one-time template configuration (Amount Format + Column Mapping), column mapping, row-level and import-level validation, derived period assignment from PostingDate, preview, saved template behavior, import results, and duplicate detection warning. On successful confirmation, validated data is handed off to the business logic layer (companion story **NXT-70777**) for journal entry creation and ledger posting.

---

## Requirements

### 1) Import Type

1.1 A new Import Monitor Import Type exists named **Payroll Import** with help text: _"Used to load payroll and other month-end journal entry lines into Financials."_

1.2 Import Type supports the following upload formats, consistent with other Import Monitor types: **xlsx, xls, csv**.

1.3 Import Type is available only to users with appropriate Financials permissions.

1.4 Files must contain a single sheet. Multi-sheet files are rejected at upload with a clear error directing the user to consolidate to a single sheet and re-upload. The system does not attempt to process any sheet from a multi-sheet file.

### 2) Import Template (one-time setup)

2.1 On the first import for this import type, the user must configure an **Import Template** consisting of:

* **Amount Format** — one of:

    * _"My file has separate Debit and Credit columns"_ → Scheme A
    * _"My file has a single Amount column (positive = debit, negative = credit)"_ → Scheme B
    
* **Column Mapping** — mapping of uploaded file columns to target fields (see Sections 5 and 6)

2.2 The Amount Format selection determines which amount target fields appear in the column mapping UI. Scheme A displays Debit and Credit as required target fields. Scheme B displays Amount as the required target field. The non-selected fields must not appear in the mapping UI.

2.3 The template is saved after the first successful configuration. The user can edit the template (amount format and column mapping) at any time.

2.4 Only one template exists per district for this import type. Configuring a new template replaces the existing one.

### 3) Period Assignment (derived from PostingDate)

3.1 Each row's fiscal year and period are **derived from its PostingDate** in the district's configured timezone. The user does not select fiscal year or period as a separate step — the date in the file is the authoritative period assignment for that row.

3.2 The derived period assignment must be displayed prominently in the preview as a per-period summary (e.g., _"This file will post to: FY2025-26 August (142 rows, $48,219.43 debit / $48,219.43 credit)"_) so the user can sanity-check the breakdown before confirming. Files that span multiple periods show one summary line per period.

3.3 If **any row's PostingDate falls in a closed period**, the entire import is blocked. The system does not accept partial imports of the rows whose dates fall in open periods. The blocking message must:

* Identify the closed period(s) affected and the count of rows whose dates fall in each.
* Route the user to the supported alternative for affecting closed-period financials:

> _"This file contains rows with posting dates in closed periods. To affect a closed period's financials, post a prior-period adjustment to the current open period. Correct the affected rows or remove them, then re-upload."_

with a link to the manual adjustment entry flow.

3.4 If any row's PostingDate falls in a fiscal year or period that does not exist in the system (e.g., the FY has not been created), the entire import is blocked with a clear error directing the user to fix the date(s) or have an administrator create the missing fiscal year before retrying.

3.5 On import confirmation, the system must re-derive period assignments and re-verify that no row's PostingDate now falls in a closed period or unconfigured period. If any period has been closed since the preview was loaded, the confirm action must be blocked with the same all-or-nothing rule. _(The business logic layer applies the same checks as a backstop — see companion story **NXT-70777**.)_

### 4) Import Flow

4.1 **First import (no template exists):** Upload → Amount Format Selection → Column Mapping → Save Template → Preview (with derived period summary) → Confirm.

4.2 **Subsequent imports (template exists):** Upload → System displays a reminder of the saved amount format (e.g., _"Your template uses separate Debit and Credit columns."_) → Preview (with derived period summary) → Confirm.

4.3 On subsequent imports, the saved column mapping and amount format are automatically applied. The user does not re-map columns or re-select the amount format unless they choose to edit the template.

### 5) Required Target Fields

| Field | Notes |
| --- | --- |
| GLAccount | Account identifier per Financials rules |
| SiteCode | Site the entry applies to. Must reference a valid site in Site Management. For district-wide entries, use the site code designated as the Central Office in Site Management. |
| Description | Free text description for the journal entry line |
| PostingDate | Date the entry should post to the ledger. Determines the row's fiscal year and period assignment (see Section 3). |

**Amount fields (determined by the Amount Format in the template):**

| Amount Format | Target Fields Shown | Convention |
| --- | --- | --- |
| Separate Debit and Credit columns | Debit, Credit | Separate positive values in each column |
| Single Amount column | Amount | Positive = debit, negative = credit |

### 6) Optional Target Fields

| Field | Notes |
| --- | --- |
| AccountName | Descriptive label for the account |
| Comments | External reference identifier; free text; max 50 characters |

### 7) Validation Rules (row-level)

7.1 GLAccount, SiteCode, Description, and PostingDate must be present for each row. Amount fields must be present per the template's amount format.

7.2 Amount validation must follow the template's amount format:

**If Scheme A (Debit/Credit):**

* Exactly one of Debit or Credit must be a positive number per row.
* Negative values are not allowed.

**If Scheme B (Amount):**

* Amount must be non-zero.
* Negative amounts are allowed (treated as credits by convention).

7.3 Zero values are not allowed in either scheme and will be flagged as invalid.

7.4 PostingDate must be a parseable date. Unparseable date values are flagged as row-level errors. The closed-period and unconfigured-period checks against PostingDate are import-level blocks (see §3.3 and §3.4) rather than row-level errors, because the all-or-nothing rule means a single offending row blocks the entire import.

7.5 GLAccount is validated against the chart of accounts. Unknown accounts are imported with a **warning** (not a blocking error) and the account is surfaced in Configuration > General Ledger > GL Accounts with a warning indicator so finance can review. _(Payroll-specific behavior, distinct from other imports that block on unknown accounts — payroll feeds legitimately reference newly-introduced accounts that need review but should not halt month-end processing.)_

7.6 SiteCode must reference a valid site in Site Management. Unknown site codes are flagged as invalid.

### 8) Validation Rules (import-level)

8.1 The system must calculate and display:

* Total rows
* Total debit-side amounts and total credit-side amounts (computed from the template's amount format), shown both overall and per derived period

8.2 Import must be blocked if the absolute difference between total debits and total credits exceeds **$0.01**. A difference of $0.01 or less is allowed to accommodate rounding across line items. The variance check applies to the file as a whole, not per period — a payroll file is a single balanced unit even when its rows span more than one period. Totals and any imbalance amount must be displayed to the user alongside the blocking message.

8.3 The system must detect potential duplicate imports for any (Fiscal Year, Period) the file would post to. If any prior successful Payroll Import exists for any of the periods this file targets, the user must explicitly acknowledge a duplicate warning before the import can proceed. The warning must identify the specific period(s) where a prior import exists. The system must not silently allow duplicate imports. _(The specific detection mechanism and any ledger-layer backstop are defined in the business logic layer — see companion story **NXT-70777**.)_

8.4 The import must be **atomic** — either all rows are imported successfully, or no rows are imported. Partial imports are not allowed. If any system-level error occurs during import processing, the entire import must roll back and the user must be notified that no data was written.

### 9) Preview and Confirm

9.1 After file upload and (on first import) template configuration, the user must see a preview showing: per-period derived assignment summary (per §3.2), the amount format from the template, totals (overall and per period), a sample of parsed rows, and any validation errors or warnings before confirming.

9.2 The user must explicitly confirm to perform the import.

9.3 If any rows contain blocking validation errors (per 7.1, 7.2, 7.3, 7.4, 7.6, 8.2) OR any row's PostingDate falls in a closed/unconfigured period (per 3.3, 3.4), the confirm action must be disabled. All blocking errors must be resolved by correcting and re-uploading the file before the import can proceed. The system must not allow the user to import only the valid rows. GLAccount warnings (per 7.5) are non-blocking and do not disable confirm.

9.4 If a duplicate is detected (per 8.3), the confirm action remains disabled until the user explicitly acknowledges the duplicate warning.

### 10) Saved Template Behavior

10.1 The user can save column mapping and amount format as a template for the Payroll Import type.

10.2 Future imports automatically apply the saved amount format and column mapping and proceed directly to preview.

10.3 The user can edit or update the saved mapping and amount format at any time. Editing the template does not affect previously-completed imports.

### 11) Import Results and Auditability

11.1 Each import must produce an import result record consistent with Import Monitor patterns (status, counts, timestamps, user).

11.2 The user can view import results including:

* Success/failed status
* Per-period breakdown (which fiscal years and periods received rows from this import, and the row count and totals for each)
* Total debits and credits
* Record counts
* Row-level error messages for invalid records
* GLAccount warnings for rows with unknown accounts

11.3 The import record must be visible in audit history so administrators can see when journal entries were imported and by whom.

### 12) Handoff to Business Logic

12.1 On successful confirmation, the data exchange layer passes the validated import data to the business logic layer (**NXT-70777**), with each row carrying its derived (Fiscal Year, Period) assignment alongside its PostingDate. The business logic layer is responsible for:

* Creating the journal entry header and lines
* Posting lines to the ledger with the correct PostingDate per row, into the period each row's date assigns it to
* Applying backstop period-status verification (defense-in-depth) — if any row's period has closed since the data exchange layer's check, the entire import is rejected
* Applying backstop duplicate detection at the ledger level for each (FY, Period) the import targets
* Ensuring the ledger write is atomic
* Handling the persistence of GLAccount warnings (rows with unknown accounts are written, but the account is flagged for review)

## Acceptance Criteria
**AC1 — Import Type Available** Given an authorized Finance user opens Import Monitor, When they view Import Types, Then Payroll Import is available.

**AC2 — First Import Requires Template Setup** Given no template exists for Payroll Import, When the user uploads a file, Then they are required to configure Amount Format and Column Mapping before reaching Fiscal Year/Period selection.

**AC3 — Subsequent Imports Skip Template Setup** Given a template exists, When the user uploads a file, Then the saved amount format and column mapping are auto-applied and the flow proceeds directly to Fiscal Year/Period selection.

**AC4 — Amount Format Selection Drives Mapping UI** Given a user selects "My file has separate Debit and Credit columns," When the column mapping UI loads, Then Debit and Credit appear as required target fields and Amount does not appear. Given a user selects "My file has a single Amount column," When the column mapping UI loads, Then Amount appears as the required target field and Debit/Credit do not appear.

**AC5 — Fiscal Year and Period Required** Given a user uploads a file, When they proceed past file upload (first or subsequent import), Then they must select a Fiscal Year and Period before reaching Preview.

**AC6 — Period Displayed as Month Name** Given the user opens the Period dropdown, When the dropdown renders, Then periods are displayed as month names corresponding to their position in the fiscal year (e.g., "July" for period 1 of a July-start FY).

**AC7 — Closed Periods Excluded from Dropdown** Given one or more periods in the selected fiscal year are closed, When the user opens the Period dropdown, Then closed periods are not available for selection.

**AC8 — Closed Period Blocked at Confirm** Given the user selected a period at dropdown load, and the period has since been closed, When the user confirms the import, Then the system blocks the import with an error directing the user to reopen the period through standard period management before retrying. _(The business logic layer enforces the same rule as a backstop.)_

**AC9 — Required Fields Enforced** Given an upload contains rows missing GLAccount, SiteCode, Description, PostingDate, or the mapped amount field(s), When preview runs, Then those rows are flagged invalid and cannot be imported until corrected.

**AC10 — Amount Validity Enforced** Given an upload using Scheme B contains rows with Amount = 0, or an upload using Scheme A contains rows where both Debit and Credit are populated or both are zero, or any row has a negative value in Scheme A, When preview runs, Then those rows are flagged invalid.

**AC11 — PostingDate Must Fall Within Selected Period** Given an upload contains rows with PostingDate outside the date range of the selected Period, When preview runs, Then those rows are flagged invalid with the error "Posting date does not fall within the selected period."

**AC12 — Unknown GLAccount Warns but Does Not Block** Given an upload contains a GLAccount not in the chart of accounts, When preview runs, Then that row is flagged with a warning (not a blocking error), the account is surfaced in Configuration > General Ledger > GL Accounts with a warning indicator, and the confirm action remains available if no other blocking errors exist.

**AC13 — SiteCode Validated Against Site Management** Given an upload contains a SiteCode that does not exist in Site Management, When preview runs, Then that row is flagged invalid with a clear error consistent with Import Monitor patterns.

**AC14 — Batch Must Balance Within $0.01 Tolerance** Given the absolute difference between total debits and total credits exceeds $0.01, When the user attempts import, Then import is blocked and totals are displayed with the imbalance amount. Given the absolute difference is $0.01 or less, When the user attempts import, Then import proceeds.

**AC15 — Blocking Row Errors Prevent Import** Given an upload contains one or more rows with blocking validation errors, When the user views the preview, Then the confirm action is disabled and the system must not allow importing only the valid rows. GLAccount warnings (per AC12) do not disable confirm.

**AC16 — Import Is Atomic** Given the user confirms an import and a system-level error occurs during processing, When the error is encountered, Then the entire import rolls back, no data is written, and the user is notified that the import failed.

**AC17 — Duplicate Detection Warning** Given a potential duplicate import for the same Fiscal Year and Period is detected, When the user reaches preview, Then the system displays a warning and the user must explicitly acknowledge before proceeding to confirm.

**AC18 — Results Visible** Given an import completes, When the user opens import results, Then they can see status, selected Fiscal Year and Period, record counts, totals, row-level errors, and any GLAccount warnings.

**AC19 — Template Edit Allowed** Given a template exists, When the user chooses to edit the template, Then they can update the amount format and column mapping at any time, and previously-completed imports are unaffected.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| Empty file or header-only | Import blocked: _"No data rows found."_ |
| Non-numeric Amount field | Row invalid with parsing error |
| Both Debit and Credit populated in same row (Scheme A) | Row invalid: _"Exactly one of Debit or Credit must have a value."_ |
| Both Debit and Credit are zero in same row (Scheme A) | Row invalid: _"Exactly one of Debit or Credit must have a positive value."_ |
| Negative value in Debit or Credit column (Scheme A) | Row invalid: _"Debit and Credit values must be positive."_ |
| Zero Amount value (Scheme B) | Row invalid; zero amounts are not allowed |
| Unknown GLAccount | Row flagged with warning (not blocking). Account surfaced in GL Accounts configuration with warning indicator. Confirm remains available. |
| Unknown SiteCode | Row invalid with clear error consistent with Import Monitor patterns |
| PostingDate outside selected Period's date range | Row invalid: _"Posting date does not fall within the selected period."_ |
| Period closed at dropdown load | Period not available for selection |
| Period closed between dropdown load and confirm | Import blocked at confirm with error directing user to reopen the period through period management before retrying |
| Total debits and credits differ by more than $0.01 | Import blocked; totals and imbalance amount displayed |
| Total debits and credits differ by $0.01 or less | Import proceeds (rounding tolerance) |
| Potential duplicate import detected for same FY + Period | Warning displayed; user must explicitly acknowledge before confirm is enabled |
| File contains mix of valid and invalid rows | Confirm action disabled; all blocking errors must be resolved before import. No partial import allowed. (Warnings alone do not block.) |
| System error during import processing | Entire import rolls back; no data written; user notified of failure |
| Multi-sheet file uploaded | Only the first sheet is processed; other sheets are ignored |

## Defaults

- **Fiscal Year and Period:** Selected at the import level after file upload. Not mapped from the file. Not saved as part of templates. Selected fresh for each import.
- **Amount format selection:** The user selects their amount format during first-import template setup (or when editing the template). The import UI must surface the appropriate guidance text based on the selection. For Scheme B: _"Positive values = debit, negative values = credit."_ For Scheme A: _"Enter a positive value in exactly one column per row."_
- For district-wide entries, the user should use the site code designated as the Central Office in their Site Management configuration. The import UI guidance text should read: _"For district-wide entries, use the site code assigned to your Central Office in Site Management."_
- **Period display:** Periods are rendered as month names relative to the fiscal year start, not as period numbers.
- AccountName and Comments default to blank/null when not mapped.

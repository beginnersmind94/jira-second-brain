# NXT-72160 — FIN — Payroll Import: Data Exchange

## Description
As a district finance user, I want to import journal entry lines into Financials via Import Monitor, so that recurring payroll and other month-end journal entries can be loaded efficiently without manual data entry.

_Updated 2026-05-06. Created 2026-04-21 as the Data Exchange companion to **NXT-70777** (Business Logic). Aligned 2026-05-05 with NXT-71249 patterns. Updated 2026-05-06 to apply DE/BL separation of concerns established in NXT-71249: Validate button moves to Field Mapping, DE does not compute totals (BL API pass/fail), sample rows → FILE DATA PREVIEW, saved mapping auto-persists (no named templates), error report generated by BL. Corrected §4: restored Pre-upload tab (standard Import Monitor entry point)._

---

## Scope

This story covers the Import Monitor pipeline for the Payroll Import type: import type setup, file upload, column mapping, validator gate, row-level and import-level validation, derived period assignment from PostingDate, Field Mapping and Review tabs, saved mapping behavior, import results, and duplicate detection warning. On successful confirmation, validated data is handed off to the business logic layer (companion story **NXT-70777**) for journal entry creation and ledger posting.

---

## Requirements

### 1) Import Type

1.1 A new Import Monitor Import Type exists named **Payroll Import** with a tooltip on the Import Type selection: _"Loads payroll and other month-end journal entries. File must have separate Debit and Credit columns."_

1.2 Import Type supports the following upload formats, consistent with other Import Monitor types: **xlsx, xls, csv, txt**.

1.3 Access to the Payroll Import is controlled via roles and permissions. The following roles have Edit access by default: **Director**, **Central Office**, and **SchoolCafe Admin**. Customer Support and Technical Support roles have full access regardless of other role configuration, consistent with platform support patterns.

1.4 Files must contain a single sheet. Multi-sheet files are rejected at upload with a clear error directing the user to consolidate to a single sheet and re-upload. The system does not attempt to process any sheet from a multi-sheet file.

### 2) Amount Format

2.1 Files use a single canonical format with separate **Debit** and **Credit** columns. Files with a signed amount column (positive = debit, negative = credit) are supported via transformation: DE detects the shape during validation (§8) and transforms signed amounts into separate Debit/Credit values before Field Mapping.

2.2 The user is not asked to choose an amount format. The Field Mapping screen always shows Debit and Credit as the canonical target fields.

### 3) Period Assignment (derived from PostingDate)

3.1 Each row's fiscal year and period are **derived from its PostingDate** in the district's configured timezone. The user does not select fiscal year or period as a separate step — the date in the file is the authoritative period assignment for that row.

3.2 The derived period assignment must be displayed prominently on the Review tab as a per-period summary (e.g., _"This file will post to: FY2025-26 August (142 rows, $48,219.43 debit / $48,219.43 credit)"_) so the user can sanity-check the breakdown before confirming. Files that span multiple periods show one summary line per period.

3.3 If **any row's PostingDate falls in a closed period**, the entire import is blocked at the validator gate (§8.1). The system does not accept partial imports of the rows whose dates fall in open periods. The blocking message must:

* Identify the closed period(s) affected and the count of rows whose dates fall in each.
* Route the user to the supported alternative for affecting closed-period financials:

> _"This file contains rows with posting dates in closed periods. To affect a closed period's financials, post a prior-period adjustment to the current open period. Correct the affected rows or remove them, then re-upload."_

with a link to the manual adjustment entry flow.

3.4 If any row's PostingDate falls in a fiscal year or period that does not exist in the system (e.g., the FY has not been created), the entire import is blocked at the validator gate with a clear error directing the user to fix the date(s) or have an administrator create the missing fiscal year before retrying.

3.5 At final confirm, the system must re-derive period assignments and re-verify that no row's PostingDate now falls in a closed period or unconfigured period. If any period has been closed since the validator gate ran, the confirm action must be blocked with the same all-or-nothing rule and the user is returned to validation. _(The business logic layer applies the same checks as a backstop — see companion story **NXT-70777**.)_

### 4) Import Flow

The import flows through four tabs, consistent with the standard Import Monitor structure:

* **Pre-upload** — user selects Import Type (Payroll Import) and File Format. No FY/Period selection — period is derived from PostingDate per row (§3).
* **File Upload** — user uploads file, detects delimiter, FILE DATA PREVIEW shows raw parsed rows.
* **Field Mapping** — user maps file columns to target fields, runs Validate (§8.1).
* **Review** — final confirm with per-period summary, totals, and duplicate warnings (§9.2).

There is no amount format selection step — files use the single canonical format with separate Debit/Credit columns (§2).

### 5) Required Target Fields

The Field Mapping screen displays the following required target fields. The same fields appear for every Payroll Import — no conditional fields.

| Field | Notes |
| --- | --- |
| GLAccount | Account identifier per Financials rules |
| SiteCode | Site the entry applies to. Must reference a valid site as configured in SchoolCafe. For district-wide entries, use the site code designated as the Central Office. |
| Description | Free text description for the journal entry line |
| PostingDate | Date the entry should post to the ledger. Determines the row's fiscal year and period assignment (see Section 3). |
| Debit | Positive value when the row is a debit; otherwise blank or zero |
| Credit | Positive value when the row is a credit; otherwise blank or zero |

### 6) Optional Target Fields

| Field | Notes |
| --- | --- |
| AccountName | Descriptive label for the account |
| Comments | External reference identifier; free text; max 50 characters |

### 7) Validation Rules (row-level)

7.1 GLAccount, SiteCode, Description, PostingDate, Debit, and Credit must be present in the column mapping for each row. Missing values in any required field are flagged as invalid.

7.2 Debit and Credit values must be **decimal numbers**. Validation peeks at the first 10 rows during the validation step (§8); if any contain non-numeric characters in the Debit or Credit columns, the entire import is rejected with a fail-fast error indicating the user likely uploaded the wrong file.

7.3 For each row, exactly one of Debit or Credit must contain a positive number. The following are invalid (block the entire import per §8.5 atomicity):

* Both Debit and Credit populated
* Both Debit and Credit blank or zero
* Negative values in either column

7.4 PostingDate must be a parseable date. Unparseable date values are flagged as row-level errors. The closed-period and unconfigured-period checks against PostingDate are import-level blocks (see §3.3 and §3.4) rather than row-level errors, because the all-or-nothing rule means a single offending row blocks the entire import.

7.5 GLAccount is validated against the chart of accounts. Unknown accounts are imported with a **warning** (not a blocking error) and the account is surfaced in Configuration > General Ledger > GL Accounts with a warning indicator so finance can review. _(Payroll-specific behavior, distinct from other imports that block on unknown accounts — payroll feeds legitimately reference newly-introduced accounts that need review but should not halt month-end processing.)_

7.6 SiteCode must reference a valid site as configured in SchoolCafe. Unknown site codes are flagged as invalid.

### 8) Validation Rules (import-level)

8.1 **Validator gate.** On the Field Mapping tab, after the user maps columns, a **Validate** button triggers a BL API call that runs all row-level (§7) and import-level (§8.2–§8.5 below) checks, including period derivation and closed/unconfigured-period blocks (§3.3–3.4). The Next button to proceed to Review is disabled until validation passes. On success, the user sees a confirmation message with row count and the per-period derived breakdown. On failure, the user sees the error reason and a download link for the error report (generated by BL) so they can correct the file and re-upload.

8.2 BL API returns pass/fail. On pass, DE displays row count confirmation and the per-period derived breakdown (which periods the file targets, row count and totals for each). On fail, DE displays the error reason and a download link for the error report (generated by BL).

8.3 BL API blocks the import if the absolute difference between total debits and total credits exceeds **$0.01**. A difference of $0.01 or less is allowed to accommodate rounding across line items. The variance check applies to the file as a whole, not per period — a payroll file is a single balanced unit even when its rows span more than one period. The blocking message is returned by BL; DE surfaces it as-is.

8.4 The system must detect potential duplicate imports for any (Fiscal Year, Period) the file would post to. If any prior successful Payroll Import exists for any of the periods this file targets, the user must explicitly acknowledge a duplicate warning before the import can proceed. The warning must identify the specific period(s) where a prior import exists. The system must not silently allow duplicate imports. _(The specific detection mechanism and any ledger-layer backstop are defined in the business logic layer — see companion story **NXT-70777**.)_

8.5 The import must be **atomic** — either all rows are imported successfully, or no rows are imported. Partial imports are not allowed. If any system-level error occurs during import processing, the entire import must roll back and the user must be notified that no data was written.

### 9) Field Mapping and Review

9.1 The **FILE DATA PREVIEW** button on the File Upload tab opens a collapsible panel showing the first few parsed rows from the uploaded file with all detected columns. This is raw file data — no mapping applied. **Field Mapping** displays only the column mapping UI and Validate.

9.2 **Review tab** is the final confirm step. After Field Mapping, the user lands on Review and sees: the per-period derived assignment summary (per §3.2), totals (overall and per period), and any duplicate warnings (§8.4) requiring acknowledgment.

9.3 No row-level validation errors appear at Review. Errors are caught and resolved at the validator gate (§8.1) on Field Mapping before the user reaches Review.

9.4 The user must explicitly confirm to perform the import. If a duplicate is detected (§8.4), the confirm action remains disabled until the user explicitly acknowledges the duplicate warning.

### 10) Saved Mapping Behavior

10.1 The column mapping for the Payroll Import type is persisted automatically when the import completes. On subsequent imports, the previous column mapping auto-loads on the Field Mapping tab. Fiscal year and period are derived per row from PostingDate (§3), so neither is part of the persisted mapping.

10.2 When using a persisted mapping, the column mapping is auto-applied — but every import must still pass through the validator gate (§8.1) using the same row-level (§7) and import-level (§8) rules as the original import. The user uploads a new file, the validator runs, and Review proceeds only after validation passes.

10.3 The Import Monitor history view exposes a **Validate** action on each prior import row, allowing the user to re-run validation against the original file without re-walking the full flow.

10.4 The user can modify the auto-loaded column mapping on any subsequent import.

### 11) Import Results and Auditability

11.1 Each import must produce an import result record consistent with Import Monitor patterns (status, counts, timestamps, user).

11.2 The Import Results view has two sections:

**Summary (top of view):**

* Success / failed status
* Per-period breakdown (which fiscal years and periods received rows from this import, and the row count and totals for each)
* Total rows
* Total debits and total credits
* Duplicate acknowledgment (when applicable) — which prior import(s) the user acknowledged before proceeding
* Timestamp and user

**Details (below summary):**

* GLAccount warnings for rows with unknown accounts
* Per-row processing details consistent with Import Monitor patterns

Row-level validation errors are not shown in Import Results — those are caught at the validator gate (§8.1) before the import runs and are available via the downloadable error report at that step.

11.3 The import record must be visible in audit history so administrators can see when journal entries were imported and by whom.

### 12) Handoff to Business Logic

12.1 On successful confirmation, the data exchange layer passes the validated import data to the business logic layer (**NXT-70777**), with each row carrying its derived (Fiscal Year, Period) assignment alongside its PostingDate. The duplicate acknowledgment flag (when present) is included in the handoff payload, scoped to the specific (FY, Period) buckets the user acknowledged. The business logic layer is responsible for:

* Creating the journal entry header and lines
* Posting lines to the ledger with the correct PostingDate per row, into the period each row's date assigns it to
* Applying backstop period-status verification (defense-in-depth) — if any row's period has closed since the data exchange layer's check, the entire import is rejected
* Applying backstop duplicate detection at the ledger level for each (FY, Period) the import targets, validated against the acknowledgment flag
* Ensuring the ledger write is atomic
* Handling the persistence of GLAccount warnings (rows with unknown accounts are written, but the account is flagged for review)

12.2 On validation failure, BL returns structured error details (affected rows and reasons) and generates the downloadable error report. DE surfaces the download link provided by BL.

## Acceptance Criteria
**AC1 — Import Type Available.** Given an authorized Finance user opens Import Monitor, When they view Import Types, Then Payroll Import is available.

**AC2 — Validator Gate.** Given the user maps columns on the Field Mapping tab, When they click Validate, Then a BL API call runs all row-level (§7) and import-level (§8) checks including period derivation and closed/unconfigured-period blocks. The Next button to proceed to Review is disabled until validation passes. On success, the user sees row count and per-period derived breakdown.

**AC3 — FILE DATA PREVIEW on File Upload.** Given the user uploads a file on the File Upload tab, When they click FILE DATA PREVIEW, Then a collapsible panel shows the first few parsed rows with all detected columns. This is raw file data — no mapping applied. Field Mapping displays only the column mapping UI and Validate button.

**AC4 — Period Derived from PostingDate.** Given the user uploads a file containing PostingDate values, When validation runs, Then BL derives each row's fiscal year and period from its PostingDate in the district's configured timezone. The user does not select FY or period as a separate step.

**AC5 — Per-Period Summary on Review.** Given validation passes, When the user reaches the Review tab, Then a per-period summary shows which FY/periods the file targets, with row count and debit/credit totals for each. Files spanning multiple periods show one summary line per period.

**AC6 — Closed-Period Rows Block Entire Import.** Given any row's PostingDate falls in a closed period, When validation runs, Then the entire import is blocked. The blocking message identifies the closed period(s), the count of affected rows, and routes the user to post a prior-period adjustment to the current open period instead.

**AC7 — Unconfigured Period Blocks Entire Import.** Given any row's PostingDate falls in a fiscal year or period that does not exist in the system, When validation runs, Then the entire import is blocked with a clear error directing the user to fix the date(s) or have an administrator create the missing fiscal year.

**AC8 — Confirm-Time Period Re-Verification.** Given the user passed validation and a period targeted by the file has since been closed, When the user attempts to confirm, Then confirm is blocked and the user is returned to validation. 

**AC9 — Required Fields Mapped.** Given an upload reaches Field Mapping, When the user views the mapping UI, Then GLAccount, SiteCode, Description, PostingDate, Debit, and Credit must be mapped. Missing values in any required field are flagged as invalid by BL.

**AC10 — Decimal Format Fail-Fast.** Given an upload contains non-numeric characters in the Debit or Credit columns in any of the first 10 rows, When validation runs, Then validation fails fast with an error indicating the user likely uploaded the wrong file. Review is not reached.

**AC11 — Debit/Credit Row Validity.** Given a row in the uploaded file, When validation runs, Then exactly one of Debit or Credit must contain a positive number. Both populated, both blank/zero, or negative values are flagged invalid and block the entire import.

**AC12 — Unknown GLAccount Warns but Does Not Block.** Given an upload contains a GLAccount not in the chart of accounts, When validation runs, Then that row is flagged with a warning (not a blocking error). The account is surfaced in GL Accounts configuration with a warning indicator. Confirm remains available if no other blocking errors exist.

**AC13 — Unknown SiteCode Blocked.** Given an upload contains a SiteCode that does not reference a valid site in SchoolCafe, When validation runs, Then that row is flagged invalid with a clear error consistent with Import Monitor patterns.

**AC14 — Batch Must Balance Within $0.01 Tolerance.** Given the absolute difference between total debits and total credits exceeds $0.01, When validation runs, Then BL returns a failure. The blocking message from BL is surfaced as-is by DE. A difference of $0.01 or less is allowed (rounding tolerance). The variance check applies to the file as a whole, not per period.

**AC15 — Duplicate Detection Warning.** Given a prior successful Payroll Import exists for any (FY, Period) this file would post to, When validation passes and the user reaches Review, Then the system displays a warning identifying the specific period(s) with prior imports. The Confirm action remains disabled until the user explicitly acknowledges the duplicate warning.

**AC16 — Downloadable Error Report on Validation Failure.** Given validation fails with one or more row-level errors, When the user views the failure message, Then a download link for the error report (generated by BL) is available listing the affected rows and reasons so the user can correct the file and re-upload.

**AC17 — Atomic Import.** Given the user confirms an import and a system-level error occurs during processing, When the error is encountered, Then the entire import rolls back, no data is written, and the user is notified that the import failed.

**AC18 — Mapping Auto-Loads but Re-Validates.** Given a previous Payroll Import has been completed, When the user starts a new Payroll Import, Then the previous column mapping auto-loads on Field Mapping. The file must still pass through the validator gate using the same row-level (§7) and import-level (§8) rules.

**AC19 — Validate Action on Import History Row.** Given a prior import exists in Import Monitor history, When the user views the history view, Then a Validate action is available on each row to re-run validation against the original file using the same rules.

**AC20 — Review is Final Confirm Only.** Given the user reaches the Review tab, When the screen loads, Then they see per-period derived assignment summary, overall totals, and any duplicate warnings requiring acknowledgment. No validation errors appear at Review (caught upstream at the validator gate on Field Mapping). The user must explicitly confirm to perform the import.

**AC21 — Results View Structure.** Given an import completes, When the user opens import results, Then a Summary section shows: success/failed status, per-period breakdown, total rows, total debits/credits, duplicate acknowledgment (when applicable), and timestamp/user. GLAccount warnings for unknown accounts appear in the Details section. Row-level validation errors do not appear here — those are caught at the validator gate before the import runs.

**AC22 — Multi-Sheet File Rejected at Upload.** Given an upload contains more than one sheet, When the file is uploaded, Then the file is rejected at upload with a clear error directing the user to consolidate to a single sheet and re-upload. The system does not attempt to process any sheet.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| Empty file or header-only | Validation fails: *"No data rows found."* |
| Multi-sheet file uploaded | Rejected at upload with error directing user to consolidate to a single sheet |
| Non-numeric characters in Debit or Credit (any of first 10 rows) | Fail-fast: validation rejects the entire import with error indicating the user likely uploaded the wrong file |
| Both Debit and Credit populated in same row | Row invalid: *"Exactly one of Debit or Credit must have a value."* |
| Both Debit and Credit blank or zero in same row | Row invalid: *"Exactly one of Debit or Credit must have a positive value."* |
| Negative value in Debit or Credit | Row invalid: *"Debit and Credit values must be positive."* |
| Unknown GLAccount | Row flagged with warning (not blocking). Account surfaced in GL Accounts configuration with warning indicator. Confirm remains available. |
| Unknown SiteCode | Row invalid with clear error consistent with Import Monitor patterns |
| Unparseable PostingDate | Row-level error; flagged invalid |
| PostingDate falls in a closed period | Entire import blocked. Message identifies closed period(s) and row counts, routes user to prior-period adjustment flow. |
| PostingDate falls in unconfigured FY/period | Entire import blocked. Error directs user to fix dates or have admin create the missing FY. |
| Period closed between validation and confirm | Confirm blocked; user returned to validation |
| Total debits and credits differ by more than $0.01 | Validation fails; blocking message from BL surfaced as-is |
| Total debits and credits differ by $0.01 or less | Import proceeds (rounding tolerance) |
| Prior Payroll Import exists for same FY + Period | Warning displayed on Review; user must explicitly acknowledge before Confirm is enabled |
| File contains mix of valid and invalid rows | Validation fails with downloadable error report; no partial import allowed. GLAccount warnings alone do not block. |
| System error during import processing | Entire import rolls back; no data written; user notified of failure |

## Defaults

- **Period assignment:** Derived from PostingDate per row. Not selected by the user. Not part of the persisted mapping.
- **District-wide entries:** Use the site code designated as the Central Office in Site Management. Import UI guidance text: *"For district-wide entries, use the site code assigned to your Central Office in Site Management."*
- **AccountName and Comments:** Optional target fields. Default to blank/null when not mapped.

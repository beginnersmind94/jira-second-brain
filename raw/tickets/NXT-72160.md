# NXT-72160 — FIN — Payroll Import: Data Exchange

## Description
As a district finance user, I want to import journal entry lines into Financials, so that recurring payroll and other month-end journal entries can be loaded efficiently without manual data entry.

## Scope

Behavior specific to **Payroll Import** in the Import Monitor pipeline: registering the import type, the Pre-upload fields, the file columns, the wizard tabs (File Upload, Field Mapping, Review), staging check, saved mapping, import results, and what DE sends to BL (**NXT-70777**).

---

## Requirements

### 1) Import Type Registration

1.1 The Import Monitor selector includes **Payroll Import** as a selectable type. Selection tooltip: _"Loads payroll and other month-end journal entries. File must have separate Debit and Credit columns."_

1.2 **Supported formats:** xlsx, xls, csv, txt (per Import Monitor patterns).

1.3 **Single sheet only.** DE reads only the first sheet. Multi-sheet files are not rejected at upload, but if the first sheet's columns do not map to the required target fields, the Field Mapping step will surface the mismatch.

1.4 **Roles with Edit access:** Director, Central Office, SchoolCafe Admin. Customer/Technical Support always full access (platform pattern).

### 2) Pre-upload Fields

**Left column:**

| Field | Required | Notes |
| --- | --- | --- |
| Import Type | Yes | "Payroll Import" |
| Entry Description | Yes | Free text, max 240 characters. Applied verbatim as the header description on every journal entry produced by this import. Entered fresh on every import; never part of saved mapping. |

No fiscal year or period selector — both are derived per row from PostedDate at processing time (BL — see **NXT-70777**).

**Right-side panel:**

2.1 **Download Template** link provides an Excel file (`.xlsx`) with the canonical headers — GLAccount, SiteCode, PostedDate, Debit, Credit — and no data rows. Users conform their file to this template before upload.

2.2 **Help text** below the Download Template link: _"Loads payroll and other month-end journal entries. Each row must have a PostedDate — entries are grouped by date. File must have separate Debit and Credit columns."_

### 3) File Columns

The file must carry these 5 columns.

| Column | Required | Type | Notes |
| --- | --- | --- | --- |
| GLAccount | Yes | String | Account identifier per Financials rules. |
| SiteCode | Yes | String | Must reference a valid SchoolCafe site. Use the Central Office site code for district-wide entries. |
| PostedDate | Yes | Date | Drives entry boundary (one entry per unique PostedDate), fiscal year, and period for the row. |
| Debit | Conditional | Decimal | Required if Credit is blank. Mutually exclusive with Credit per row. |
| Credit | Conditional | Decimal | Required if Debit is blank. Mutually exclusive with Debit per row. |

No line-level Description column — Financials 2.0 carries description at the entry header only.

### 4) File Upload Tab

4.1 Standard file picker per Import Monitor pattern.

4.2 **FILE DATA PREVIEW** — a collapsible panel showing the first few parsed rows and all detected columns from the uploaded file. Raw data, no mapping applied. Follows the existing Import Monitor preview pattern.

### 5) Field Mapping

5.1 Column mapping UI. The target fields are GLAccount, SiteCode, PostedDate, Debit, and Credit. No Validate button — all validation runs at BL Commit.

5.2 **Auto-match.** When the file's column headers match target field names (case-insensitive), DE auto-maps them and shows a toast notification: _"Columns have been automatically matched. Review and adjust if needed."_ The user can modify any auto-matched mapping before proceeding.

### 6) Review Tab

6.1 Final confirmation before Save. Displays:

* Entry Description (from Pre-upload)
* Total rows in the file
* Total Debits (sum, formatted as currency)
* Total Credits (sum, formatted as currency)
* Count of unique PostedDates (= number of journal entries that will be created)

6.2 No validation runs on Review. Save → BL Commit.

6.3 User must explicitly confirm.

### 7) DE Staging Check

After the user clicks Save and before BL Commit runs, DE performs one check:

**File-level Dr = Cr within $0.01 tolerance.** If `|sum Debits − sum Credits| > $0.01`, the file is rejected before BL runs. A single exception of type `FileBalance` appears in Import Monitor → Exceptions, naming total Debits, total Credits, and the difference.

All other validation belongs to BL (**NXT-70777**).

### 8) Saved Mapping

8.1 Column mapping auto-persists on successful import completion. Subsequent Payroll Imports auto-load the previous mapping on the Field Mapping tab.

8.2 **Entry Description is entered fresh per import** — not persisted with the mapping.

8.3 Persisted mapping auto-applies, but every import still goes through BL Commit under the same business rules per **NXT-70777**.

8.4 User can modify the auto-loaded mapping on any subsequent import.

### 9) What DE Sends to BL

After the staging check passes, DE sends to BL Commit:

* Rows from the file (5 columns per row, as uploaded)
* Entry Description from Pre-upload
* File details: original filename, file hash (SHA-256), uploaded by, uploaded at

### 10) Import Monitor Statistics

The Import Monitor dashboard for a completed Payroll Import displays customizable statistics cards:

* Total records in file
* Records processed (on success = total records)
* Records with exceptions (on failure)
* Total Debit amount
* Total Credit amount

Card layout and ordering follow the existing Import Monitor pattern (customizable per import type).

### 11) Import Results and Auditability

11.1 Each import produces a result record per Import Monitor pattern (status, counts, timestamps, user).

11.2 **Summary (top):**

* Status (success or failure)
* Entry Description
* Total rows
* Total Debits and Credits
* Count of unique PostedDates
* Count of journal entries created (on success)
* Timestamp and user

**Details (below):**

* Per-row processing details (Import Monitor pattern)

Row-level errors live in the **Exceptions tab** (per Import Monitor pattern) — not in the Results Summary.

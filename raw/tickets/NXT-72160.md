# NXT-72160 — FIN — Payroll Import: Data Exchange

## Description
As a district finance user, I want to import journal entry lines into Financials via Import Monitor, so that recurring payroll and other month-end journal entries can be loaded efficiently without manual data entry.

_Updated 2026-05-07. Compressed for readability (\~30% trim, no semantic changes). Earlier 2026-05-07: added TL;DR section. Earlier 2026-05-06: applied DE/BL separation of concerns established in NXT-71249 — Validate button moved to Field Mapping, DE does not compute totals (BL API pass/fail), sample rows → FILE DATA PREVIEW, saved mapping auto-persists (no named templates), error report generated by BL. Created 2026-04-21 as the Data Exchange companion to **NXT-70777** (Business Logic). Aligned 2026-05-05 with NXT-71249 patterns. Corrected §4: restored Pre-upload tab._

## TL;DR

Build a new **Payroll Import** type in Import Monitor — standard 4-tab flow. _BL owns validation and writes; DE owns UI and orchestration._

**Flow:**

* **Pre-upload:** Import Type + File Format only. **No FY/Period selector** — period is **derived per row from PostingDate** _(key difference vs Opening Balance)_.
* **File Upload:** `xlsx / xls / csv / txt`, **single sheet only**. **FILE DATA PREVIEW** button shows raw first rows.
* **Field Mapping:** Map 6 required targets (`GLAccount`, `SiteCode`, `Description`, `PostingDate`, `Debit`, `Credit`) + 2 optional (`AccountName`, `Comments`). **Validate** button calls BL API. Next disabled until pass.
* **Review:** **Per-period summary** showing which `(FY, Period)` the file targets with row count and totals per period — multi-period files get one summary line each. Plus overall totals and any duplicate warnings.

**Validator gate.** BL runs every check. On pass: row count + per-period breakdown. On fail: surface BL's error + error report download link.

**Closed period or unconfigured FY** anywhere in the file = **entire import blocks** (all-or-nothing). Closed-period error routes user to the manual prior-period adjustment flow. _No partial imports of just the open-period rows._

**At confirm**, BL re-derives periods and re-checks closure. If a period closed since Validate, block and return the user to validation.

**Duplicate detection:** if a prior successful Payroll Import exists for any `(FY, Period)` the file targets, show a warning on Review naming the affected periods. Confirm stays disabled until the user **explicitly acknowledges**. Acknowledgment passes through to BL.

**Unknown GLAccount = warning, NOT blocking** _(differs from Opening Balance, which blocks)_. Row imports; account flagged in GL Accounts config for finance review.

**Balance:** BL allows **±$0.01 tolerance** _file-wide_, not per period _(differs from OB's exact $0)_.

**Atomic.** All rows or none.

**Saved mapping** auto-persists on success — auto-loads next time; file _still goes through Validate_. **Period is never part of the mapping** (derived per row).

**What NOT to build:**

* An FY/Period selector — _PostingDate is authoritative._
* Any balancing/period/account validation in DE — BL owns it.

## Scope

Import Monitor pipeline for Payroll Import: import type setup, file upload, column mapping, validator gate, row + import-level validation, period derivation from PostingDate, Field Mapping + Review tabs, saved mapping, import results, duplicate detection warning. On confirm, validated data hands off to BL (**NXT-70777**) for journal entry creation and ledger posting.

## Requirements

### 1) Import Type

1.1 New Import Monitor Import Type **Payroll Import**. Tooltip on Import Type selection: _"Loads payroll and other month-end journal entries. File must have separate Debit and Credit columns."_

1.2 Supported formats: **xlsx, xls, csv, txt** (per Import Monitor patterns).

1.3 **Roles with Edit access:** Director, Central Office, SchoolCafe Admin. Customer/Technical Support always full access (platform pattern).

1.4 **Single sheet only.** Multi-sheet files rejected at upload with an error directing the user to consolidate and re-upload. No sheet processed.

### 2) Amount Format

2.1 **Single canonical format:** separate **Debit** and **Credit** columns. Signed-amount files (positive = debit, negative = credit) are transformed by DE during validation (§8), upstream of Field Mapping.

2.2 User does not pick an amount format. Field Mapping always shows Debit and Credit as canonical target fields.

### 3) Period Assignment (derived from PostingDate)

3.1 Each row's FY and period are **derived from its PostingDate** in the district's timezone. User does not select FY or period — PostingDate is authoritative.

3.2 Derived period assignment is displayed on Review as a per-period summary (e.g., _"This file will post to: FY2025-26 August (142 rows, $48,219.43 debit / $48,219.43 credit)"_) for sanity-check. Multi-period files show one line per period.

3.3 If **any row's PostingDate falls in a closed period**, the entire import is blocked at the validator gate (§8.1). No partial imports. Blocking message:

* Identifies closed period(s) and row count per period.
* Routes the user to the alternative:

> _"This file contains rows with posting dates in closed periods. To affect a closed period's financials, post a prior-period adjustment to the current open period. Correct the affected rows or remove them, then re-upload."_

with a link to the manual adjustment entry flow.

3.4 If any row's PostingDate is in an unconfigured FY/period, the entire import is blocked at the validator gate with an error directing the user to fix dates or have an admin create the missing FY.

3.5 At confirm, BL re-derives periods and re-verifies no row's PostingDate is in a closed/unconfigured period. If a period closed since Validate, confirm is blocked (all-or-nothing); user returned to validation. _(BL applies same checks as backstop — see_ **NXT-70777**_.)_

### 4) Import Flow

Standard 4-tab Import Monitor flow:

* **Pre-upload** — Import Type + File Format. No FY/Period selection (derived from PostingDate per §3).
* **File Upload** — upload, delimiter detection, FILE DATA PREVIEW.
* **Field Mapping** — column mapping + Validate (§8.1).
* **Review** — per-period summary, totals, duplicate warnings (§9.2). Final confirm.

No amount format step — single canonical format (§2).

### 5) Required Target Fields

Required target fields on Field Mapping. Same set every import — no conditional fields.

| Field | Notes |
| --- | --- |
| GLAccount | Account identifier per Financials rules |
| SiteCode | Site the entry applies to. Must reference a valid SchoolCafe site. For district-wide entries, use the site code designated as the Central Office. |
| Description | Free text description for the journal entry line |
| PostingDate | Date the entry posts to the ledger. Determines the row's FY and period (§3). |
| Debit | Positive value when the row is a debit; otherwise blank or zero |
| Credit | Positive value when the row is a credit; otherwise blank or zero |

### 6) Optional Target Fields

| Field | Notes |
| --- | --- |
| AccountName | Descriptive label for the account |
| Comments | External reference identifier; free text; max 50 characters |

### 7) Validation Rules (row-level)

7.1 GLAccount, SiteCode, Description, PostingDate, Debit, Credit must be mapped. Missing values are flagged invalid.

7.2 Debit and Credit must be **decimal numbers**. Validation peeks at the first 10 rows (§8); any non-numeric character in Debit/Credit fail-fasts the entire import with a wrong-file error.

7.3 Each row must have **exactly one** of Debit or Credit positive. Invalid (blocks entire import per §8.5):

* Both Debit and Credit populated
* Both Debit and Credit blank or zero
* Negative values in either column

7.4 PostingDate must be parseable. Unparseable dates are row-level errors. Closed-period and unconfigured-period checks are import-level blocks (§3.3, §3.4) — one offending row blocks the entire import per the all-or-nothing rule.

7.5 GLAccount is validated against the chart of accounts. Unknown accounts import with a **warning** (not blocking); the account is surfaced in Configuration > General Ledger > GL Accounts for finance review. _(Payroll-specific — distinct from imports that block on unknown accounts; payroll feeds legitimately reference newly-introduced accounts that should not halt month-end.)_

7.6 SiteCode must reference a valid SchoolCafe site. Unknown SiteCodes are flagged invalid.

### 8) Validation Rules (import-level)

8.1 **Validator gate.** On Field Mapping, after column mapping, **Validate** triggers a BL API call running row-level (§7) and import-level (§8.2–§8.5) checks, including period derivation and closed/unconfigured-period blocks (§3.3–3.4). Next stays disabled until pass. On success: row count + per-period derived breakdown. On failure: error reason + downloadable error report (generated by BL).

8.2 BL returns pass/fail. DE surfaces row count + per-period derived breakdown on pass; error reason + report download link on fail.

8.3 BL blocks if |total Debits − total Credits| > **$0.01** (rounding tolerance). File-wide check, not per period — a payroll file is a single balanced unit even across periods. BL returns the blocking message; DE surfaces as-is.

8.4 Duplicate detection runs for every (FY, Period) the file targets. If a prior successful Payroll Import exists for any targeted period, the user must explicitly acknowledge a warning identifying the affected period(s) before proceeding. No silent allowance. _(Detection mechanism and ledger-layer backstop in BL — see_ **NXT-70777**_.)_

8.5 **Atomic** — all rows or none. On any system error, the entire import rolls back; user is notified that no data was written.

### 9) Field Mapping and Review

9.1 **FILE DATA PREVIEW** on File Upload opens a collapsible panel with the first few parsed rows + all detected columns. Raw data, no mapping. **Field Mapping** is column mapping + Validate only.

9.2 **Review** is the final confirm. User sees: per-period derived summary (§3.2), overall + per-period totals, and any duplicate warnings (§8.4) requiring acknowledgment.

9.3 No validation errors on Review — caught upstream at the validator gate (§8.1).

9.4 User must explicitly confirm. If a duplicate is detected (§8.4), Confirm is disabled until the user acknowledges.

### 10) Saved Mapping Behavior

10.1 Column mapping auto-persists on completion. Subsequent imports auto-load the previous mapping on Field Mapping. **FY and period are derived per row from PostingDate (§3)** — never part of the persisted mapping.

10.2 Persisted mapping auto-applies, but every import still passes through the validator gate (§8.1) under the same rules. Upload → validate (§7, §8) → Review only after pass.

10.3 Import Monitor history exposes a **Validate** action on each prior row to re-run validation against the original file.

10.4 User can modify the auto-loaded mapping on any subsequent import.

### 11) Import Results and Auditability

11.1 Each import produces a result record (status, counts, timestamps, user — Import Monitor pattern).

11.2 Import Results has two sections:

**Summary (top):**

* Success / failed status
* Per-period breakdown (FY/period, row count, totals for each)
* Total rows
* Total debits and credits
* Duplicate acknowledgment (when applicable) — which prior import(s) the user acknowledged
* Timestamp and user

**Details (below):**

* GLAccount warnings for unknown-account rows
* Per-row processing details (Import Monitor pattern)

No row-level validation errors here — caught at the validator gate (§8.1) and available via the downloadable error report.

11.3 Import record appears in audit history showing when journal entries were imported and by whom.

### 12) Handoff to Business Logic

12.1 On confirm, DE passes validated data to BL (**NXT-70777**). Each row carries its derived (FY, Period) alongside its PostingDate. Duplicate acknowledgment flag (when present) is in the payload, scoped to acknowledged (FY, Period) buckets. BL responsibilities:

* Create journal entry header and lines
* Post lines to the ledger with the correct PostingDate per row, into the assigned period
* Backstop period-status verification — entire import rejected if any row's period closed since DE's check
* Backstop duplicate detection at the ledger level for each (FY, Period), validated against the acknowledgment flag
* Atomic ledger write
* Persist GLAccount warnings (unknown-account rows are written; account flagged for review)

12.2 On failure, BL returns structured error details (rows + reasons) + downloadable error report. DE surfaces the link.

## Acceptance Criteria
**AC1 — Import Type Available.** When a Finance user views Import Types in Import Monitor, Payroll Import appears.

**AC2 — Validator Gate.** Clicking Validate on Field Mapping triggers a BL API call running all row-level (§7) and import-level (§8) checks including period derivation and closed/unconfigured-period blocks. Next stays disabled until pass. On success, row count + per-period derived breakdown surface.

**AC3 — FILE DATA PREVIEW on File Upload.** Clicking FILE DATA PREVIEW on File Upload opens a collapsible panel showing the first few parsed rows with all detected columns. Raw data, no mapping. Field Mapping shows only the mapping UI + Validate.

**AC4 — Period Derived from PostingDate.** BL derives each row's FY and period from PostingDate in the district's timezone. The user does not select FY or period.

**AC5 — Per-Period Summary on Review.** On Review, a per-period summary shows the FY/periods the file targets with row count and debit/credit totals per period. Multi-period files show one line per period.

**AC6 — Closed-Period Rows Block Entire Import.** If any row's PostingDate falls in a closed period, the entire import is blocked. The message identifies the closed period(s), affected row count, and routes the user to post a prior-period adjustment to the current open period.

**AC7 — Unconfigured Period Blocks Entire Import.** If any row's PostingDate falls in a fiscal year or period that doesn't exist, the entire import is blocked with an error directing the user to fix the date(s) or have an admin create the missing FY.

**AC8 — Confirm-Time Period Re-Verification.** If a period targeted by the file closes between Validate and Confirm, Confirm is blocked and the user is returned to validation.

**AC9 — Required Fields Mapped.** GLAccount, SiteCode, Description, PostingDate, Debit, and Credit must be mapped. Missing values in any required field are flagged invalid by BL.

**AC10 — Decimal Format Fail-Fast.** Non-numeric characters in Debit or Credit in any of the first 10 rows fail-fast validation with a wrong-file error. Review is not reached.

**AC11 — Debit/Credit Row Validity.** Each row must have exactly one of Debit or Credit positive. Both populated, both blank/zero, or negative values block the entire import.

**AC12 — Unknown GLAccount Warns but Does Not Block.** Rows with a GLAccount not in the chart of accounts are flagged with a warning (not blocking). The account is surfaced in GL Accounts configuration with a warning indicator. Confirm remains available if no other blocking errors exist.

**AC13 — Unknown SiteCode Blocked.** Rows with a SiteCode that doesn't reference a valid SchoolCafe site are flagged invalid with a clear error.

**AC14 — Batch Must Balance Within $0.01 Tolerance.** If |total debits − total credits| > $0.01, BL returns failure. The blocking message from BL is surfaced as-is. ≤ $0.01 is allowed (rounding tolerance). Check is file-wide, not per period.

**AC15 — Duplicate Detection Warning.** If a prior successful Payroll Import exists for any (FY, Period) the file targets, Review displays a warning identifying the period(s). Confirm is disabled until the user explicitly acknowledges.

**AC16 — Downloadable Error Report on Validation Failure.** On validation failure with row-level errors, a download link for the BL-generated error report is available listing affected rows + reasons.

**AC17 — Atomic Import.** On any system error during processing, the entire import rolls back, no data is written, and the user is notified.

**AC18 — Mapping Auto-Loads but Re-Validates.** On a new Payroll Import, the previous column mapping auto-loads on Field Mapping. The file must still pass through the validator gate using the same row-level (§7) and import-level (§8) rules.

**AC19 — Validate Action on Import History Row.** Each row in Import Monitor history exposes a Validate action to re-run validation against the original file using the same rules.

**AC20 — Review is Final Confirm Only.** Review shows per-period derived assignment summary, overall totals, and any duplicate warnings requiring acknowledgment. No validation errors appear here (caught upstream at the validator gate). The user must explicitly confirm.

**AC21 — Results View Structure.** On import completion, Summary shows: success/failed status, per-period breakdown, total rows, total debits/credits, duplicate acknowledgment (when applicable), timestamp/user. GLAccount warnings appear in Details. Row-level validation errors do not appear here.

**AC22 — Multi-Sheet File Rejected at Upload.** Multi-sheet files are rejected at upload with an error directing the user to consolidate to a single sheet. No sheet is processed.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| Empty file or header-only | Validation fails: _"No data rows found."_ |
| Multi-sheet file uploaded | Rejected at upload; user prompted to consolidate to single sheet |
| Non-numeric characters in Debit or Credit (any of first 10 rows) | Fail-fast: validation rejects entire import; wrong-file error |
| Both Debit and Credit populated in same row | Row invalid: _"Exactly one of Debit or Credit must have a value."_ |
| Both Debit and Credit blank or zero in same row | Row invalid: _"Exactly one of Debit or Credit must have a positive value."_ |
| Negative value in Debit or Credit | Row invalid: _"Debit and Credit values must be positive."_ |
| Unknown GLAccount | Row flagged with warning (not blocking); account surfaced in GL Accounts config; Confirm remains available |
| Unknown SiteCode | Row invalid; clear error per Import Monitor patterns |
| Unparseable PostingDate | Row-level error; flagged invalid |
| PostingDate falls in a closed period | Entire import blocked; identifies closed period(s) + row counts; routes user to prior-period adjustment flow |
| PostingDate falls in unconfigured FY/period | Entire import blocked; user fixes dates or admin creates missing FY |
| Period closed between validation and confirm | Confirm blocked; user returned to validation |
| Total debits and credits differ by more than $0.01 | Validation fails; BL blocking message surfaced as-is |
| Total debits and credits differ by $0.01 or less | Import proceeds (rounding tolerance) |
| Prior Payroll Import exists for same FY + Period | Warning displayed on Review; user must explicitly acknowledge before Confirm enables |
| File contains mix of valid and invalid rows | Validation fails with downloadable error report; no partial import. GLAccount warnings alone do not block. |
| System error during import processing | Entire import rolls back; no data written; user notified |

## Defaults

False. **Period assignment:** Derived from PostingDate per row. Not selected by the user. Not part of the persisted mapping.
False. **District-wide entries:** Use the site code designated as the Central Office in Site Management. UI guidance: _"For district-wide entries, use the site code assigned to your Central Office in Site Management."_
False. **AccountName and Comments:** Optional target fields. Default to blank/null when not mapped.

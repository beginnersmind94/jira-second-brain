# NXT-44510 — IM - Item Import - Make Inventory Ready fields optional

## Description
As a District Admin I want the ability to import items in a non-Inventory Ready state so I can import items into an incomplete state and finish the inventory readiness in the UI.

Example of an item being imported into Inventory Ineligible status (serving is not linked to base unit):

!image-20231031-142940.png|width=91.66666666666667%!

## Acceptance Criteria
* Item Import should allow for optional input of all Inventory Ready data points
** Item Category, Storage Category, Valuation Group
** Pack Size unit fields
** Fair Market Value
** pack-serving linkage
* Items should import into the UI in the appropriate Inventory Readiness status (Ineligible, Incomplete, etc.)
* Checks for Pack Size-serving size relationships for dual unit possibility
** If the *serving size quantity is 1* AND the *serving description is NOT a standard measure*, perform the following checks:
**# If the serving size weight (in grams +/- 15%) equals the lowest unit weight/measure (in gram/ounces/pounds) and has the *same* unit/measure description, use the serving size weight and same description
**# If the serving size weight (in grams +/- 15%) equals the lowest unit weight/measure (in grams/ounces/pounds) but has a *different* unit/measure description, use the serving size weight and the serving size description.
**# If the serving size weight (in grams +/- 15%) does *not* equal the lowest unit weight/measure (in ounces or grams) and has the same unit/measure description, generate an error as duplicate serving size/lowest unit measure is not allowed, and do not import.
**# If the serving size weight (in grams +/- 15%) does *not* equal the lowest unit weight/measure (in ounces or grams) and has a different unit/measure description, import it as a pack unit and serving unit like normal.
** When converting between different units, we should account for rounding with a margin of +/- 15%. If the units are the same and weight is not within threshold, we should flag them as an error when the descriptions match; otherwise, treat them as normal. (Explaining point #3)

# NXT-15895 — Verification - Sampling - Snapshot

## Description
As a district admin I want the sample to be created automatically, based on the options I picked from the wizard.

*mockup*

n/a

*requirements*

* Generate the sample pool based on the selected options from the wizard.
** Depending on their selected Sample Process, we have rules we need to follow. The percentages for these rules should be pulled from system settings to allow for future enhancement.
*** There was talk of increasing these percentages in the near-mid future - better to keep these malleable via system settings.
*** !image-20201208-231425.png|width=1149,height=802!
** Applications should be randomly selected from the sample pool and added to the sample, fitting the below requirement per the selected process:
*** !image-20201224-200914.png|width=651,height=553!
***Be careful with rounding. Use the new system setting below. We’ll need to add that setting.*
*** If there aren’t enough applications to reach the sample size, then the remainder should be randomly pulled per the following:
**** !image-20201224-201021.png|width=646,height=183!
* New system setting for Verification Rounding
** Name: Verification Rounding for Alternate Two
*** Description: Controls whether the rounding for alternate two should be performed individually (twice) for categorical and error-prone applications, or only on the sum of both values (once). States like Texas will issue a finding if rounding happens twice.
*** Options: Once, Twice
*** Default: Once
** This impacts the calculation done for the total number of apps for Alternate Two only.
* Check date ranges while generating samples, with a special provision for rolling verification’s final sample:
** If multiples samples have been configured, check from the end date of the one prior up to the selected sampling date.
** Scenarios:
*** Two rolling + 1 final sample:
**** 8/1 - check for apps from 7/1 to 8/1
**** 9/1 - check for apps from 8/2 - 9/1
**** Final Sample - 10/1 
***** First check for apps from 9/2 - 10/1
***** Calculate how many apps should have been selected from 7/1 to Final Date (10/1 in this example)
***** Do we have enough apps? Great, just add the first check.
***** If we don’t have enough apps, perform another random select from 7/1 - Final Date to add additional apps.

## Acceptance Criteria
Background mechanism to automate sample creation created and working.
System setting for time added, and used for sample creation.
Future samples should be triggered at the appropriate date.
Past samples should also work properly.

If we're doing this with Rolling Verification in mind, then we need to be sure that we're not oversampling applications - the gathered app number would need to check for a date range starting at the previous sample date + 1, if one exists, up until the new sample date at the timestamp.

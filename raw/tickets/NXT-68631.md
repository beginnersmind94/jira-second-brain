# NXT-68631 — SC - MT - District Dashboard

## Description
As a Customer Support role user, I want to be able to create a state realm instead of a district so that we can sell our product to states.

*Requirements*

* On the District page, do the following:
* Data for the grid, add these fields:
** Realm Type
*** Reflects the new realm type (State, District, etc.)
*** Ensure it is in the managed view columns
** Realms
*** Count of children realms
*** Ensure it is in the managed view columns
*** If no children, then value is 0
** If a Central Realm:
*** Realms, Sites, Users and Students counts should roll up (count all for all children districts)
* 
* Grid Columns:
** Add Realm Type as a column
*** Make it the next column after the name
** Add Realms as a column
*** Put after the type
*** Add to grid as part of the standard view
* 
* Actions column
* In the vertical ellipses:
** If the realm is a Central one:
** Add button:
*** Label: Change District
*** On click, redirect user to the district login page
* 
* Security
** Allow district users in roles with View access to view this page

## Acceptance Criteria
h2. *1. Grid Data Enhancements*

h3. New Data Fields

* *Realm Type*
** Displays the organizational level for the realm (e.g., _State, District, Area, Sub Area, County_).
** Must accurately reflect the new realm type.
** Included in the *managed view columns* (so it can be toggled on/off).
* *Realms (Count of Children)*
** Displays the count of child realms for the current realm.
** If there are no children, show *0*.
** Included in the *managed view columns*.
** For *Central realms*, the value represents *all rolled-up counts* across children and their descendants.

h3. Roll-Up Logic for Central Realms

* For realms designated as *Central*:
** *Realms*, *Sites*, *Users*, and *Students* counts should *aggregate all children* recursively.
** Example: A Central realm containing three districts should show the sum of their counts for each respective metric.

----

h2. *2. Grid Columns*

h3. Column Additions & Order

* Add *Realm Type* column:
** Position: Immediately after the *Name* column.
* Add *Realms* column:
** Position: After the *Realm Type* column.
* Both columns must appear by default as part of the *standard grid view*.

h3. Column Behavior

* Both columns must support:
** Sorting
** Filtering
** Inclusion in saved grid configurations / managed view

----

h2. *3. Actions Column Enhancements*

h3. Vertical Ellipsis Menu

* For any realm identified as a *Central realm*, add a new option under the vertical ellipsis (*⋮*) menu:
** *Label:* {{Change District}}
** *On Click:*
*** Redirect the user to the *district login page*.
*** Behavior must mimic the existing system’s district-switching workflow.
** This action must only appear for *Central realms*.

----

h2. *4. Security*

* Access Control:
** *District users* in roles with *View* permission can access and view this page.
** Users without View permission should not see or be able to navigate to the page.
** Only authorized Customer Support or Technical Support roles may modify data (if applicable under other related stories).

----

h2. *5. Validation & Behavior*

* Realm Type and Realms data should correctly refresh when:
** A new realm is created or linked.
** Child realms are added or removed.
** Counts should update dynamically or on grid refresh.
* If data retrieval fails, show a standard system error message (e.g., “Unable to load realm data. Please try again.”).

----

h2. *6. Non-Functional*

* Grid loads and updates within *3 seconds* under normal conditions.
* Sorting, filtering, and column management retain user preferences between sessions.
* The design should remain consistent with existing *System > Districts* grid styling and behavior.

# NXT-71510 — SC - PBIRD - Dashboard – History

## Description
As a Director, I want to view the history of configuration changes for a Power BI report so that I can understand what changes were made and by whom.

----

h2. *Description*

Allow users to view the *configuration history* of a Power BI report from the dashboard.

This history will include only *configuration-level events* (add, edit, delete) and will follow the same *data and business rules* as Managed View Reports history, but will use the *new Tailwind CSS history UI format* instead of the legacy format.

----

h2. *User Flow*

# User navigates to:
#* *Reports → Power BI Reports*
# User clicks the *ellipsis (⋮)* on a report row
# User selects *View History*
# System opens the *History drawer/modal*
# User reviews history records
# User views details of a selected event
# User closes the history view

----

h2. *Out of Scope*

* Execution history (run, email, distribution events) _(covered in separate story)_
* Multi-level history
* Other ellipsis actions
* Editing from history

----

h2. *Requirements*

h3. *History Access*

* “View History” is available as an option in the row ellipsis menu _(covered in separate story)_
* Only a *single history option* should exist

----

h3. *History Scope*

* History must include only *configuration-related events*:
** Create (Add)
** Edit
** Delete
* Do NOT include:
** Report executions
** Email/distribution events
** Scheduling runs

----

h3. *History Data*

* History must capture:
** Action type (Create, Edit, Delete)
** User who performed the action
** Timestamp
** Summary/details of the change
* Data must align with audit logging implemented in prior stories

----

h3. *UI – Tailwind CSS History Format (Deviation)*

* This implementation must use the *new Tailwind CSS history UI format*
* This is a *deliberate deviation* from Managed View Reports:
** Do NOT use the legacy history UI

----

h3. *History Display Behavior*

* Display a *list of history events*
* When a user selects an event:
** Show *details of that event* in a separate section of the UI
* No further click-through or navigation beyond this level

----

h3. *Read-Only Behavior*

* History is view-only
* No actions (edit, revert, etc.) are available

----

h3. *Consistency*

* While UI differs, the following must align with Managed View Reports:
** Data captured
** Events tracked
** Business rules

## Acceptance Criteria
* Given a user selects View History, when the history view opens, then only configuration-related events (create, edit, delete) are displayed
* Given the history is displayed, when the user views the list, then each event includes user, timestamp, and action type
* Given the user selects a history event, when the selection occurs, then the details of that event are displayed in a separate section
* Given the history view is open, when the user interacts with it, then no additional navigation or click-through is available
* Given the history feature is implemented, when compared to Managed View Reports, then the data is consistent but the UI uses the Tailwind CSS format

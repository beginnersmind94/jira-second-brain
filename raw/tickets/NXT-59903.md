# NXT-59903 — TH - Sec - Login Screen

## Description
*As a*: TeamHub user *I want to*: log into the application *So that*: I can access my account and features

*Comments*:

* Screen: Login screen ({{app/(auth)/login.tsx}})
* Handles user authentication and access control

*User Flow*:

* User opens the application and is directed to login screen
* App checks if they have configured the Platform and District in the District Setup screen before
* If no, app redirects them to the District Setup screen
* If yes, app allows them to stay on the login screen
* User enters email and password
* User clicks "Sign In" button
* App connects to the correct platform and authenticates against the correct district
* On success, user is redirected to home screen
* On failure, error message is displayed
* Optionally, the user can click on Forget their password and be redirected to that screen

*Requirements*:

_Front End_:

* On the Login screen, do the following:
* Ensure the screen supports dark mode from the app settings
* On screen load,
** Go to asynchronous state management datastore and get the Platform, District and Email values from the last time this app on this device saved
** If never have saved, use defaults
* Fields:
* Email/Username
** Textbox
** Required
** If platform is PE, change the label to Username
** Else leave label as Email
** If platform is SC, put an email validator
* Password
** Textbox
** Required
** Put the password mask on it with the eye to show it
* Sign In
** Button
** Enable only after they have entered in values for Email and Password
** On click:
*** Validate required fields
*** Verify Platform and District
*** Using Platform and District, authenticate to the correct platform and district
**** Remember, SC is SaaS, so only one server, but PE has n number of servers
**** Note: the authentication code needed for this is covered in another story (linked below)
*** If errors:
**** Show them in user friendly error message
**** If system errors (not login failure, but like bugs), log in error log
* Forgot Password
** Link button
** On click, redirect to Forgot Password screen (another story)
* District Setup
** Link button
** On click, redirect to District Setup screen (another story)
* Error message display
** Show all error messages in standardized error component
* Loading state indicator
** If needed, show the loading indicator until the screen renders
** If needed, show the loading indicator while trying to authenticate

_Back End_:

* APIS to validate credentials against auth service covered in other stories
** Connect them to Login button
** Encrypt data, especially password
* When successful auth:
** Generate and return auth token for app to use
** Encrypt if needed
* Store user session
** If not already done, create the user session object as needed
** Ensure you get licenses, roles and permissions
** Encrypt if needed
* Rate limit login attempts
** Allow up to 5 failed attempts and then start limiting by time
** Add 5 minutes for every failed attempt after that

_Security_:

* Screen is visible to any user, no security trimming

## Acceptance Criteria
h4. *User Authentication & Access Control*

* User must be directed to the login screen upon opening the application.
* If the Platform and District are not set up, the user must be redirected to the District Setup screen.
* If the Platform and District are configured, the user remains on the login screen.
* User must be able to enter their email/username and password.
* User must be able to click the "Sign In" button to authenticate.
* Upon successful authentication, the user is redirected to the home screen.
* Upon authentication failure, a user-friendly error message is displayed.
* User must have an option to reset their password by clicking the "Forgot Password" link, redirecting them to the Forgot Password screen.

h4. *Front-End Requirements*

* Ensure the screen supports dark mode from the app settings

h5. *On Screen Load*

* Fetch Platform, District, and Email values from the asynchronous state management datastore.
* If values do not exist, use default values.

h5. *Fields*

* *Email/Username*
** Display as a textbox.
** Mark as a required field.
** If the platform is *PE*, change the label to *Username*.
** If the platform is *SC*, apply an email validation rule.
* *Password*
** Display as a masked textbox.
** Mark as a required field.
** Include an eye icon to toggle password visibility.
* *Sign In Button*
** Must be disabled until both the Email/Username and Password fields contain values.
** On click:
*** Validate required fields.
*** Verify Platform and District.
*** Authenticate the user against the correct platform and district.
*** Display error messages if authentication fails.
*** Log system errors if necessary.
* *Forgot Password Link*
** On click, redirect the user to the Forgot Password screen.
* *District Setup Link*
** On click, redirect the user to the District Setup screen.

h5. *Error Message Display*

* All error messages must be shown using the standardized error component.

h5. *Loading State Indicator*

* Display a loading indicator:
** Until the login screen is fully rendered.
** While authentication is in progress.

h4. *Back-End Requirements*

* APIs for authentication are covered in separate stories.
* APIs must be connected to the Login button.
* User credentials must be encrypted before transmission.
* On successful authentication:
** Generate and return an authentication token.
** Encrypt data as needed.
** Create and store the user session, if not already created.
** Fetch and store user licenses, roles, and permissions.
* Implement rate limiting:
** Allow up to *5* failed login attempts before imposing a cooldown.
** For each additional failed attempt, increase wait time by *5 minutes*.

h4. *Security Considerations*

* The login screen must be accessible to all users (no security trimming).
* Sensitive data, including passwords, must always be encrypted.

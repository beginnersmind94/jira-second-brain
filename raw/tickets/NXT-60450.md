# NXT-60450 — TH - WS - Tasks Screen

## Description
*As a* TeamHub user *I want* a screen that provides quick access to my SchoolCafé Task features *So that* I can manage my daily activities

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the task screens from SC
* This story covers the Task dashboard; new and edit and async data management will be in other stories

h2. User Flow

* User opens the application and is directed to login screen
* On login success, user is redirected to home screen
* User clicks Workspace and is redirected there
* User clicks on Task and is redirected here
* User is able to view a list of tasks assigned to themselves
* User is able to click on a tasks and edit it
* User is able to assign a new task

h2. Requirements

* Ensure the screen supports dark mode

_Header Section_

* Have a back button to Workspace
* Page Title: Tasks



_Filters Section_

* Search textbox
** Filters Tasks
** Filters against the title and description
* Status chips
** Acts as a filter on the status field
** If click it, filter the tasks to that status
* Sync
** Missing from Bolt code. Add just like in Inbox
** Show last sync time
** have a Sync button
*** On click:
*** Run the asynchronous Inbox job to sync the mobile app to the server’s tasks (another story)



_Body Section_

* List of Tasks
** Be able to view a list of tasks
*** View: 
**** Title
**** Beginning of the description (length trimmed)
**** Due Date
**** Status
**** Assigned By
*** View a visual indicator of the status
** Be able to click on a task and view its details (another story)
* Compose button
** When clicked, redirects the user to the New Task screen (another story)

h4. *Data Requirements*

* There should be no API calls on this screen
* All data should be pulled from the asynchronous storage on the phone
* There should be separate code (another story) that runs independently from this screen that syncs the tasks to the server and this screen should be agnostic to that.

h4. *Performance & Usability*

* The screen must load within 2 seconds and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use try/catch in all asynchronous calls and use error boundaries for any unexpected render failures.
* Errors must be communicated using user-friendly messages and use toast notifications, modals, or inline messages without disrupting the user experience
* Log any API or rendering errors to the error log system/database for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *Theme Support*

* The screen must fully support dark mode, including:
** Backgrounds, text, icons, and interactive elements.
** All visual elements must remain accessible and readable in both light and dark modes.

----

h4. *Header Section*

* The header displays a Back button that returns the user to the Workspace screen.
* The page title is "Tasks" and is clearly visible.

----

h4. *Filters Section*

* A search textbox is present at the top of the screen.
** Filters tasks in real time based on the task title and task description.
* Status filter chips are displayed below the search bar.
** Each chip represents a different task status (e.g., To Do, In Progress, Done).
** Tapping a chip filters the task list to show only tasks with that status.
** Only one status filter is active at a time.
* A Sync section is displayed:
** Shows the timestamp of the last sync with the server.
** Includes a Sync button:
*** When tapped, it triggers the asynchronous task sync job (handled in another story).
*** The screen itself remains agnostic to sync logic and only displays sync status.

----

h4. *Body Section*

* A scrollable list of tasks is displayed.
* Each task entry includes:
** Title of the task.
** Beginning of the description (truncated to fit the card).
** Due date, formatted for readability.
** Status of the task, with a visual status indicator (e.g., color or icon).
** Assigned By information (name of the assigning user).
* Tapping on a task navigates to the Task Details screen (defined in another story).
* A Compose button is present:
** When tapped, redirects the user to the New Task screen (handled in another story).

----

h4. *Data Requirements*

* The screen must not make any API calls directly.
* All data (task list, sync timestamps) is pulled from asynchronous local storage on the device.
* The screen writes no updates and only displays information.
* A separate sync process (another story) handles server communication and local cache updates.

----

h4. *Performance & Usability*

* The screen must load in under 2 seconds under normal conditions.
* It must display and function correctly across:
** Mobile
** Tablet
** Desktop
** Compact and expanded views
* All tap and swipe interactions must be:
** Visually responsive
** Smooth and intuitive

----

h4. *Error Handling*

* All asynchronous local storage interactions use try/catch blocks.
* Error boundaries are used to catch and gracefully handle any rendering issues.
* User-facing errors are communicated using:
** Toast notifications
** Modals
** Inline messages
* All critical errors are logged to the centralized error log system for further investigation.

----

h4. *Security*

* The screen and its features are accessible to any authenticated user.
* No additional role-based permissions are required to view or interact with the task list.

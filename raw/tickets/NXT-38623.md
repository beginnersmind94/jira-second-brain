# NXT-38623 — Monitoring Scheduler – Framework

## Description
*Notes:*

* This is part 2 of the Framework. In part 1, Syed did the spike. Below are his comments on the work that needs to be done.



*Requirements*:

Syed Abiddin:

After analyzing the job monitor related effort which has already been done below work is anticipated to complete the desired effort.

Database: We would need a new table called JM_JobMonitor_History which will be similar to NT_ConfigSchedule_History which was used for adding the notifications that were already executed to avoid rerunning them.

*Publishers*

_Executing Scheduled Job Monitor_

Add a new azure timer trigger function in the notification scheduler [repo|https://dev.azure.com/Cybersoft-Technologies-Inc/Platform/_git/PrimeroEdge.Administration?path=/PrimeroEdge.NotificationScheduler/PrimeroEdge.NotificationScheduler.Functions/Functions] that will run every 5 minutes and :

# Fetch all the job monitors that are scheduled to be executed.
# Iterate through each job monitor and check if the job monitor is currently scheduled to be executed.
# Make an entry to the newly created JM_JobMonitor_History indicating the job monitor has been executed.
# Publish out a message with the routing key “PrimeroEdge.NotificationScheduler.Domain.JobMonitors.RequestInitiateMessage“. The message will contain job monitor details.
# Complete executing the function and repeat.
# Would require devops help for creating a new resource in each environment and promote the code.

_Executing Job Monitor On Demand_

# The Front End will have a context menu item called “Run“ available on the job monitor table for each row.
# We need to create a new POST web api action in the JobMonitorController that will take a JobMonitorId.
# Internally fetch the job monitor’s details and serialize it to the model and send it to the rabbit mq using the routing key “PrimeroEdge.NotificationScheduler.Domain.JobMonitors.RequestInitiateMessage“.
# Make an entry to the newly created JM_JobMonitor_History indicating the job monitor has been executed.

 

*Subscribers(Listeners)*

# All the modules who are offering the Job Monitor functionality will be listening to the routing key “PrimeroEdge.NotificationScheduler.Domain.JobMonitors.RequestInitiateMessage“.
# Upon receiving the message they will extract the message, determine what functionality needs to be executed and process it.
# Upon completion they will send the message via the routing key “PrimeroEdge.NotificationScheduler.Domain.JobMonitors.ResponseMessage“ and a message indicating success or failure.
# Administration repo needs to be enhanced to include a sync agent such that it listens to the routing key “PrimeroEdge.NotificationScheduler.Domain.JobMonitors.RequestInitiateMessage“ and update the JM_JobMonitor_History table indicating the response received, success or failed.
# Would need devops help creating new CI and CD pipelines for the new sync agent for each environment.

# NXT-60344 — TH - WS - Inbox Sent Screen

## Description
*As a* TeamHub user *I want* a screen that provides quick access to my SchoolCafé Inbox and Sent features *So that* I can stay informed about my current tasks and notifications.

h2. Comments

* This story is for SchoolCafé (SC) districts
** PrimeroEdge districts screen stories will come in another story
* Basically, we want to mimic the Inbox Sent screens from SC
* This story covers the Sent section, Inbox will be covered in another story

h2. User Flow

* User opens the application and is directed to login screen
* On login success, user is redirected to home screen
* User clicks Workspace and is redirected there
* User clicks on Inbox and is redirected there
* User is able to view a list of sent messages
* User is able to click on a message and see the details
* User is able to reply and forward a message
* User is able to delete a message

h2. Requirements

* Ensure the screen supports dark mode

_Header Section_

* Covered in the inbox story

_Filters Section_

* All covered in the inbox story
* Ensure the Sent chip opens the sent section

_Body Section_

* List of sent messages
** Be able to view a list of sent messages
*** View: Avatar of sender, From, Date, Subject, Beginning of the body
*** Note: Star is not needed for sent items, but it is ok to have if it is easier to keep it consistent with Inbox message
*** Note: a visual indicator of if it is not read or not is not needed for sent items, but it is ok to have if it is easier to keep it consistent with Inbox message; if it shares code, it can stay
*** Indicate if there is an attachment
** Be able to click on a message and view its details (another story)
** Be able to swipe the message to the left and be able to delete it (another story)
* Compose button
** when clicked, redirects the user to the compose inbox screen (another story)

h4. *Data Requirements*

* There should be no API calls on this screen
* All data should be pulled from the asynchronous storage on the phone
* The Delete (and Star if keep) should write to the asynchronous storage on the phone
* There should be separate code (another story) that runs independently from this page that syncs the sent messages to the server and this screen should be agnostic to that.

h4. *Performance & Usability*

* The screen must load within *2 seconds* and the layout be responsive and render correctly on: Mobile, Tablet, Desktop, Compact and expanded views
* All tap interactions must be responsive, with clear visual feedback

h4. *Error Handling*

* Use *try/catch* in all asynchronous calls
* Use *error boundaries* for any unexpected render failures.
* Errors must be communicated using *user-friendly messages*:
** Use *toast notifications, modals, or inline messages* without disrupting the user experience
* Log any API or rendering errors to the *error log system/database* for further analysis



_Security_

* Any authenticated user can view this screen and take the actions on it

## Acceptance Criteria
h4. *Theme Support*

* The screen must fully support dark mode, including:
** Header, background, and text colors adjusting properly.
** Message cards and icons remaining readable and accessible.

----

h4. *Header & Filters Section*

* The header section and filter section behavior is inherited from the Inbox screen and must function identically.
* Tapping the Sent chip:
** Highlights the chip to indicate selection.
** Displays the Sent messages view in the body section.

----

h4. *Body Section: Sent Messages*

* The screen displays a scrollable list of sent messages pulled from asynchronous storage.
* Each message card includes:
** Avatar of the sender.
** "From" field (user who sent the message).
** Date the message was sent.
** Subject line.
** Beginning of the message body.
** Attachment indicator, if applicable.
* Optional UI elements:
** A star icon may be included, but is not required. It may be shown if reusing the Inbox component.
** A read/unread visual indicator may be shown for consistency with the Inbox card UI, though it is not necessary for Sent messages.
* Tapping a message opens the message details view (covered in another story).
* Swiping a message to the left allows the user to delete it (covered in another story).
* A Compose button is visible and persistent:
** Tapping it redirects the user to the Compose Message screen (covered in another story).

----

h4. *Data Requirements*

* All data must be pulled from local asynchronous storage on the device.
* There are no API calls on this screen.
* Any delete or star (if kept) actions must update the local async storage directly.
* Message syncing with the server is handled by a separate process (another story), and this screen must remain agnostic to that process.

----

h4. *Performance & Usability*

* The screen must load within 2 seconds under normal conditions.
* The UI must render correctly and responsively across:
** Mobile
** Tablet
** Desktop
** Compact and expanded views
* All tap, swipe, and interaction actions must be:
** Responsive
** Provide clear visual feedback
** Maintain consistent behavior across light and dark modes

----

h4. *Security*

* Any authenticated user can view the Sent screen and perform allowed actions.
* No additional permissions are required.

----

h4. *Error Handling*

* All asynchronous actions (e.g., accessing local storage) must use try/catch blocks.
* Error boundaries must be implemented to catch unexpected rendering issues.
* Any errors must be communicated to the user using non-disruptive, user-friendly messages:
** Use toast notifications, modals, or inline messages.
* All errors (data, rendering, interaction) must be logged to the error log system/database for analysis and debugging.

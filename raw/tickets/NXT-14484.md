# NXT-14484 — [Duplicate] Data Exchange - SIS Import using API

## Description
This story is to cover API based Student Information Import – SIS Import from Vendors @ API.

*A platform has to be designed and developed where all the API’s stored (API Configuration Page) and the user can trigger various API’s based on the requirements.*

*Once the API’s are triggered and the data is imported, the data is mapped to the respective template, transformed per requirement and  stored locally per business rules.*

Here in this case, Student information is imported from external vendors.

# As an end user, I would like to have a screen where all the API information (import types, end point URL’s, Credentials, API versions) is stored.
#* Ideally the new screen will be under System Configuration API Configuration.
# Student information will be imported based on filtering criteria State-ISD. 
## Accommodate changes needed to reflect
### File Import Set up as Import Set up
### File Import Dashboard as Import Dash board
### File Import Set up tab as Import Set up
## Once Import Tab is selected, the user must be given an option of selecting the import type (Student, Items etc) and next the type of processing (API or File based). Here in this case we will select API.
## Now that the API related information is already stored per the requirement #1, the user can fire and the importing of data starts.
### Note: User must be given an option of downloading the data via file based or API but only one can be active ant any given time.
### Also, The API versions must be displayed in this screen which are captured in the API configuration screen.
## Once the data is imported, mapping to templates can be started based on the import type selected (predefined) and rest of the process continues.
## Status of the process must be available via Import Monitor.
# User must have an option of selecting the data at state level and ISD level (filtering criteria).
# Here this triggering must be done in two ways.
#* Scheduler for regular intervals
#* One time.
# The users who access or trigger this must be the decision makers – School District IT Director and Nutrition Director.
# For Student Imports , Currently we have contracts with all three major vendors and they must be contacted so that the student import template is designed considering the data offered by them.



*_Note : The platform designed and developed based on #1 and #2 must be leveraged to any type of imports in future with very minimal changes._*

## Acceptance Criteria

# Verify and validate that the user can import/download the information based on his access levels.
# Verify and Validate that all the tool tips and messages are in place.
# The imported data must address all the required fields in the mapping spec.
# The API can invoked as per scheduler and also as needed (ad-hoc).

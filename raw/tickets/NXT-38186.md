# NXT-38186 — Device Management – FO – Device Management APIs (API ONLY)

## Description
_This is Device Management Phase 1 for Front Office team_



We need to create multiple APIs to be used by Device Manager so that Device Manager can perform the relevant CRUD operations.

*Notes:*

* This is part of a multi-story work scope for Device Manager
** 1^st^, we need to update the terminals table to include the relevant columns
** 2^nd^, we need to create the History table and its stored procedures
** In this story, we cover the CRUD operations
* System team needs these APIs in order to complete the POS Device Manager feature
* APIs will be grouped together in order of necessity and priority
* Work with System team as needed

*Requirements*:

* Multiple APIs need to be created to help the device management team build the page and its associated operations.
* APIs are split into these groupings:
** Group 1:
*** POS Devices – handled in [https://primeroedgenext.atlassian.net/browse/NXT-36442|https://primeroedgenext.atlassian.net/browse/NXT-36442|smart-link] 
*** Site IDs – might already exist
*** Site Names – might already exist
*** POS Device Types
*** POS Status
*** POS Versions
** Group 2:
*** -Add New POS Device-
*** Edit POS Device
*** Delete POS Device
*** Suspend POS Device
*** Activate POS Device
** Group 3:
*** POS Device History
*** POS Device Details



*Group 1*

* *POS Devices API (handled in* [https://primeroedgenext.atlassian.net/browse/NXT-36442|https://primeroedgenext.atlassian.net/browse/NXT-36442|smart-link] )
** Used on the Device Manager page which is the first thing System team will work on
** This will be used by the grid in main body
** It is a list of all the POS devices per district
** Columns:
*** SchoolID
*** SiteName
*** TerminalType – the name, not the id, get via join
*** TerminalName
*** Status
*** Description
*** LastCommunicationDate
*** AppVersion

Note: The next 5 APIs are used in the Filters section of the Device Manager. If any of these already exist (like Site and site Name), provide the API names.

* *Site ID API*
** List of unique Site IDs for the district
** May already exist
* *Site Name API*
** List of unique school names for the district
** May already exist
* *Device Type*
** List of terminal types, not ID, but name
** Used on Device Manager, Add New and Edit screens
* *Status*
** List of statuses
* *POS Version*
** List of unique app versions

 

*Group 2*

These are the 2nd ones that need to be created in order of importance

* *Edit POS Device API*
** Used to update an existing record
*** Fields receive:
**** Device ID
**** Device Name
**** School ID
**** Device type
**** Description – may be blank
**** User – account logged in
** Edit POS Device Stored Procedure
*** Create or update existing stored procedure to update a record with this data
** Insert into History Table:
*** After updating the terminals table, insert a record in the History table created for POS terminal devices
*** Call the History table insert stored procedure
**** Pass in Edit for type
**** Format the description as noted in the History table story
**** Pass in the rest of the required fields
* *Delete POS Device API*
** Used to delete an existing record
*** Fields receive:
**** Device ID
**** User – account logged in
**  Delete POS Device Stored Procedure
*** Create or update existing stored procedure to delete a record with this data
** Insert into History Table:
*** After deleting from terminals table, insert a record in the History table created for POS terminal devices
*** Call the History table insert stored procedure
**** Pass in Delete for event type
**** Pass in Device ID
* *Suspend POS Device API*
** Used to update status of an existing record to Suspended
*** Fields receive:
**** Device ID
**** User – account logged in
** Suspend POS Device Stored Procedure
*** Create or update existing stored procedure to set the status to suspended on record with that Device ID
** Insert into History Table:
*** After updating terminals table, insert a record in the History table created for POS terminal devices
*** Call the History table insert stored procedure
**** Pass in Suspend for event type
**** Pass in Device ID
** Independent of this API, you need to create code to suspend the device and call it when you receive this, or in same way handle suspending a device
* *Activate POS Device API*
** Used to update status of an existing record to Activated
*** Fields receive:
**** Device ID
**** User – account logged in
** Activate POS Device Stored Procedure
*** Create or update existing stored procedure to set the status to Activated on record with that Device ID
** Insert into History Table:
*** After updating terminals table, insert a record in the History table created for POS terminal devices
*** Call the History table insert stored procedure
**** Pass in Activate for event type
**** Pass in Device ID
** Independent of this API, you need to create code to activate the device and call it when you receive this, or in same way handle suspending a device

 

*Group 3*

* *POS Device Details API*
** Used to show the full details of the POS Device
*** Parameters:
**** Device ID
*** Fields provide:
**** All the columns in the terminals table for that device id
** Get POS Device Details Stored Procedure
*** Create or update existing stored procedure to fetch the whole record for that Device ID



*POS*

* Rename / Save Terminal updated to account for new need to track history on a single terminal.
** User should be able to see that the machine has been renamed, and the prior history of that machine.

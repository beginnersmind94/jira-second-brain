# NXT-3195 — Accounts - Student Info - Return Check (Single)

## Description
*User Story*

As an administrator, I need to be able to process a returned check in the system for a single account so that there is a record that can be tracked for future auditing.

*Mock-up*

Return check with a single student is on the check payment.

!image-20210316-125112.png|width=1301,height=755!

*Requirements*

* In the Account Details page of Accounts, where there is a ‘More Actions’ drop-down menu, when a user selects the menu, show an option for *Return Check*.
* When that button is selected, open an action slide out panel should appear where the user can process a returned check.
* Details:
** {color:#00b8d9}Check field{color}
*** {color:#00b8d9}Displays the checks currently applied to the student account.{color}
*** {color:#00b8d9}Displayed as “\[Check Number] for $\[Amount]”{color}
** {color:#ff5630}-Check Number field-{color}
*** {color:#ff5630}-Allow a user to enter a check number.-{color}
**** {color:#ff5630}-Only numerical values should be allowed.-{color}
*** {color:#ff5630}-Validate the check number. It should match a check that was paid for the selected patron. (No unmatched checks allowed).-{color}
** {color:#ff5630}-Payment Amount field-{color}
*** {color:#ff5630}-Based on the Check Number entered above, get the amount of the check and automatically display it in this field.-{color}
*** {color:#ff5630}-This field is read-only-{color}
** {color:#00b8d9}Total Check Amount{color}
*** {color:#00b8d9}Display the amount of the check payment{color}
** Previous Balance
*** Display the selected account’s current balance
** Payment Amount {color:#00b8d9}Applied{color}
*** Based on the amount of the payment that was applied to the selected account. This amount could be different from the Total Check Amount if more than one student was part of the payment (see related story).
** Bonus Amount {color:#00b8d9}Applied{color}
*** Based on the bonus amount that was applied to the account (if any).
** Return Check Fee
*** Based on the check return fee settings for the district (BKCHKFEE and ISDCHKFEE), display the total (SUM of the two setting values in the screenshot below). 
*** {color:#ff5630}-Display in red if it is > 0.00. Display it in black if it = 0.00.-{color}
!Capture 2021-02-25 at 10.01.40 PM.png|width=1121,height=513!
** New Balance
*** Sum of Previous Balance - (Payment Amount Applied + Bonus Amount Applied + Return Check Fee)
*** {color:#00b8d9}If balance is negative, display as red text. Otherwise, text will display as black.{color}
** Comments
*** A comment is required for any transaction.
** “Do not accept subsequent checks on this account” checkbox
*** When checked, after the user selects apply, update the selected patron’s account to not be able to accept future check payments. Account “Restrictions” should be updated on the account(s) related to this.
!image-20210315-144930.png|width=1020,height=274!
* Once the requirements above are met, the user should be allow to select the Save button to complete the {color:#00b8d9}Returned Check{color}.
* ‘Close’ and/or ‘Cancel’ buttons that return them to the Account Info page without saving anything.
** Display dialog prompt if canceling with information entered.
** {color:#00b8d9}“You have not saved your changes. Are you sure you want to exit?” - Provide Yes and No options.{color}

Error handling:

* IF Comments IS NULL
** Display “A comment is required{color:#ff5630} -in order to return checks-{color}.”
** {color:#ff5630}-IF Check Number is invalid OR was paid for a patron that is different than the one selected-{color}
*** {color:#ff5630}-Display “Invalid check number.”-{color}
* {color:#ff5630}-IF Check Number is valid but matches multiple patrons-{color}
** {color:#ff5630}-Display “This check number matches multiple patrons. Please enter the check amount as well.”-{color}
** {color:#ff5630}-Can be displayed in amber/yellow color rather than red if possible.-{color}
* When ‘Save’ is selected, multiple un-editable *transactions* should then be entered into the patron’s account, corresponding to each fee (check amount, bonus amount (if > 0.00), return check fee (if > 0.00)).
** Return Check Amount should be displayed as a negative PAYMENT transaction for the amount of the original check amount
** Return Check Fee should be displayed as a CHARGE transaction for the amount of the combined district and bank fee settings
** Bonus should be displayed as a negative PAYMENT transaction (if applicable to the original sale) for the amount of the original bonus applied
* A *session* is also created if an open session for that site and user is not open, and a single transaction is entered into that session.
* If an *OPEN session* exists for that site and user, the transaction is added to the open session.
* This story is complete when a user can navigate to the Return Check menu from the Student Info page and perform a check return for that account and have the option to prevent future check payments for them, with balance/check amount/bonus amount/check fees display changes and error handling where expected, and transactions are created for the account.

## Acceptance Criteria
The user can arrive from the 'More Actions' section of Student Info page

The user sees the fields and data above, matching the mock-up

Process:

* The user is able to see all checks applied to the account for the current academic year. User is able to select the check from the dropdown list.
* After selection, the return check information is listed: Total Check Amount, Previous Balance, Payment Amount Applied, Bonus Amount Applied, and Return Check Fee.
* The Return Check Fee is the sum of BKCHKFEE and ISDCHKFEE in system settings
* A comment must be entered
* Transaction is entered to reverse the Payment Amount and Bonus on any students associated with the original payment. These are to be entered as negative CREDIT entries.
* Transaction is entered to record the fees on the student that the return check action is performed if the Fees are not 0 transaction type is Check Return Fee. Fees should be entered as negative CREDIT entries. 
* The transaction is added to reverse the check transaction type is Return Check. It should be entered as a negative CREDIT transaction.
* The transaction date is the current date/time.
* User can back out of the transaction by selecting 'Close' and/or 'Cancel' to return to Account Info. A prompt should warn the user before exiting to avoid accidental actions.
* The Do not accept check flag is set to false for the account (we should require the user to make a decision to not accept by checking).
* A *session* is also created if an open session for that site and user is not open, and a single transaction is entered into that session.
* If an *OPEN session* exists for that site and user, the transaction is added to the open session.

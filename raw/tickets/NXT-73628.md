# NXT-73628 — FIN — Custom Reports Data Set: Financial Ledger Entries

## Description
As a district finance user, I want my Financials journal entry lines available as a data set in Custom Reports, so that after each period close I can build an export in the exact file layout my district's ERP needs and deliver it on demand — or on a fixed schedule for fixed-window exports.

---

## TL;DR

Register **one** new data set in the existing Custom Reports framework: **Financial Ledger Entries**. One row per journal entry line. The columns below cover what Oracle Fusion (FBDI Journal Import) and Tyler Munis (Import Journals) need from our side — the district (or implementation) arranges and renames columns in Custom Reports to match their ERP's import file. Sharing, Distribution, Scheduling, and FTP delivery come from the Custom Reports framework — no Financials-specific delivery work. This rides the same registration path as the Adams 12 Oracle export datasets ([NXT-69492](https://primeroedgenext.atlassian.net/browse/NXT-69492)).

**Why this matters:** A district will not treat Financials as viable if their ledger data cannot get back to district finance (their ERP). This data set is the last mile, and the same data set serves every ERP — including the formats North Carolina PrimeroEdge customers will need as they migrate to Oracle and Munis.

---

## Requirements

### 1) Data set registration

1.1 Add **Financial Ledger Entries** to the Custom Reports Data Source list, surfaced under the Financials module, available only to districts with Financials enabled. (Mirror the registration mechanics of the Adams 12 datasets — confirm with Matt at refinement.)

1.2 Reports built on this data set get all standard Custom Reports behavior with no extra work: column selection, renaming, reordering, file format options, Sharing, Distribution, Scheduling, and FTP delivery.

### 2) Grain

2.1 One output row per journal entry **line**. A journal entry with 6 lines produces 6 rows.

2.2 Entry-level values (Transaction Number, Posted Date, Description, Source, Entered Date, Entered By) repeat on every row of that entry.

### 3) Columns

Columns the data set makes available. Output order is chosen by the user in Custom Reports (drag order) and preserved on export — this list is not a fixed column order.

| Column | What it is |
| --- | --- |
| Fiscal Year | Fiscal year of the entry (e.g., 2026) |
| Period | The Financial period whose configured date range contains the Posted Date (per the Periods config — not a raw derive off the date) |
| Period Status | Status (Open / Closed) of that period at the time the report runs |
| Posted Date | The entry's posted date, delivered as YYYY/MM/DD text (see 3.2) |
| Transaction Number | The journal entry number |
| Description | The journal entry's description |
| Source | What created the entry (e.g., POS Posting, Inventory Posting, Payroll Import, Opening Balance Import, Manual Journal Entry) |
| Site Code | The line's site code — named as a school-code field so the framework's Location parameter activates (see 5.1) |
| Site Name | The line's site name |
| Account Number | The line's full GL account code, exactly as stored (Munis uses this single column) |
| Account Name | The account's name |
| Segment 1 … Segment 30 | The account code split into individual segments per the district's Account Code Structure (Oracle uses these; Munis does not). Fixed set of 30 columns; each district's report enables only the segments it uses. |
| Debit | Debit amount. Blank on credit lines. |
| Credit | Credit amount. Blank on debit lines. |
| Signed Amount | One-column amount: debit positive, credit negative. |
| Entered Date | System timestamp when the entry was created |
| Entered By | User or system that created the entry |

3.1 Amount format: two decimals, no currency symbols, no thousands separators. ERP imports reject formatted numbers like `$1,234.56`.

3.2 Date format: Posted Date is delivered as YYYY/MM/DD text inside the data set, so it exports that way regardless of a report's display settings. Trade-off (intentional): because the value is text rather than a true date, the framework's Time parameter will not attach to it ([NXT-43230](https://primeroedgenext.atlassian.net/browse/NXT-43230)) and it sorts and filters alphabetically. Date-range selection is therefore done through the Period filter, not Posted Date.

### 4) Why these columns (ERP coverage)

Oracle FBDI Journal Import is fed by: ACCOUNTING_DATE ← Posted Date · SEGMENT1–n ← Segment 1–N · ENTERED_DR / ENTERED_CR ← Debit / Credit · REFERENCE4 (journal name) ← Transaction Number · REFERENCE5 (journal description) ← Description.

Tyler Munis Import Journals is fed by: account string ← Account Number · amounts ← Debit / Credit (or Signed Amount) · description ← Description.

Constants an ERP needs (Oracle STATUS=NEW, ACTUAL_FLAG=A, CURRENCY_CODE, LEDGER_ID, journal source, journal category, plus DATE_CREATED / CREATED_BY) are not ledger data and are **not** produced by this data set. Custom Reports has no self-service constant-value column — only a Customer Support–authored Dynamic Query can emit one ([NXT-40342](https://primeroedgenext.atlassian.net/browse/NXT-40342)). For Oracle, the district supplies them in Oracle's own FBDI Journal Import spreadsheet when packaging the file. Munis needs no constants. This keeps the data set pure ledger data and ERP-agnostic.

### 5) Filtering (how it works in Custom Reports)

The framework provides two filtering surfaces; this data set is designed so both land cleanly. There is no dataset-level "required" filter — see Open questions.

5.1 **Sites → Location parameter.** Naming the site column as a school-code field makes the framework's Location parameter auto-appear at the bottom of Configuration ([NXT-43230](https://primeroedgenext.atlassian.net/browse/NXT-43230)), letting districts reuse saved location groupings. Default: all sites.

5.2 **Fiscal Year, Period, Source → per-column filter.** Exposed as filterable columns; the user sets Operator and Value on the Source Table tab ([NXT-42458](https://primeroedgenext.atlassian.net/browse/NXT-42458)). A per-source file (e.g., payroll-only) is one report with Source filtered — see AC9.

5.3 **Date range → Period, not Posted Date.** Because Posted Date ships as pre-formatted text (3.2), the Time parameter will not attach to it; date-range selection runs through the Period column.

### 6) Data freshness

6.1 The data set reads the ledger as it is at run time. If a period is reopened and an entry changes, the next run returns the updated data (consistent with the period-state handling in [NXT-71184](https://primeroedgenext.atlassian.net/browse/NXT-71184)).

### 7) Period targeting & cadence (constraint)

7.1 The Period filter is a static value, set per report. The framework cannot target "the period that just closed": its only dynamic parameters are Location and Time, the Time values are calendar-relative (This Month, etc.), and fiscal periods are not guaranteed calendar-aligned — so "last closed fiscal period" is unexpressible regardless of how Posted Date is formatted.

7.2 Consequence: a recurring schedule re-runs the same static period every cycle. The supported model is therefore **on demand after each period close** — close the period, set or confirm the Period value, then run and deliver (More Actions → Distribute, with Distribution configured and Scheduling = No). Reserve Scheduling for **fixed-window exports** (a specific historical period or semester) where the target period never changes.

7.3 A static recurring schedule wired to hands-off ERP delivery would re-deliver the same journals each cycle. That risk belongs to the delivery automation (out of scope here), not to this data set, where a person packages each file per run.

## Edge / Empty / Error States

| Case | Behavior |
| --- | --- |
| Selected period has no journal entries | Output file contains headers only |
| District structure has N segments | District enables Segment 1…N in its report; unused segment columns (N+1…30) are left off |
| Reversal entries (e.g., created by a re-import) | Appear as their own rows, same as any other entry |
| Lines at the Central Office site | Included; Site Code / Site Name show the Central Office site |

---

## Defaults

* New report on this data set: all columns included; user removes, renames, reorders in Custom Reports.
* Sites: all. Source: all.
* Amounts: two decimals, no symbols, no separators. Dates: YYYY/MM/DD.

## Decisions

* **Constants out of scope (decided).** Custom Reports has no self-service constant-value column — only a Customer Support–authored Dynamic Query can emit one. The data set stays pure ledger data; Oracle constants and DATE_CREATED / CREATED_BY are supplied by the district in Oracle's FBDI Journal Import spreadsheet at packaging time. Munis needs none. Column order is user-controlled and preserved on export, so any target layout can be matched.
* **Segment columns — fixed (decided; Hussain).** The data set exposes a fixed set, Segment 1 through Segment 30 (the most an Oracle chart of accounts can have); each district's report enables only the segments it uses. The structure is per-district and stable within a fiscal year, so the count is known at setup.
* **Required filters — accept + guidance (decided).** The framework cannot make a dataset filter required; a user could leave a filter empty and export everything. We accept this: the report is built for the district at implementation with Fiscal Year and Period already set, so the district edits the Period value rather than building from scratch. No baked-in default period — it would constrain every report on the shared data set and would usually be the wrong period given the run-after-close cadence.
* **Export cadence — on demand after close (decided).** Run-after-close on demand is the supported model for recurring exports; Scheduling is reserved for fixed-window exports whose target period never changes (see 7).

## Open questions

* **Site-level restriction (feasible; separate dependency).** Per Hussain, restricting a site-level user to only their own site's data is possible but requires work with the system team — out of this data set's scope. The data set carries Site Code, so the scoping can be applied there when that work is done. Track as a separate dependency.
* **Permission gating (open — refinement).** Whether financial ledger data should sit behind a Financials-specific permission rather than the standard Custom Reports permission.
* **Scheduling suppression (dependency, not a gate).** To prevent a naive recurring schedule on a static period — relevant only once automated ERP delivery exists — ask the framework owners whether per-data-set suppression or a warning on the Scheduling tab is feasible; the MVR per-context tab suppression ([NXT-48640](https://primeroedgenext.atlassian.net/browse/NXT-48640) / [NXT-49469](https://primeroedgenext.atlassian.net/browse/NXT-49469)) suggests it may be without core framework work. Gates enabling automated ERP delivery, not registration of this data set. Deferred risk-acceptance: if suppression is infeasible, whether to ship ERP delivery with guidance-only mitigation.
* **Output delimiter (open).** Distribution offers formats (CSV, Excel, HTML, PDF, JSON) but no configurable delimiter ([NXT-40345](https://primeroedgenext.atlassian.net/browse/NXT-40345)). If Oracle FBDI or Munis need a non-comma delimiter, that's a framework gap, not something this data set sets — confirm target import formats (both are expected to accept comma CSV).

## Acceptance Criteria
**AC1 — Data set is available**

**Given **a district with Financials enabled,
**When **a user creates a new report in Custom Reports,
**Then **“Financial Ledger Entries” is available as a data set to build the report on.

**AC2 — One row per journal entry line**

**Given **a journal entry with 6 lines in the selected period,
**When **the report runs,
**Then **the output has exactly 6 rows for that entry, and each row repeats the entry-level values (Fiscal Year, Period, Posted Date, Transaction Number, Description, Source, Entered Date, Entered By).

**AC3 — Segment columns (fixed Segment 1–30)**

**Given **a district whose Account Code Structure has 5 segments,
**When **it builds a report on this data set,
**Then **Segment 1 through Segment 5 are populated with that line's account code split per the structure, and Segment 6 through Segment 30 exist in the data set but are left off the report.

**AC4 — Debit, Credit, and Signed Amount**

**Given **a debit line of $250.00,
**When **the report runs,
**Then **Debit shows 250.00, Credit is blank, and Signed Amount shows 250.00.
**Given **a credit line of $250.00,
**When **the report runs,
**Then **Credit shows 250.00, Debit is blank, and Signed Amount shows -250.00. Amounts never contain currency symbols or thousands separators.

**AC5 — Filters**

**Given **Fiscal Year 2026 and one period are selected,
**When **the report runs,
**Then **only lines whose Posted Date falls inside that period are returned.
**Given **two sites are selected via the Location parameter,
**When **the report runs,
**Then **only lines for those two sites are returned. The Source filter restricts results the same way.

**AC6 — Standard Custom Reports features work**

**Given **a report built on this data set,
**When **the user configures it,
**Then **column selection, renaming, reordering, file format options, scheduling, and FTP delivery all work the same as for existing Custom Reports data sets.

**AC7 — Data is read live**

**Given **a period was exported, then reopened, and an entry was corrected,
**When **the report runs again,
**Then **the output reflects the corrected ledger.

**AC8 — Empty period**

**Given **a selected period with no journal entries,
**When **the report runs,
**Then **the file contains headers only.

**AC9 — Per-source / per-period files**

**Given **a district needs one journal source per import file,
**When **it creates a separate report per Source (or per Period) using the Source or Period filter,
**Then **each report produces its own file. The framework does not auto-split one export into multiple files; separation is by separate reports, and output order within a file is user-set on the Default Sort tab.

**AC10 — Export cadence**

**Given **a fixed-window export (a specific historical period or semester) on a recurring schedule,
**When **each scheduled run fires,
**Then **it delivers the same defined period every run (expected).
**Given **a recurring close-cycle export targeting the latest closed period,
**When **a new period closes,
**Then **the Period filter does not auto-advance; the export is run on demand after close with the Period value set for that close.

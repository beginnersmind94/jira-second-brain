# NXT-49413 — Temp User - District Settings

## Description
As a SchoolCafe admin, I would like to be able to add a new temporary User so that we can have auditors in the site

*Notes*:

* This is the district configuration story; user account updates in other stories
* We need to add in the concept of temp users to each authentication type
* The admin can then configure a scenario like this:
** All permanent users must use SSO
** Temp users like auditors can use SchoolCafe password or 2FA
* To do this, we will add a field to designate if the authentication method is for permanent, temporary or both users

*Requirements*:

*Dashboard*

* On the System > Security> Authentication, do the following:
* Grid updates:
** Add a column for Account Type
*** Values: Permanent, Temporary, All
** Ensure all previously created auth methods show a value

*Add New*

* On the System > Security> Authentication > Add New buttons, do the following:
* For each authentication method when click the Add New button, do the following on that drawer:
* Add a Field:
** Account Type
*** Multi-Select list
*** Values: Permanent, Temporary
*** Default selected: Both
*** Required
* 
* On save:
** Add this value to the DB
** Close the drawer and refresh the grid on the dashboard ensuring the value is in the row

*Edit Drawer*

* On the System > Security> Authentication > Edit buttons, do the following:
* For each authentication method when click the Edit button, do the following on that drawer:
* Add a Field:
** Account Type
*** Multi-Select list
*** Values: Permanent, Temporary
*** Default selected: values from DB
*** Required
* 
* New Validator:
** Rule: all temp users must be assigned to at least one auth method that has temp user account type
** Note: If the admin removes Temporary as the account type and there is no other method with Temporary, then they will orphan temp user accounts, meaning there will be no way for temp users to authenticate
** Same thing for permanent users
** To prevent orphaning users, we need to put in a validator here
** Validate:
*** If the admin is removing an Account Type value (either perm or temp), check if any other authentication method has this account type and is enabled
*** If not, check if there are any users with this account type
*** If there are, then this would orphan them, so stop the Admin from doing this:
**** Show this message: Removing this account type would cause some users to no longer be able to authenticate. Enable the account type in another authentication method first and then you can proceed.
**** Do not allow them to save their changes unless they undo this change
* 
* On save:
** Update this value to the DB
** Close the drawer and refresh the grid on the dashboard ensuring the value is in the row



*Backend*

* We need to ensure this concept is applied retroactively to already existing authentication methods
* So when ready to promote to a new environment, run a script that sets the Account Type to All for all existing authentication methods



*Imports*

* Temporary accounts are not affected by import
** ex: they are not disabled when user import is run
* All imported accounts are considered Permanent

## Acceptance Criteria
* Permanent and Temporary account type is added to Add New for each authentication method
* Permanent and Temporary are added to Edit drawer for each authentication method
* Dashboard has the account type column added to it and it shows the word All if both are selected
* When move to higher environment, previously created methods have the value set to All for them 
* A user can create an authentication method that is only for temp users

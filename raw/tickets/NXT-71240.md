# NXT-71240 — FIN — Opening Balance Import: Business Logic

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-06. Added consolidated API Validations section reflecting the DE/BL contract established with Jonathan Smith — DE hands off the mapped file, BL runs all validations and returns pass/fail with totals or a structured error report._

---

## Scope

This story covers the business logic layer for the Opening Balance Import: API Validations (validate-only and commit modes), ledger entry creation, effective date assignment, entry type behavior, amount handling, sub-account validation, hard reversal on re-import, persistence of restatement audit metadata, and downstream report behavior. The data exchange layer (companion story **NXT-71249**) handles file upload, column mapping UI, validator gate (which calls the BL Validate API), preview, and the Restatement required confirmation gate (typed FY confirmation + restatement reason). This story receives validated data with separate Debit and Credit columns, the selected Fiscal Year, and (when applicable) the restatement audit metadata after the user confirms the import.

---

## API Validations

The BL API exposes a **Validate** operation (called by DE on the Field Mapping tab per **NXT-71249** §8.1) and a **Commit** operation (called by DE on Confirm). Both run the same validation pass; Commit additionally writes the ledger entry. All checks below run on every call.

If any check fails, the API returns failure with a structured error list (row number, field, error code, message) and a generated error report download URL. DE surfaces the result. No partial work is performed.

### 1) Row-level structural

1.1 **Required fields.** GLAccount, SiteCode, and Description are present on every row.

1.2 **Decimal numbers.** Debit and Credit are decimal numbers across all rows. (DE peeks at the first 10 rows for fail-fast detection per **NXT-71249** §7.2; BL re-checks every row.)

1.3 **Single-side rule.** Each row carries a positive value in exactly one of Debit or Credit. Rows with both populated, both blank, both zero, or any negative value are invalid.

1.4 **Description length.** Description ≤ 50 characters when provided.

### 2) Reference data

2.1 **GLAccount exists** in the chart of accounts. Unknown accounts are **blocking** for Opening Balance — chart hygiene wins over throughput because OB is one-time setup. (This differs from Payroll Import, which writes unknown accounts with a warning per **NXT-70777** §5.)

2.2 **GLAccount base type is Asset, Liability, or Equity.** Revenue and Expense rows are rejected with the message: _"Opening balances cannot be set for Revenue or Expense accounts."_

2.3 **SiteCode exists** as configured in SchoolCafe Site Management.

2.4 **Sub-account configured** for each (GLAccount, SiteCode) combination per §4. Central Office is treated as just another site (no special handling).

2.5 **Uniqueness.** Each (GLAccount, SiteCode) combination appears only once in the file. Duplicate combinations are rejected with the user-facing message defined in **NXT-71249** §7.8.

### 3) Aggregate

3.1 **Balanced.** Sum of Debits = Sum of Credits across all rows. Tolerance is **exactly $0** — opening balance is a clean carryover, no rounding allowance. (This differs from Payroll, which permits a $0.01 tolerance.)

### 4) Restatement gate enforcement

4.1 If any period in the target FY has been closed, the restatement gate metadata (typed FY confirmation, reason text, confirmation timestamp) must be present in the request payload. If missing, reject. If present, accept — the user has acknowledged the restatement.

4.2 If a prior Opening Balance entry exists for the target FY, the restatement gate metadata must be present. If missing, reject. This is the **authoritative replacement check** per §6.6 — it prevents gate bypass via URL manipulation, concurrent submissions, or client-side tampering.

4.3 On Commit, the metadata is persisted to the ledger entry's audit trail per §1.1.

### 5) Period status re-check (Commit only, defense-in-depth)

5.1 At Commit, BL re-verifies that no period in the target FY has been closed since the Validate call. If a period closed in the meantime and the gate metadata does not cover the new closed-period state, Commit is rejected and the user is returned to validation.

### Response contract

**Success** — all checks pass:

* Status: `pass`
* Total rows
* Total Debits
* Total Credits
* Effective date (computed per §2 — day before FY start)
* Restatement classification: `Eligible` / `Restatement: closed periods` / `Restatement: prior import` / `Restatement: both`

**Failure** — any check fails:

* Status: `fail`
* Error list — row number, field, error code, message
* Error report download URL (generated by BL; DE surfaces the link per **NXT-71249** §8.1)

## Requirements

### 1) Ledger Entry Creation

1.1 On Commit, after all validations pass per the API Validations section, the system must create **one ledger entry** with:

* **Entry type:** "Opening Balance" (or equivalent dedicated type)
* **Effective date:** The day before the first day of the specified fiscal year (see Section 2)
* **Description:** The Description value from each row (defaulted to "Opening Balance Import" if blank)
* **Restatement audit metadata** (when the import was triggered through the data exchange layer's Restatement required gate per **NXT-71249** §3.4): the user-entered restatement reason text, the confirmation timestamp, and the trigger classification (closed periods present, prior import present, or both). For Eligible imports (no gate triggered), these fields are null. The metadata must be visible in audit history alongside the entry and queryable so administrators can review the rationale for any restatement.

1.2 Each row in the import file becomes a **line** within that single ledger entry — one account, one side (debit or credit) per line.

1.3 Each GLAccount + SiteCode combination appears only once (enforced by the API Validations §2.5).

### 2) Effective Date

2.1 The effective date of the opening balance entry must be the **day before the first day of the specified fiscal year** (e.g., if the fiscal year starts July 1, the effective date is June 30).

2.2 This places the opening balance before any fiscal year activity in the ledger, ensuring date-based beginning balance calculations (e.g., `date < period start`) work without special-case logic.

2.3 The effective date is computed by BL and returned in the Validate response so DE can display it on Review.

2.4 The closed-period rule is enforced by API Validations §4 — see that section for the authoritative behavior. The Restatement required gate (DE side, **NXT-71249** §3.4) is the user-facing acknowledgment; this BL story is the authoritative backstop and persists the metadata.

### 3) Amount Handling

3.1 The data exchange layer enforces a fixed Debit/Credit amount format (separate Debit and Credit columns; signed-amount layouts are transformed by DE upstream of mapping per **NXT-71249** §4). Each validated row carries a positive value in exactly one of the two columns.

3.2 The Debit and Credit values from the validated row map directly to the ledger entry line. The opposite side is stored as zero.

3.3 The system does **not** enforce normal balance direction by base type. Assets may carry credit balances and liabilities may carry debit balances (abnormal balances are legitimate). Correctness is enforced by the batch-level balance check in API Validations §3.

### 4) Sub-account Validation

4.1 Sub-account validation is part of the API Validations pass — see §2.4 of that section for the authoritative rule. The sub-account is the routing target for posting site-level activity to a given GL account.

4.2 If any line's (GLAccount, SiteCode) combination has no configured sub-account, the **entire import is rejected** at the validator gate (DE never reaches Commit). Per the atomicity rule (§6 of this story), partial writes are not allowed.

4.3 The Central Office is treated as just another site for this check. A row with the Central Office SiteCode requires a configured sub-account on the targeted GL account, the same as any other site. BL does not apply special handling based on whether the SiteCode happens to be the Central Office.

4.4 On rejection, the import result record must surface the specific offending row(s) — the GLAccount, SiteCode, and a clear message indicating that no sub-account exists for that combination. The user resolves by either correcting the file (changing the SiteCode or GLAccount on the offending rows) or configuring the missing sub-account in Configuration > General Ledger and re-uploading.

4.5 Sub-account validation runs in the BL Validate call (and again at Commit for defense-in-depth). The data exchange layer (**NXT-71249**) does not pre-validate sub-account configuration — DE only validates that the SiteCode itself exists as configured in SchoolCafe. Sub-account routing is a BL concern.

### 5) Entry Behavior

5.1 Opening balance ledger entries are **read-only**. They cannot be edited or reversed through the UI.

5.2 If the user needs to correct individual accounts after import, they can create manual adjustment entries. The opening balance entry itself is immutable.

5.3 The "Opening Balance" entry type must be identifiable in the ledger so that reports, calculations, and the replacement/reversal logic can distinguish opening balance entries from period transaction activity.

### 6) Hard Reversal on Re-Import

6.1 If an opening balance import already exists for the selected fiscal year (detected and acknowledged through the data exchange layer's Restatement required gate per **NXT-71249** §3.4), the business logic layer must **reverse all ledger entry lines** from the previous import before writing the new import's entries.

6.2 This is a **hard reversal**, not a soft delete. Reversal entries are written to the ledger to maintain a complete audit trail.

6.3 The reversal entries must be dated the **same date as the original entries** (day before fiscal year start), so that the original and reversal net to zero at the same point in the ledger.

6.4 After reversal, the new import's entry and lines are written per Sections 1–4, including the restatement audit metadata from §1.1 and the sub-account validation from §4.

6.5 The entire reversal + re-creation must be **atomic** — if any part fails (including a sub-account validation failure on the new entry), no data is changed and the user is notified. The previous import remains intact.

6.6 The business logic layer must reject any re-import for a fiscal year that already has an opening balance entry if the restatement audit metadata is missing from the handoff payload. This prevents the gate from being bypassed by any means (URL manipulation, concurrent submissions, client-side tampering) and is the authoritative replacement check. (Enforced by API Validations §4.2.)

### 7) Downstream Report Behavior

7.1 Imported opening balances establish the **fiscal year starting position** for permanent accounts. This is the position as of the first day of the specified fiscal year.

7.2 **Trial Balance:** Imported opening balances must populate the Opening Balance column for the corresponding fiscal year. This column always reflects the fiscal year starting position, regardless of which period the user is viewing.

7.3 **Balance Sheet:** The imported opening balance feeds into the beginning balance calculation. The relationship depends on the reporting period:

* If the reporting period begins on the first day of the fiscal year (e.g., Period 1 or fiscal YTD), the beginning balance equals the imported opening balance.
* If the reporting period begins after the first period (e.g., October in a July fiscal year), the beginning balance equals the imported opening balance plus all cumulative transaction activity from prior periods within that fiscal year.

7.4 Opening balances must be clearly labeled as imported opening balances in all reports and must be visually distinguishable from period transaction activity.

7.5 If no opening balance import exists for a fiscal year, the Opening Balance column in affected reports must display `$0.00` with a visual indicator (e.g., icon, highlight, or footnote) communicating that no opening balance has been imported for this fiscal year. The indicator must not interfere with numeric sorting or Excel export behavior — the underlying value must always be zero, not a text string.

### 8) Audit History

8.1 Audit history views for opening balance entries must surface, when present, the restatement reason text, confirmation timestamp, and trigger classification from §1.1. This makes the rationale for any restatement queryable directly from the entry's audit trail without requiring a separate query against the import history.

8.2 Audit history must distinguish between:

* **Eligible imports** (no gate triggered, no restatement metadata)
* **Restatement imports triggered by closed periods** (gate triggered, reason persisted, prior import may or may not have existed)
* **Restatement imports triggered by prior import only** (gate triggered, reason persisted, no closed periods)
* **Restatement imports triggered by both conditions**

## Acceptance Criteria
**AC1 — Ledger Entry Created with Correct Entry Type** Given the data exchange layer passes validated import data after user confirmation, When the business logic layer processes the data, Then one ledger entry is created with entry type "Opening Balance" containing one line per imported row.

**AC2 — Effective Date Is Day Before Fiscal Year Start** Given the user imports opening balances for a fiscal year starting July 1, When the ledger entry is created, Then the effective date is June 30.

**AC3 — Debit/Credit Amounts Mapped Directly** Given the validated import data carries separate Debit and Credit columns (single canonical contract; signed-amount layouts are transformed by DE upstream of mapping per NXT-71249 §4), When the ledger entry lines are created, Then each line carries the debit or credit value from the row and zero on the opposite side.

**AC4 — Entry Is Read-Only** Given an opening balance ledger entry exists, When a user attempts to edit or reverse it through the UI, Then the system blocks the action — the entry can only be replaced by a new opening balance import.

**AC5 — Hard Reversal on Re-Import** Given an opening balance import already exists for the selected fiscal year and the user confirms a re-import, When the business logic layer processes the new import, Then all previous opening balance entry lines are reversed (hard reversal, not soft delete) with reversal entries dated the same day as the originals, and new entry lines are written from the new import.

**AC6 — Reversal + Re-Creation Is Atomic** Given a re-import is in progress and a system error occurs during the reversal or re-creation, When the error is encountered, Then no data is changed — the previous opening balance entries remain intact and the user is notified.

**AC7 — Closed-Period FY Rejected Without Restatement Metadata** Given any period within the target fiscal year has been closed and the request payload does not include the restatement gate metadata, When the business logic layer receives the import, Then BL rejects the import with an error indicating the affected period(s) must be reopened or the restatement gate must be satisfied. (Defense-in-depth — DE enforces the same rule upstream.)

**AC8 — Trial Balance Reflects Imported Opening Balance** Given an opening balance import has been completed for a fiscal year, When the user runs a Trial Balance for any period in that fiscal year, Then the Opening Balance column reflects the imported fiscal year starting position.

**AC9 — Balance Sheet Beginning Balance Computed Correctly for Period 1** Given an opening balance import has been completed for a fiscal year, When the user runs a Balance Sheet for the first period of the fiscal year (or fiscal YTD), Then the beginning balance equals the imported opening balance.

**AC10 — Balance Sheet Beginning Balance Computed Correctly for Later Periods** Given an opening balance import has been completed for a fiscal year and transactions exist in prior periods, When the user runs a Balance Sheet for a period after Period 1, Then the beginning balance equals the imported opening balance plus cumulative transaction activity from all prior periods in that fiscal year.

**AC11 — Missing Import Surfaced in Reports** Given no opening balance import exists for a fiscal year, When the user runs a Trial Balance or Balance Sheet, Then the Opening Balance column shows $0.00 with a visual indicator that no opening balance has been imported — the column must not contain a text string.

**AC12 — Normal Balance Not Enforced by Base Type** Given an import contains an asset account with a credit balance (abnormal balance), When the business logic layer processes the import, Then the entry is created as-is — the system does not reject or modify the balance direction based on account type.

**AC13 — API Validate Operation Returns Pass or Fail** Given DE calls the BL Validate API with a mapped file and target Fiscal Year, When BL runs the validation pass, Then BL returns either pass with total rows, total Debits, total Credits, computed effective date, and restatement classification — or fail with a structured error list (row number, field, error code, message) and an error report download URL. No partial work is performed.

**AC14 — Row-Level Structural Validation** Given the file contains rows with missing required fields (GLAccount, SiteCode, Description), non-decimal Debit or Credit values, both Debit and Credit populated on a row, both blank, both zero, any negative value, or a Description longer than 50 characters, When BL runs validation, Then BL returns fail with the affected rows listed.

**AC15 — GLAccount Existence and Type Restriction** Given a row references a GLAccount that does not exist in the chart of accounts or that is a Revenue or Expense account, When BL runs validation, Then BL returns fail. Unknown accounts are blocking for Opening Balance (chart hygiene wins because OB is one-time setup); Revenue/Expense rows are rejected with the message "Opening balances cannot be set for Revenue or Expense accounts."

**AC16 — Sub-Account Required for Every (GLAccount, SiteCode)** Given any line's (GLAccount, SiteCode) combination has no configured sub-account, When BL runs validation, Then BL returns fail with the offending rows surfaced. Central Office is treated as just another site for this check (no special handling).

**AC17 — Uniqueness of (GLAccount, SiteCode)** Given two or more rows in the file share the same (GLAccount, SiteCode) combination, When BL runs validation, Then BL returns fail with the duplicate combinations listed.

**AC18 — Balance Check Exact ($0 Tolerance)** Given the sum of Debits does not equal the sum of Credits across all rows, When BL runs validation, Then BL returns fail. Tolerance is exactly $0 — no rounding allowance.

**AC19 — Authoritative Replacement Check** Given a prior Opening Balance entry already exists for the target Fiscal Year and the request payload does not include the restatement gate metadata (typed FY confirmation, reason text, confirmation timestamp), When BL runs Commit, Then BL rejects the request. This prevents the DE-side gate from being bypassed via URL manipulation, concurrent submissions, or client-side tampering.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| BL receives import for an FY where any period is closed AND restatement gate metadata is absent | BL rejects with error message identifying the closed period(s); no ledger entry written |
| BL receives import for an FY where any period is closed AND restatement gate metadata is present | Accepted; previous report numbers will be revised; metadata persisted to entry audit trail |
| Re-import: prior OB entry exists AND restatement gate metadata is absent | BL rejects (authoritative replacement check); previous entry remains intact |
| Re-import: reversal fails mid-process | Entire operation rolls back; previous entries remain intact; user notified |
| Re-import: new entry creation fails after successful reversal | Entire operation rolls back; previous entries are restored; user notified |
| Opening balance entry already exists and user re-imports with fewer accounts | Previous entry fully reversed; new entry has only the accounts in the new file — missing accounts from the previous import are not carried forward |
| Validate-only call: any check fails | BL returns structured error list + error report download URL; no ledger write attempted |
| Commit: any check fails (including period status re-check) | BL returns structured error list; no ledger write performed |
| Validate succeeds, but a period closes between Validate and Commit and metadata does not cover it | Commit rejected; user returned to validation |
| No opening balance import exists for a fiscal year | Reports show $0.00 with visual indicator |
| Abnormal balance (e.g., asset with credit balance) | Accepted — system does not enforce normal balance direction |

## Data Model Summary

| Attribute | Value |
| --- | --- |
| Entry type | "Opening Balance" (dedicated, identifiable) |
| Effective date | Day before first day of fiscal year (e.g., June 30 for July 1 FY) |
| Structure | One entry, N lines (one per account per site) |
| Amount storage | Debit or credit per line; opposite side = zero |
| Editable | No — read-only, immutable |
| Reversible via UI | No — only via re-import (hard reversal) |
| Reversal date | Same date as original entries |
| Period constraints | Rejected if any period in target FY is closed AND restatement gate metadata is absent |
| Sub-account constraints | Rejected if any (GLAccount, SiteCode) lacks a configured sub-account |
| Replacement constraints | Rejected if a prior OB entry exists for target FY AND restatement gate metadata is absent |
| Atomicity | All-or-nothing — partial writes never permitted |
| Affects opening balances | Yes — sole source of opening balance values for permanent accounts |

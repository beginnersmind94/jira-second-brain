# NXT-71240 — FIN — Opening Balance Import: Business Logic

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

----

h2. Scope

This story covers the business logic layer for the Opening Balance Import: ledger entry creation, effective date assignment, entry type behavior, amount conversion, hard reversal on re-import, and downstream report behavior. The data exchange layer (companion story) handles file upload, validation, and preview, including the Fiscal Year selection (chosen at the import level, not mapped from the file). This story receives validated data and the selected Fiscal Year after the user confirms the import.

----

h2. Requirements

h3. 1) Ledger Entry Creation

1.1 On receiving validated import data from the data exchange layer, the system must create *one ledger entry* with:

* *Entry type:* "Opening Balance" (or equivalent dedicated type)
* *Effective date:* The day before the first day of the specified fiscal year (see Section 2)
* *Description:* The Comments value from each row (defaulted to "Opening Balance Import" if blank)

1.2 Each row in the import file becomes a *line* within that single ledger entry — one account, one side (debit or credit) per line.

1.3 Each GLAccount + SiteCode combination appears only once (enforced by the data exchange layer).

h3. 2) Effective Date

2.1 The effective date of the opening balance entry must be the *day before the first day of the specified fiscal year* (e.g., if the fiscal year starts July 1, the effective date is June 30).

2.2 This places the opening balance before any fiscal year activity in the ledger, ensuring date-based beginning balance calculations (e.g., {{date < period start}}) work without special-case logic.

2.3 The effective date must be communicated back to the data exchange layer for display in the preview.

2.4 Opening balance entries are *exempt from financial period open/close constraints*. The import must succeed regardless of whether any period is open, closed, or has never been opened.

h3. 3) Amount Conversion

3.1 For *Scheme A* (Debit/Credit): the debit and credit values from the validated data map directly to the ledger entry line. The opposite side is stored as zero.

3.2 For *Scheme B* (signed Amount): positive values are stored as debits, negative values are stored as credits (absolute value). The opposite side is stored as zero.

3.3 The system does *not* enforce normal balance direction by base type. Assets may carry credit balances and liabilities may carry debit balances (abnormal balances are legitimate). Correctness is enforced by the batch-level balance check in the data exchange layer.

h3. 4) Entry Behavior

4.1 Opening balance ledger entries are *read-only*. They cannot be edited or reversed through the UI.

4.2 If the user needs to correct individual accounts after import, they can create manual adjustment entries. The opening balance entry itself is immutable.

4.3 The "Opening Balance" entry type must be identifiable in the ledger so that reports, calculations, and the replacement/reversal logic can distinguish opening balance entries from period transaction activity.

h3. 5) Hard Reversal on Re-Import

5.1 If an opening balance import already exists for the selected fiscal year (detected by the data exchange layer, confirmed by the user), the business logic layer must *reverse all ledger entry lines* from the previous import before writing the new import's entries.

5.2 This is a *hard reversal*, not a soft delete. Reversal entries are written to the ledger to maintain a complete audit trail.

5.3 The reversal entries must be dated the *same date as the original entries* (day before fiscal year start), so that the original and reversal net to zero at the same point in the ledger.

5.4 After reversal, the new import's entry and lines are written per Sections 1–3.

5.5 The entire reversal + re-creation must be *atomic* — if any part fails, no data is changed and the user is notified.

h3. 6) Downstream Report Behavior

6.1 Imported opening balances establish the *fiscal year starting position* for permanent accounts. This is the position as of the first day of the specified fiscal year.

6.2 *Trial Balance:* Imported opening balances must populate the Opening Balance column for the corresponding fiscal year. This column always reflects the fiscal year starting position, regardless of which period the user is viewing.

6.3 *Balance Sheet:* The imported opening balance feeds into the beginning balance calculation. The relationship depends on the reporting period:

* If the reporting period begins on the first day of the fiscal year (e.g., Period 1 or fiscal YTD), the beginning balance equals the imported opening balance.
* If the reporting period begins after the first period (e.g., October in a July fiscal year), the beginning balance equals the imported opening balance plus all cumulative transaction activity from prior periods within that fiscal year.

6.4 Opening balances must be clearly labeled as imported opening balances in all reports and must be visually distinguishable from period transaction activity.

6.5 If no opening balance import exists for a fiscal year, the Opening Balance column in affected reports must display {{$0.00}} with a visual indicator (e.g., icon, highlight, or footnote) communicating that no opening balance has been imported for this fiscal year. The indicator must not interfere with numeric sorting or Excel export behavior — the underlying value must always be zero, not a text string.

## Acceptance Criteria
*AC1 — Ledger Entry Created with Correct Entry Type* Given the data exchange layer passes validated import data after user confirmation, When the business logic layer processes the data, Then one ledger entry is created with entry type "Opening Balance" containing one line per imported row.

*AC2 — Effective Date Is Day Before Fiscal Year Start* Given the user imports opening balances for a fiscal year starting July 1, When the ledger entry is created, Then the effective date is June 30.

*AC3 — Scheme A Amounts Mapped Directly* Given the import uses Scheme A (Debit/Credit), When the ledger entry lines are created, Then each line carries the debit or credit value from the file and zero on the opposite side.

*AC4 — Scheme B Amounts Converted Correctly* Given the import uses Scheme B (signed Amount), When the ledger entry lines are created, Then positive values are stored as debits and negative values are stored as credits (absolute value), with zero on the opposite side.

*AC5 — Entry Is Read-Only* Given an opening balance ledger entry exists, When a user attempts to edit or reverse it through the UI, Then the system blocks the action — the entry can only be replaced by a new opening balance import.

*AC6 — Hard Reversal on Re-Import* Given an opening balance import already exists for the selected fiscal year and the user confirms a re-import, When the business logic layer processes the new import, Then all previous opening balance entry lines are reversed (hard reversal, not soft delete) with reversal entries dated the same day as the originals, and new entry lines are written from the new import.

*AC7 — Reversal + Re-Creation Is Atomic* Given a re-import is in progress and a system error occurs during the reversal or re-creation, When the error is encountered, Then no data is changed — the previous opening balance entries remain intact and the user is notified.

*AC8 — Opening Balances Exempt from Financial Period Constraints* Given the first period of the specified fiscal year is closed or has never been opened, When the business logic layer processes the import, Then the ledger entry is created successfully regardless of the period's open/close status.

*AC9 — Trial Balance Reflects Imported Opening Balance* Given an opening balance import has been completed for a fiscal year, When the user runs a Trial Balance for any period in that fiscal year, Then the Opening Balance column reflects the imported fiscal year starting position.

*AC10 — Balance Sheet Beginning Balance Computed Correctly for Period 1* Given an opening balance import has been completed for a fiscal year, When the user runs a Balance Sheet for the first period of the fiscal year (or fiscal YTD), Then the beginning balance equals the imported opening balance.

*AC11 — Balance Sheet Beginning Balance Computed Correctly for Later Periods* Given an opening balance import has been completed for a fiscal year and transactions exist in prior periods, When the user runs a Balance Sheet for a period after Period 1, Then the beginning balance equals the imported opening balance plus cumulative transaction activity from all prior periods in that fiscal year.

*AC12 — Missing Import Surfaced in Reports* Given no opening balance import exists for a fiscal year, When the user runs a Trial Balance or Balance Sheet, Then the Opening Balance column shows {{$0.00}} with a visual indicator that no opening balance has been imported — the column must not contain a text string.

*AC13 — Normal Balance Not Enforced by Base Type* Given an import contains an asset account with a credit balance (abnormal balance), When the business logic layer processes the import, Then the entry is created as-is — the system does not reject or modify the balance direction based on account type.

h2. Edge / Empty / Error States

||Scenario||Behavior||
|Re-import: reversal fails mid-process|Entire operation rolls back; previous entries remain intact; user notified|
|Re-import: new entry creation fails after successful reversal|Entire operation rolls back; previous entries are restored; user notified|
|Opening balance entry already exists and user re-imports with fewer accounts|Previous entry fully reversed; new entry has only the accounts in the new file — missing accounts from the previous import are not carried forward|
|No opening balance import exists for a fiscal year|Reports show $0.00 with visual indicator|
|Abnormal balance (e.g., asset with credit balance)|Accepted — system does not enforce normal balance direction|

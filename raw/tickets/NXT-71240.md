# NXT-71240 — FIN — Opening Balance Import: Business Logic

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-04._

---

## Scope

This story covers the business logic layer for the Opening Balance Import: ledger entry creation, effective date assignment, entry type behavior, amount handling, sub-account validation, hard reversal on re-import, persistence of restatement audit metadata, and downstream report behavior. The data exchange layer (companion story **NXT-71249**) handles file upload, validation, preview, and the Restatement required confirmation gate (typed FY confirmation + restatement reason). This story receives validated data with separate Debit and Credit columns, the selected Fiscal Year, and (when applicable) the restatement audit metadata after the user confirms the import.

---

## Requirements

### 1) Ledger Entry Creation

1.1 On receiving validated import data from the data exchange layer, the system must create **one ledger entry** with:

* **Entry type:** "Opening Balance" (or equivalent dedicated type)
* **Effective date:** The day before the first day of the specified fiscal year (see Section 2)
* **Description:** The Description value from each row (defaulted to "Opening Balance Import" if blank)
* **Restatement audit metadata** (when the import was triggered through the data exchange layer's Restatement required gate per **NXT-71249** §3.3): the user-entered restatement reason text, the confirmation timestamp, and the trigger classification (closed periods present, prior import present, or both). For Eligible imports (no gate triggered), these fields are null. The metadata must be visible in audit history alongside the entry and queryable so administrators can review the rationale for any restatement.

1.2 Each row in the import file becomes a **line** within that single ledger entry — one account, one side (debit or credit) per line.

1.3 Each GLAccount + SiteCode combination appears only once (enforced by the data exchange layer).

### 2) Effective Date

2.1 The effective date of the opening balance entry must be the **day before the first day of the specified fiscal year** (e.g., if the fiscal year starts July 1, the effective date is June 30).

2.2 This places the opening balance before any fiscal year activity in the ledger, ensuring date-based beginning balance calculations (e.g., `date < period start`) work without special-case logic.

2.3 The effective date must be communicated back to the data exchange layer for display in the preview.

2.4 The business logic layer must **reject the import if any period in the target fiscal year has been closed** AND the data exchange layer did not pass through a satisfied Restatement required gate acknowledgment for the closed-period trigger. With the gate acknowledgment present, the import is accepted because the user has explicitly acknowledged that financial reports for the closed periods will be revised. Without the gate acknowledgment present, the import is rejected to prevent silent invalidation of already-closed financial reports. The data exchange layer (**NXT-71249**) applies the gate upstream; BL applies the same check as a defense-in-depth backstop. Imports for fiscal years where no period has been closed (go-live, new FY, mid-year correction before first close) are accepted without a gate acknowledgment.

### 3) Amount Handling

3.1 The data exchange layer enforces a fixed Debit/Credit amount format (separate Debit and Credit columns; signed-amount layouts are not supported per **NXT-71249** §1.5). Each validated row carries a positive value in exactly one of the two columns.

3.2 The Debit and Credit values from the validated row map directly to the ledger entry line. The opposite side is stored as zero.

3.3 The system does **not** enforce normal balance direction by base type. Assets may carry credit balances and liabilities may carry debit balances (abnormal balances are legitimate). Correctness is enforced by the batch-level balance check in the data exchange layer.

### 4) Sub-account Validation

4.1 For each line, BL must verify that a sub-account exists for the (GLAccount, SiteCode) combination. The sub-account is the routing target for posting site-level activity to a given GL account.

4.2 If any line's (GLAccount, SiteCode) combination has no configured sub-account, the **entire import is rejected**. Per the atomicity rule (§6 of this story), partial writes are not allowed — a single missing sub-account blocks the whole import.

4.3 The Central Office is treated as just another site for this check. A row with the Central Office SiteCode requires a configured sub-account on the targeted GL account, the same as any other site. BL does not apply special handling based on whether the SiteCode happens to be the Central Office.

4.4 On rejection, the import result record must surface the specific offending row(s) — the GLAccount, SiteCode, and a clear message indicating that no sub-account exists for that combination. The user resolves by either correcting the file (changing the SiteCode or GLAccount on the offending rows) or configuring the missing sub-account in Configuration > General Ledger and re-uploading.

4.5 This check runs at write time. The data exchange layer (**NXT-71249**) does not pre-validate sub-account configuration — DE only validates that the SiteCode itself exists in Site Management. Sub-account routing is a BL concern.

### 5) Entry Behavior

5.1 Opening balance ledger entries are **read-only**. They cannot be edited or reversed through the UI.

5.2 If the user needs to correct individual accounts after import, they can create manual adjustment entries. The opening balance entry itself is immutable.

5.3 The "Opening Balance" entry type must be identifiable in the ledger so that reports, calculations, and the replacement/reversal logic can distinguish opening balance entries from period transaction activity.

### 6) Hard Reversal on Re-Import

6.1 If an opening balance import already exists for the selected fiscal year (detected and acknowledged through the data exchange layer's Restatement required gate per **NXT-71249** §3.3), the business logic layer must **reverse all ledger entry lines** from the previous import before writing the new import's entries.

6.2 This is a **hard reversal**, not a soft delete. Reversal entries are written to the ledger to maintain a complete audit trail.

6.3 The reversal entries must be dated the **same date as the original entries** (day before fiscal year start), so that the original and reversal net to zero at the same point in the ledger.

6.4 After reversal, the new import's entry and lines are written per Sections 1–4, including the restatement audit metadata from §1.1 and the sub-account validation from §4.

6.5 The entire reversal + re-creation must be **atomic** — if any part fails (including a sub-account validation failure on the new entry), no data is changed and the user is notified. The previous import remains intact.

6.6 The business logic layer must reject any re-import for a fiscal year that already has an opening balance entry if the restatement audit metadata is missing from the handoff payload. This prevents the gate from being bypassed by any means (URL manipulation, concurrent submissions, client-side tampering) and is the authoritative replacement check.

### 7) Downstream Report Behavior

7.1 Imported opening balances establish the **fiscal year starting position** for permanent accounts. This is the position as of the first day of the specified fiscal year.

7.2 **Trial Balance:** Imported opening balances must populate the Opening Balance column for the corresponding fiscal year. This column always reflects the fiscal year starting position, regardless of which period the user is viewing.

7.3 **Balance Sheet:** The imported opening balance feeds into the beginning balance calculation. The relationship depends on the reporting period:

* If the reporting period begins on the first day of the fiscal year (e.g., Period 1 or fiscal YTD), the beginning balance equals the imported opening balance.
* If the reporting period begins after the first period (e.g., October in a July fiscal year), the beginning balance equals the imported opening balance plus all cumulative transaction activity from prior periods within that fiscal year.

7.4 Opening balances must be clearly labeled as imported opening balances in all reports and must be visually distinguishable from period transaction activity.

7.5 If no opening balance import exists for a fiscal year, the Opening Balance column in affected reports must display `$0.00` with a visual indicator (e.g., icon, highlight, or footnote) communicating that no opening balance has been imported for this fiscal year. The indicator must not interfere with numeric sorting or Excel export behavior — the underlying value must always be zero, not a text string.

### 8) Audit History

8.1 Audit history views for opening balance entries must surface, when present, the restatement reason text, confirmation timestamp, and trigger classification from §1.1. This makes the rationale for any restatement queryable directly from the entry's audit trail without requiring a separate query against the import history.

8.2 Audit history must distinguish between:

* **Eligible imports** (no gate triggered, no restatement metadata)
* **Restatement imports triggered by closed periods** (gate triggered, reason persisted, prior import may or may not have existed)
* **Restatement imports triggered by prior import only** (gate triggered, reason persisted, no closed periods)
* **Restatement imports triggered by both conditions**

## Acceptance Criteria
**AC1 — Ledger Entry Created with Correct Entry Type** Given the data exchange layer passes validated import data after user confirmation, When the business logic layer processes the data, Then one ledger entry is created with entry type "Opening Balance" containing one line per imported row.

**AC2 — Effective Date Is Day Before Fiscal Year Start** Given the user imports opening balances for a fiscal year starting July 1, When the ledger entry is created, Then the effective date is June 30.

**AC3 — Scheme A Amounts Mapped Directly** Given the import uses Scheme A (Debit/Credit), When the ledger entry lines are created, Then each line carries the debit or credit value from the file and zero on the opposite side.

**AC4 — Scheme B Amounts Converted Correctly** Given the import uses Scheme B (signed Amount), When the ledger entry lines are created, Then positive values are stored as debits and negative values are stored as credits (absolute value), with zero on the opposite side.

**AC5 — Entry Is Read-Only** Given an opening balance ledger entry exists, When a user attempts to edit or reverse it through the UI, Then the system blocks the action — the entry can only be replaced by a new opening balance import.

**AC6 — Hard Reversal on Re-Import** Given an opening balance import already exists for the selected fiscal year and the user confirms a re-import, When the business logic layer processes the new import, Then all previous opening balance entry lines are reversed (hard reversal, not soft delete) with reversal entries dated the same day as the originals, and new entry lines are written from the new import.

**AC7 — Reversal + Re-Creation Is Atomic** Given a re-import is in progress and a system error occurs during the reversal or re-creation, When the error is encountered, Then no data is changed — the previous opening balance entries remain intact and the user is notified.

**AC8 — Closed-Period FY Rejected** Given any period within the target fiscal year has been closed, When the business logic layer receives the import, Then BL rejects the import with an error indicating the affected period(s) must be reopened before the opening balance can be set. _(This is a defense-in-depth check — the data exchange layer enforces the same rule upstream.)_

**AC9 — Trial Balance Reflects Imported Opening Balance** Given an opening balance import has been completed for a fiscal year, When the user runs a Trial Balance for any period in that fiscal year, Then the Opening Balance column reflects the imported fiscal year starting position.

**AC10 — Balance Sheet Beginning Balance Computed Correctly for Period 1** Given an opening balance import has been completed for a fiscal year, When the user runs a Balance Sheet for the first period of the fiscal year (or fiscal YTD), Then the beginning balance equals the imported opening balance.

**AC11 — Balance Sheet Beginning Balance Computed Correctly for Later Periods** Given an opening balance import has been completed for a fiscal year and transactions exist in prior periods, When the user runs a Balance Sheet for a period after Period 1, Then the beginning balance equals the imported opening balance plus cumulative transaction activity from all prior periods in that fiscal year.

**AC12 — Missing Import Surfaced in Reports** Given no opening balance import exists for a fiscal year, When the user runs a Trial Balance or Balance Sheet, Then the Opening Balance column shows `$0.00` with a visual indicator that no opening balance has been imported — the column must not contain a text string.

**AC13 — Normal Balance Not Enforced by Base Type** Given an import contains an asset account with a credit balance (abnormal balance), When the business logic layer processes the import, Then the entry is created as-is — the system does not reject or modify the balance direction based on account type.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| BL receives import for an FY where any period is closed | BL rejects with error message identifying the closed period(s); no ledger entry written |
| Re-import: reversal fails mid-process | Entire operation rolls back; previous entries remain intact; user notified |
| Re-import: new entry creation fails after successful reversal | Entire operation rolls back; previous entries are restored; user notified |
| Opening balance entry already exists and user re-imports with fewer accounts | Previous entry fully reversed; new entry has only the accounts in the new file — missing accounts from the previous import are not carried forward |
| No opening balance import exists for a fiscal year | Reports show $0.00 with visual indicator |
| Abnormal balance (e.g., asset with credit balance) | Accepted — system does not enforce normal balance direction |

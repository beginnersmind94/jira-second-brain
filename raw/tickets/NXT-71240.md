# NXT-71240 — FIN — Opening Balance Import: Business Logic

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-12. Reworked per Opening Balance Import edge case resolutions: replaced the Restatement gate / metadata model with batch-level overwrite, added explicit Step 0 eligibility checks (Prior FY periods, CoA minimum, concurrency lock), added Mandatory / Optional exception model, formalized GL account and Site validation refinements (inactive, format, casing, AccountName mismatch), added uniqueness duplicate handling, all-zero file and decimal-precision rules, three-label Posting Activity Log convention. Removed Sub-account Validation as a separate concern (not addressed by edge case resolutions). 2026-05-07: TL;DR added; description trimmed \~30%. 2026-05-06: API Validations contract established with Jonathan Smith (DE/BL split)._

## TL;DR

Expose two API ops to DE: **Validate** (run every check; write nothing) and **Commit** (run every check + write). _DE calls Validate at the Field Mapping validator gate; Commit at the user's final confirm._

**Step 0 eligibility (before file upload):**

* Prior FY periods all closed (else block with deep link to Period Management).
* CoA has ≥1 Asset, Liability, or Equity account (else block).
* If a prior OB import exists for this FY, user must choose overwrite. **Overwrite is allowed only if there is not a single closed period in the associated Fiscal Year.**
* Server-side lock per District + Fiscal Year — first job wins; concurrent attempts blocked.

**On every Validate / Commit call, BL runs:**

* **Row structure:** required fields present; Debit/Credit positive decimals; _exactly one_ of Debit/Credit populated and non-zero; max 2 decimal places (do not round or truncate).
* **GLAccount:** exists in CoA, type is Asset/Liability/Equity, is active, format is valid. AccountName mismatch logged as Optional (CoA name wins).
* **SiteCode:** exists in Site Management. Case-insensitive match, leading/trailing spaces trimmed. Site type irrelevant. Inactive sites allowed. Blank SiteCode blocks (no defaulting).
* **Uniqueness:** each (GLAccount, SiteCode) appears once only. Duplicates block (do not drop, do not sum).
* **File integrity:** Sum Debits = Sum Credits, exactly $0 tolerance. At least one non-zero amount. No negative values.

**Exception model:**

* **Mandatory** — any present blocks the full commit. No journal entries written.
* **Optional** — logged for review; commit may still proceed if no Mandatory exceptions exist.

**Response on pass:** rows, Debits, Credits, **effective date** (= day before FY start; e.g., June 30 for July 1 FY), overwrite eligibility flag.
**Response on fail:** structured exception list (Column Name, Type, Description). DE surfaces.

**On Commit success:** create one ledger entry, type **"Opening Balance"**, one line per row. Each line stores the row's Debit or Credit with **zero on the opposite side**. Entry is **read-only** — no edit or reverse via UI.

**Overwrite = hard reversal by batch.** When user opts to overwrite and FY has no closed periods, reverse the entire prior OB batch (reversal entries dated same as originals), then post the replacement batch. Both atomic — any failure rolls everything back. Posting Activity Log uses three distinct labels: `Opening Balance Import`, `Opening Balance Import Reversal`, `Opening Balance Import Replacement`.

**Reports:**

* Trial Balance Opening Balance column = imported OB for the FY.
* Balance Sheet beginning balance = OB for Period 1 / fiscal YTD; OB + cumulative prior-period activity for later periods.
* **No OB for FY?** Display `$0.00` with a visual indicator — _never a text string_ (must not break sort/Excel).

**Abnormal balances are accepted** (e.g., asset with credit balance). We don't enforce normal balance direction by base type.

DE layer (**NXT-71249**) handles upload, mapping UI, validator gate (calls BL Validate), preview, and the user-facing overwrite confirmation dialog. This story receives the mapped file with separate Debit/Credit columns and the selected FY.

**Out of scope (other owners):**

* File parsing, encoding, multi-sheet, oversized file, duplicate column headers — owned by DX.
* Import Monitor status taxonomy, notifications, retention, deletion — owned by DX. Available statuses: `DONE`, `FAILED`.
* Session recovery, browser refresh, duplicate-click handling, retry behavior, idempotency — platform concerns.
* Permissions — Platform. Only users with the highest level of access can run OB Import; that same access level includes overwrite. No separate Import / Overwrite permissions. No row-level site permission checks because site-scoped users cannot access OB Import.

## Exception Format

When a check fails, BL returns the exception using:

* **Column Name** — the field or pair (e.g., `GLAccount`, `SiteCode`, `Debit/Credit`, `GLAccount/SiteCode`, `Fiscal Year`, `AccountName`, `Description`)
* **Type** — `Mandatory` or `Optional`
* **Description** — human-readable text. If a source row matters, include `Source row {RowNumber}: ...`.

**Mandatory** = any present blocks the full commit. No journal entries written.
**Optional** = logged but commit may proceed if no Mandatory exceptions exist.

## Step 0 — Eligibility (before file processing)

### 0.1 Prior FY periods open

BL eligibility API returns `priorPeriodsOpen`. FE blocks Step 0 with a deep link to Period Management. No Import Monitor exception. User cannot proceed.

### 0.2 Prior OB import exists for this FY

BL eligibility API returns `priorImportExists`. FE prompts the user to overwrite.

* Overwrite is allowed only if there is not a single closed period in the associated Fiscal Year.
* If user chooses **No**, import cannot proceed.
* If user chooses **Yes** and all FY periods are open, overwrite proceeds.
* If user chooses **Yes** and one or more FY periods are closed, overwrite is blocked at Commit (see §6 below).

### 0.3 Chart of Accounts missing required account types

BL blocks at Step 0 if the CoA has zero Asset, Liability, or Equity accounts. Message:

```
Configure at least one Asset, Liability, or Equity account in Chart of Accounts before importing Opening Balances.
```

No Import Monitor exception. User cannot proceed.

### 0.4 Selected FY is past, future, or unknown

* Past FY allowed (§0.1 still enforces required period closure on prior FY).
* Future FY start date allowed (normal setup before FY starts).
* Unknown FY not selectable — dropdown is sourced from configured fiscal years.

### 0.5 Concurrent submissions for the same District + Fiscal Year

Server-side lock per District + Fiscal Year. First processing job wins. Any second job for the same District/FY is blocked while the first is in progress. Do not use last-writer-wins.

```
An Opening Balance import is already in progress for this fiscal year. Wait for it to finish, then refresh Import Monitor before trying again.
```

No row-level exception. Job-level concurrency block.

## API Validations

BL exposes **Validate** (called by DE at the Field Mapping validator gate per **NXT-71249** §8.1) and **Commit** (called by DE on Confirm). Both run the same validation pass; Commit additionally writes. All checks below run on every call.

On any Mandatory failure, the API returns the structured exception list. DE surfaces. **No partial work.**

### 1) Row-level structural

1.1 **Required fields.** GLAccount, SiteCode, Debit/Credit present on every row. Missing required field → Mandatory exception (`{ColumnName} is required`).

1.2 **Numeric.** Debit and Credit are numeric on every row. Non-numeric → Mandatory exception.

1.3 **Positive only.** Debit and Credit columns are positive-only. Negative values → Mandatory exception.

1.4 **Single-side rule.** Exactly one of Debit/Credit populated and non-zero per row. Both populated with non-zero, or both empty/zero → Mandatory exception.

1.5 **Decimal precision.** Amount must have ≤ 2 decimal places. **Do not round or truncate.** > 2 decimal places → Mandatory exception.

1.6 **Description length.** Description > 50 characters → truncate to 50, log Optional exception. Commit may proceed.

1.7 **Description sanitization.** Special characters / formula-like content sanitized silently at BL. No exception unless the sanitized value differs meaningfully from input.

### 2) Reference data — GL Account

2.1 **Exists in CoA.** Unknown account → Mandatory exception. _(Differs from Payroll, which writes unknown accounts with a warning per_ **NXT-70777** _§5.)_

2.2 **Type is Asset, Liability, or Equity.** Revenue/Expense rows rejected with the message: _"Opening balances cannot be set for Revenue or Expense accounts."_

2.3 **Active.** Inactive GL account → Mandatory exception.

2.4 **Format valid.** Malformed GL account value → Mandatory exception (`GLAccount {GLAccount} is not in a valid account format`).

2.5 **AccountName mismatch.** If imported AccountName ≠ CoA AccountName for the same GLAccount → Optional exception. CoA name wins. Commit may proceed.

### 3) Reference data — Site

3.1 **Exists.** SiteCode must exist in SchoolCafe Site Management. Unknown → Mandatory exception (`SiteNotFound`).

3.2 **Active status irrelevant.** Inactive sites are valid posting targets for opening balances. No exception.

3.3 **Site type irrelevant.** The system must not inspect or block based on site type (meal-program, accounting, Central Office, or any other). No exception.

3.4 **Case insensitive.** SiteCode matching is case-insensitive and trims leading/trailing spaces. Posted entries use the canonical SiteCode from Site Management. No exception.

3.5 **Blank.** Blank SiteCode → Mandatory exception. No defaulting (including no default to Central Office).

3.6 **Central Office not present in district.** Treated as `SiteNotFound` per row. Do not block at Step 0 based on an assumed Central Office code. Do not default unknown or blank values to Central Office.

### 4) Uniqueness

4.1 **(GLAccount, SiteCode) pair unique within file.** Duplicates (same pair, any amounts — including literal duplicate rows that keep the file balanced) → Mandatory exception. Do not drop either row, do not keep first/last, do not sum.

4.2 **Same GLAccount across multiple sites.** Allowed. Uniqueness is the pair, not GLAccount alone.

### 5) File-level integrity

5.1 **Balanced.** Sum Debits = Sum Credits. Tolerance **exactly $0**. _(Differs from Payroll's $0.01 tolerance.)_ Out of balance → Mandatory exception with `Debits: {DebitTotal}; Credits: {CreditTotal}; Difference: {Difference}`.

5.2 **Non-zero file.** At least one Debit or Credit must be > 0. All-zero file → Mandatory exception.

5.3 **Single-row file.** No special rule — standard row + balance checks apply. A single row inevitably fails §1.4 or §5.1.

5.4 **Monetary precision.** Process amounts using decimal-safe logic. Floating-point precision loss is not acceptable for ledger import.

> **BL confirmation pending:** maximum supported amount and decimal scale.

### 6) Overwrite eligibility

6.1 **Overwrite = Yes, all FY periods open** → overwrite proceeds.

6.2 **Overwrite = Yes, one or more FY periods closed** → Mandatory exception (`Fiscal Year`):

```
Opening Balance overwrite cannot be completed because one or more periods in Fiscal Year {FiscalYear} are closed.
```

No reversal entries are posted. No replacement entries are posted. The system must not reverse the prior batch, post the replacement, reopen periods automatically, or partially process the overwrite. The condition is not limited to the original Opening Balance posting period — any closed period in the FY blocks.

6.3 **Overwrite = No, prior import exists** → Step 0 blocks. No exception (import has not started).

6.4 **Period status re-check at Commit.** BL re-verifies no period in the target FY closed between Validate and Commit. If any closed, Commit is rejected.

6.5 **User selects Overwrite = Yes but cancels before Start Import** → overwrite choice is UI state only; no entries written.

## Requirements

### 1) Ledger Entry Creation

1.1 On Commit (after all API Validations pass), create **one ledger entry**:

* **Entry type:** `Opening Balance`
* **Effective date:** Day before fiscal year start (see §2)
* **Description:** Each row's Description, sanitized and truncated to 50 chars where applicable

1.2 Each file row becomes a **line** in the entry — one account, one side per line.

1.3 Each (GLAccount, SiteCode) appears only once (enforced by API Validations §4.1).

### 2) Effective Date

2.1 Effective date is the **day before the first day of the specified FY** (e.g., July 1 FY → June 30).

2.2 BL computes the effective date and returns it in Validate so DE can display on Review.

2.3 Period-closure enforcement — see API Validations §6.

### 3) Amount Handling

3.1 DE enforces a fixed Debit/Credit format (separate columns; signed-amount transformed upstream per **NXT-71249** §4). Each validated row carries a positive value in exactly one column.

3.2 Debit/Credit values map directly to the ledger line; opposite side stored as zero.

3.3 **Normal balance direction is not enforced.** Assets may carry credit balances; liabilities may carry debit balances. Correctness is enforced by the batch balance check (API Validations §5.1).

### 4) Entry Behavior

4.1 OB entries are **read-only** — no edit or reverse via UI.

4.2 Corrections after import = manual adjustment entries. The OB entry is immutable.

4.3 The `Opening Balance` entry type must be identifiable in the ledger so reports, calculations, and overwrite/reversal logic can distinguish it from period activity.

### 5) Overwrite — Hard Reversal by Batch

5.1 Opening Balance imports are treated as **batches**.

5.2 When the user overwrites a prior OB import (and eligibility passes per API Validations §6), the system **reverses every entry from the prior OB import batch**, then posts the new file as a new OB import batch.

5.3 **Hard reversal**, not soft delete. Reversal entries are written to the ledger for a complete audit trail. Deletion is not possible in the system.

5.4 Reversal entries are dated **same as the originals** (day before FY start) — original and reversal net to zero at the same ledger point.

5.5 Reversal + new batch posting is **atomic**. Any failure → no data changed, previous import intact, user notified.

5.6 The system does not compare old rows to new rows, does not selectively preserve entries from the prior file, does not merge the new file into the old batch. Accounts present in the prior batch but absent from the new file are simply not reposted in the replacement batch.

5.7 Later transactions (posted after the original OB) remain posted as-is. Reports show the latest ledger state based on posted entries. No special handling solely because prior OB entries appeared in reports or later activity exists.

5.8 Validation against the **current** CoA. Prior-file history does not allow posting to a now-missing account. Accounts that existed in the prior file but no longer exist in CoA fail per API Validations §2.1.

### 6) Posting Activity Log Labels

Three distinct labels for audit readability:

* `Opening Balance Import` — original batch
* `Opening Balance Import Reversal` — reversal of a prior batch on overwrite
* `Opening Balance Import Replacement` — new batch posted after overwrite

Minimum audit details per entry:

* Fiscal Year
* Original Import ID
* Replacement Import ID (if applicable)
* Batch ID
* User
* Timestamp
* Source file name

Posting Activity Log shows the original, reversal, and replacement entries as separate auditable activity.

### 7) Downstream Report Behavior

7.1 Imported OBs establish the **FY starting position** for permanent accounts (as of day 1 of the FY).

7.2 **Trial Balance:** Imported OBs populate the Opening Balance column for the FY. Column always reflects the FY starting position regardless of period viewed.

7.3 **Balance Sheet:** OB feeds the beginning balance calculation:

* Period begins on day 1 of FY (Period 1 or fiscal YTD) → beginning balance = imported OB.
* Period begins after Period 1 (e.g., October in a July FY) → beginning balance = imported OB + cumulative prior-period activity within the FY.

7.4 OBs are clearly labeled in all reports and visually distinguishable from period activity.

7.5 If no OB import exists for an FY, the Opening Balance column shows `$0.00` + a visual indicator (icon, highlight, or footnote) that no OB has been imported. Indicator must not interfere with sort or Excel export — underlying value is always zero, never a text string.

## Acceptance Criteria
**AC1 — Ledger Entry Created with Correct Entry Type** On Commit, BL creates one ledger entry with entry type "Opening Balance", one line per imported row.

**AC2 — Effective Date Is Day Before Fiscal Year Start** For an FY starting July 1, the entry's effective date is June 30.

**AC3 — Debit/Credit Amounts Mapped Directly** Validated data carries separate Debit/Credit columns (signed-amount layouts transformed upstream by DE per NXT-71249 §4). Each ledger line carries the row's debit or credit; opposite side = zero.

**AC4 — Entry Is Read-Only** Edit or reverse via UI is blocked. The entry can only be replaced by a new OB import.

**AC5 — Hard Reversal on Re-Import** On a re-import for an FY with a prior OB, all prior entry lines are hard-reversed (reversal entries dated same as originals); new entry lines written from the new import.

**AC6 — Reversal + Re-Creation Is Atomic** Any system error during reversal or re-creation → no data changed; previous OB entries remain intact; user notified.

**AC7 — Closed-Period FY Rejected Without Restatement Metadata** If any period in the target FY is closed and the payload lacks restatement gate metadata, BL rejects with an error naming the affected period(s) and required action. (Defense-in-depth — DE enforces upstream.)

**AC8 — Trial Balance Reflects Imported Opening Balance** After an OB import for an FY, Trial Balance for any period in that FY shows the imported FY starting position in the Opening Balance column.

**AC9 — Balance Sheet Beginning Balance Computed Correctly for Period 1** After an OB import, Balance Sheet for Period 1 (or fiscal YTD) shows beginning balance = imported OB.

**AC10 — Balance Sheet Beginning Balance Computed Correctly for Later Periods** After an OB import (with prior-period activity), Balance Sheet for a period after Period 1 shows beginning balance = imported OB + cumulative prior-period activity within the FY.

**AC11 — Missing Import Surfaced in Reports** If no OB exists for an FY, Trial Balance / Balance Sheet shows $0.00 + a visual indicator that no OB has been imported. Column must not contain a text string.

**AC12 — Normal Balance Not Enforced by Base Type** An asset row with a credit balance (abnormal) is processed as-is. System does not reject or modify direction by account type.

**AC13 — API Validate Operation Returns Pass or Fail** BL Validate returns either pass (rows, Debits, Credits, computed effective date, restatement classification) or fail (structured error list — row#, field, code, message — + error report download URL). No partial work.

**AC14 — Row-Level Structural Validation** BL returns fail with affected rows for: missing required fields (GLAccount, SiteCode, Description), non-decimal Debit/Credit, both populated/both blank/both zero/negative, or Description > 50 chars.

**AC15 — GLAccount Existence and Type Restriction** BL returns fail for unknown GLAccount or Revenue/Expense base type. Unknown is blocking for OB (chart hygiene wins; OB is one-time). Revenue/Expense message: _"Opening balances cannot be set for Revenue or Expense accounts."_

**AC16 — Sub-Account Required for Every (GLAccount, SiteCode)** BL returns fail with offending rows when any (GLAccount, SiteCode) lacks a configured sub-account. Central Office is just another site.

**AC17 — Uniqueness of (GLAccount, SiteCode)** BL returns fail with duplicate combinations listed when two+ rows share the same (GLAccount, SiteCode).

**AC18 — Balance Check Exact ($0 Tolerance)** BL returns fail when sum Debits ≠ sum Credits. Tolerance exactly $0 — no rounding allowance.

**AC19 — Authoritative Replacement Check** BL Commit rejects when a prior OB exists for the target FY and the payload lacks restatement gate metadata (typed FY confirmation, reason, timestamp). Prevents DE gate bypass via URL manipulation, concurrent submissions, or client-side tampering.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| BL receives import for an FY where any period is closed AND restatement gate metadata is absent | BL rejects identifying closed period(s); no entry written |
| BL receives import for an FY where any period is closed AND restatement gate metadata is present | Accepted; prior reports will be revised; metadata persisted to audit trail |
| Re-import: prior OB entry exists AND restatement gate metadata is absent | BL rejects (authoritative replacement check); previous entry intact |
| Re-import: reversal fails mid-process | Operation rolls back; previous entries intact; user notified |
| Re-import: new entry creation fails after successful reversal | Operation rolls back; previous entries restored; user notified |
| Opening balance entry already exists and user re-imports with fewer accounts | Previous entry fully reversed; new entry has only the new file's accounts — missing accounts not carried forward |
| Validate-only call: any check fails | BL returns structured error list + error report URL; no ledger write attempted |
| Commit: any check fails (including period status re-check) | BL returns structured error list; no ledger write performed |
| Validate succeeds, but a period closes between Validate and Commit and metadata does not cover it | Commit rejected; user returned to validation |
| No opening balance import exists for a fiscal year | Reports show $0.00 with visual indicator |
| Abnormal balance (e.g., asset with credit balance) | Accepted — system does not enforce normal balance direction |

## Data Model Summary

| Attribute | Value |
| --- | --- |
| Entry type | "Opening Balance" (dedicated, identifiable) |
| Effective date | Day before first day of fiscal year (e.g., June 30 for July 1 FY) |
| Structure | One entry, N lines (one per account per site) |
| Amount storage | Debit or credit per line; opposite side = zero |
| Editable | No — read-only, immutable |
| Reversible via UI | No — only via re-import (hard reversal) |
| Reversal date | Same date as original entries |
| Period constraints | Rejected if any period in target FY is closed AND restatement gate metadata is absent |
| Sub-account constraints | Rejected if any (GLAccount, SiteCode) lacks a configured sub-account |
| Replacement constraints | Rejected if a prior OB entry exists for target FY AND restatement gate metadata is absent |
| Atomicity | All-or-nothing — partial writes never permitted |
| Affects opening balances | Yes — sole source of opening balance values for permanent accounts |

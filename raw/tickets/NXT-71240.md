# NXT-71240 — FIN — Opening Balance Import: Business Logic

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions for permanent accounts before any period transactions are recorded.

_Updated 2026-05-07. Compressed for readability (\~30% trim, no semantic changes). Earlier 2026-05-07: added TL;DR section. Earlier 2026-05-06: added consolidated API Validations section reflecting the DE/BL contract established with Jonathan Smith — DE hands off the mapped file, BL runs all validations and returns pass/fail with totals or a structured error report._

## TL;DR

Expose two API ops to DE: **Validate** (run every check; write nothing) and **Commit** (run every check + write). _DE calls Validate at the Field Mapping validator gate; Commit at the user's final confirm._

**On every call, BL runs:**

* **Row structure:** required fields present; Debit/Credit are decimals; _exactly one_ of Debit/Credit positive; Description ≤ 50 chars.
* **GLAccount exists** in chart of accounts — **unknown = blocking** _(differs from Payroll)_.
* **GLAccount base type is Asset, Liability, or Equity** — Revenue/Expense rejected.
* **SiteCode exists.**
* Each `(GLAccount, SiteCode)` has a configured **sub-account**. Central Office is just another site.
* Each `(GLAccount, SiteCode)` appears **once only** — no duplicates.
* Sum of Debits = Sum of Credits, **exactly $0** tolerance _(differs from Payroll's $0.01)_.
* If any period in target FY is closed OR a prior OB exists → **restatement gate metadata** (FY confirmation, reason, timestamp) **must be in the payload**. Missing = reject. _This is the authoritative replacement check._

**Commit-only:** re-verify no period closed between Validate and Commit.

**Response on pass:** rows, Debits, Credits, **effective date** (= day before FY start; e.g., June 30 for July 1 FY), restatement classification.
**Response on fail:** structured error list (`row#, field, code, message`) + error report download URL. DE just surfaces it.

**On Commit success:** create one ledger entry, type **"Opening Balance"**, one line per row. Each line stores the row's Debit or Credit with **zero on the opposite side**. Persist restatement metadata on the entry's audit trail. Entry is **read-only** — no edit or reverse via UI.

**Re-import = hard reversal.** If prior OB exists for FY (and metadata present), reverse all prior lines — _reversal entries dated same date as originals_ — then write the new entry. Reversal + re-creation is **atomic** — any failure rolls everything back.

**Reports:**

* Trial Balance Opening Balance column = imported OB for the FY.
* Balance Sheet beginning balance = OB for Period 1 / fiscal YTD; OB + cumulative prior-period activity for later periods.
* **No OB for FY?** Display `$0.00` with a visual indicator — _never a text string_ (must not break sort/Excel).

**Abnormal balances are accepted** (e.g., asset with credit balance). We don't enforce normal balance direction by base type.

## Scope

BL layer for Opening Balance Import: API Validations (Validate + Commit), ledger entry creation, effective date, entry type behavior, amount handling, sub-account validation, hard reversal on re-import, restatement audit metadata persistence, downstream report behavior. DE layer (**NXT-71249**) handles upload, mapping UI, validator gate (calls BL Validate), preview, and the Restatement required confirmation gate. This story receives validated data with separate Debit/Credit columns, the selected FY, and restatement audit metadata when applicable.

## API Validations

BL exposes **Validate** (called by DE at the Field Mapping validator gate per **NXT-71249** §8.1) and **Commit** (called by DE on Confirm). Both run the same validation pass; Commit additionally writes. All checks below run on every call.

On any failure, the API returns a structured error list (row#, field, code, message) + error report download URL. DE surfaces. **No partial work.**

### 1) Row-level structural

1.1 **Required fields.** GLAccount, SiteCode, and Description are present on every row.

1.2 **Decimal numbers.** Debit and Credit are decimals on every row. (DE fail-fasts the first 10 rows per **NXT-71249** §7.2; BL re-checks every row.)

1.3 **Single-side rule.** Exactly one of Debit/Credit positive per row. Both populated, both blank/zero, or any negative = invalid.

1.4 **Description length.** Description ≤ 50 characters when provided.

### 2) Reference data

2.1 **GLAccount exists** in the chart of accounts. Unknown = **blocking** for OB (chart hygiene wins; OB is one-time setup). _(Differs from Payroll, which writes unknown accounts with a warning per_ **NXT-70777** _§5.)_

2.2 **GLAccount base type is Asset, Liability, or Equity.** Revenue and Expense rows are rejected with the message: _"Opening balances cannot be set for Revenue or Expense accounts."_

2.3 **SiteCode exists** in SchoolCafe Site Management.

2.4 **Sub-account configured** for each (GLAccount, SiteCode) per §4. Central Office is just another site.

2.5 **Uniqueness.** Each (GLAccount, SiteCode) once only. Duplicates rejected with the message in **NXT-71249** §7.8.

### 3) Aggregate

3.1 **Balanced.** Sum Debits = Sum Credits. Tolerance **exactly $0** — OB is a clean carryover. _(Differs from Payroll's $0.01 tolerance.)_

### 4) Restatement gate enforcement

4.1 If any period in the target FY is closed, restatement gate metadata (typed FY confirmation, reason, timestamp) must be in the payload. Missing = reject. Present = accept (user has acknowledged).

4.2 If a prior OB entry exists for the target FY, the metadata must be present. Missing = reject. **Authoritative replacement check** per §6.6 — prevents gate bypass via URL manipulation, concurrent submissions, or client-side tampering.

4.3 On Commit, the metadata is persisted to the ledger entry's audit trail per §1.1.

### 5) Period status re-check (Commit only, defense-in-depth)

5.1 At Commit, BL re-verifies no period in the target FY closed since Validate. If a period closed and gate metadata doesn't cover it, Commit is rejected; user returns to validation.

### Response contract

**Success** — all checks pass:

* Status: `pass`
* Total rows
* Total Debits
* Total Credits
* Effective date (day before FY start)
* Restatement classification: `Eligible` / `Restatement: closed periods` / `Restatement: prior import` / `Restatement: both`

**Failure** — any check fails:

* Status: `fail`
* Error list — row#, field, code, message
* Error report download URL (generated by BL; DE surfaces the link per **NXT-71249** §8.1)

## Requirements

### 1) Ledger Entry Creation

1.1 On Commit (after all API Validations pass), create **one ledger entry**:

* **Entry type:** "Opening Balance"
* **Effective date:** Day before fiscal year start (see §2)
* **Description:** Each row's Description (defaulted to "Opening Balance Import" if blank)
* **Restatement audit metadata** (when triggered through the DE Restatement gate per **NXT-71249** §3.4): reason text, confirmation timestamp, trigger classification (closed periods, prior import, or both). For Eligible imports, these fields are null. Metadata appears in audit history and is queryable.

1.2 Each file row becomes a **line** in the entry — one account, one side per line.

1.3 Each GLAccount + SiteCode combination appears only once (enforced by API Validations §2.5).

### 2) Effective Date

2.1 Effective date is the **day before the first day of the specified FY** (e.g., July 1 FY → June 30).

2.2 BL computes the effective date and returns it in Validate so DE can display on Review.

2.3 Closed-period rule is enforced by API Validations §4. The Restatement gate (**NXT-71249** §3.4) is the user-facing acknowledgment; this story is the authoritative backstop and persists the metadata.

### 3) Amount Handling

3.1 DE enforces a fixed Debit/Credit format (separate columns; signed-amount transformed upstream per **NXT-71249** §4). Each validated row carries a positive value in exactly one column.

3.2 Debit/Credit values map directly to the ledger line; opposite side stored as zero.

3.3 **Normal balance direction is not enforced.** Assets may carry credit balances; liabilities may carry debit balances (abnormal balances are legitimate). Correctness is enforced by the batch balance check (API Validations §3).

### 4) Sub-account Validation

4.1 Sub-account validation runs in API Validations §2.4 (authoritative rule). The sub-account routes site-level activity to a GL account.

4.2 Any line lacking a configured sub-account for its (GLAccount, SiteCode) → **entire import rejected** at the validator gate (DE never reaches Commit). Per §6 atomicity, no partial writes.

4.3 Central Office is just another site — same sub-account requirement as any other SiteCode. No special handling.

4.4 On rejection, the result surfaces offending rows (GLAccount, SiteCode, clear no-sub-account message). User fixes the file or configures the missing sub-account in Configuration > General Ledger, then re-uploads.

4.5 Validation runs in BL Validate (and re-runs at Commit for defense-in-depth). DE doesn't pre-validate sub-accounts — only that the SiteCode exists. Sub-account routing is a BL concern.

### 5) Entry Behavior

5.1 OB entries are **read-only** — no edit or reverse via UI.

5.2 Corrections after import = manual adjustment entries. The OB entry is immutable.

5.3 The "Opening Balance" entry type must be identifiable in the ledger so reports, calculations, and replacement/reversal logic can distinguish it from period activity.

### 6) Hard Reversal on Re-Import

6.1 If a prior OB exists for the selected FY (detected and acknowledged via the DE Restatement gate per **NXT-71249** §3.4), BL **reverses all prior entry lines** before writing the new entry.

6.2 **Hard reversal**, not soft delete. Reversal entries are written to the ledger for a complete audit trail.

6.3 Reversal entries are dated **same as the originals** (day before FY start) — original and reversal net to zero at the same ledger point.

6.4 After reversal, the new entry and lines are written per §1–§4, including restatement audit metadata (§1.1) and sub-account validation (§4).

6.5 Reversal + re-creation is **atomic**. Any failure → no data changed, previous import intact, user notified.

6.6 BL rejects any re-import for a FY with a prior OB entry if restatement audit metadata is missing — prevents gate bypass via URL manipulation, concurrent submissions, or client-side tampering. **Authoritative replacement check.** _(Enforced by API Validations §4.2.)_

### 7) Downstream Report Behavior

7.1 Imported OBs establish the **FY starting position** for permanent accounts (as of day 1 of the FY).

7.2 **Trial Balance:** Imported OBs populate the Opening Balance column for the FY. Column always reflects the FY starting position regardless of period viewed.

7.3 **Balance Sheet:** OB feeds the beginning balance calculation. Relationship depends on reporting period:

* Period begins on day 1 of FY (Period 1 or fiscal YTD) → beginning balance = imported OB.
* Period begins after Period 1 (e.g., October in a July FY) → beginning balance = imported OB + cumulative prior-period activity within the FY.

7.4 OBs are clearly labeled in all reports and visually distinguishable from period activity.

7.5 If no OB import exists for an FY, the Opening Balance column shows `$0.00` + a visual indicator (icon, highlight, or footnote) that no OB has been imported. Indicator must not interfere with sort or Excel export — underlying value is always zero, never a text string.

### 8) Audit History

8.1 Audit history surfaces (when present) restatement reason, confirmation timestamp, and trigger classification from §1.1 — restatement rationale is queryable from the entry's audit trail without a separate query against import history.

8.2 Audit history distinguishes:

* **Eligible** — no gate, no metadata
* **Restatement: closed periods** — gate triggered, reason persisted (prior import may or may not exist)
* **Restatement: prior import only** — gate triggered, reason persisted, no closed periods
* **Restatement: both**

## Acceptance Criteria
**AC1 — Ledger Entry Created with Correct Entry Type** On Commit, BL creates one ledger entry with entry type "Opening Balance", one line per imported row.

**AC2 — Effective Date Is Day Before Fiscal Year Start** For an FY starting July 1, the entry's effective date is June 30.

**AC3 — Debit/Credit Amounts Mapped Directly** Validated data carries separate Debit/Credit columns (signed-amount layouts transformed upstream by DE per NXT-71249 §4). Each ledger line carries the row's debit or credit; opposite side = zero.

**AC4 — Entry Is Read-Only** Edit or reverse via UI is blocked. The entry can only be replaced by a new OB import.

**AC5 — Hard Reversal on Re-Import** On a re-import for an FY with a prior OB, all prior entry lines are hard-reversed (reversal entries dated same as originals); new entry lines written from the new import.

**AC6 — Reversal + Re-Creation Is Atomic** Any system error during reversal or re-creation → no data changed; previous OB entries remain intact; user notified.

**AC7 — Closed-Period FY Rejected Without Restatement Metadata** If any period in the target FY is closed and the payload lacks restatement gate metadata, BL rejects with an error naming the affected period(s) and required action. (Defense-in-depth — DE enforces upstream.)

**AC8 — Trial Balance Reflects Imported Opening Balance** After an OB import for an FY, Trial Balance for any period in that FY shows the imported FY starting position in the Opening Balance column.

**AC9 — Balance Sheet Beginning Balance Computed Correctly for Period 1** After an OB import, Balance Sheet for Period 1 (or fiscal YTD) shows beginning balance = imported OB.

**AC10 — Balance Sheet Beginning Balance Computed Correctly for Later Periods** After an OB import (with prior-period activity), Balance Sheet for a period after Period 1 shows beginning balance = imported OB + cumulative prior-period activity within the FY.

**AC11 — Missing Import Surfaced in Reports** If no OB exists for an FY, Trial Balance / Balance Sheet shows $0.00 + a visual indicator that no OB has been imported. Column must not contain a text string.

**AC12 — Normal Balance Not Enforced by Base Type** An asset row with a credit balance (abnormal) is processed as-is. System does not reject or modify direction by account type.

**AC13 — API Validate Operation Returns Pass or Fail** BL Validate returns either pass (rows, Debits, Credits, computed effective date, restatement classification) or fail (structured error list — row#, field, code, message — + error report download URL). No partial work.

**AC14 — Row-Level Structural Validation** BL returns fail with affected rows for: missing required fields (GLAccount, SiteCode, Description), non-decimal Debit/Credit, both populated/both blank/both zero/negative, or Description > 50 chars.

**AC15 — GLAccount Existence and Type Restriction** BL returns fail for unknown GLAccount or Revenue/Expense base type. Unknown is blocking for OB (chart hygiene wins; OB is one-time). Revenue/Expense message: *"Opening balances cannot be set for Revenue or Expense accounts."*

**AC16 — Sub-Account Required for Every (GLAccount, SiteCode)** BL returns fail with offending rows when any (GLAccount, SiteCode) lacks a configured sub-account. Central Office is just another site.

**AC17 — Uniqueness of (GLAccount, SiteCode)** BL returns fail with duplicate combinations listed when two+ rows share the same (GLAccount, SiteCode).

**AC18 — Balance Check Exact ($0 Tolerance)** BL returns fail when sum Debits ≠ sum Credits. Tolerance exactly $0 — no rounding allowance.

**AC19 — Authoritative Replacement Check** BL Commit rejects when a prior OB exists for the target FY and the payload lacks restatement gate metadata (typed FY confirmation, reason, timestamp). Prevents DE gate bypass via URL manipulation, concurrent submissions, or client-side tampering.

## Edge / Empty / Error States

| Scenario | Behavior |
| --- | --- |
| BL receives import for an FY where any period is closed AND restatement gate metadata is absent | BL rejects identifying closed period(s); no entry written |
| BL receives import for an FY where any period is closed AND restatement gate metadata is present | Accepted; prior reports will be revised; metadata persisted to audit trail |
| Re-import: prior OB entry exists AND restatement gate metadata is absent | BL rejects (authoritative replacement check); previous entry intact |
| Re-import: reversal fails mid-process | Operation rolls back; previous entries intact; user notified |
| Re-import: new entry creation fails after successful reversal | Operation rolls back; previous entries restored; user notified |
| Opening balance entry already exists and user re-imports with fewer accounts | Previous entry fully reversed; new entry has only the new file's accounts — missing accounts not carried forward |
| Validate-only call: any check fails | BL returns structured error list + error report URL; no ledger write attempted |
| Commit: any check fails (including period status re-check) | BL returns structured error list; no ledger write performed |
| Validate succeeds, but a period closes between Validate and Commit and metadata does not cover it | Commit rejected; user returned to validation |
| No opening balance import exists for a fiscal year | Reports show $0.00 with visual indicator |
| Abnormal balance (e.g., asset with credit balance) | Accepted — system does not enforce normal balance direction |

# NXT-71240 — FIN — Opening Balance Import: Business Logic

## Description
As a district finance user, I want to import opening account balances into Financials at the start of a fiscal year or when going live on the system, so that my balance sheet and trial balance reflect accurate starting positions before any period transactions are recorded.

---

## TL;DR

The wizard (owned by DE in **NXT-71249**) collects the fiscal year, file, and field mapping. On Save, DE hands the mapped file and selected fiscal year to BL's Commit operation. BL runs every business rule in one pass — if anything fails, the entire file is rejected atomically. Results (success or row-level exceptions) surface in Import Monitor's Exceptions tab.

**Step 0 eligibility (when the user selects a fiscal year):**

* If a prior FY is configured and any of its periods are still open, block with a deep link to Period Management. If no prior FY is configured (e.g., district going live), this check passes automatically.
* If the Chart of Accounts has no Asset, Liability, or Equity accounts, block.
* If a prior opening balance import exists for this FY, prompt the user to choose overwrite or abandon.
* If either a prior import exists or any period in the target FY is closed (or both), the Restatement Gate is triggered — the user must satisfy it before proceeding (gate UX per **NXT-71249** AC4).
* Server-side lock per District + Fiscal Year — first job wins; concurrent attempts are blocked.

**On Commit, BL checks every row against these rules:**

* Required fields present; Debit/Credit are positive decimals; exactly one of Debit/Credit populated and non-zero per row; max 2 decimal places (no rounding or truncation).
* GLAccount exists in the Chart of Accounts, is of type Asset/Liability/Equity, is active, is in a valid format, and AccountName must match the CoA.
* SiteCode exists in Site Management. Case-insensitive, leading/trailing spaces trimmed. Site type and active status are irrelevant. Blank SiteCode blocks the row (no defaulting).
* The (GLAccount, SiteCode) combination has a configured sub-account.
* Each (GLAccount, SiteCode) pair appears only once. Duplicates block (no dropping, no summing).
* Total Debits = Total Credits with exactly $0 tolerance. At least one non-zero amount in the file.

All exceptions are blocking — if any check fails, the entire file is rejected and nothing is written to the ledger.

**On success:** BL creates one ledger entry of type "Opening Balance," one line per row. Each line stores the row's Debit or Credit with zero on the opposite side. The entry is read-only — no edit or reverse through the UI. The effective date is the day before the fiscal year starts (e.g., June 30 for a July 1 FY).

**On overwrite:** BL hard-reverses the entire prior opening balance batch (reversal entries dated same as the originals), then posts the replacement batch. The entry description on the replacement serves as the restatement reason. Both steps are atomic — any failure rolls everything back and the prior import stays intact. Three distinct Posting Activity Log labels: `Opening Balance Import`, `Opening Balance Import Reversal`, `Opening Balance Import Replacement`.

## Scope

**This story (BL):** Step 0 eligibility checks, all business rules at Commit, ledger entry creation, effective date computation, entry type behavior, amount handling, hard reversal on overwrite, Restatement Gate enforcement, Posting Activity Log labels, downstream report behavior.

**Companion story — NXT-71249 (DE):** The wizard (Pre-upload → File Upload → Field Mapping → Review → Save), file acceptance and format validation, field mapping UI, **the downloadable Debit/Credit template (NXT-71249 §4.3) that users conform their file to before upload — no transformation logic in DE**, file-level staging checks, the Restatement Gate UI (per **NXT-71249** AC4), Review tab display, and calling BL's Commit on Save. This story receives the mapped file with separate Debit/Credit columns and the selected FY.

**User-facing error microcopy:** owned by **NXT-71249** (the DE story surfaces strings to the user). This story identifies which business rule failed and returns a structured exception; the user-facing message lives in the corresponding NXT-71249 AC. Rules with no NXT-71249 equivalent keep their microcopy here.

**Out of scope:** Permissions — only users with the highest level of access can run OB Import; that same access level includes overwrite. No separate Import / Overwrite permissions. No row-level site permission checks because site-scoped users cannot access OB Import.

## Exception Format

When a check fails, BL returns:

* **Column Name** — the field or pair involved (e.g., `GLAccount`, `SiteCode`, `Debit/Credit`, `GLAccount/SiteCode`, `Fiscal Year`, `AccountName`)
* **Description** — human-readable text. If a specific row matters, include `Source row {RowNumber}: ...`

All exceptions are blocking. If any check fails, the entire file is rejected and no journal entries are written.

Exceptions surface in **Import Monitor → Exceptions tab**.

---

## Step 0 — Eligibility (when the user selects a fiscal year)

### 0.1 Prior FY periods still open

If a prior fiscal year is configured and any of its periods are still open, the user cannot proceed. The system shows a message with a deep link to Period Management so they can close the open periods first. No Import Monitor exception — the import has not started.

If no prior fiscal year is configured (e.g., a district going live on the system for the first time), this check passes automatically — there are no prior periods to evaluate.

### 0.2 Prior opening balance import exists for this FY

If BL detects that an opening balance has already been imported for the selected FY, the user is prompted to choose: overwrite the prior import or abandon. If the user chooses to abandon, the import stops.

If the user chooses to overwrite, this triggers the **Restatement Gate** (see §0.6).

### 0.3 Chart of Accounts missing required account types

If the Chart of Accounts has zero Asset, Liability, or Equity accounts, the user cannot proceed:

```
Configure at least one Asset, Liability, or Equity account in Chart of Accounts before importing Opening Balances.
```

No Import Monitor exception. The import has not started.

### 0.4 Past and future fiscal years

Both past and future fiscal years are valid selections. A past FY is allowed (§0.1 still enforces prior-year period closure). A future FY start date is allowed (normal setup before a FY begins).

### 0.5 Concurrent submissions for the same District + Fiscal Year

If another opening balance import is already in progress for the same District and Fiscal Year, the second attempt is blocked. First job wins.

```
An Opening Balance import is already in progress for this fiscal year. Wait for it to finish, then refresh Import Monitor before trying again.
```

No row-level exception. This is a job-level block.

### 0.6 Restatement Gate

The Restatement Gate is triggered if **either** of these conditions is true (or both):

1. A prior opening balance import already exists for the selected FY.
2. Any period in the target FY is closed.

Gate UX is defined in **NXT-71249** AC4. BL's responsibilities:

* Detect the trigger conditions during eligibility and tell DE the gate is required.
* On Commit, require the confirmation acknowledgment returned by the gate. If absent → reject (see §7.1).
* Use the entry description (always required — see §1.1 under Ledger Entry Creation) as the restatement reason for overwrites.
* Preserve the confirmation timestamp on the entry's audit trail.

## Business Rules (run on every Commit)

BL runs every check below in a single pass. If any check fails, the entire file is rejected — nothing is written.

### 1) Row-level structure

1.1 **Required fields.** Every row must have GLAccount, SiteCode, and Debit/Credit. If any required field is missing → blocked (`{ColumnName} is required`).

1.2 **Numeric.** Debit and Credit values must be numeric on every row. If non-numeric → blocked. User-facing message per **NXT-71249** AC6.

1.3 **Positive only.** Debit and Credit must be positive. If negative → blocked. User-facing message per **NXT-71249** AC7.

1.4 **Single-side rule.** Exactly one of Debit or Credit must be populated and non-zero per row. If both are populated with non-zero values, or both are empty/zero → blocked. User-facing message per **NXT-71249** AC7.

1.5 **Decimal precision.** Amounts must have 2 or fewer decimal places. Do not round or truncate. If more than 2 decimal places → blocked.

### 2) GL Account

2.1 **Must exist in the Chart of Accounts.** If unknown → blocked. User-facing message per **NXT-71249** AC9.

2.2 **Must be Asset, Liability, or Equity.** If Revenue or Expense → blocked. User-facing message per **NXT-71249** AC8.

2.3 **Must be active.** If inactive → blocked.

2.4 **Must be in a valid format.** If malformed → blocked (`GLAccount {GLAccount} is not in a valid account format`).

2.5 **AccountName must match.** If the imported AccountName does not match the CoA's AccountName for the same GLAccount → blocked. The user must correct the file or update the Chart of Accounts before importing. User-facing message per **NXT-71249** AC22.

### 3) Site

3.1 **Must exist in Site Management.** If unknown → blocked. User-facing message per **NXT-71249** AC9.

3.2 **Active status does not matter.** Inactive sites are valid posting targets for opening balances. No exception.

3.3 **Site type does not matter.** The system must not inspect or block based on site type (meal-program, accounting, Central Office, or any other). No exception.

3.4 **Case-insensitive matching.** SiteCode matching ignores case and trims leading/trailing spaces. The posted entry uses the canonical SiteCode from Site Management. No exception.

3.5 **Blank SiteCode blocks the row.** No defaulting — including no default to Central Office. → blocked.

3.6 **Central Office not in district.** Treated as unknown SiteCode per row (per §3.1). The system does not block at Step 0 based on an assumed Central Office code and does not default unknown or blank values to Central Office.

### 4) Sub-account

4.1 **Configured sub-account required.** Every row's (GLAccount, SiteCode) combination must have a configured sub-account in the system. If even one row fails this check → blocked. The exception identifies the offending rows so the user knows which sub-accounts to configure.

### 5) Uniqueness

5.1 **(GLAccount, SiteCode) must be unique within the file.** If duplicates exist (same pair, any amounts — including literal duplicate rows that keep the file balanced) → blocked. Do not drop either row, do not keep first/last, do not sum. User-facing message per **NXT-71249** AC10.

5.2 **Same GLAccount across multiple sites is allowed.** Uniqueness is on the pair, not on GLAccount alone.

### 6) File-level integrity

6.1 **Balanced.** Total Debits must equal Total Credits. Tolerance is exactly $0. If out of balance → blocked. User-facing message per **NXT-71249** AC11.

6.2 **Non-zero file.** At least one Debit or Credit must be greater than zero. If all amounts are zero → blocked. User-facing message per **NXT-71249** AC26.

6.3 **Monetary precision.** Amounts must be processed using decimal-safe logic. Floating-point precision loss is not acceptable for ledger data.

### 7) Restatement enforcement

7.1 If a prior opening balance exists for the selected FY, or any period in the target FY is closed, BL checks that the commit includes the confirmation acknowledgment from the Restatement Gate (per **NXT-71249** AC4). If missing → blocked.

## On Success — Ledger Entry Creation

### 1) Entry structure

1.1 On a successful Commit (all business rules pass), BL creates **one ledger entry**:

* Entry type: `Opening Balance`
* Effective date: day before the fiscal year starts (see §2 below)
* Description: required. Entered by the user in the wizard (one description for the entire entry, not from the file). **Minimum 10 characters, maximum 50 characters.** Special characters and formula-like content are sanitized. For overwrites, this description serves as the restatement reason.

1.2 Each file row becomes one **line** in the entry — one account, one site, one side per line.

1.3 Each (GLAccount, SiteCode) appears only once (enforced by §5.1).

### 2) Effective date

2.1 The effective date is the **day before the first day of the selected fiscal year** (e.g., if the FY starts July 1, the effective date is June 30).

2.2 BL computes the effective date and returns it so DE can display it on the Review tab.

### 3) Amount handling

3.1 DE delivers separate Debit/Credit columns per the canonical template (**NXT-71249** §4.3). Users conform their file to the template before upload; **there is no signed-amount transformation in DE**. Each validated row carries a positive value in exactly one column.

3.2 Debit/Credit values map directly to the ledger line; the opposite side is stored as zero.

3.3 **Normal balance direction is not enforced.** Assets may carry credit balances; liabilities may carry debit balances. Correctness is enforced by the batch balance check (§6.1), not by the account's base type.

### 4) Entry behavior

4.1 Opening Balance entries are **read-only** — no edit or reverse through the UI.

4.2 If a correction is needed after import, the user creates manual adjustment entries. The OB entry itself is immutable.

4.3 The `Opening Balance` entry type must be identifiable in the ledger so reports, calculations, and overwrite/reversal logic can distinguish it from period activity.

4.4 The only way to replace an opening balance is to import a new one for the same fiscal year, which triggers the Restatement Gate and hard reversal.

---

## Overwrite — Hard Reversal by Batch

### When overwrite applies

If a prior opening balance import exists for the selected FY and the user passes the Restatement Gate (per **NXT-71249** AC4), the system overwrites by reversing the entire prior batch and posting the replacement.

### How it works

5.1 Opening Balance imports are treated as **batches**.

5.2 The system **reverses every entry from the prior opening balance batch**, then posts the new file as a new batch.

5.3 This is a **hard reversal** — reversal entries are written to the ledger for a complete audit trail.

5.4 Reversal entries are dated **the same day as the originals** (day before FY start). Original and reversal net to zero at the same ledger point.

5.5 Reversal + new batch posting is **atomic**. If anything fails, no data changes — the previous import stays intact and the user is notified.

5.6 The system does not compare old rows to new rows, does not selectively preserve entries from the prior file, and does not merge the new file into the old batch. Accounts present in the prior batch but absent from the new file are simply not reposted.

5.7 Later transactions (posted after the original OB) remain as-is. Reports show the latest ledger state.

5.8 The new file is validated against the **current** Chart of Accounts. An account that existed in the prior file but no longer exists in the CoA will fail per §2.1.

5.9 The confirmation timestamp from the gate is preserved on the entry's audit trail. The entry description serves as the restatement reason.

---

## Posting Activity Log

Three distinct labels for audit readability:

* `Opening Balance Import` — original batch
* `Opening Balance Import Reversal` — reversal of a prior batch on overwrite
* `Opening Balance Import Replacement` — new batch posted after overwrite

Audit details per entry: Fiscal Year, Transaction ID, Replacement Transaction ID (if applicable).

The Posting Activity Log shows the original, reversal, and replacement entries as separate auditable activity.

## Downstream Report Behavior

7.1 Imported opening balances establish the **fiscal year starting position** (as of day 1 of the FY).

7.2 **Trial Balance:** The Opening Balance column for any period in the fiscal year reflects the imported opening balance for that FY. It always reflects the FY starting position regardless of which period is viewed.

7.3 **Balance Sheet:**

* If the period begins on day 1 of the FY (Period 1 or fiscal YTD) → beginning balance = imported OB.
* If the period begins after Period 1 (e.g., October in a July FY) → beginning balance = imported OB + cumulative prior-period activity within the FY.

7.4 Opening balances are clearly labeled in all reports and visually distinguishable from period activity.

7.5 If no opening balance import exists for an FY, the Opening Balance column shows `$0.00` with a visual indicator (icon, highlight, or footnote) that no OB has been imported. The indicator must not interfere with sort or Excel export — the underlying value is always zero, never a text string.

## Acceptance Criteria
**AC1 — Ledger Entry Created with Correct Entry Type** On Commit, BL creates one ledger entry with entry type "Opening Balance", one line per imported row.
**AC2 — Effective Date Is Day Before Fiscal Year Start** For an FY starting July 1, the entry's effective date is June 30.
**AC3 — Description Required** The user enters a single description for the entire entry in the wizard (not from the file). Required, minimum 10 characters, maximum 50 characters, sanitized. For overwrites, the description serves as the restatement reason.
**AC4 — Debit/Credit Amounts Mapped Directly** Each ledger line carries the row's Debit or Credit; opposite side = zero.
**AC5 — Entry Is Read-Only** Edit or reverse via UI is blocked. The entry can only be replaced by a new OB import.
**AC6 — Hard Reversal on Overwrite** When the user overwrites a prior OB for the same FY (after passing the Restatement Gate), all prior entry lines are hard-reversed (reversal entries dated same as originals); new entry lines written from the new file.
**AC7 — Reversal + Replacement Is Atomic** Any failure during reversal or replacement → no data changed; previous OB entries remain intact; user notified.
**AC8 — Restatement Gate Enforcement** If a prior OB exists for the target FY or any period in the target FY is closed, BL checks that the commit includes the confirmation acknowledgment from the Restatement Gate (per NXT-71249 AC4). If missing → blocked. The entry description serves as the restatement reason.
**AC9 — All Exceptions Are Blocking** If any business rule check fails, the entire file is rejected. No partial writes.
**AC10 — Row-Level Validation** BL blocks the file if any row has: missing GLAccount, SiteCode, or Debit/Credit; non-numeric amounts; negative amounts; both Debit and Credit populated or both empty/zero; more than 2 decimal places.
**AC11 — GLAccount Must Exist, Be Active, Be Asset/Liability/Equity, Be Valid Format** Unknown, inactive, Revenue/Expense, or malformed GLAccount → blocked.
**AC12 — AccountName Must Match CoA** If the imported AccountName does not match the CoA's AccountName for the same GLAccount → blocked.
**AC13 — SiteCode Must Exist in Site Management** Unknown SiteCode → blocked. Blank SiteCode → blocked (no defaulting). Inactive sites and all site types are accepted.
**AC14 — Sub-Account Required** Every (GLAccount, SiteCode) must have a configured sub-account. If any row fails → blocked, with offending rows identified.
**AC15 — Uniqueness of (GLAccount, SiteCode)** Duplicate pairs → blocked. Do not drop, keep first/last, or sum.
**AC16 — Balance Check Exact ($0 Tolerance)** If total Debits ≠ total Credits → blocked. At least one non-zero amount required.
**AC17 — Normal Balance Direction Not Enforced** An asset with a credit balance (abnormal) is accepted. System does not reject or modify direction by account type.
**AC18 — Trial Balance Reflects Imported Opening Balance** After an OB import, the Opening Balance column for any period in that FY shows the imported FY starting position.
**AC19 — Balance Sheet Beginning Balance Correct for Period 1** Beginning balance = imported OB.
**AC20 — Balance Sheet Beginning Balance Correct for Later Periods** Beginning balance = imported OB + cumulative prior-period activity within the FY.
**AC21 — Missing Import Surfaced in Reports** If no OB exists for an FY, reports show $0.00 with a visual indicator. No text strings — underlying value is always zero.
**AC22 — Posting Activity Log Labels** Three distinct labels: Opening Balance Import, Opening Balance Import Reversal, Opening Balance Import Replacement. Audit details per entry: Fiscal Year, Transaction ID, Replacement Transaction ID (if applicable).

## Edge / Empty / Error States
| Scenario | Behavior |
| --- | --- |
| Prior FY configured with open periods | User cannot proceed; deep link to Period Management |
| No prior FY configured (go-live) | Prior-period check passes automatically |
| CoA has no Asset/Liability/Equity accounts | User cannot proceed |
| Prior OB exists, user chooses overwrite, confirmation acknowledgment provided | Overwrite proceeds: hard reversal + replacement, atomic |
| Prior OB exists, user chooses overwrite, confirmation acknowledgment missing | BL rejects the commit |
| Any period in target FY is closed, confirmation acknowledgment missing | BL rejects the commit |
| Any period in target FY is closed, confirmation acknowledgment provided | Accepted; description serves as restatement reason |
| Concurrent import for same District + FY | Second attempt blocked; first job wins |
| Reversal fails mid-process | Operation rolls back; previous entries intact; user notified |
| New entry creation fails after successful reversal | Operation rolls back; previous entries restored; user notified |
| Re-import with fewer accounts than prior file | Previous entry fully reversed; new entry has only the new file's accounts |
| Commit: any check fails | Entire file rejected; no ledger write; exceptions surface in Import Monitor |
| No OB imported for a fiscal year | Reports show $0.00 with visual indicator |
| Abnormal balance (e.g., asset with credit balance) | Accepted — system does not enforce normal balance direction |

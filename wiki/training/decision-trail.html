Perseus / NXT · Decision Trail

Decisions embedded in shipped work.

Living scope discipline, not a graveyard. The corpus contains 721 tickets where the description records "we did X instead of Y" in the middle of otherwise-shipped work — five clusters worth surfacing, drawn from the 8,847-ticket archive.

What this page is. Most teams produce a decision log only when something blows up. The rest of the time, the choices live inside the description fields of normal tickets — the option that wasn't taken, the scope that was deferred, the default that was overridden — and nobody ever pulls them back out. The five clusters below were detected automatically by phrase-scanning all 8,847 ticket descriptions for "instead of," "out of scope," "will not," and similar markers. Each card is paraphrased from real ticket text and tagged for SME review. Treat as observation, not record. wiki/decisions/ stays empty by design — when an SME confirms one of these, that's where the curated version goes.
Item Management marked for review

Ship narrow first, defer the cross-module variants

The corpus suggests a recurring scope-discipline pattern in Item Management: when a feature could span the catalog and the menu, the team ships the catalog-only version and defers the menu-aware modes. The clearest example is the standard allergen chart in NXT-13229 — the original spec proposed three report scopes (all active menu items, items on active menus, items within a time frame); the latter two were explicitly struck through with "out of scope for now" before shipping, leaving only the all-items mode. The note in the ticket cites a dependency on Menu Planning — meaning the cross-module versions would have required coordinated work that wasn't budgeted. The same restraint shows up in NXT-17759, the Catalog Integration landing page, where the team deliberately chose explicit-search-on-click over auto-loaded results — keeping the global-catalog query under user control rather than letting the page hammer the integration on every visit. Pattern: when in doubt, build the controlled subset.

Inventory marked for review

View-only ships first; editable comes as a separate story

The Receipts surface in Inventory shipped read-only before it shipped editable, and the corpus shows this was deliberate phasing rather than a delayed feature. NXT-16188 introduces a "View Receipt" button that opens the Receipt Details screen in view-only state, with explicit AC saying "fields will not be editable by default; user has to click Edit to make changes." The Edit story is a separate ticket entirely — NXT-15379 — gated on a distinct "Edit Receipt" permission. The view-only-first approach extended to other components on the same screen: NXT-16335 added a view-only Comments popup specifically for non-editable contexts, with no save or cancel buttons, just a Close. The corpus suggests a deliberate stance: read paths and write paths are independent stories with independent permissions, and the read path always ships first.

Platform — System marked for review

Permissions reorganized around categories, not individual resources

Two tickets together suggest an architectural choice in how Roles & Permissions evolved. NXT-15171 redesigned the Roles & Permissions screen to display a consolidated single-pane view of role-to-permission mappings, explicitly instead of the multi-level hierarchical navigation it replaced. NXT-20158 then introduced a Resource Type called "Functions" that lets resources be granted by category and module — meaning a permission no longer maps to a single individual resource but to a group of them defined by function and category. The corpus suggests the team prioritized administrative ergonomics — granting access to a category at once — over per-resource auditability. The trade-off: easier to give access broadly, harder to reason about exactly which resources a given role can touch. An SME would know whether that trade-off has held up in practice.

Platform — Data Exchange marked for review

Build the configuration first; make the configuration legible later

The Data Exchange module shipped its API-endpoint configuration page well before it shipped UI clarity for what was being configured. NXT-15142 established the scaffolding: an API Configuration page where district admins plug in endpoint URLs, aliases, user IDs, passwords, certificates, and tie each connection to a vendor (the ticket names third-party SIS systems by example). The story shipped with raw fields. Months later, NXT-16375 noted that the resulting Code Mapping screen was showing internal codes like "F" / "M" / "N" without any context, and added human-readable descriptions next to each — "F (Female), M (Male), N (Not Provided)." The corpus suggests a recurring shape: in this module, the connection-and-mapping infrastructure tends to ship before the connection-and-mapping UX, with the latter following as a separate story once enough districts have hit the rough edges.

Global Catalog marked for review

Pull what moves the kitchen; defer what moves the procurement office

External-source integration in Global Catalog has a recurring scope shape — the team consistently pulls in the data that the kitchen uses and defers the data that the procurement or finance side would. NXT-39283 is the GDSN data-processing API; the AC explicitly scopes coverage to ingredients and stock items, with vendor details and pricing updates marked "out of scope" and routed to a separate story. NXT-45142 makes a different kind of choice — USDA Branded Foods data arrives normalized to per-100-gram nutrition values, but the team chose to override that default and display values per actual serving size (e.g., per 30 g), with a scaling formula calculated at render time. Two clusters of decision-shape work: which fields to pull from the integration at all, and which units to display when the upstream's defaults don't match the user. Both choices favor the kitchen-facing user over the upstream contract.

Prerequisites

Before beginning work in the Eligibility module each school year, one system setting must be reviewed and confirmed. Skipping this step can cause income surveys to be processed incorrectly before the season opens.

Auto-Process Income Surveys setting

A system setting governs whether online income surveys are automatically processed after submission. Confirm the value is appropriate for your district's workflow before applications go live.

Setting value What happens to submitted surveys
Yes All surveys are auto-processed regardless of student match status.
No No surveys are auto-processed; all remain in a pending state.
Only if all students identified (default) Only surveys where every listed student is matched are auto-processed.

The default value is set to the most conservative option. If your district changed this setting in a prior year, verify it is still correct before the new year begins.

Note: manually entered income surveys are not affected by this setting — manual entry behavior remains unchanged regardless of which option is selected.

Workflow: Adding and Submitting an Application

Application entry follows a fixed seven-section wizard. Complete each section in order before moving to the next; the wizard does not allow skipping ahead.

The Seven-Section Wizard

# Section Key actions
1StudentsIdentify the student(s) tied to this application
2BenefitsSelect the benefit type(s) being applied for
3Household MembersAdd all members; enter household size manually if auto-count is overridden
4Contact InfoVerify or enter contact details for the household
5DetailsProvide case number (required) and any supplemental information
6ReviewConfirm all entries before submission; last chance to edit
7SubmitEnter SSN and signature certification; set Received Date

Rules to Know Before You Start

  • Categorical status override: A confirmed categorical status can be manually overridden to unconfirmed (or vice versa). Use this only when documentation conflicts with the system-resolved status. [TO VERIFY — confirm exact field label and permission level required]
  • Manual household size: If the system count does not reflect the actual household, enter the size directly in the Household Members section. [TO VERIFY — confirm whether this triggers a supervisor flag]
  • Case number: Required. The wizard will not advance past the Details section without a valid case number.

Submit Section — What's Required

  • SSN — must be entered at submission; not collected in earlier sections.
  • Signature certification — staff certify that the household member signed the application.
  • Received Date — the date the physical or digital application was received. This field drives turnaround-time calculations, so accuracy is critical; backdating or leaving it blank affects compliance reporting. [TO VERIFY — confirm exact turnaround window tied to this field]

Workflow: Managing and Processing Applications

The Applications Page

The Applications page organises work across three tabs :

  • ALL — every application regardless of state
  • Pending Applications — submitted applications awaiting a decision
  • Pending Students — applicants whose student status requires resolution before processing can continue

Use the filter controls at the top of the page to narrow results before acting .

Each row exposes two icon actions — View (open the application record) and History (audit trail of prior activity) — plus a full action menu :

Workflow: Handling Pending Students

The Pending Students tab surfaces applications where a student record could not be automatically matched to a SIS account. Three matching scenarios are supported: system single-match, system multiple-match, and manual match. In every scenario, the outcome is the same — the pending student is linked to an actual account record.

Steps

  1. Find Matches — Run the automated search from the tab. Contextual instruction text is provided on the tab, in the View window, and as a success message to orient you at each stage.
  2. Review candidates — When multiple candidates are returned, a Match button appears for each one. The Compare Details view shows each candidate's status (Active or Inactive) and their eligibility along with the qualifying basis.
  3. Select one candidate — Exactly one candidate must be selected to proceed. A link is created at the point of selection. If you need to change your selection, you can remove the linked student and re-select.
  4. Process Matches — Click Process Matches to commit the selection. Application Details are not updated until this step is explicitly taken — searching or editing a match alone does not update the application. After processing, the student is linked to the application and their eligibility is updated accordingly.

Scenario reference

ScenarioWhat to watch for
System single-matchOne candidate returned; review and proceed to Process Matches.
System multiple-matchMatch button appears per candidate; select exactly one before processing.
Manual matchUse manual search or Edit to locate and link the record when auto-match finds nothing.

Workflow: Adding and Processing Income Surveys

Income surveys for Special Provision Program districts are submitted online and then processed by the system. The key system behavior to understand before adding surveys is how auto-processing is controlled — this determines whether a submitted survey moves to a processed status automatically or waits in a pending queue.

Auto-Processing Behavior

A system setting governs whether submitted online income surveys are automatically processed. The setting offers three values:

Setting Value What Happens to a Submitted Survey
Yes All online surveys are auto-processed (existing behavior preserved).
No No surveys are auto-processed; all remain in Pending status.
Only if all students identified Surveys auto-process only when every student on the survey is matched and identified.

The default value for new environments is Only if all students identified. This default prevents surveys with unmatched students from being silently auto-processed.

Manual Entry Is Unaffected

This setting applies to online survey submissions only. Manually entered income surveys follow their existing processing path regardless of the setting value.

Key Takeaway

  • Check the auto-process setting before a survey intake period begins — a value of No means all surveys will require manual processing.
  • If surveys are stacking in Pending status unexpectedly, the setting value is the first place to check.

Key Fields & Statuses

The Eligibility module surfaces specific fields across applications, student matching, and verification. Knowing each field's behavior prevents common processing errors.

Match Details: Status & Eligibility

The Match Details screen displays two informational fields for every candidate student account.

  • Status — indicates whether the student account is Active or Inactive.
  • Eligibility — displays the current determination (Free / Reduced / Paid) together with its basis. A student with categorical eligibility through DC Medicaid, for example, displays as Free (DC MEDICAID).

Application Details: When the Field Updates

Application Details do not update during search or manual matching. The fields commit only after Process Matches is clicked.

Surveys Stuck in Pending

If online income surveys remain in a Pending status unexpectedly, the auto-process system setting controls this behavior. Three values are available. The default processes surveys only when every student on the survey has been identified. Setting it to process none means surveys will always stay in Pending until manually actioned. Manual survey entry is not affected by this setting regardless of the value chosen.

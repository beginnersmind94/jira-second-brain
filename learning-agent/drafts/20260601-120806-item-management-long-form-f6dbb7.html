Item Management: Learning Guide

Overview

Item Management is the starting point for all records in SchoolCafé 2.0. Items must be configured here before they can appear in menus or be tracked through inventory. The classification choices made during item creation determine which fields are active throughout setup and which data populates downstream modules.

Two item types, one shared framework

At creation, a question sequence classifies every item before the main configuration grid becomes active. Classification controls which fields are enabled for that item:

Classification Serving sizes Measure column label
Food — non-fluid Enabled Weight
Food — fluid Enabled; volume units apply Weight / Volume
Non-food Disabled —

The Units card

All serving size and pack size configuration is consolidated into a single Units card in the item header. When pack sizes are added to an item, the item becomes inventory-ready.

Downstream impact

Menu items draw nutrition, contribution, and allergen data directly from the underlying item record. Because these fields originate in Item Management, an incomplete or misconfigured item record surfaces as a gap in every module that references it.

Roles & Permissions

Item Management is accessed by two distinct role types. Understanding which role owns which task prevents workflow errors and misdirected support requests.

Role Primary Responsibility Module Area
Menu Planner Add and manage menu items from the Menu Item Management page Menu Item Management
User (item configuration) Configure item units, pack sizes, and serving size data points Item Units / Item Creation

Key Distinctions

  • Menu Planners work within the Menu Item Management page and initiate item creation from the Add button on the landing page.
  • Users configuring items work at the units level — defining pack sizes, linking servings, and confirming inventory readiness before a save can proceed.
  • These two roles operate in separate parts of the module. A Menu Planner creating a menu item is not the same workflow as a configuration user building the underlying item record.

Note: Only local data source is used when creating menu items. Confirm with your administrator which role accounts are provisioned for your site before beginning item setup.

Prerequisites

Before you begin creating items, confirm that the following conditions and configuration elements are in place. Requirements differ depending on whether your deployment uses Inventory features.

Configuration Elements

Element POS-Only Deployment Inventory-Enabled Deployment
Item Categorizations Optional Required
Global Info (Brand, Manufacturer, Product Code) Optional Optional — but ideally available
At least one unit configured Required before saving Required before saving
Pack Size units Not required Triggers Inventory flag when added

Key Concepts to Understand First

  • Inventory Readiness — The system enforces a concept of Inventory Readiness as part of item status and item creation flow logic. You will encounter readiness checks when saving the Units card for the first time.
  • Minimum unit requirement — At least one serving size or pack size unit must be configured before you can proceed past the Units card.
  • Pack Size and Inventory linkage — Adding a pack size unit is what activates Inventory behavior on an item. If your site does not use Inventory, you do not need to configure pack sizes.

If you are unsure whether your deployment has Inventory enabled, check with your system administrator before creating items — the required fields and available workflows differ significantly between the two configurations.

Saving the Item Card Activates the Units Card

Once you save the Item card, the Units card — displayed in the item creation header — becomes editable. The Units card is where pack sizes and serving sizes are configured; it is intentionally locked until the Item card is complete.

Do not navigate away before saving: an unsaved Item card keeps the Units card disabled and no unit configuration can proceed.

Non-Food Items: Building a Pack Size Hierarchy

Pack sizes for non-food items are configured directly in the Units grid. The system provides the ability to add pack size units in the combined Units grid , with each new tier inserted via the + Pack button.

Adding Your First Pack

When you click + Pack, the row requires a Description. This field is a hybrid smart text field that recommends common measures such as Case or Bag. Sub-Qty is intentionally disabled for the first pack size being added.

Sub-Qty: When It Appears and Why

The system brings attention to the appropriate Sub-Qty fields as required to link pack sizes. The rule depends on where the new tier sits relative to what already exists:

ScenarioSub-Qty behavior
First pack added Sub-Qty not enabled
Second pack added below (child of) existing Sub-Qty required on the child row
Second pack added above (parent of) existing Sub-Qty required on the same row

Weight Field

Weight is optional for most pack tiers but is required for linking a pack size to a serving size. The Weight field is emphasized for rare scenarios where it is required.

Tier Limit and the Base Unit

Inventory readiness logic applies specifically when fewer than three pack sizes are configured. Once you proceed to save, the system prompts you to confirm whether the lowest-tier unit is the smallest inventory-able unit. Confirming Yes locks that unit as the Base Unit and proceeds. If you select No, the system emphasizes the + Pack button to prompt adding a lower tier.

Adding pack sizes triggers the Inventory flag for the item. At least one pack size or serving size unit must be configured before you can move on.

Costing, GTIN/UPC & Inventory Settings

Cost entry is a finish-step task — the cost fields are not available until pack size units have been configured. The costing check is one of two required prompts the system runs before the Units card can be saved for the first time.

Correct Setup Sequence

  1. Add all pack sizes first. Adding pack sizes triggers the inventory flag on the item.
  2. Link pack sizes via Sub-Qty. This establishes the hierarchy the system needs before costing is meaningful.
  3. Enter cost last. Select the specific pack size or inventory-eligible serving unit on which to enter the pricing.

Costing on Subsequent Edits

After initial save, costing checks continue to run on every save. However, they are no longer presented as step-by-step prompts — instead they surface through icons and banners.

⚠ Permanent Lock: Base Unit & Pack Size

Setting Editable before first inventory transaction? Editable after?
Base Unit Yes No
Pack Size configuration Yes No

Once an item appears in Inventory tables, pack size and Base Unit selections are locked. Add and verify every pack size before recording the first inventory transaction — there is no recovery path after that point.

Food Items: Serving Sizes, Fluid Selection & Linking to Pack Sizes

The Units card sits in the item creation header alongside the general info card and combines serving size and pack size configuration into one place. The card becomes editable only after the general card is saved.

Step 1 — Answer the header questions

Before the grid enables, the card presents two sequential questions.

  • Food Item? — If No, serving sizes are disabled entirely.
  • Fluid Item? — Shown only when Food Item is Yes. Selecting Yes switches the measurement column from Weight (grams) to Weight / Volume to accommodate fluid measures for servings while keeping weight for pack sizes.

Step 2 — Add serving sizes

After the questions are answered, the grid loads with Add Serving Size and Add Pack Size actions. Each serving size row contains three key fields:

FieldKey behaviour
Amount When set to 1, the serving becomes "potentially inventory-able" and is eligible for Sub-Qty linking or promotion to a base unit. Any other value makes it a serving-only unit.
Description A smart text field that matches against standard measures.
Weight / Volume Column label is Weight for non-fluid items and Weight / Volume for fluid items.

Step 3 — Resolve the Inventory Ready alert

If pack sizes exist but are not linked to servings, an Inventory Ready banner appears with a specific corrective action. Two paths resolve the alert:

  • Weight path — Enter the pack's weight; the system links pack to serving via the existing weight-based rule.
  • Sub-Qty path — Enter a Sub-Qty value on an inventory-able serving (Amount = 1); this links the serving directly to the base unit. When multiple inventory-able servings exist, entering a value in one field disables the others.

Step 4 — Confirm the base unit (optional promotion)

During save, the system checks inventory readiness and may ask whether the current lowest pack unit or a linked inventory-able serving should be the base unit. If the user selects the serving size, it is promoted — becoming both a serving and a pack size base unit. This promotion is unavailable once the item is active in inventory tables.

Auto-Populated Information

When recipe links are attached, the system automatically populates nutrition information for the item. Allergen information is also auto-populated in a similar fashion. Contribution information can be populated manually, following the same validation rules used during item creation.

Saving and Cancelling

  • Save: Confirms creation; the system assigns and displays a Menu Item Code on the detail page.
  • Cancel: Discards all entries — no item is created — and returns you to the Menu Items landing page.

Key Fields, Statuses, Exceptions & Troubleshooting

Inventory Ready vs. Not Ready

An item's Inventory Ready status is driven by pack size configuration. The flag is triggered as soon as pack sizes are added to the item. During the initial save flow, the system presents guided prompts to confirm Inventory readiness and lock in the Base Unit. After the first save, readiness checks shift from wizard steps to inline icons and banners.

The Inventory Ready banner icon is expandable — clicking it surfaces the specific corrective action required to resolve the incomplete state.

Common "Not Inventory Ready" Conditions

ConditionWhat to Fix
No pack sizes configured Add at least one pack size unit using the + Pack button.
Pack sizes exist but are not linked to servings Enter a Weight on the base unit or a Sub-Qty on the inventory-eligible serving size to establish the link.
Pack size configuration incomplete or invalid The system will prompt: confirm whether you intend to proceed without inventory eligibility, or return to fix the configuration.

Key Rules & Exceptions

  • Sub-Qty applies to pack sizes. The one exception: a serving size linked to the base unit via sub-quantity.
  • Item Categorizations are only required when Inventory is enabled on the item.
  • Pack Size lock: once an item exists in Inventory tables, the option to convert a serving size into a pack size and redefine the Base Unit is no longer available.
  • Fewer than 3 pack sizes triggers additional Base Unit confirmation prompts to ensure the correct lowest unit is identified.
